## 11. Реализация процесса исполнения. Синхронизированный цикл. Управление через прерывания. Достоинства и недостатки. Пример.
*Источник: rtos.md, Глава 3.2 «Реализация управляющей логики в приложении» (3.2.3 «Синхронизированный цикл», 3.2.4 «Прерывания»)*

**Синхронизированный цикл.** Главный цикл и циклический планировщик имеют общий недостаток — они постоянно крутятся, расходуя энергию даже при отсутствии событий. Синхронизированный цикл устраняет это, добавляя ожидание прерывания между итерациями. Идея: использовать прерывания, чтобы ограничить частоту опроса.

Простейшая реализация — фиксированная задержка между итерациями:

```c
// Источник: rtos.md, раздел 3.2.3 «Синхронизированный цикл»
do {
    // ... обработка ...
    delay(100);
    state = schedule();
} while(1);
```

Функция `delay` программирует таймер и переводит процессор в режим пониженного энергопотребления командой типа ARM `WFI` (Wait For Interrupt). Чаще, однако, точное время ожидания неизвестно, поэтому:

```c
// Источник: rtos.md, раздел 3.2.3 «Синхронизированный цикл»
do {
    // ... обработка событий ...
    stop();
    state = schedule();
} while(1);
```

Обработчики прерываний сами устанавливают флаги готовности своих устройств (атомарными инструкциями), а цикл останавливается на `stop()` до прихода очередного прерывания.

**Тонкость реализации `stop()`.** Между проверкой флагов «нечего делать» и фактическим выполнением `WFI` всегда есть зазор. Если прерывание придёт в этот момент, оно будет «пропущено». Запрет прерываний перед проверкой не решает проблему — между разрешением и инструкцией остановки тоже зазор. Это требует специальной аппаратной поддержки: `WFI` должна выходить из состояния ожидания даже при запрещённых прерываниях, откладывая вызов обработчика:

```c
// Источник: rtos.md, раздел 3.2.3 «Синхронизированный цикл»
void stop(void) {
    disable_interrupts();
    if (nothing_to_do)
        wait_for_interrupts();
    enable_interrupts();
}
```

**Достоинства синхронизированного цикла:** существенно ниже энергопотребление по сравнению с простым суперциклом; предсказуемость работы сохраняется; обработчики прерываний могут быть предельно простыми.

**Недостатки:** скорость реакции ограничена скоростью работы самой длинной функции-обработчика в цикле; наличие прерываний привносит элемент неопределённости (атомарная установка флагов) и усложняет тестирование.

**Управление через прерывания.** Если процессор уже умеет вызывать обработчики в ответ на прерывания, можно использовать это как каркас для приложения. Код обработки события размещается в обработчике соответствующего прерывания, а главная программа содержит пустой бесконечный цикл:

```c
// Источник: rtos.md, раздел 3.2.4 «Прерывания»
void handler1(void) { /* обработка события 1 */ }
void handler2(void) { /* обработка события 2 */ }

int main(void) {
    do { wait_for_interrupt(); } while(1);
}
```

Для работы метода нужен процессор-специфичный слой, готовящий таблицу прерываний. Например, ARM Cortex-M позволяет писать обработчики на C — при прерывании в качестве адреса возврата сохраняется особое значение (например, `0xFFFFFFFD`), а x86 требует ассемблерных обёрток с инструкцией `IRET`.

**Достоинства управления через прерывания:** при наличии вытесняемых обработчиков (сохранение/восстановление контекста на входе и выходе) достигается **фиксированное время реакции** для самого приоритетного прерывания — наилучший результат среди рассмотренных подходов. Хорошее энергопотребление.

**Недостатки управления через прерывания:** очень тесная интеграция с аппаратурой — непереносимо; приоритеты обработчиков прерываний не всегда совпадают с приоритетами реальных задач; ограниченное число векторов прерываний; на многих процессорах нельзя программно инициировать аппаратное прерывание; сложно реализовать общую процедуру обработки данных из разных источников; при переносе на другую платформу — существенная модификация.

Этот подход хорош, если процессор поддерживает гибкую настройку приоритетов прерываний и обработчики короткие. С ростом сложности обработки становится менее привлекательным — что и приводит к идее акторов и потоков.

## 12. Реализация процесса исполнения. Энергосберегающий режим.
*Источник: rtos.md, Глава 2.1.4 «Прерывания», Глава 3.2.3 «Синхронизированный цикл»; rtos.md, разделы про планировщик и поток простоя*

**Энергосбережение в процессоре.** Помимо реагирования на события, прерывания дают возможность реализовать в процессоре различные режимы энергосбережения. Система команд может содержать **специальную инструкцию останова**, приостанавливающую выборку и исполнение инструкций до получения прерывания. Примеры: `WFI` (Wait For Interrupt) у ARM, `HLT` у x86. Пока такой инструкции не отдана команда, процессор продолжает крутить цикл, расходуя энергию впустую при отсутствии работы.

**Главный цикл vs синхронизированный цикл.** В простом главном цикле и циклическом планировщике нет никаких механизмов остановки процессора — опрос датчиков происходит постоянно. Если события случаются редко (например, раз в секунду), такой подход расходует огромное количество энергии. В синхронизированном цикле в момент, когда работы нет, выполняется `wait_for_interrupts()` — процессор выключает выборку команд и остаётся в энергосберегающем режиме до прихода прерывания.

**Корректная реализация энергосбережения.** Между проверкой «работы нет» и собственно остановкой процессора всегда есть временной зазор. Если в этот момент придёт прерывание, оно может быть «потеряно» в смысле, что флаг работы будет установлен, но процессор уже встанет на `WFI` до следующего события. Для решения нужно, чтобы `WFI` выходила из состояния ожидания **даже при запрещённых прерываниях** (вызов обработчика откладывается до их разрешения):

```c
// Источник: rtos.md, раздел 3.2.3 «Синхронизированный цикл»
void stop(void) {
    disable_interrupts();
    if (nothing_to_do)
        wait_for_interrupts();   // WFI выйдет, если за это время пришло прерывание
    enable_interrupts();
}
```

**Энергосбережение в ОСРВ.** В системе с потоками и планировщиком вводится специальный **поток простоя** (idle thread). Когда нет ни одного готового к выполнению потока (и не работает ни один обработчик прерывания), планировщик передаёт управление потоку простоя, который в цикле выполняет `wait_for_interrupts()`. Любое прерывание выводит процессор из состояния ожидания, и если оно делает какой-нибудь поток готовым, планировщик передаст управление этому потоку. В противном случае поток простоя снова уходит в ожидание.

**Управление через прерывания и энергосбережение.** В подходе с прерываниями в качестве «движка» приложения вся главная программа — это бесконечный цикл `do { wait_for_interrupt(); } while(1);`. Процессор большую часть времени находится в энергосберегающем режиме, и просыпается только при наступлении событий.

**Управление частотой и динамическое масштабирование.** Современные процессоры (включая встроенные) поддерживают несколько уровней энергосбережения: понижение тактовой частоты, отключение отдельных периферийных модулей, перевод в глубокий сон (deep sleep) с сохранением минимального состояния. ОС или прикладное ПО может выбирать уровень в зависимости от ожидаемой длительности простоя.

**Сравнительная таблица энергоэффективности (из rtos.md):**

| Подход | Энергоэффективность |
|---|---|
| Главный цикл | – |
| Циклический планировщик | – |
| Синхронизированный цикл | + |
| Прерывания | + |
| Акторы | + |
| Потоки | + |

Энергосберегающим режимом является любой подход, в котором процессор останавливается между событиями (последние четыре строки).

## 13. Виртуальное адресное пространство. Защита памяти с использованием MPU.
*Источник: rtos.md, Глава 2.3.3 «Права доступа к памяти»; Глава 4.7.1 «MPU»*

**Проблема защиты памяти.** Привилегированная программа (ОС — операционная система) находится в той же памяти, что и непривилегированная. Без аппаратной защиты ошибки в пользовательской программе могут испортить данные ОС или других процессов.

**Простейшая защита — пара регистров «база/предел».** Процессор содержит два регистра: базовый адрес и размер региона, разрешённого для непривилегированной программы. Любое обращение по адресу за границы этого региона вызывает исключение. Регистры доступны только привилегированной программе. При переключении процессов их содержимое меняется — процессы изолированы и от ОС, и друг от друга. **Недостаток:** возможен только один разрешённый регион.

**MPU (Memory Protection Unit — устройство защиты памяти).** Развитие подхода. MPU тесно связан с ядром процессора и содержит набор регистров, описывающих **несколько регионов** (обычно 8–16) и права доступа к каждому. Этого достаточно для большинства приложений: можно отдельно настроить права для кода (RX), данных (RW), стека, периферии, и т.д.

**Что задаётся для региона:**
- Базовый адрес и размер.
- Права доступа (R/W/X) отдельно для привилегированного и непривилегированного режимов.
- Атрибуты кэширования (cacheable, bufferable).
- Признак разделяемой памяти (для многоядерных систем).

**Схема работы MPU:**

```
        Программа                          Физическая память
       обращение по                        (адрес = физический)
        адресу A
            │
            ▼
    ┌───────────────┐
    │     MPU       │
    │ ┌───────────┐ │       ┌─────────────────────────┐
    │ │ Регион 0  │ │       │ 0x0800_0000 .text (RX)  │
    │ │ base,size,│─┼──────▶│ 0x2000_0000 .data (RW)  │
    │ │ rights    │ │       │ 0x2000_4000 stack (RW)  │
    │ ├───────────┤ │       │ 0x4000_0000 периферия   │
    │ │ Регион 1  │ │       │            (RW, priv)   │
    │ │ ...       │ │       └─────────────────────────┘
    │ └───────────┘ │
    │   проверка    │
    │   прав ──────►│ ✓ разрешено → доступ к памяти
    │               │ ✗ нарушение  → исключение MemManage
    └───────────────┘
```

Адрес A передаётся напрямую в память (без трансляции), но **только если** он попадает в один из настроенных регионов с подходящими правами.

**Особенности MPU:**
- Виртуальное и физическое адресное пространство **совпадают** — MPU не выполняет трансляцию адресов, а только проверяет права. Это критическое отличие от MMU (Memory Management Unit — устройство управления памятью).
- Нет таблиц трансляции в памяти — все настройки в регистрах процессора.
- Детерминированное поведение: проверка прав занимает фиксированное время, в отличие от MMU с TLB (Translation Lookaside Buffer — буфер ассоциативной трансляции) и подкачкой.
- Не требует поддержки страничного механизма — отлично подходит для микроконтроллеров.

**Применение в ОСРВ (операционной системе реального времени).** При переключении потоков ядро перепрограммирует регистры MPU так, чтобы текущий поток имел доступ только к разрешённой ему памяти (свой стек, своя секция данных, общие зоны). Обращение по другим адресам вызывает исключение, которое обычно завершает аварийно поток или процесс.

**Промежуточный вариант — FMT (Fixed Mapping Translation — трансляция с фиксированным отображением).** Чип содержит жёстко заданные диапазоны доступа с разными правами. Границы задаются регистрами, как у MPU, но виртуальное и физическое адресные пространства уже разделены. Например, адресное пространство делится на нижний (с проверками границ) и верхний (без проверок) диапазоны, при этом адрес `0x80000000` отображается на тот же физический адрес, что и `0x00000000`.

**Достоинства MPU:**
- Малые накладные расходы по памяти (нет таблиц).
- Детерминированное время доступа к памяти — критично для жёсткого реального времени.
- Простая реализация в аппаратуре.
- Низкое энергопотребление.

**Недостатки:**
- Ограниченное число регионов.
- Нет виртуальной памяти (адресное пространство процесса не может превышать физическое).
- Нет страничной подкачки.
- Регионы должны быть выровнены по степеням двойки, что приводит к фрагментации.

## 14. Виртуальное адресное пространство. MMU. Страничная и странично-сегментная виртуализация. Таблица страниц. Пример. Где хранится таблица страниц.
*Источник: rtos.md, Глава 2.3.3 «Права доступа к памяти»; tanenbaum.md, Глава 3 «Управление памятью», разделы про страничную организацию и сегментацию*

**MMU (Memory Management Unit — устройство управления памятью).** Гораздо более продвинутое устройство, чем MPU (Memory Protection Unit). Идея: отделить **виртуальное адресное пространство**, с которым работают программы, от **физического адресного пространства**. Каждой программе предоставляется иллюзия владения большим непрерывным куском памяти, тогда как физически данные могут быть размещены произвольно.

**Страничная виртуализация.** Память разбивается на блоки фиксированного размера — **страницы** (обычно 4 КБ). Физическая память также разбивается на блоки того же размера — **страничные блоки** (page frames). MMU задаёт отображение виртуальных страниц на физические через **таблицу страниц**.

Запись в таблице страниц (PTE — Page Table Entry) для одной виртуальной страницы содержит:
- Бит присутствия (страница загружена в физическую память или выгружена).
- Физический адрес страничного блока (если присутствует).
- Биты прав доступа (R/W/X) для пользователя и ядра.
- Бит «изменена» (dirty) — записывали ли в страницу.
- Бит «обращалась» (accessed) — обращались ли к странице.
- Бит запрета кэширования.

**Схема трансляции виртуального адреса в физический:**

```
   Виртуальный адрес (32 бита):
   ┌──────────────────────┬──────────────────┐
   │ Номер вирт. страницы │  Смещение (12)   │
   │     (старшие 20)     │                  │
   └──────────┬───────────┴────────┬─────────┘
              │                    │
              ▼                    │
   ┌─────────────────────┐         │
   │  CR3 / TTBR         │         │
   │ (адрес таблицы)     │         │
   └─────────┬───────────┘         │
             │                     │
             ▼                     │
   ┌────────────────────┐          │
   │  Таблица страниц   │          │
   │  ┌──────────────┐  │          │
   │  │ PTE[0]       │  │          │
   │  │ PTE[1]       │  │          │
   │  │ ...          │  │          │
   │  │ PTE[N] ──────┼──┼──┐       │
   │  │ присутствие  │  │  │       │
   │  │ права R/W/X  │  │  │       │
   │  │ физ. блок ───┼──┼──┼──┐    │
   │  └──────────────┘  │  │  │    │
   └────────────────────┘  │  │    │
                           │  │    │
              ┌────────────┘  │    │
              │ если P=0:     │    │
              ▼               │    │
       Page Fault             │    │
       (обработчик ОС         │    │
        подгружает с диска)   │    │
                              ▼    ▼
                  ┌──────────────────────────┐
                  │ Физический адрес:        │
                  │ (физ. блок << 12) | смещ.│
                  └──────────────────────────┘
                              │
                              ▼
                       Физическая память
```

**Пример (Tanenbaum, гл. 3).** Виртуальное пространство 64 КБ при размере страницы 4 КБ имеет 16 виртуальных страниц. Физической памяти 32 КБ — 8 страничных блоков. Программа делает запрос `MOV REG, 8196`. Адрес 8196 = `0010000000000100` — старшие 4 бита (0010 = 2) определяют номер виртуальной страницы, младшие 12 бит (000000000100 = 4) — смещение в странице. По таблице страниц виртуальная страница 2 отображается на физический блок, например, 6 — итоговый физический адрес: `6 * 4096 + 4 = 24580`.

Если виртуальная страница помечена как отсутствующая (бит присутствия = 0), процессор генерирует **страничное прерывание** (page fault). Обработчик ОС загружает нужную страницу с диска, при необходимости вытеснив какую-то другую страницу, обновляет таблицу страниц и возобновляет команду.

**Где хранится таблица страниц.** Таблица страниц находится **в оперативной памяти**. Процессор содержит специальный регистр (`CR3` на x86 — Control Register 3, `TTBR` на ARM — Translation Table Base Register), указывающий на физический адрес начала таблицы. При переключении процесса этот регистр обновляется.

Для ускорения трансляции используется **TLB (Translation Lookaside Buffer — буфер ассоциативной трансляции)** — кэш недавних трансляций виртуальный → физический адрес. При обращении к памяти процессор сначала ищет в TLB, и только если не нашёл — обращается к таблице страниц в памяти (это значительно медленнее).

**Размер таблицы.** Например, таблица для 4 ГБ адресного пространства со страницами по 4 КБ содержит 1 048 576 записей; при 4 байта на запись — 4 МБ только на таблицу. Для каждого процесса! Поэтому применяются **многоуровневые таблицы** (см. вопрос 16).

**Странично-сегментная виртуализация.** Гибрид страничной и сегментной организаций. Виртуальный адрес делится на:
- **номер сегмента** — индекс в таблице сегментов; каждая запись описывает свой сегмент и содержит указатель на собственную таблицу страниц этого сегмента;
- **номер страницы внутри сегмента**;
- **смещение в странице**.

Применялась в MULTICS, реализована в Intel x86. Позволяет иметь у каждого сегмента собственный размер и права доступа, при этом физическая память не страдает от фрагментации благодаря страничной организации.

**Чистая сегментация** (Tanenbaum, 3.7) — память делится на сегменты переменной длины, каждый со своими атрибутами и адресацией от 0. Адрес — пара (сегмент, смещение). Сегмент представляет логический объект (процедуру, массив, стек), его размер может меняться независимо. Сейчас в чистом виде не применяется.

**Применение MMU в ОСРВ.** В системах реального времени свопинг не используется из-за непредсказуемости. Используется только защита и трансляция — то есть «всегда присутствующие в памяти» страницы.

## 15. Виртуальное адресное пространство. Инвертированные таблицы страниц. Пример. Быстрый поиск виртуального адреса.
*Источник: rtos.md, Глава 2.3.4 «Инвертированные таблицы страниц»; tanenbaum.md, Глава 3 «Управление памятью», раздел про инвертированные таблицы*

**Проблема обычных таблиц страниц.** Обычная (прямая) таблица содержит запись для каждой виртуальной страницы. Для 64-разрядных систем виртуальное адресное пространство огромно: например, при 48-разрядных адресах и страницах по 4 КБ — 2³⁶ виртуальных страниц, что даёт сотни ГБ только на таблицу страниц. Даже многоуровневые таблицы тратят значительный объём памяти.

**Идея инвертированной таблицы.** Вместо записи на каждую виртуальную страницу хранятся записи на каждую **физическую** страницу (отсюда название). Размер таблицы пропорционален физической памяти, которой обычно гораздо меньше, чем виртуального пространства.

**Структура.** Каждая запись инвертированной таблицы содержит:
- идентификатор процесса (PID — Process Identifier, или ASID — Address Space Identifier);
- виртуальную страницу, отображённую в этот физический блок;
- права доступа и другие флаги.

**Проблема поиска.** Виртуальный адрес больше нельзя использовать как индекс в таблице — нужно перебирать все записи, пока не найдётся пара `(PID, virtual page)`, совпадающая с искомой. Прямой линейный поиск был бы катастрофически медленным.

**Быстрый поиск — хэширование.** Применяется хэш-таблица: по `(PID, virtual page)` вычисляется хэш, указывающий на цепочку записей в инвертированной таблице. Каждая запись содержит указатель «следующая запись с тем же хэшом», поэтому при коллизиях достаточно пройти короткий список.

**Схема поиска по инвертированной таблице:**

```
   Виртуальный адрес + PID
              │
              ▼
        ┌──────────┐
        │  hash()  │
        └────┬─────┘
             │
             ▼
   ┌─────────────────────┐         Инвертированная таблица
   │   Хэш-таблица       │         (запись на физ. блок)
   │  ┌───────────────┐  │       ┌─────────────────────────┐
   │  │ [0] ──────────┼──┼──────▶│ блок 0: PID=2, vpn=5    │
   │  │ [1]   NULL    │  │       │         next ───────────┼─┐
   │  │ [2] ──────────┼──┼──┐    │ блок 1: PID=7, vpn=42   │ │
   │  │ ...           │  │  │    │         next = NULL     │ │
   │  │ [H] ──────────┼──┼──┼──▶ │ блок 2: PID=PID*, vpn=V*│ │
   │  └───────────────┘  │  │    │         next = NULL  ✓  │ │
   └─────────────────────┘  │    │ ...                     │ │
                            │    │ блок N: PID=3, vpn=10   │ │
              ┌─────────────┘    │         next = NULL     │ │
              ▼                  └─────────────────────────┘ │
   Перебор цепочки с тем               ▲                     │
   же хэшом до совпадения              └─────────────────────┘
   (PID, vpn) с искомым.

   При совпадении → индекс записи = номер физ. блока
   Физический адрес = (физ. блок << page_shift) | смещение
```

**Алгоритм:**
1. Виртуальный адрес делится на номер виртуальной страницы (VPN — Virtual Page Number) и смещение.
2. Из `(PID, VPN)` вычисляется хэш.
3. По хэшу — указатель на цепочку записей.
4. Перебор цепочки до нахождения нужной записи.
5. Физический адрес = `(физический блок << page_shift) | смещение`.

**TLB как сокращение пути.** Поскольку поиск даже с хэшированием медленнее обычной таблицы, особенно важен TLB. Большая часть обращений попадает в TLB, и весь сложный механизм инвертированной таблицы запускается только при TLB miss.

**Пример применения.** PowerPC, IA-64 (Itanium), Ultra SPARC, IBM RT (исторически). Используются преимущественно в системах с 64-разрядными адресами, где обычные таблицы были бы слишком большими.

**Программная трансляция.** В rtos.md описано, что при программном управлении TLB (TLB-miss обрабатывается ПО) ОС может вообще не хранить таблицу: если правило отображения задаётся простой функцией (например, прямое отображение от 0 до `0x80000000`), логику трансляции можно реализовать прямо в обработчике исключения.

**Достоинства инвертированных таблиц:** малый размер таблицы (пропорционален физической памяти, а не виртуальной); удобно для больших виртуальных пространств.

**Недостатки:** менее детерминированное поведение (хэш-коллизии), более низкая производительность по сравнению с прямыми многоуровневыми таблицами; сложнее поддерживать совместно используемые страницы между процессами.

## 16. Виртуальное адресное пространство. Многоуровневые таблицы страниц. Пример. Где хранятся таблицы страниц.
*Источник: rtos.md, Глава 2.3.3 «Права доступа к памяти» (раздел про многоуровневые таблицы); tanenbaum.md, Глава 3 «Управление памятью»*

**Проблема одноуровневой таблицы.** Таблица для 4 ГБ адресного пространства со страницами 4 КБ содержит 1 048 576 записей; при 4 байта на запись — 4 МБ на каждый процесс. При сотнях процессов и сравнительно небольшом объёме физической памяти это неприемлемо. К тому же большая часть адресного пространства процесса не используется — хранить таблицу для пустых регионов бессмысленно.

**Идея многоуровневых таблиц.** Виртуальный адрес делится не на 2, а на **N + 1 частей**: N индексов и смещение в странице. Каждый индекс используется как индекс в таблице соответствующего уровня. Каждая запись в таблице уровня i (кроме последнего) указывает на таблицу уровня i+1. Записи последнего уровня указывают на физические страничные блоки.

Запись каждого уровня содержит бит присутствия. Если на промежуточном уровне таблица отмечена как отсутствующая, нет нужды хранить таблицы более низких уровней для этого региона.

**Пример из rtos.md (двухуровневая).** Адрес 32 бита делится:
- 10 бит — **индекс в каталоге страниц** (Page Directory);
- 10 бит — **индекс в таблице страниц** (Page Table);
- 12 бит — смещение в странице (4 КБ).

**Схема двухуровневой трансляции:**

```
   Виртуальный адрес (32 бита):
   ┌────────────┬────────────┬──────────────┐
   │ PD index   │ PT index   │   Offset     │
   │ (10 бит)   │ (10 бит)   │  (12 бит)    │
   └─────┬──────┴─────┬──────┴──────┬───────┘
         │            │             │
         │            │             │
   CR3 ──┼─▶ Каталог страниц        │
         │   ┌─────────────┐         │
         │   │ PDE[0]      │         │
         └──▶│ PDE[1]      │         │
             │ ...         │         │
             │ PDE[k] ─────┼──┐      │
             │ ...         │  │      │
             │ PDE[1023]   │  │      │
             └─────────────┘  │      │
                              ▼      │
                  Таблица страниц    │
                  ┌─────────────┐    │
                  │ PTE[0]      │    │
              ┌──▶│ PTE[1]      │    │
              │   │ ...         │    │
              │   │ PTE[j] ─────┼──┐ │
              │   │ ...         │  │ │
              │   │ PTE[1023]   │  │ │
              │   └─────────────┘  │ │
              │                    ▼ ▼
              │           ┌──────────────────────┐
              │           │ Физический адрес:    │
              │           │ (физ. блок << 12)    │
              │           │      | offset        │
              │           └──────────────────────┘
              │                       │
              │                       ▼
              │              Физическая память
              │
              └── PT index выбирает запись
                  в таблице страниц
```

Каталог страниц содержит 1024 записи. Каждая указывает либо на таблицу страниц (тоже 1024 записи), либо отмечена как отсутствующая. Полностью описанное пространство содержало бы каталог + 1024 таблицы по 1024 записи = 4 МБ. Но если физической памяти всего 256 МБ (65 536 страниц), то достаточно 64 таблиц по 1024 записи + каталог = 64 КБ + 4 КБ = ~68 КБ + сам каталог.

В тексте rtos.md приведён пример: для описания 256 МБ нужно 64 таблицы + каталог = **260 КБ вместо 4 МБ** — ощутимая экономия.

**Алгоритм трансляции:**
1. Из CR3 (или аналога) берётся физический адрес каталога.
2. Старшие 10 бит виртуального адреса — индекс в каталоге → физический адрес таблицы страниц.
3. Средние 10 бит — индекс в таблице страниц → физический адрес страничного блока.
4. Младшие 12 бит — смещение в странице.

**Где хранятся таблицы страниц.** Все таблицы (каталог и таблицы всех уровней) находятся **в оперативной памяти**. В процессоре есть лишь один регистр, указывающий на физический адрес корневой таблицы (каталога). При переключении процесса этот регистр перепрограммируется.

Поскольку при каждом обращении к памяти потребовалось бы N дополнительных обращений за таблицами разных уровней (для 64-битной системы до 4–5 уровней), используется **TLB (Translation Lookaside Buffer)**, кэширующий результаты трансляций. После первого обращения трансляция попадает в TLB, и дальнейшие обращения к страницам в той же области идут без обращений к таблицам.

**64-битные системы.** Размер виртуального адресного пространства таков, что описание полной таблицы потребовало бы тысячи терабайт. Используется 3, 4 или даже 5 уровней страничной трансляции. x86-64 использует 4 (PML4 → PDPT → PD → PT — Page Map Level 4, Page Directory Pointer Table, Page Directory, Page Table) или 5 (PML5) уровней при страницах 4 КБ. При больших страницах (2 МБ, 1 ГБ) глубина меньше.

**Схема 4-уровневой трансляции x86-64 (48-битный адрес):**

```
   Виртуальный адрес:
   ┌─────┬─────┬─────┬─────┬────────────┐
   │PML4 │PDPT │ PD  │ PT  │  Offset    │
   │ idx │ idx │ idx │ idx │  (12 бит)  │
   │(9)  │(9)  │(9)  │(9)  │            │
   └──┬──┴──┬──┴──┬──┴──┬──┴─────┬──────┘
      │     │     │     │        │
 CR3 ─┘     │     │     │        │
      ▼     │     │     │        │
   ┌─────┐  │     │     │        │
   │PML4 ├──┼─────┼─────┼────────┼──▶ выбор записи (512 записей)
   └──┬──┘  │     │     │        │
      ▼     ▼     │     │        │
   ┌─────────┐    │     │        │
   │  PDPT   ├────┼─────┼────────┼──▶ 512 записей
   └────┬────┘    │     │        │
        ▼     ▼   │     │        │
     ┌─────────┐  │     │        │
     │   PD    ├──┼─────┼────────┼──▶ 512 записей
     └────┬────┘  │     │        │
          ▼     ▼ │     │        │
        ┌─────────┐     │        │
        │   PT    ├─────┼────────┼──▶ 512 записей
        └────┬────┘     │        │
             ▼          ▼        ▼
          ┌──────────────────────────┐
          │ физ. адрес страничного   │
          │ блока + offset           │
          └──────────────────────────┘
```

**Большие страницы.** ARM позволяет страницы 4 КБ, 64 КБ, 1 МБ и 16 МБ. При страницах 16 МБ таблица содержит всего 256 записей по 4 байта = 1 КБ. Большие страницы хороши для уменьшения числа промахов TLB при работе с большими структурами данных, но плохо подходят для тонкой настройки прав доступа.

**Достоинства многоуровневых таблиц:**
- Существенная экономия памяти при разреженном использовании адресного пространства.
- Можно помечать как отсутствующие целые регионы (через записи каталога).
- Удобно для совместного использования памяти между процессами (можно разделять таблицы нижнего уровня).

**Недостатки:**
- Каждый промах TLB требует N обращений к памяти за таблицами.
- Более сложная реализация в ОС.

## 17. Размещение данных в виртуальном адресном пространстве. Код, данные, стек и куча. Пример.
*Источник: tanenbaum.md, Глава 1 «Введение», раздел про сегменты памяти процесса (рис. 1.18); rtos.md, Глава 4.7.3 «Распределение адресного пространства»*

**Сегменты процесса в UNIX/Linux.** Память каждого процесса делится на три классических сегмента:

1. **Сегмент кода (text segment).** Содержит исполняемый код программы — последовательность инструкций. Помечен как read-only и executable. Обычно отображается напрямую с исполняемого файла (используется демандовая страничная загрузка) и разделяется между процессами, выполняющими ту же программу. Размер фиксирован.

2. **Сегмент данных (data segment).** Содержит глобальные и статические переменные программы. Делится на:
   - **`.data`** — инициализированные данные (`int x = 5;`);
   - **`.bss`** — неинициализированные данные (`int y;`), при загрузке заполняется нулями (физически не хранится в файле).

   Может расти вверх по мере выделения динамической памяти. Расширение через системный вызов `brk` (или `sbrk`), который устанавливает новый адрес конца сегмента данных.

3. **Сегмент стека (stack segment).** Хранит активационные записи (стек-фреймы) функций: локальные переменные, адреса возврата, сохранённые регистры. Растёт вниз (от старших адресов к младшим) автоматически по мере вызова функций. Между стеком и сегментом данных есть **большой пустой промежуток** виртуального адресного пространства.

**Куча (heap).** Между сегментом данных и стеком в современных программах располагается **куча** — область для динамического выделения памяти через `malloc()`/`new`. С точки зрения ОС куча — это расширение сегмента данных (управляется через `brk`/`sbrk`) или отдельные регионы, созданные через `mmap()`. Куча растёт вверх. С точки зрения программы, между ней и стеком сохраняется свободное пустое пространство для роста обоих.

**Пример типичной карты адресного пространства Linux (32-битного процесса):**

```
0xFFFFFFFF  ┌──────────────────────┐
            │  Область ядра (KGD)  │  → отображена в каждом процессе
0xC0000000  ├──────────────────────┤
            │                      │
            │  Стек (растёт вниз ↓)│
            ├──────────────────────┤
            │       свободно       │
            │     ↑ mmap, .so     │
            │  Куча (растёт ↑)     │
            ├──────────────────────┤
            │  .bss (неинит. данные)│
            ├──────────────────────┤
            │  .data (инит. данные)│
            ├──────────────────────┤
            │  .text (код)         │
0x08048000  ├──────────────────────┤
            │  зарезервировано     │
0x00000000  └──────────────────────┘
```

**Распределение в ОСРВ (rtos.md, 4.7.3).** В системе с MMU код ОС и HAL должны присутствовать в адресном пространстве каждого процесса (для системных вызовов), причём по фиксированным виртуальным адресам. Поэтому **ядро располагается в выделенной области виртуальных адресов**, обычно в **старших адресах** (например, от `0x80000000` до `0xFFFFFFFF`), не пересекающейся с адресами пользовательских процессов (от `0x00000000` до `0x80000000`).

Преимущество: по самому указателю можно сразу определить его права доступа. Указатель в верхней половине — для привилегированного режима, в нижней — для пользователя.

При размещении устройств, отображённых на память, доступ к ним со стороны пользовательских приложений закрывается. Обычно устройства располагаются в старших физических адресах, и таким образом автоматически попадают в зону ОС.

**Пример сегментов на простой программе:**

```c
int g_inited = 5;     // .data
int g_uninited;       // .bss
const char *msg = "hi"; // указатель в .data, строка в .rodata (часть .text)

int main() {
    int local = 10;       // стек
    int *p = malloc(100); // p в стеке, выделенные 100 байт — в куче
    return 0;
}
```

## 18. Ограничения доступа к памяти. TLB. Своппинг.
*Источник: rtos.md, Глава 2.3.3 «Права доступа к памяти»; tanenbaum.md, Глава 3.2.2 «Свопинг»*

**Ограничения доступа к памяти.** При использовании MMU (Memory Management Unit) для каждой страницы виртуального адресного пространства задаются:
- Бит присутствия — страница в физической памяти или выгружена.
- Права доступа: R (чтение), W (запись), X (выполнение) — отдельно для привилегированного и непривилегированного режимов.
- Бит «обращалась» — устанавливается при любом доступе.
- Бит «изменена» (dirty) — устанавливается при записи; нужен для решения, надо ли сбрасывать на диск при вытеснении.
- Атрибуты кэширования.
- Бит запрета доступа из непривилегированного режима.

Нарушение прав вызывает **исключение** (page fault). ОС-обработчик решает: ошибка приложения (сегфолт) или нормальная ситуация (страница выгружена, надо подгрузить; copy-on-write при записи в разделяемую страницу).

**TLB (Translation Lookaside Buffer — буфер ассоциативной трансляции).** Кэш недавних трансляций виртуальный → физический адрес. Физически реализован в процессоре. Структура похожа на ассоциативный кэш: по ключу (виртуальный адрес) ищется значение (физический адрес и флаги). Объём TLB ограничен — обычно десятки–сотни записей, поскольку поиск должен быть быстрым (параллельным по всему содержимому).

**Схема работы TLB:**

```
   Виртуальный адрес от программы
                │
                ▼
         ┌───────────┐
         │    TLB    │   ◄── параллельный поиск по всем записям
         │ ┌───────┐ │
         │ │vpn→pfn│ │
         │ │vpn→pfn│ │     hit
         │ │  ...  │ ├──────────┐
         │ └───────┘ │          │
         └─────┬─────┘          │
               │ miss            │
               ▼                 │
       ┌──────────────┐          │
       │ Page table   │          │
       │ walk (HW/SW) │          │
       │ из CR3/TTBR  │          │
       └──────┬───────┘          │
              │                  │
              ▼                  │
       Если PTE присутствует:    │
         загрузить в TLB ────────┤
       Если нет:                 │
         Page fault → ОС         │
                                 ▼
                       ┌──────────────────┐
                       │ Физический адрес │
                       │ → доступ к RAM   │
                       └──────────────────┘
```

**Работа TLB:**
1. При обращении к памяти процессор сначала ищет запись в TLB.
2. **TLB hit** — есть запись для нужной виртуальной страницы. Берётся физический адрес, проверяются права, выполняется обращение к памяти.
3. **TLB miss** — записи нет. Запускается процедура заполнения:
   - **Аппаратная**: процессор сам читает таблицу страниц по адресу из регистра типа CR3, проходит уровни таблиц, находит запись и загружает её в TLB.
   - **Программная**: возникает исключение TLB miss, обработчик ОС находит нужную запись и явно загружает в TLB через специальные команды.

**Особенности TLB:**
- При переключении процессов TLB нужно сбросить (или использовать ASID — Address Space Identifier, идентификатор адресного пространства) — иначе старые трансляции применятся к новому процессу.
- На многопроцессорной системе при изменении таблицы страниц одним CPU (Central Processing Unit) остальные надо уведомить о необходимости обновить свои TLB (TLB shootdown).
- В системах реального времени можно создавать **невыгружаемые записи** TLB для критических регионов, чтобы исключить промахи и сделать время доступа детерминированным.

**Своппинг (swapping).** Способ преодоления нехватки физической памяти. Возможны два подхода:

**Классический своппинг (Tanenbaum, 3.2.2).** В памяти содержится весь процесс целиком. При нехватке памяти бездействующий процесс целиком выгружается на диск (в область подкачки, swap area), его место занимают другие процессы. Когда процесс снова надо выполнить — его читают с диска и помещают в свободную область.

Проблема: процессы могут оказаться в разных местах памяти при разных запусках, поэтому нужны механизмы перемещения (base/limit регистры или MMU).

**Подкачка страниц (paging) — современный подход.** Программа может частично находиться в памяти. ОС подкачивает только активно используемые страницы, остальные хранятся на диске. Когда процесс обращается к выгруженной странице, MMU генерирует page fault, ОС загружает страницу с диска (возможно вытеснив другую) и продолжает выполнение.

**Алгоритмы вытеснения** (какую страницу выгрузить): LRU (least recently used), Clock (часы), NFU, и их вариации. В Linux — kswapd, демон, который при нехватке памяти инициирует алгоритм востребования страниц (PFRA).

**В системах реального времени своппинг не применяется** — он вносит непредсказуемые задержки. Используется только защитная и трансляционная функциональность MMU.

**Атрибут swappable.** В современных Linux-системах для каждой страницы есть атрибут: может ли она быть выгружена. Закреплённые (pinned) страницы — например, страницы ядра, DMA-буферы — не выгружаются никогда. Атрибут `vm.swappiness` управляет агрессивностью свопа (0..100).

## 19. Планировщик. Точка входа в планировщик. Вызов планировщика. События + акторы/потоки.
*Источник: rtos.md, Глава 4.4.3 «Переключение контекста»; Глава 5.3.5 «Планирование»; Глава 5.8.1 «Реализация планировщика»*

**Точка входа в планировщик.** Планировщик ОС — это функция, которая:
1. Анализирует состояние всех потоков/акторов;
2. Выбирает наиболее приоритетный из готовых к выполнению;
3. При необходимости вызывает переключение контекста на этот поток.

В классической реализации (например, Linux) точка входа — функция `schedule()`. В реализации ОСРВ (rtos.md, гл. 4–5) роль точки входа выполняет обработчик специального программного прерывания «DISPATCH», что позволяет вызывать планировщик асинхронно после выхода из всех вложенных аппаратных прерываний.

**Когда вызывается планировщик:**

1. **После обработки аппаратного прерывания.** Если прерывание сделало готовым более приоритетный поток, по выходу из всех вложенных обработчиков должен сработать планировщик и переключить контекст. Реализация — счётчик вложения прерываний с логикой «кто сделал последний декремент и достиг нуля, тот вызывает планировщик».
2. **Поток вызывает блокирующую функцию** (например, ожидание семафора, сообщения). Поток переходит в состояние ожидания, планировщик ищет следующий.
3. **Поток освобождает занятый ресурс**, разблокирующий более приоритетный поток.
4. **Поток явно отдаёт CPU** (`yield`, `sched_yield`).
5. **Истечение кванта времени** (для round-robin между равноприоритетными потоками).
6. **Поток завершается.**
7. **Возврат из системного вызова** в пользовательский режим (на некоторых архитектурах проверяется флаг `need_resched`).

**Реализация в ОСРВ через программное прерывание (rtos.md, 4.4.3).** Все события, требующие переключения контекста, **не выполняют переключение сразу**. Они запрашивают программное прерывание определённого приоритета (`DISPATCH`). Обработчик аппаратного прерывания, после установки флагов и запроса dispatch, заканчивается. Когда все аппаратные прерывания обработаны, начинает выполняться dispatch-обработчик, который:
1. Выбирает наиболее приоритетный готовый поток.
2. Сравнивает с текущим: если другой — выполняет переключение контекста.
3. Возврат из dispatch уже в новом потоке.

```c
// Источник: rtos.md, раздел 4.8 «Пример реализации приложения на интерфейсах HAL»
void interrupt_handler(void) {
    if (get_current_vector() == SIGNAL_VECTOR) {
        thread_ready = true;
        request_software_interrupt(DISPATCH);
    }
}
```

**Преимущества такого подхода:**
- Планировщик не вызывается рекурсивно из вложенных прерываний.
- Прерывания могут быть короткими, переключение контекста не задерживает их обработку.
- Один централизованный механизм для всех событий: аппаратных, программных, синхронных.

**События и акторы/потоки.** В архитектуре с акторами событие (прерывание или вызов из другого актора) приводит к активации актора через `actor_activate()`. Активация не запускает обработчик немедленно, а только переводит актор в состояние «готов». Планировщик акторов выбирает самый приоритетный из активных акторов и запускает его обработчик на общем стеке.

В архитектуре с потоками событие приводит к разблокированию ожидающего на нём потока. Поток переводится в состояние готовности. Планировщик потоков выбирает самый приоритетный из готовых потоков (или продолжает выполнение текущего, если он самый приоритетный) и при необходимости выполняет переключение контекста.

**Различия с точки зрения планировщика:**
- Акторы: один стек, минимальные накладные расходы, обработчик всегда запускается с начала.
- Потоки: каждому потоку свой стек, переключение контекста полное (сохранение/восстановление регистров и стека), но поток может возобновиться с любого места.

## 20. Алгоритмы планирования FCFS, RR. Примеры.
*Источник: tanenbaum.md, Глава 2.4.2 «Планирование в системах пакетной обработки» (First Come First Served); Глава 2.4.3 «Планирование в интерактивных системах» (Циклическое планирование)*

**FCFS (First Come First Served — «первым пришёл — первым обслужен»).** Простейший непреемптивный алгоритм. CPU выделяется процессам в порядке поступления их запросов. Все готовые процессы находятся в единой очереди FIFO. Когда работающий процесс блокируется или завершается, на выполнение запускается первый процесс из очереди. Заблокированный процесс при пробуждении встаёт в конец очереди.

**Достоинства FCFS:**
- Простота понимания и реализации.
- Справедливость в смысле «кто раньше пришёл, того раньше обслужили».
- Низкие накладные расходы (один связанный список).
- Нет голодания: каждый процесс рано или поздно дождётся CPU.

**Недостатки FCFS:**
- Большое среднее время ожидания при разной длительности задач.
- **Эффект конвоя** — короткие процессы блокируются за длинным. Пример из Tanenbaum: один CPU-bound процесс работает по 1 секунде, и множество I/O-bound, каждый из которых должен сделать 1000 чтений с диска. При FCFS I/O-bound процессы выполнят одно чтение, а потом ждут целую секунду, пока CPU-bound снова отработает. В итоге каждый I/O-bound заканчивается за 1000 с. Преемптивный алгоритм с квантом 10 мс делает то же самое за 10 с.
- Не подходит для интерактивных систем.

**Пример FCFS.** Поступили задания A (8 мин), B (4 мин), C (4 мин), D (4 мин) в порядке A, B, C, D. Оборотные времена: A=8, B=12, C=16, D=20; среднее = 14 мин.

**RR (Round Robin — циклическое планирование).** Один из самых старых, простых и наиболее часто используемых алгоритмов для интерактивных систем. Каждому процессу выделяется фиксированный **квант времени** (time slice/quantum). По истечении кванта, если процесс ещё работает, его принудительно вытесняют и помещают в конец очереди готовых, а CPU отдают следующему процессу из начала очереди. Если процесс блокируется или завершается раньше — переключение происходит немедленно.

**Реализация.** Связный список процессов в состоянии готовности. Когда квант истекает, процесс перемещается из начала в конец списка.

**Выбор кванта.** Ключевой параметр. Переключение контекста занимает время (сброс/перезагрузка кэша, обновление таблиц, ~1 мс в Tanenbaum):
- Слишком короткий квант → много переключений → большие накладные расходы. При 4 мс кванте и 1 мс переключения 20 % времени тратится впустую.
- Слишком длинный квант → плохая интерактивность. При 100 мс кванте и 50 процессах в очереди последний ждёт 5 секунд.

Разумный компромисс — **квант 20–50 мс**.

**Особенности RR:**
- Считаются все процессы равнозначными.
- Простая реализация.
- Хорошая отзывчивость для коротких интерактивных задач.
- Справедливость в смысле «всем по равной доле CPU».
- Не учитывает приоритеты и поведение процесса (I/O-bound vs CPU-bound).

**Пример RR.** Три процесса A, B, C готовы к выполнению. Квант = 4 мс. Очередь: A → B → C → A → B → C → … Через каждые 4 мс CPU переключается. Если C блокируется на середине своего кванта, очередь становится: A → B → A → B → … пока C не разблокируется.

**FCFS vs RR:**

| Параметр | FCFS | RR |
|---|---|---|
| Вытеснение | Нет | Да (по кванту) |
| Эффект конвоя | Сильный | Подавляется |
| Сложность | Очень низкая | Низкая |
| Подходит для | Пакетных систем | Интерактивных систем |
| Учёт приоритетов | Нет | Нет |
| Справедливость | По порядку поступления | По доле CPU |

Реальные ОС обычно используют **приоритетный round robin** — RR внутри каждого класса приоритетов плюс приоритетный выбор класса (см. вопросы 22, 27).
