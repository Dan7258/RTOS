## 11. Реализация процесса исполнения. Синхронизированный цикл. Управление через прерывания. Достоинства и недостатки. Пример.
*Источник: rtos.md, Глава 3.2.3 «Синхронизированный цикл»; Глава 3.2.4 «Прерывания»*

> «Суть этого подхода в использовании прерываний для ограничения частоты цикла».

**Синхронизированный цикл.** Развитие главного цикла и циклического планировщика. Простой цикл при низкой частоте событий тратит энергию на бесполезную «прокрутку». Идея — использовать прерывания, чтобы ограничить частоту цикла: цикл выполняет всю готовую работу, а затем останавливает процессор до прихода очередного прерывания.

```c
// Источник: rtos.md, Глава 3.2.3 «Синхронизированный цикл»
do {
    ...
    stop();           // ждать прерывания
    state = schedule();
} while (1);
```

Функция `stop()` использует специальную процессорную инструкцию (например, `WFI` — Wait For Interrupt у ARM), которая переводит процессор в режим пониженного энергопотребления до прихода прерывания. Обработчики прерываний устанавливают флаги готовности своих устройств; цикл анализирует флаги и выполняет нужную работу.

**Тонкость с гонкой.** Между проверкой «нет работы» и инструкцией `WFI` есть зазор: если прерывание придёт в этот момент, оно будет потеряно (точнее, реакция задержится до следующего).

> «Если прерывание придёт в момент между проверкой и остановкой процессора, оно будет пропущено и время реакции на него возрастёт до времени прихода следующего прерывания». Чтобы этого избежать, требуется аппаратная поддержка: `WFI` должна выводить процессор из ожидания и при запрещённых прерываниях, откладывая вызов обработчика до их разрешения. Тогда корректный `stop()` выглядит так:

```c
// Источник: rtos.md, Глава 3.2.3 «Синхронизированный цикл»
void stop(void) {
    disable_interrupts();
    if (nothing_to_do)
        wait_for_interrupts();
    enable_interrupts();
}
```

**Управление через прерывания.** Если приоритет — время реакции, цикл становится неприемлемым: любая ветвь должна доработать до конца, и высокоприоритетная функция не запускается сразу. Альтернатива — построить приложение на обработчиках прерываний: в `main()` пустой цикл `wait_for_interrupt()`, а вся логика — в обработчиках, привязанных к таблице векторов.

```c
// Источник: rtos.md, Глава 3.2.4 «Прерывания»
void handler1(void) { /* обработка события 1 */ }
void handler2(void) { /* обработка события 2 */ }

int main(void) {
    do { wait_for_interrupt(); } while (1);
}
```

Если обработчики вытесняемы (сохраняют/восстанавливают контекст), достигается лучшая скорость реакции — для самого приоритетного обработчика она становится фиксированной.

**Достоинства и недостатки:**

| Подход | Плюсы | Минусы |
|---|---|---|
| Синхронизированный цикл | Простая отладка; низкое энергопотребление; предсказуем | Реакция ограничена самой длинной ветвью цикла; недетерминизм из-за атомарных флагов |
| Управление через прерывания | Лучшая скорость реакции; фиксированная задержка для приоритетного обработчика | Тесная связь с железом; ограничено число векторов; сложно расширять (общая обработка, очереди, асинхронность); сложен перенос |

## 12. Реализация процесса исполнения. Энергосберегающий режим.
*Источник: rtos.md, Глава 2.1.4 «Прерывания»; Глава 3.2.3 «Синхронизированный цикл»*

> «Прерывания дают возможность реализовать в процессоре различные режимы энергосбережения. Например, система команд может содержать специальную инструкцию останова, которая приостанавливает выборку и исполнение инструкций процессором до получения прерывания».

**Энергосберегающий режим** — состояние процессора с пониженным энергопотреблением, в которое он переводится при отсутствии работы и из которого выходит по внешнему событию (прерыванию). Этот режим — естественное продолжение механизма прерываний: процессор не «крутится» в ожидании событий, а буквально приостанавливает выборку и исполнение инструкций.

**Аппаратная база.** Система команд содержит специальную инструкцию останова. В архитектуре ARM это `WFI` (Wait For Interrupt) — переводит процессор в режим ожидания до прихода прерывания. У других процессоров — аналогичные `HLT`, `STOP` и т. п.

**Использование в ОСРВ.** Если планировщик не нашёл готовых к выполнению потоков, ОС запускает «холостой» поток (idle thread), который выполняет инструкцию останова. После прихода прерывания процессор просыпается, обработчик переводит какой-либо поток в готовое состояние, планировщик его выбирает.

```c
// Источник: rtos.md, idle-поток в FX-RTOS (Глава 6)
void idle_thread(void* arg) {
    for (;;) {
        wait_for_interrupt();   // WFI / HLT
    }
}
```

**Проблема гонки между проверкой и остановкой.** Корректная реализация останова требует, чтобы инструкция `WFI` выводила процессор из ожидания и при запрещённых прерываниях (с отложенным вызовом обработчика). Это позволяет атомарно проверить «работы нет» и остановиться — см. реализацию `stop()` в вопросе 11.

**Достоинства:** существенно ниже энергопотребление, чем у простого суперцикла; естественно сочетается с реакцией на внешние события.

**Недостатки:** требуется аппаратная поддержка корректной семантики `WFI`; время выхода из глубоких энергосберегающих состояний может быть значительным, что вносит непредсказуемость в системы жёсткого реального времени.

## 13. Виртуальное адресное пространство. Защита памяти с использованием MPU.
*Источник: rtos.md, Глава 2.3.3 «Права доступа к памяти»; Глава 4.7.1 «MPU»*

> «Развитие этого метода — устройство защиты памяти (memory protection unit, MPU). MPU тесно связан с ядром процессора и содержит набор регистров, описывающих несколько регионов и права доступа к ним. Количество регионов обычно невелико, в пределах 8–16, но этого достаточно для большинства приложений, поскольку позволяет установить различные права для кода, данных и прочего».

**Виртуальное адресное пространство** — диапазон адресов, которыми оперирует программа; он может отличаться от физического. Защита памяти — это аппаратный контроль того, какие обращения программы допустимы.

**MPU (Memory Protection Unit — устройство защиты памяти).** Тесно связан с ядром процессора, содержит набор регистров, описывающих несколько регионов памяти и права доступа к ним (чтение/запись/исполнение). Количество регионов невелико — обычно 8–16, но этого хватает: достаточно отдельно описать код, данные, стек и устройства ввода-вывода.

**Простейший вариант** — два регистра: базовый адрес и размер. Доступ непривилегированного приложения за пределы региона вызывает исключение. Сами регистры программируются только привилегированной программой. Главный недостаток такого варианта — один регион, тогда как MPU позволяет несколько.

**Настройка MPU при одном процессе.** MPU программируется один раз при инициализации:

```c
// Источник: rtos.md, Глава 4.7.1 «MPU», Рис. 40
Регион 0: вся память,            права = привилегированный (код и данные ОС)
Регион 1: код приложения,        права = read+exec (непривилегированный)
Регион 2: данные приложения,     права = read+write (непривилегированный)
```

**Несколько процессов.** При переключении на другой процесс MPU перепрограммируется — обычно в обработчике программного прерывания, как часть переключения контекста.

**Ключевые свойства:**
- Виртуальное адресное пространство совпадает с физическим — MPU только проверяет права, не транслирует адреса.
- Нарушение прав → исключение → обработчик ОС, который локализует ошибку.
- Высокая предсказуемость по времени (нет таблиц трансляции, нет TLB-промахов) → подходит для систем жёсткого реального времени.

## 14. Виртуальное адресное пространство. MMU. Страничная и странично-сегментная виртуализация. Таблица страниц. Пример. Где хранится таблица страниц.
*Источник: rtos.md, Глава 2.3.3 «Права доступа к памяти»; Глава 4.7.2 «MMU»*

> «MMU — гораздо более продвинутое устройство, по сравнению с MPU. Идея заключается в отделении виртуального адресного пространства памяти, с которым работают программы, от физического адресного пространства. Память разбивается на набор блоков, фиксированного размера, называемых страницами. MMU задаёт отображение страниц виртуального адресного пространства на физические страницы».

**MMU (Memory Management Unit — устройство управления памятью).** Существенно мощнее MPU. Идея — отделить виртуальное адресное пространство программы от физического. Память разбивается на блоки фиксированного размера — **страницы**. MMU задаёт отображение виртуальных страниц на физические и хранит для каждой страницы права доступа и флаг наличия.

**Назначение MMU:**
- Каждой программе — собственное изолированное адресное пространство.
- Виртуальная память: программе доступно больше адресов, чем физической памяти. Часто используемые страницы — в ОЗУ, остальные — на диске; обращение к отсутствующей странице вызывает исключение, ОС подгружает её (свопинг, swapping).
- Гибкая настройка прав доступа на уровне страницы.

В системах реального времени свопинг не применяется (вносит непредсказуемость) — используется только защита и трансляция адресов.

**Страничная виртуализация.** Виртуальный адрес = `(номер виртуальной страницы, смещение)`. MMU подменяет номер виртуальной страницы на номер физической, смещение остаётся прежним.

**Странично-сегментная виртуализация.** Адресное пространство сначала делится на сегменты (код, данные, стек), каждый сегмент — на страницы. Сегмент задаёт логическую область с общими правами, страница — единица отображения. Применялась в Multics и x86 (16-битный режим), сейчас почти вытеснена чистой страничной моделью.

**Таблица страниц** — структура, хранящая отображение «виртуальный номер страницы → физический номер страницы + флаги». Часть виртуального адреса — индекс в таблице.

```
Формирование физического адреса (одноуровневая таблица):
// Источник: rtos.md, Глава 2.3.3, Рис. 18
виртуальный адрес = [индекс | смещение]
PTE = таблица[индекс]                    // PTE — Page Table Entry, запись таблицы
физический адрес = (PTE.phys_addr << k) | смещение
```

**Пример.** 32-битное пространство, страницы по 4 КБ → 20 бит индекса, 12 бит смещения. Таблица содержит 2²⁰ = 1 048 576 записей по 4 байта = 4 МБ.

**Где хранится таблица страниц.** В физической памяти. Процессор содержит специальный регистр с указателем на начало таблицы (CR3 у x86, TTBR — Translation Table Base Register — у ARM). При переключении контекста ОС меняет содержимое этого регистра — так каждому процессу даётся своё адресное пространство.

**TLB (Translation Lookaside Buffer)** — аппаратный кеш переводов внутри MMU, чтобы не обращаться к таблице страниц при каждом доступе.

## 15. Виртуальное адресное пространство. Инвертированные таблицы страниц. Пример. Быстрый поиск виртуального адреса.
*Источник: rtos.md, Глава 2.3.4 «Инвертированные таблицы страниц»*

> «Для систем с большим адресным пространством может использоваться так называемая инвертированная таблица страниц — хранение записей описывающих не виртуальные, а физические страницы (откуда и происходит название). В таком случае, виртуальный адрес уже не может использоваться для поиска трансляции и приходится пользоваться различными вариантами хеширования».

**Обычная (прямая) таблица** хранит записи по индексу = «номер виртуальной страницы». Размер таблицы пропорционален размеру виртуального пространства — для 64-битных систем это нереалистично велико.

**Инвертированная таблица** хранит записи о **физических** страницах: одна запись на каждую физическую страницу в системе. Запись содержит, какой виртуальный адрес (и какого процесса) отображается в эту физическую страницу. Размер таблицы пропорционален физической памяти, а не виртуальной — гораздо меньше.

**Проблема поиска.** При обращении программы у нас есть виртуальный адрес, а индексом таблицы является физический. Линейный перебор недопустим — слишком медленно.

**Решение: хеширование.** Виртуальный адрес (плюс PID — Process Identifier, идентификатор процесса) хешируется, результат — индекс в хеш-таблице. Бакет хеш-таблицы содержит указатель на цепочку записей инвертированной таблицы; в цепочке ищется запись с совпадающим виртуальным адресом и PID.

```
// Источник: rtos.md, Глава 2.3.4
1. h = hash(VPN, PID)                      // VPN — Virtual Page Number
2. бакет = hash_table[h]
3. идти по цепочке записей, пока не найдём (VPN, PID)
4. физическая страница = индекс найденной записи
```

**Пример.** Используется в PowerPC, ранних IBM AS/400, в Itanium. На встроенных RTOS почти не применяется.

**Быстрый поиск.** Несмотря на хеш-схему, на «горячих» путях используется тот же TLB: если перевод найден в TLB, инвертированная таблица не трогается. К таблице обращается только обработчик промаха TLB.

**Достоинства и недостатки:**
- (+) Размер таблицы не зависит от размера виртуального пространства.
- (–) Поиск — через хеш и цепочку коллизий → менее детерминирован, чем индексация по таблице.
- (–) Производительность ниже, чем у многоуровневых таблиц на средних задачах.

## 16. Виртуальное адресное пространство. Многоуровневые таблицы страниц. Пример. Где хранятся таблицы страниц.
*Источник: rtos.md, Глава 2.3.3 «Права доступа к памяти», Рис. 19*

> «Верхняя часть адреса используется как индекс в таблице верхнего уровня, называемой каталогом страниц. Средняя часть — индекс в таблице страниц, базовый адрес которого определяется с помощью записи каталога. Такой подход позволяет помечать как отсутствующие не только страницы, но и таблицы, через записи каталога страниц».

**Проблема одноуровневой таблицы.** Для 32-битного пространства со страницами 4 КБ нужна таблица размером 4 МБ — для каждого процесса. На большинстве задач большая её часть пуста.

**Идея многоуровневой таблицы.** Виртуальный адрес делится на несколько частей. Старшая часть — индекс в **каталоге страниц** (page directory), запись которого ссылается на **таблицу страниц** второго уровня. Средняя часть — индекс в этой таблице, дающий запись с физическим адресом страницы.

```
// Источник: rtos.md, Глава 2.3.3, Рис. 19
вирт. адрес = [индекс в каталоге | индекс в таблице | смещение]
                       │                  │
                       ▼                  ▼
             каталог страниц       таблица страниц
                       │                  │
                       ▼                  │
              физ. адрес таблицы    физ. адрес страницы → + смещение
```

**Экономия.** Если в системе 256 МБ физической памяти и страницы по 4 КБ: 65 536 страниц. Поделим 20 бит ВПН поровну: каталог 1024 записи (4 КБ), таблиц — 64 шт. по 1024 записи (4 КБ каждая) = 256 КБ. Итого 260 КБ вместо 4 МБ. Записи каталога могут быть помечены как отсутствующие — целая таблица не нужна, если соответствующий диапазон адресов не используется.

**64-битные системы.** Уровней может быть больше. x86-64 использует **четырёхуровневую** схему: PML4 (Page Map Level 4) → PDPT (Page Directory Pointer Table) → PD (Page Directory) → PT (Page Table). ARM позволяет страницы 4 КБ, 64 КБ, 1 МБ, 16 МБ — крупная страница уменьшает глубину и размер таблиц.

**Где хранятся.** Все таблицы — в физической памяти. Процессор знает только физический адрес корневой таблицы (каталога), который хранится в специальном регистре (CR3 у x86, TTBR у ARM). При промахе TLB процессор сам обходит таблицы и заполняет TLB; формат таблиц задаётся архитектурой и фиксирован — ОС обязана подготавливать таблицы согласно этому формату.

**Программная альтернатива.** Если процессор поддерживает программный обход, ОС сама задаёт структуру таблиц и обрабатывает промах TLB — это медленнее, но даёт гибкость и используется во встроенных системах с детерминированным отображением.

## 17. Размещение данных в виртуальном адресном пространстве. Код, данные, стек и куча. Пример.
*Источник: tanenbaum.md, Глава 1.6.1 «Системные вызовы для управления процессами»; Глава 1.7.1 «Структура операционной системы»; rtos.md, Глава 4.7.3 «Распределение адресного пространства»*

> «В UNIX память каждого процесса делится на три сегмента: текстовый сегмент (то есть код программы), сегмент данных (переменные) и сегмент стека. Сегмент данных растёт вверх, а стек растёт вниз. Между ними существует часть неиспользованного адресного пространства. Стек заполняет пустое пространство автоматически по мере надобности».

В UNIX (и большинстве современных ОС) виртуальная память процесса делится на сегменты с разным назначением:

- **Текстовый сегмент (`.text`)** — машинный код программы. Постоянный, не меняется во время выполнения, права = read+execute. Обычно расположен в районе младших адресов.
- **Сегмент данных (`.data`, `.bss`)** — глобальные и статические переменные. Имеет начальный размер; может расти вверх через системный вызов `brk`/`sbrk` или библиотеку `malloc()`.
- **Куча (heap)** — растущая вверх область над сегментом данных, из которой `malloc()` выдаёт блоки динамической памяти.
- **Сегмент стека (stack)** — расположен у верхней границы пользовательской области и растёт вниз. Заполняется автоматически по мере вложенных вызовов функций; содержит локальные переменные и адреса возврата. Между стеком и кучей — пустое неотображённое пространство.

**Пример карты процесса (32 бита, традиционный UNIX):**

```
// Источник: tanenbaum.md, Рис. 1.18
0xFFFFFFFF  ┌────────────────────────┐
            │ Область ядра           │ ← привилегированный режим
0xC0000000  ├────────────────────────┤
            │ Стек (растёт вниз)     │ ↓
            ├────────────────────────┤
            │ ... свободно ...       │
            ├────────────────────────┤
            │ Куча (растёт вверх)    │ ↑
            ├────────────────────────┤
            │ .bss (неинициализир.)  │
            │ .data (инициализир.)   │
            │ .text (код)            │
0x00000000  └────────────────────────┘
```

**Размещение ядра.** В системах с MMU ядро размещается в выделенной области виртуальных адресов (обычно старшие адреса), не пересекающейся с пользовательской. Эта область отображается в каждом процессе — чтобы системные вызовы выполнялись без переключения таблицы страниц. По старшему биту адреса можно сразу сказать, относится он к ядру или к пользователю.

## 18. Ограничения доступа к памяти. TLB. Своппинг.
*Источник: rtos.md, Глава 2.3.3 «Права доступа к памяти»*

> «Любой адрес, по которому обращается программа, это может быть явное обращение к данным или выборка инструкций, проходит процедуру трансляции — поиска записи в TLB, которая по данному виртуальному адресу определяет физический и далее этот физический адрес используется для доступа к памяти».

**Ограничения доступа к памяти.** Для каждой страницы (или региона MPU) задаются права: чтение, запись, исполнение, режим (привилегированный/пользовательский), флаг наличия (present). Любое обращение, нарушающее права, вызывает исключение, которое перехватывается ОС. Это и есть основной механизм защиты памяти и изоляции процессов.

**TLB (Translation Lookaside Buffer — буфер ассоциативной трансляции).** Аппаратный кеш переводов «виртуальный адрес → физический» внутри MMU. По структуре похож на кеш: по ключу (виртуальному адресу) сразу возвращает физический. Поиск параллельный по всему содержимому, поэтому объём ограничен — обычно несколько десятков записей.

```
// Источник: rtos.md, Глава 2.3.3, Рис. 16
запись TLB = | флаги | VPN | физ. номер страницы |
```

**Промах TLB (TLB miss).** Если запись для нужного виртуального адреса в TLB отсутствует, начинается заполнение:
- *Аппаратный обход таблиц* — процессор сам читает таблицы страниц из памяти и загружает запись в TLB. Прозрачно для ПО, но непредсказуемо по времени.
- *Программный обход* — возникает исключение, обработчик ОС загружает нужную запись в TLB через специальные инструкции. Медленнее, но даёт детерминизм; применяется во встраиваемых системах.

**Многопроцессорная согласованность.** У каждого процессора свой TLB. При изменении таблиц страниц на одном ядре нужно прерыванием (IPI — Inter-Processor Interrupt) уведомить остальные ядра, чтобы они инвалидировали соответствующие записи TLB (так называемый «TLB shootdown»). Аппаратная когерентность TLB существует, но менее распространена.

> «Помимо прав доступа, для каждой страницы указывается её наличие или отсутствие. Попытки обращения по адресу, который находится в странице, помеченной как отсутствующая, вызывают исключение, поэтому ОС оставляет в памяти только наиболее часто используемые приложениями страницы, а все остальные выгружает на внешние носители, такие как диски… Этот механизм называется свопингом (swapping)».

**Своппинг (swapping)** — механизм виртуальной памяти, при котором редко используемые страницы выгружаются на диск, а в ОЗУ остаются только активные. При обращении к выгруженной странице:
1. MMU обнаруживает, что флаг наличия = 0, и вызывает исключение page fault.
2. ОС находит свободную физическую страницу (при необходимости выгружая на диск чужую).
3. ОС читает нужную страницу с диска в эту физическую страницу.
4. ОС обновляет таблицу и возобновляет инструкцию, вызвавшую исключение.

**Почему в ОСРВ не применяется.** Свопинг вносит непредсказуемые задержки (миллисекунды и более при чтении с диска). В системах жёсткого реального времени используется только защита и трансляция; либо страницы с RT-кодом и данными помечаются как «прибитые» к физической памяти и не выгружаются, либо в TLB создаются невыгружаемые записи на эти регионы.

## 19. Планировщик. Точка входа в планировщик. Вызов планировщика. События + акторы/потоки.
*Источник: rtos.md, Глава 5.3.5 «Планирование»; Глава 4.4.3 «Переключение контекста»; Глава 3.3.1 «Акторы»; Глава 3.3.2 «Потоки»*

> «Понятие планировщика в ядре операционной системы включает в себя два аспекта: во-первых, планировщик понимается как некоторый алгоритм, который используется для выбора из нескольких готовых к выполнению потоков того, который будет исполняться следующим; во-вторых, планировщик также должен решать задачу синхронизации доступа к хранилищу готовых к выполнению потоков и предоставлять интерфейсы для изменения различных параметров потоков, таких как приоритеты».

**Планировщик** — компонент ОС, выбирающий из множества готовых к выполнению потоков следующий, который получит процессор. В понятие планировщика входят два аспекта: алгоритм выбора и слой синхронизации, управляющий контейнером готовых потоков и обеспечивающий потокобезопасный доступ к нему.

**Точки планирования (scheduling points).** Места в коде, где планировщику разрешено переключать поток:
- Поток заблокировался (ожидание семафора, очереди, ввода-вывода).
- Поток явно отдал процессор (`yield`).
- Истёк квант времени в round-robin.
- Появился готовый поток с более высоким приоритетом (например, его разбудил обработчик прерывания).
- Поток завершился.

**Вызов планировщика.** Возможны два подхода:
1. *Синхронный вызов* — планировщик работает на стеке текущего потока. Войдя в функцию переключения контекста в одном потоке, «выход» происходит уже в другом. Простая модель, но требует места под стек планировщика в каждом потоке и трудно отлаживается.
2. *Асинхронный (отложенный) вызов* — планировщик рассматривается как низкоприоритетное прерывание со своим стеком прерываний. Если обработчик аппаратного прерывания сделал что-то, что может потребовать переключения контекста, он устанавливает запрос «планировщик»; после выхода из всех вложенных обработчиков прерываний этот запрос автоматически выполнится.

```
// Источник: rtos.md, Глава 4.4.3, Рис. 32
ISR ────┐
        ▼
       ISR
        │
        ▼
Поток 1 → Ядро (планировщик) → Поток 2
```

**События + акторы/потоки.** В ОСРВ существуют два взаимосвязанных уровня:
- **Акторы (события)** — обработчики, выполняющиеся до конца (run-to-completion) на общем стеке. Активируются программно или из прерывания. Не имеют сохраняемого контекста и стека — лёгкие, но не могут блокироваться.
- **Потоки** — имеют собственный стек, состояние и могут блокироваться в ожидании событий. Между точками планирования их выполнение приостанавливается, состояние сохраняется и затем восстанавливается.

Планировщик потоков обычно выбирает поток с самым высоким приоритетом среди готовых; внутри одного приоритета — кооперативно или по round-robin. Акторы, как более лёгкие сущности, обычно имеют приоритет выше любого потока и обрабатываются перед возвратом в планировщик потоков.

## 20. Алгоритмы планирования FCFS, RR. Примеры.
*Источник: tanenbaum.md, Глава 2.4.2 «Планирование в системах пакетной обработки»; Глава 2.4.3 «Планирование в интерактивных системах»*

> «Наверное, наипростейшим из всех алгоритмов планирования будет неприоритетный алгоритм, следующий принципу „первым пришёл — первым обслужен". При использовании этого алгоритма центральный процессор выделяется процессам в порядке поступления их запросов».

**FCFS (First-Come, First-Served — первым пришёл, первым обслужен).** Простейший неприоритетный алгоритм. Готовые процессы хранятся в одной очереди в порядке поступления; центральный процессор выделяется первому в очереди. Процесс выполняется столько, сколько хочет (либо до блокировки). Разблокированный процесс попадает в конец очереди.

**Достоинства FCFS:** простота понимания и реализации (односвязный список); справедливость в смысле «кто первый встал, того и тапки».

**Недостатки FCFS:** один длинный CPU-bound процесс задерживает все I/O-bound. Пример (Tanenbaum): один процесс ограничен скоростью вычислений и работает 1 с; множество I/O-процессов делают по 1000 чтений с диска. При FCFS каждый I/O-процесс получает один блок за секунду и завершится за 1000 с. При вытесняющем планировании каждые 10 мс — за 10 с, почти не замедляя CPU-bound процесс.

**Пример FCFS.** Три задания A (8 с), B (4 с), C (4 с) пришли в порядке A, B, C.
- Оборотные времена: A=8, B=12, C=16. Среднее = 12 с.

> «Каждому процессу назначается определённый интервал времени, называемый его квантом, в течение которого ему предоставляется возможность выполнения. Если процесс к завершению кванта времени всё ещё выполняется, то ресурс центрального процессора у него отбирается и передаётся другому процессу».

**RR (Round-Robin — циклическое планирование).** Каждому процессу назначается квант времени. Процесс выполняется до конца кванта или до блокировки; по истечении кванта вытесняется и помещается в конец очереди готовых, процессор передаётся следующему.

```
// Источник: tanenbaum.md, Рис. 2.22
Очередь готовых: [A] → [B] → [C] → [D]
После исчерпания кванта B:
Очередь готовых: [C] → [D] → [A] → [B]
```

**Выбор кванта.** Слишком маленький → много переключений контекста, накладные расходы. Tanenbaum: при кванте 4 мс и переключении 1 мс теряется 20% процессорного времени. Слишком большой → плохая интерактивность: если квант 100 мс и в очереди 50 задач, последняя ждёт 5 с. Разумный компромисс — обычно 20–50 мс.

**Достоинства RR:** справедливое распределение времени; ограниченное время ожидания; подходит для интерактивных систем.

**Недостатки RR:** не учитывает приоритеты и характер задачи; накладные расходы на переключение контекста; в системах реального времени применяется только внутри одного приоритета (как часть приоритетной схемы).

**Сравнение:**

| Параметр | FCFS | RR |
|---|---|---|
| Вытеснение | Нет | По истечении кванта |
| Структура данных | FIFO-очередь | FIFO-очередь + таймер |
| Среднее время ожидания | Зависит от порядка | Равномерное |
| Применение | Пакетные системы | Интерактивные, ОСРВ (внутри приоритета) |
