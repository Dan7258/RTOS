## 101. Файловые системы. i-узлы. Достоинства и недостатки. Пример.
*Источник: tanenbaum.md, Глава 4.3.2; Глава 4.5.2 «Файловая система UNIX V7»*

### Зачем нужны i-узлы

Предыдущие методы:
- **Непрерывное:** простота, но фрагментация (вопрос 99).
- **Связный список / FAT:** медленный random access, FAT требует много RAM (вопрос 100).

Для **больших** дисков с **многими** файлами нужна схема, которая:
- Не требует много RAM (масштабируется).
- Быстрый random access.
- Удобные атрибуты.

**i-узлы** — решение.

### Понятие i-узла

Из tanenbaum.md (4.3.2):
> «Последним из рассматриваемых методов отслеживания принадлежности конкретного блока конкретному файлу является связь с каждым файлом структуры данных, называемой i-узлом (index-node — индекс-узел), содержащей атрибуты файла и дисковые адреса его блоков».

**i-узел** = структура, описывающая **один файл**:
- Атрибуты файла.
- Массив адресов блоков данных.

Каждый файл имеет ровно один i-узел.

### Структура UNIX V7

Из tanenbaum.md (рис. 4.10, 4.28):

```c
struct inode_v7 {
    // Атрибуты
    u16 mode;       // тип файла + права доступа
    u16 uid;        // ID владельца
    u16 gid;        // ID группы
    u32 size;       // размер в байтах
    u32 atime;      // время последнего доступа
    u32 mtime;      // время последней модификации
    u32 ctime;      // время последнего изменения метаданных
    u16 nlink;      // количество жёстких ссылок

    // Адреса блоков (13 элементов)
    u32 direct[10];          // прямые адреса первых 10 блоков
    u32 single_indirect;     // адрес однократного косвенного блока
    u32 double_indirect;     // двукратного косвенного
    u32 triple_indirect;     // трёхкратного косвенного
};
```

### Косвенные блоки — суть

Прямых 10 адресов хватает на маленькие файлы. Для больших — **косвенные блоки**.

**Однократный косвенный (single indirect):**
- `i-узел.single_indirect` указывает на блок диска.
- Этот блок содержит **массив адресов** блоков данных.

При размере блока 1 КБ и адресе 4 байта:
- Блок содержит 256 адресов.
- → 256 КБ дополнительных данных.

**Двукратный косвенный:**
- `i-узел.double_indirect` → блок с 256 адресами.
- Каждый из этих 256 адресов → блок с 256 адресами.
- 256 × 256 = 65 536 блоков → 64 МБ данных.

**Трёхкратный косвенный:**
- Ещё уровень.
- 256³ = 16M блоков → 16 ГБ.

### Иллюстрация

```
i-узел:
  атрибуты
  direct[0..9] →→→ 10 блоков данных
  single_ind  → [256 адресов] → 256 блоков данных
  double_ind  → [256 адресов] → 256 × [256 адресов] → 256² блоков
  triple_ind  → [256 адресов] → 256² × [256 адресов] → 256³ блоков
```

### Расчёт максимального размера файла

При блоке 1 КБ:
- Прямые: 10 КБ.
- Single indirect: 256 КБ.
- Double indirect: 64 МБ.
- Triple indirect: 16 ГБ.

Итого: ~16.06 ГБ. Для UNIX V7 1970-х — невероятно много.

В современных ext4 (блок 4 КБ, 32-bit адрес): максимум 4 ТБ на файл.

### Маленький файл — быстро

Для файла 5 КБ (5 блоков):
- Все адреса в `direct[]` самого i-узла.
- При open: 1 чтение i-узла. Все адреса блоков в памяти.
- При read: прямое чтение.

### Большой файл — больше I/O

Для файла 100 КБ (100 блоков):
- 10 в direct.
- 90 через single_indirect.
- При обращении к блоку 50: чтение single_indirect блока + блок 50.
- **2 I/O вместо 1.**

### Очень большой файл

Файл 100 МБ:
- Через double_indirect.
- При обращении: чтение double_indirect + single_indirect + сам блок.
- **3 I/O.**

### Главное преимущество — экономия RAM

Из tanenbaum.md (4.3.2):
> «Большим преимуществом этой схемы перед связанными списками, использующими таблицу в памяти, является то, что i-узел должен быть в памяти только в том случае, когда открыт соответствующий файл».

**Сравнение для 1 ТБ диска:**

| Метод | RAM на метаданные |
|---|---|
| FAT | ~4 ГБ (пропорц. диску) |
| i-узлы | k × 128 байт, где k = число открытых файлов |

При 10 000 открытых файлов и i-узле 128 байт → 1.3 МБ. На любом диске.

### Достоинства i-узлов

**1. Эффективное использование RAM.**

Только открытые файлы.

**2. Быстрый доступ к маленьким файлам.**

Прямые адреса достаточны.

**3. Поддержка больших файлов.**

Через косвенные блоки.

**4. Жёсткие ссылки.**

Несколько имён → один i-узел.

**5. Атрибуты централизованы.**

Изменение атрибутов — изменение одного i-узла.

### Недостатки

**1. Дополнительные I/O для больших файлов.**

При доступе через косвенные блоки — лишние чтения.

Решение в современных ФС:
- **Кэширование** косвенных блоков.
- **Экстенты** вместо списка отдельных блоков.

**2. Сложнее реализация.**

3. **Размер i-узла зависит от архитектуры.**

В ext2 i-узел 128 байт, в ext4 — 256.

### Пример: открытие /usr/ast/mbox в UNIX V7

Из tanenbaum.md (рис. 4.29):

1. **Найти i-узел корня.** В фиксированном месте на диске.
2. **Прочитать корневой каталог** через i-узел корня.
3. **Найти запись "usr"** → номер i-узла /usr.
4. **Прочитать i-узел /usr** (по номеру — фиксированное место для всех i-узлов).
5. **Прочитать каталог /usr.**
6. **Найти "ast"** → номер i-узла /usr/ast.
7. **Прочитать i-узел /usr/ast** → каталог.
8. **Найти "mbox"** → i-узел файла.
9. **Прочитать i-узел /usr/ast/mbox** → атрибуты + адреса блоков.

i-узел остаётся в памяти, пока файл открыт.

### Применение в современных ФС

i-узлы — основа большинства Unix/Linux ФС:
- **ext2/3/4** — классические.
- **XFS** — высокопроизводительная.
- **JFS** — IBM.
- **ZFS** — Sun (адаптация концепции).
- **Btrfs** — Linux (copy-on-write, B-tree).
- **NTFS** — Windows (MFT records аналогично).

### Современные расширения

**Экстенты** вместо массива адресов:

```c
struct extent {
    u32 logical_start;   // где в файле
    u32 length;          // сколько блоков
    u64 physical_start;  // где на диске
};
```

Преимущества:
- Меньше метаданных для непрерывных регионов.
- Эффективная random I/O.

ext4 использует экстенты вместо прямых/косвенных адресов.

### Резюме

- i-узел = атрибуты + массив адресов блоков.
- Адреса организованы по схеме direct/single/double/triple indirect.
- Маленькие файлы — все адреса в i-узле, быстро.
- Большие файлы — косвенные блоки, дополнительные I/O.
- RAM используется только для открытых файлов.
- Основа всех современных Unix ФС.

## 102. Файловые системы. Реализация каталогов в FAT и i-узлах. Атрибуты файлов при FAT и i-узлах.
*Источник: tanenbaum.md, Глава 4.3.3*

### Каталог — это что?

Из tanenbaum.md (4.3.3):
> «Основной функцией системы каталогов является преобразование ASCII-имени файла в информацию, необходимую для определения местоположения данных».

**Каталог** = таблица «имя файла» → «информация о файле».

Что именно хранится — зависит от файловой системы.

### Каталоги в FAT

Из tanenbaum.md (рис. 4.11, а):

В FAT (MS-DOS) запись каталога **самодостаточна** — содержит и имя, и атрибуты, и адрес.

```
Запись каталога MS-DOS — 32 байта:
┌────────────────────────────────────────┐
│ Имя файла (8 байт) — заполнено пробелами│
├────────────────────────────────────────┤
│ Расширение (3 байта)                    │
├────────────────────────────────────────┤
│ Атрибуты (1 байт)                       │
├────────────────────────────────────────┤
│ Зарезервированно для NT (1 байт)        │
├────────────────────────────────────────┤
│ Время создания, мс (1 байт)             │
├────────────────────────────────────────┤
│ Время создания (2 байта)                 │
├────────────────────────────────────────┤
│ Дата создания (2 байта)                  │
├────────────────────────────────────────┤
│ Дата последнего доступа (2 байта)        │
├────────────────────────────────────────┤
│ Старшие 16 бит первого кластера (2 байта)│
├────────────────────────────────────────┤
│ Время изменения (2 байта)                │
├────────────────────────────────────────┤
│ Дата изменения (2 байта)                 │
├────────────────────────────────────────┤
│ Младшие 16 бит первого кластера (2 байта)│
├────────────────────────────────────────┤
│ Размер файла (4 байта)                   │
└────────────────────────────────────────┘
```

### Атрибуты в FAT — 1 байт

| Бит | Атрибут | Назначение |
|---|---|---|
| 0 | R | Read-only |
| 1 | H | Hidden |
| 2 | S | System |
| 3 | V | Volume label |
| 4 | D | Directory |
| 5 | A | Archive (изменён) |

### Имя файла в FAT

Формат **8.3**:
- 8 символов основное имя.
- 3 символа расширение.
- Без точки в самом имени.
- Только верхний регистр.
- Ограниченный набор символов (A-Z, 0-9, некоторые знаки).

### Длинные имена в FAT (VFAT, LFN)

Microsoft расширила FAT для имён до 255 символов:
- Длинное имя хранится в **дополнительных записях** каталога.
- Каждая дополнительная запись маркируется атрибутами `R+H+S+V` (что для старых ОС выглядит как «странная запись» и игнорируется).
- Связано с обычной 8.3-записью.

Это обеспечивает обратную совместимость со старыми FAT-реализациями.

### Каталог в UNIX (i-узлы)

Из tanenbaum.md (рис. 4.11, б):

В UNIX каталог — **минимальный**. Только имя и номер i-узла:

```
Запись каталога UNIX V7 — 16 байт:
┌─────────────────────┐
│ Номер i-узла (2 б)  │
├─────────────────────┤
│ Имя файла (14 байт) │
└─────────────────────┘
```

Атрибуты — **в i-узле**, не в записи.

### Современный Linux (ext2/3/4)

Запись каталога переменной длины:

```c
struct ext2_dir_entry {
    __u32 inode;        // номер i-узла
    __u16 rec_len;      // длина записи (для перехода к следующей)
    __u8  name_len;     // длина имени
    __u8  file_type;    // тип файла (для оптимизации)
    char  name[];       // имя переменной длины
};
```

`file_type` — оптимизация: можно понять, что файл — каталог, не читая i-узел.

### Сравнение

| Параметр | FAT | i-узлы (UNIX) |
|---|---|---|
| Атрибуты в записи каталога | Да | Нет |
| Размер записи | Фиксированный (32 б) | Переменный |
| Имя | 8.3 или LFN | До 255 символов |
| Получить атрибуты | Из записи каталога | Прочитать i-узел |
| Жёсткие ссылки | Невозможны | Возможны |
| Размер каталога | Больше (атрибуты в каждой) | Меньше |
| Стандартный набор символов | Ограниченный | Любые, кроме `/` и `\0` |

### Жёсткие ссылки — преимущество i-узлов

**Жёсткая ссылка (hard link)** — несколько записей каталогов указывают на один i-узел.

```bash
$ ln existing_file new_link
```

В каталоге появляется запись `new_link` → тот же номер i-узла, что у `existing_file`.

В i-узле счётчик `nlink` увеличивается на 1.

После этого:
- `existing_file` и `new_link` — два имени **одного и того же** файла.
- Атрибуты, содержимое, размер — общие.
- Изменения видны через любое имя.

При удалении одной ссылки — `nlink` уменьшается. Файл реально удалится только когда `nlink = 0`.

### Почему в FAT жёсткие ссылки невозможны

Атрибуты (включая размер, время) — в каждой записи каталога. Если две записи на один файл:
- При изменении файла размер обновится в одной записи, не в другой.
- Несогласованность.

Symlinks (символические ссылки) — это **отдельная семантика**, поддерживается даже в FAT через специальные «файлы-ссылки».

### Эффективность переименования

В FAT:
- Поменять имя в одной записи каталога.

В UNIX:
- Удалить старую запись, добавить новую с тем же inode.
- Содержимое не трогается.

В обоих случаях — быстро.

### Поиск файла по пути

UNIX `/a/b/c.txt`:
1. i-узел корня.
2. Найти "a" → i-узел /a.
3. По нему → каталог /a.
4. Найти "b" → i-узел /a/b.
5. По нему → каталог /a/b.
6. Найти "c.txt" → i-узел /a/b/c.txt.

Каждый шаг: чтение каталога + чтение i-узла. Кэшируется в **dentry cache** (dcache).

### Атрибуты — где?

**FAT:**
- В записи каталога.
- Изменение прав = изменение записи.

**i-узлы:**
- В i-узле.
- Изменение прав = изменение i-узла.

### Хранение каталога

**FAT:**
- Корневой каталог: фиксированное место (для FAT12/16) или обычный файл (FAT32).
- Подкаталоги — обычные файлы с особым контентом.

**i-узлы:**
- Каталог — обычный файл, но i-узел помечен как тип «каталог».
- Содержимое — массив записей каталога.

### Резюме

- FAT: всё в записи каталога (имя, атрибуты, адрес первого блока).
- i-узлы: запись каталога минимальна (имя, номер i-узла); атрибуты в i-узле.
- i-узлы поддерживают жёсткие ссылки, FAT — нет.
- Имена в FAT ограничены 8.3 (LFN — расширение); в i-узлах — произвольные.

## 103. Журналируемые файловые системы. Иденпотентность операций.
*Источник: tanenbaum.md, Глава 4.3.4, 4.4.3*

### Проблема — сбои при записи

При записи файла нужно обновить несколько структур на диске:
1. Найти свободный блок.
2. Обновить bitmap или список свободных блоков.
3. Записать данные в найденный блок.
4. Обновить i-узел: размер, адрес нового блока.
5. Записать i-узел на диск.

Если на любом шаге выключается питание — **противоречивое состояние**.

Примеры противоречий:
- Блок выделен, но i-узел не знает о нём → потерянный блок.
- Блок в bitmap свободный, но используется в файле.
- Размер i-узла не соответствует реальному количеству блоков.

### Традиционное решение — fsck

Из tanenbaum.md (4.4.3):
> «В системе UNIX есть `fsck`… Эта утилита может быть запущена при каждой загрузке системы, особенно после сбоя».

`fsck`:
1. Сканирует все i-узлы.
2. Считает, какие блоки использованы.
3. Сравнивает с bitmap.
4. Сканирует все каталоги.
5. Считает ссылки.
6. Сравнивает с `nlink` i-узлов.
7. Исправляет несоответствия.

Проблема — **очень долго** для больших дисков. Час и больше для терабайтных.

### Журналируемые файловые системы

**Идея:** перед изменением структуры ФС записать **намерение** в специальный регион — **журнал** (log, journal).

Этапы:
1. **Запись в журнал:** «собираюсь сделать операции X, Y, Z».
2. **Sync журнала** на диск.
3. **Выполнение** X, Y, Z.
4. **Отметка** в журнале «выполнено».

При сбое:
- Если в журнале **есть незавершённая запись** → повторить операции из журнала.
- Если **выполнено** → ничего не делать.

### Восстановление вместо fsck

После сбоя:
1. Читать журнал.
2. Найти незавершённые транзакции.
3. **Повторить** их.

Это быстро — журнал маленький (десятки МБ).

### Идемпотентность операций

**Идемпотентная операция:** повторение даёт тот же результат.

Примеры:

**Идемпотентные:**
- ✅ `Set block X = data D`.
- ✅ `i-node Y: size = 1000, block[0] = X`.
- ✅ `Set bitmap[Z] = 1`.

**НЕ идемпотентные:**
- ❌ `Add 100 to i-node size` (после двух повторений размер вырастет на 200).
- ❌ `Append block to file` (добавит блок дважды).
- ❌ `Increment nlink` (вырастет на 2).

### Требование к журналируемой ФС

Все логируемые операции должны быть **идемпотентными**.

Это позволяет **безопасно повторить** их при восстановлении, даже если они уже частично/полностью выполнены до сбоя.

### Пример логируемой операции

Не «увеличить размер», а:
```
i-node 42: size = 5120
```

Повторение → тот же результат.

Не «добавить блок к файлу», а:
```
i-node 42: block[5] = 1000
i-node 42: size = 6144
```

### Уровни журналирования

**1. Full journaling** (полное).

В журнал пишутся **и данные, и метаданные**.

- Самое безопасное: после сбоя ничего не теряется.
- Самое медленное: всё пишется **дважды** (журнал + диск).

**2. Metadata journaling** (только метаданные).

В журнал пишутся **только** метаданные. Данные пишутся как обычно.

- Быстрее.
- Метаданные ФС всегда целостны.
- Но **данные могут быть потеряны** или несогласованы с метаданными.

Пример: i-узел может говорить «размер 1000», а блоки данных не записаны → читаем мусор.

**3. Ordered mode.**

Компромисс: метаданные в журнале, но **данные записываются перед** метаданными в журнал.

Гарантия: i-узел всегда указывает на **уже записанные** данные.

ext3/4 по умолчанию — ordered.

### Журналируемые ФС

- **ext3, ext4** (Linux).
- **NTFS** (Windows).
- **JFS** (IBM, Linux).
- **XFS** (SGI, Linux).
- **HFS+ Journaled** (macOS до APFS).
- **ReiserFS** (Linux).
- **Btrfs** (через snapshots/COW).

### Транзакции

Группа связанных операций — **транзакция**. Записывается в журнал атомарно.

Пример: запись файла:
- Транзакция включает изменения bitmap, i-узла, директории.
- Либо вся транзакция применяется, либо никакая.

### Алгоритм журналируемой записи

```
1. Begin transaction
2. Write operations to journal
3. Sync journal to disk (commit)
4. Apply operations to actual locations on disk
5. Mark transaction complete in journal
```

Если сбой между 3 и 5:
- Транзакция в журнале, но не на диске.
- При recovery → повторить операции из журнала.

Если сбой до 3:
- Транзакция не закоммичена.
- При recovery → отбросить.

### Накладные расходы

- **Двойная запись** для метаданных.
- **Износ** SSD (особенно).
- На современных дисках накладные расходы ~5-10%.

### Log-structured FS (LFS)

Радикальный подход: **вся ФС** — это журнал.

Из tanenbaum.md (4.3.5):
> «Идея была в том, чтобы превратить весь диск в один большой журнал».

Все записи добавляются в конец журнала. Старые версии остаются, пока не будут собраны мусорщиком.

Применение:
- **NILFS** (Linux).
- **F2FS** (для SSD).
- **ZFS, Btrfs** (copy-on-write).

### Резюме

- Журналируемые ФС записывают намерения перед выполнением.
- Быстрое восстановление после сбоя.
- Требуют **идемпотентности** операций.
- Уровни: full / metadata / ordered.
- ext3/4, NTFS, XFS используют ordered.

## 104. Ввод-вывод, отображаемый на пространство памяти.
*Источник: tanenbaum.md, Глава 5.1.3*

### Способы доступа к устройствам

Из tanenbaum.md (5.1.3):
> «На некоторых компьютерах регистры устройств отображаются в адресное пространство памяти».

Два подхода:

**1. Port-mapped I/O (отдельные I/O порты).**

Специальные инструкции:
```asm
in  al, dx     ; прочитать из порта DX
out dx, al     ; записать в порт
```

Адресное пространство I/O — отдельное от памяти. На x86: 65 536 16-битных портов.

**2. Memory-mapped I/O.**

Регистры устройств отображены на **адреса памяти**. Обычные load/store.

### Преимущества memory-mapped I/O

**1. Не нужны специальные инструкции.**

RISC-процессоры (ARM, MIPS, RISC-V) **не имеют** инструкций IN/OUT — только load/store. Все устройства через memory-mapped.

**2. C/C++ поддержка.**

Доступ через указатели:
```c
volatile uint32_t *reg = (volatile uint32_t *)0xF0001000;
*reg = 0x42;
uint32_t val = *reg;
```

`volatile` — нужно, чтобы компилятор не оптимизировал доступ.

**3. Защита через MMU.**

ОС может настроить таблицы страниц так, что регионы устройств доступны только ядру.

**4. Унификация.**

Память и I/O — одно адресное пространство, упрощает архитектуру.

### Недостатки

**1. Расход адресного пространства.**

На 32-bit с 4 ГБ ОЗУ и устройствами 1 ГБ → реально ОЗУ только 3 ГБ.

(На 64-bit проблема не актуальна — 256 ТБ + АП.)

**2. Кэширование может вызвать проблемы.**

Если регион кэшируется:
- Запись в регистр устройства попадёт в кэш.
- На устройство не дойдёт, пока кэш не флэшнут.

Решение: **отключить кэширование** для I/O регионов.

### Пример: UART

```c
#define UART_BASE 0xF0010000

typedef struct {
    volatile uint32_t data;       // 0x00
    volatile uint32_t status;     // 0x04
    volatile uint32_t control;    // 0x08
    volatile uint32_t baud_rate;  // 0x0C
} uart_regs_t;

uart_regs_t *uart = (uart_regs_t *)UART_BASE;

// Установить скорость
uart->baud_rate = 115200;

// Отправить байт
while (!(uart->status & TX_READY))
    ;
uart->data = 'A';

// Принять байт
while (!(uart->status & RX_READY))
    ;
char c = uart->data;
```

### В Linux

Драйвер не знает физический адрес заранее — он узнаётся при probe (для PCI) или из device tree (для ARM).

Чтобы получить доступ:
```c
void __iomem *regs;
regs = ioremap(physical_addr, size);
```

`ioremap`:
- Резервирует виртуальный регион в АП ядра.
- Настраивает таблицы страниц.
- Отключает кэширование (как правило).

Использование:
```c
u32 val = readl(regs + REG_STATUS);
writel(0x42, regs + REG_DATA);
```

`readl`/`writel` инкапсулируют:
- `volatile` доступ.
- Барьеры памяти (где нужны).
- Endian conversion (если нужно).

### Барьеры памяти

Если порядок записи в регистры важен:

```c
writel(setup, regs + REG_SETUP);
wmb();   // wait until previous writes complete
writel(GO, regs + REG_CONTROL);
```

Без `wmb()` процессор может переупорядочить → устройство получит команду до настройки.

### Освобождение

```c
iounmap(regs);
```

### __iomem аннотация

```c
void __iomem *regs;
```

`__iomem` — для sparse: «это I/O память, не RAM».

Sparse предупредит, если попытаться использовать как обычный указатель:
```c
*regs = 0x42;   // ОШИБКА: использовать writel
```

### Примеры применения

- **PCI/PCIe устройства.** BAR'ы.
- **Контроллер прерываний** (APIC, NVIC, GIC).
- **Системные таймеры.**
- **DMA контроллеры.**
- **Любые встроенные периферийные модули** на SoC.

### Резюме

- Memory-mapped I/O: регистры устройств в адресном пространстве памяти.
- Доступ через обычные load/store или `readl/writel`.
- Преимущества: простота, C/C++ поддержка, защита через MMU.
- Недостатки: расход АП, кэширование (отключается для регионов).
- В Linux: `ioremap` для маппинга.

## 105. Программный ввод-вывод. Опрос устройства.
*Источник: tanenbaum.md, Глава 5.2.2*

### Программный I/O — определение

Из tanenbaum.md (5.2.2):
> «Простейшая форма ввода-вывода — programmed I/O — заключается в том, что центральный процессор делает всю работу».

CPU **сам** управляет всеми этапами I/O:
- Инициирует операцию.
- Ждёт готовности (polling).
- Передаёт данные.
- Завершает.

Альтернативы:
- I/O с прерываниями.
- DMA.

### Полный пример

Принтер. Каждый символ:
1. Записать в регистр данных.
2. Подождать, пока принтер обработает (busy).
3. Когда принтер готов — следующий символ.

```c
void print(char *str, int n)
{
    for (int i = 0; i < n; i++) {
        // Polling — ждём, пока принтер готов
        while ((*printer_status & READY) == 0)
            ;
        // Записываем символ
        *printer_data = str[i];
    }
}
```

CPU занят 100% — ждёт в цикле, потом пишет.

### Опрос (polling)

**Опрос:** периодически проверять регистр статуса.

```c
// Busy wait
while (device_busy())
    ;
read_data();
```

Это **активное ожидание** — никакой полезной работы.

### Достоинства программного I/O

**1. Простота.**

Минимальный код, никакой инфраструктуры (нет прерываний, нет DMA).

**2. Низкая латентность.**

Сразу после готовности устройства — CPU реагирует.

**3. Предсказуемость.**

Точное время реакции (зависит от частоты опроса).

**4. Не требует прерываний.**

В простых системах без поддержки прерываний — единственный вариант.

### Недостатки

**1. CPU занят 100%.**

Никакой другой задачей.

**2. Очень неэффективно для медленных устройств.**

Принтер 10 символов/с → CPU 99.99% времени ждёт.

**3. Не масштабируется** на много устройств.

### Аналогия

CPU = студент, ждущий пиццу:
- Каждые 30 секунд звонит в пиццерию: «Уже едут?»
- Не делает уроки.
- Не смотрит фильм.
- Час спустя пицца наконец приедет.

### Когда применяется

**1. Микроконтроллеры** без ОС.

В простых embedded — нет планировщика, нет процессов, программный I/O — норма.

**2. Очень быстрые устройства.**

Если операция занимает наносекунды, polling быстрее прерывания (которое занимает микросекунды).

Пример: чтение CPU-локального регистра, проверка кэша.

**3. Системы реального времени.**

Когда **предсказуемое** время реакции важнее экономии CPU.

**4. Внутри обработчиков прерываний.**

Когда нужно подождать готовность какого-то регистра — polling.

```c
void my_isr() {
    while (!(reg_status & READY))
        ;
    // ...
}
```

### Pollable interface в Linux

В Linux есть **`poll`** и **`select`** — пользовательский API для опроса нескольких fd:

```c
struct pollfd fds[2] = {
    { .fd = sock_fd, .events = POLLIN },
    { .fd = stdin_fd, .events = POLLIN }
};
poll(fds, 2, -1);   // блокируется до готовности
```

Это не «активное» ожидание — внутри poll процесс спит.

Драйвер реализует `poll` метод:
```c
unsigned int (*poll)(struct file *, struct poll_table_struct *);
```

### Сравнение polling vs interrupts

| Параметр | Polling | Interrupts |
|---|---|---|
| Простота | Очень | Средне |
| Эффективность CPU | Плохо | Хорошо |
| Латентность | Зависит от частоты опроса | Низкая |
| Подходит для быстрых устройств | Да | Дорого (overhead) |
| Подходит для медленных | Нет (расход CPU) | Да |
| Гибкость | Минимальная | Высокая |

### Гибрид: polling после прерывания

Современные сетевые драйверы используют **NAPI** (New API):
1. Первое прерывание сигнализирует о пакете.
2. Драйвер выключает прерывания.
3. Polling-обработка нескольких пакетов.
4. Когда пакеты кончились — снова включить прерывания.

Это даёт: реакция как у прерываний + эффективность как у polling.

### Резюме

- Программный I/O — CPU полностью управляет операцией.
- Polling — активное ожидание готовности.
- Просто, но неэффективно для CPU.
- Применяется в микроконтроллерах, быстрых устройствах, RT-системах, внутри ISR.
- В Linux: poll/select для пользователя; NAPI для сетевых драйверов.

## 106. Ввод-вывод, управляемый прерываниями. Точные и неточные прерывания. Достоинства и недостатки точных и неточных прерываний. Синхронизация устройств.
*Источник: tanenbaum.md, Глава 5.2.3, 5.1.4*

### I/O с прерываниями

Из tanenbaum.md (5.2.3):
> «Чтобы позволить центральному процессору заниматься другой работой во время ожидания готовности печати… можно использовать прерывания».

Идея:
1. CPU **инициирует** операцию.
2. CPU делает **другую работу.**
3. Устройство, готовое, шлёт **прерывание.**
4. Обработчик прерывания обслуживает устройство.

### Пример: печать с прерываниями

```c
char *str;
int strpos, strlen;

void start_print(char *s, int n)
{
    str = s;
    strpos = 0;
    strlen = n;
    *printer_data = str[strpos++];
    enable_printer_irq();
}

void printer_isr()
{
    if (strpos < strlen) {
        *printer_data = str[strpos++];
    } else {
        disable_printer_irq();
        signal_completion();
    }
    acknowledge_irq();
}
```

CPU свободен между прерываниями.

### Достоинства I/O с прерываниями

**1. CPU занят полезной работой.**

**2. Эффективное использование** ресурсов.

**3. Хорошая отзывчивость.**

**4. Несколько устройств обслуживаются параллельно.**

### Недостатки

**1. Накладные расходы.**

Каждое прерывание:
- Сохранение/восстановление контекста.
- Влияние на конвейер, TLB, кэш.

Из tanenbaum.md:
> «При использовании современных центральных процессоров прерывания приводят к существенным издержкам в силу необходимости переключения контекста и их влияния на конвейерную обработку, буфер TLB и кэш».

**2. Сложнее код** — нужны очереди, синхронизация.

**3. Возможны гонки.**

### Точные прерывания (precise interrupts)

Из tanenbaum.md (5.1.4):

**Точное прерывание** — состояние процессора в момент прерывания **строго определено**:
- **PC указывает на следующую** ещё не выполненную инструкцию.
- **Все инструкции до PC** выполнены полностью.
- **Никакие инструкции после PC** не начаты.
- **Регистры** имеют согласованное состояние.

### Зачем нужны точные прерывания

**1. Page fault.**

Обработчик подгружает страницу и **возвращается** к той же инструкции для повтора. Это работает только если PC указывает точно на эту инструкцию.

**2. System call.**

Возврат — после `INT`/`SYSCALL`.

**3. Отладка.**

Дебаггер хочет знать точно, где остановка.

**4. Виртуальная память.**

Без точных прерываний демандовая подкачка невозможна.

### Достоинства точных прерываний

- **Простота программирования** обработчиков.
- **Корректность** VM, syscall, debug.
- **Прозрачность** для прикладной программы.

### Недостатки

- **Сложность реализации** в out-of-order процессорах.
- **Возможное снижение производительности** — нужно дождаться завершения «в полёте» операций.

### Неточные прерывания

В **out-of-order** процессорах инструкции могут быть в разной стадии:
- Некоторые **после** PC уже выполнены спекулятивно.
- Некоторые **до** PC ещё в конвейере.

При неточном прерывании:
- Сохраняется «snapshot» состояния, но неясно, какие именно операции уже произошли.
- Обработчик не может полагаться на точное состояние.

### Достоинства

- **Производительность процессора** — не нужно «осушать» конвейер при прерывании.

### Недостатки

- **Сложно реализовать ОС.**
- **Нельзя использовать для page fault.**
- **Сложно отлаживать.**

### Современные процессоры

x86, ARM, RISC-V — все поддерживают **точные прерывания** для архитектурно важных событий:
- Page fault.
- Syscall.
- Trap (debug).

Для **machine check** (аппаратная ошибка) могут быть неточные — там предположение, что система не восстанавливаема.

### Реализация точных прерываний в out-of-order

Через **reorder buffer** и **register renaming**:
- Инструкции выполняются в произвольном порядке.
- Но **завершаются** (commit) в программном порядке.
- При прерывании сбрасываются все спекулятивные результаты.

Накладные расходы — потенциальные потери конвейера.

### Синхронизация устройств

После прерывания нужно **синхронизировать** драйвер с устройством:

**1. Acknowledge.**

Подтвердить прерывание. Иначе устройство будет повторно генерировать его.

```c
writel(IRQ_ACK, regs + REG_IRQ_STATUS);
```

**2. Сброс флагов.**

Если устройство имеет флаги «готово», «ошибка» — обработать и сбросить.

**3. Подготовка к следующей операции.**

Загрузить новые данные или команду.

**4. Уведомление ожидающих.**

```c
wake_up_interruptible(&dev->wait);
```

### Гонки

При работе с прерываниями возможны гонки:
- Драйвер читает статус.
- Между чтением и обработкой статус меняется.

Решения:
- **Запрет прерываний** на короткое время.
- **Спинлоки** с `spin_lock_irqsave`.
- **Атомарные операции.**

### Полудуплексная синхронизация устройств

Сложные устройства имеют несколько каналов (TX, RX), которые могут прерывать одновременно. Драйвер должен корректно обрабатывать комбинации.

```c
irqreturn_t my_isr(int irq, void *dev_id) {
    u32 status = readl(regs + REG_STATUS);

    if (status & RX_READY) {
        handle_rx();
    }
    if (status & TX_DONE) {
        handle_tx_complete();
    }
    if (status & ERROR) {
        handle_error();
    }

    writel(status, regs + REG_STATUS_ACK);   // ACK всех битов
    return IRQ_HANDLED;
}
```

### Резюме

- I/O с прерываниями: CPU инициирует, продолжает работу, обработчик завершает.
- Точные прерывания: чёткое состояние процессора; нужны для VM, debug, syscall.
- Неточные: проще для out-of-order, но ограниченное применение.
- Синхронизация после прерывания: ACK, очистка флагов, уведомление ожидающих.

## 107. Устройства ввода-вывода. Ввод-вывод с использованием DMA.
*Источник: tanenbaum.md, Глава 5.1.4, 5.2.4*

### Зачем DMA

Для быстрых устройств обработка каждого пакета через прерывание + копирование данных CPU — катастрофа:
- Сетевая карта 10 Гбит/с, пакеты 1500 байт → 800 000 пакетов/с.
- 800 000 прерываний/с.
- CPU занят только их обработкой.

Нужна **разгрузка** CPU.

### DMA — Direct Memory Access

Из tanenbaum.md (5.1.4):
> «Многие компьютеры используют для оптимизации работы возможности прямого доступа к памяти (DMA), который может управлять потоком битов между памятью и некоторыми контроллерами без постоянного вмешательства центрального процессора».

**DMA-контроллер** — отдельная подсистема, передающая данные **между устройством и памятью** напрямую, без CPU.

### Простой DMA-обмен

Из tanenbaum.md (5.2.4):

1. CPU **настраивает** DMA-контроллер:
   - Адрес источника.
   - Адрес назначения.
   - Размер передачи.
   - Направление.

2. CPU **запускает** DMA.

3. CPU **делает другие задачи.**

4. DMA-контроллер **передаёт** данные **байт за байтом** (или большими порциями), вмешиваясь в шину.

5. По завершении DMA шлёт **одно** прерывание.

### Иллюстрация

```
Без DMA:
  Устройство → CPU → RAM        (CPU занят)
  Устройство → CPU → RAM        (CPU занят)
  ...

С DMA:
  CPU → DMA: "передать N байт"  (CPU свободен)
  Устройство → DMA → RAM        (CPU делает другое)
  Устройство → DMA → RAM
  ...
  DMA → CPU: "готово"           (одно прерывание)
```

### Преимущества DMA

**1. CPU свободен** во время передачи.

**2. Меньше прерываний** — одно на блок, а не на байт.

**3. Эффективнее** на больших объёмах.

**4. Параллелизм** — CPU и устройство работают одновременно.

### Недостатки

**1. Сложность реализации.**

Нужны:
- DMA-контроллер.
- Поддержка в драйверах.
- Координация с CPU.

**2. Cache coherency проблемы** (см. ниже).

**3. Расход памяти** на DMA-буферы.

### Типы DMA

**1. Block (burst) transfer.**

Передаётся блок данных, потом одно прерывание.

**2. Cycle stealing.**

DMA «крадёт» циклы шины у CPU — передаёт данные понемногу. CPU замедляется, но работает.

**3. Burst transfer.**

DMA блокирует шину на короткое время и передаёт много данных. Быстро.

**4. Scatter-gather.**

Один DMA-запрос на несколько участков памяти (например, фрагментированный пакет).

### Cache coherency

**Проблема:**
- CPU читал данные → они в кэше.
- DMA пишет новые данные в RAM.
- CPU при чтении видит **старые** данные из кэша.

Симметричная проблема при записи:
- CPU модифицирует данные → они в кэше (write-back).
- DMA читает из RAM → старые данные.

**Решения:**

**1. Snooping.**

Контроллер кэша «подслушивает» транзакции DMA и инвалидирует свои строки.

**2. Software cache management.**

Драйвер вручную:
- Перед DMA-чтением: инвалидировать кэш-строки получателя.
- После DMA-записи: инвалидировать кэш у CPU.
- Перед DMA-записью устройства из памяти: flush кэша.

**3. Non-cacheable регион.**

DMA-буферы — в специальной области без кэширования.

### DMA в Linux

```c
#include <linux/dma-mapping.h>
```

### Когерентный DMA

```c
dma_addr_t dma_handle;
void *cpu_addr;

cpu_addr = dma_alloc_coherent(&pdev->dev, size, &dma_handle, GFP_KERNEL);
```

Возвращает:
- `cpu_addr` — виртуальный адрес для CPU.
- `dma_handle` — физический адрес для устройства.

Память **гарантированно когерентна** — не нужно ручное управление кэшем.

Использование:
```c
// CPU пишет
strcpy(cpu_addr, "Hello DMA");

// Устройство читает (по dma_handle)
writel(dma_handle, regs + REG_DMA_SOURCE);
writel(size, regs + REG_DMA_LEN);
writel(GO, regs + REG_DMA_CMD);
```

Освобождение:
```c
dma_free_coherent(&pdev->dev, size, cpu_addr, dma_handle);
```

### Streaming DMA

Для однократных передач:

```c
// Один раз настроить
dma_addr_t handle = dma_map_single(&pdev->dev, buffer, size, DMA_TO_DEVICE);

// Устройство читает из buffer
writel(handle, regs + REG_DMA);

// Подождать завершения...

// Освободить
dma_unmap_single(&pdev->dev, handle, size, DMA_TO_DEVICE);
```

Направления:
- `DMA_TO_DEVICE` — память → устройство.
- `DMA_FROM_DEVICE` — устройство → память.
- `DMA_BIDIRECTIONAL` — оба.

### Применение

- **Сетевые карты** (NIC).
- **SATA/SCSI/NVMe контроллеры.**
- **Видеокарты.**
- **Звуковые карты.**
- **USB контроллеры.**

Практически все современные устройства высокой производительности.

### Bus mastering vs Third-party DMA

**Third-party DMA** (старый ISA-DMA):
- Отдельный DMA-контроллер.
- Устройство просит его о передаче.

**Bus mastering** (современный PCI/PCIe):
- Само устройство «захватывает шину» и инициирует передачу.
- Не нужен отдельный контроллер.

### IOMMU

**IOMMU** (I/O MMU) — аналог MMU для устройств:
- Устройства видят **виртуальные** I/O адреса.
- IOMMU транслирует в физические.

Преимущества:
- **Безопасность:** устройство не может писать «куда попало».
- **Виртуализация:** разные ВМ могут иметь свои DMA-буферы.

В Linux: VT-d (Intel), AMD-Vi.

### Резюме

- DMA — передача данных без CPU.
- Контроллер настраивается, передаёт, шлёт одно прерывание.
- Снижает нагрузку на CPU.
- Проблемы: cache coherency, расход RAM.
- Решения: snooping, software management, non-cacheable регионы.
- В Linux: `dma_alloc_coherent`, `dma_map_single`.

## 108. Уровни программного обеспечения ввода-вывода. Реентерабельность драйверов.
*Источник: tanenbaum.md, Глава 5.3*

### Слои ввода-вывода

Из tanenbaum.md (5.3):

Программное обеспечение I/O организовано слоями:

```
┌────────────────────────────────────────┐
│  Уровень 4: Пользовательские программы  │
├────────────────────────────────────────┤
│  Уровень 3: Device-independent software │
├────────────────────────────────────────┤
│  Уровень 2: Драйверы устройств           │
├────────────────────────────────────────┤
│  Уровень 1: Обработчики прерываний       │
├────────────────────────────────────────┤
│  Уровень 0: Аппаратное обеспечение       │
└────────────────────────────────────────┘
```

Каждый уровень:
- **Скрывает детали** уровня ниже.
- **Предоставляет абстракции** уровню выше.

### Уровень 1: Обработчики прерываний

Самый низкий программный уровень.

Задачи:
- Реакция на аппаратные сигналы.
- Сохранение контекста.
- Вызов соответствующего драйвера.

**Скрывает** от верхних слоёв факт прерывания.

См. вопросы 72-75.

### Уровень 2: Драйверы устройств

**Драйвер** управляет конкретным типом железа.

Задачи:
- Принимать абстрактные запросы от верхнего слоя.
- Преобразовывать в команды устройства.
- Обрабатывать ответы.
- Возвращать результат.

Каждое устройство — свой драйвер.

**В Linux** драйверы — модули ядра. Реализуют `file_operations` (для символьных), `block_device_operations` (для блочных) и т.д.

### Уровень 3: Device-independent software

**Универсальный** код, общий для всех устройств:

Из tanenbaum.md (5.3.3):
- Единое именование устройств.
- Защита.
- Буферизация (см. вопрос 109).
- Распределение/освобождение устройств.
- Сообщение об ошибках.
- Управление размером блока.

### Уровень 4: Пользовательские программы

Программы используют I/O через стандартные библиотеки:
- libc: `printf`, `fopen`, `read`, `write`.
- C++: iostream.
- glibc: дополнительные обёртки.

### Преимущества слоистой архитектуры

**1. Изоляция.**

Изменения в одном слое не затрагивают другие.

**2. Переносимость.**

Программы используют **device-independent** интерфейс — работают на разном железе.

**3. Простота добавления** новых устройств.

Достаточно написать драйвер.

### Реентерабельность драйверов

Из tanenbaum.md (5.3.2):
> «Драйверы должны быть реентерабельны (повторновходимы)».

**Реентерабельная функция** — может быть **вызвана повторно** до завершения первого вызова, без проблем.

### Зачем реентерабельность

**1. Многопоточность.**

Несколько процессов могут одновременно открыть устройство и делать I/O.

**2. SMP.**

На разных CPU драйвер может выполняться параллельно.

**3. Вытеснение.**

Один поток выполняет драйвер, прерывание разблокировало другой более приоритетный → он тоже идёт в драйвер.

**4. Прерывания на одном уровне.**

Вложенные обработчики (на разных приоритетах).

### Требования к реентерабельной функции

**1. Не использует глобальные переменные** без синхронизации.

**2. Не использует статические локальные переменные.**

```c
// НЕ реентерабельно
int format(int n) {
    static char buf[64];   // ПЛОХО
    sprintf(buf, "%d", n);
    return process(buf);
}
```

**3. Не возвращает указатель на статические данные.**

```c
// НЕ реентерабельно
char *itoa(int n) {
    static char buf[64];   // ПЛОХО
    sprintf(buf, "%d", n);
    return buf;
}
```

**4. Не вызывает нереентерабельных функций.**

### Способы сделать функцию реентерабельной

**1. Локальные переменные на стеке.**

```c
// Реентерабельно
int format(int n) {
    char buf[64];   // на стеке — у каждого вызова свой
    sprintf(buf, "%d", n);
    return process(buf);
}
```

**2. Возвращать через параметр.**

```c
void itoa(int n, char *buf, size_t size) {
    snprintf(buf, size, "%d", n);
}
```

**3. Защита глобальных данных синхронизацией.**

```c
struct device_state {
    spinlock_t lock;
    /* shared data */
};

ssize_t my_read(...) {
    unsigned long flags;
    spin_lock_irqsave(&dev->lock, flags);
    // работа с общими данными
    spin_unlock_irqrestore(&dev->lock, flags);
}
```

### Пример реентерабельного драйвера

```c
struct mydev {
    spinlock_t  lock;
    char        buffer[BUFSIZE];
    size_t      head, tail;
    wait_queue_head_t wait;
};

ssize_t my_read(struct file *filp, char __user *user_buf,
                size_t count, loff_t *ppos)
{
    struct mydev *dev = filp->private_data;
    unsigned long flags;
    char tmp[256];
    size_t to_copy;

    // Локальный буфер на стеке (реентерабельно)
    spin_lock_irqsave(&dev->lock, flags);
    to_copy = min(count, available(dev));
    memcpy(tmp, &dev->buffer[dev->tail], to_copy);
    dev->tail = (dev->tail + to_copy) % BUFSIZE;
    spin_unlock_irqrestore(&dev->lock, flags);

    // copy_to_user без удержания спинлока
    if (copy_to_user(user_buf, tmp, to_copy))
        return -EFAULT;

    return to_copy;
}
```

Особенности:
- Локальный буфер `tmp` на стеке.
- Спинлок защищает общие данные.
- `copy_to_user` **вне** спинлока (может блокироваться).

### Резюме

- 4 уровня I/O: пользователь, device-independent, драйверы, обработчики прерываний.
- Каждый уровень скрывает детали и предоставляет абстракции.
- Драйверы должны быть реентерабельны для SMP и многопоточности.
- Реентерабельность — локальные данные на стеке + синхронизация общих.

## 109. Уровни программного обеспечения ввода-вывода. Программное обеспечение, не зависящее от конкретных устройств. Двойная буферизация.
*Источник: tanenbaum.md, Глава 5.3.3*

### Device-independent layer — обзор

Из tanenbaum.md (5.3.3):
> «Программное обеспечение, не зависящее от конкретных устройств, выполняет ту работу для ввода-вывода данных, которая является общей для всех устройств, и предоставляет единый интерфейс программному обеспечению уровня пользователя».

Функции:
- Единое именование.
- Защита.
- Буферизация.
- Контроль размера блока.
- Распределение/освобождение.
- Обработка ошибок.

### Единое именование

В Unix: устройства видны как **файлы** в `/dev`:
- `/dev/sda`, `/dev/sda1` — диски, разделы.
- `/dev/tty1`, `/dev/ttyS0` — терминалы.
- `/dev/null` — null-устройство.
- `/dev/random`, `/dev/urandom` — источники случайных чисел.

Все они доступны через `open/read/write/close`.

В Windows: `C:`, `D:`, или специальные имена `\\.\COM1`, `\\.\PhysicalDrive0`.

### Защита

Через **права файлов** в `/dev`:
```
$ ls -l /dev/sda
brw-rw---- 1 root disk 8, 0 ... /dev/sda
```

Только root и пользователи группы `disk` могут открыть.

Это единый механизм для всех устройств.

### Буферизация — зачем

Из tanenbaum.md:

**1. Уменьшить число системных вызовов.**

Прямая запись каждого байта = очень много syscalls. Буферизация — собрать вместе, отправить пачкой.

**2. Сгладить разницу скоростей.**

CPU быстрее устройства. Без буфера CPU «крутится» при каждой записи.

**3. Принять асинхронные данные.**

Устройство шлёт данные, когда хочет. Без буфера данные могут быть потеряны, если приёмник не готов.

### Без буферизации

Каждый байт от пользователя → syscall → I/O → ожидание:
- Очень медленно.
- Огромный overhead.

### Одинарная буферизация

В ядре буфер. Пользователь → буфер → когда заполнен или явный flush → отправка.

```
User space:
  write(fd, byte) → syscall

Kernel:
  put byte in buffer
  if buffer full or sync requested:
      send buffer to device
```

### Проблема одинарной буферизации

Пока буфер передаётся на устройство, пользователь не может писать (ждёт освобождения).

```
Время:
  Буфер: [заполнить][передать][заполнить][передать]
  User:  [пишет   ][простой][пишет   ][простой]
```

CPU и устройство работают по очереди — не параллельно.

### Двойная буферизация

Из tanenbaum.md (5.3.3):
> «Можно использовать двойную буферизацию: пока заполняется один буфер, опустошается второй, и наоборот».

Два буфера, роли меняются:

```
Время:
  Буфер A: [заполн][передача][заполн][передача]...
  Буфер B: [передача][заполн][передача][заполн]...
  User:    [пишет А][пишет B][пишет A][пишет B]
  Device:  [читает B][читает A][читает B][читает A]
```

**Пользователь и устройство работают параллельно** — нет простоя.

### Применение двойной буферизации

**1. Видеовывод.**

Front buffer — то, что видно. Back buffer — рисуется следующий кадр. После полной отрисовки — swap.

Используется во всех графических системах.

**2. Аудио.**

Один буфер играется ЦАП'ом, второй заполняется кодеком.

**3. Сетевые буферы.**

Один отправляется, второй принимает следующий пакет.

**4. Принтеры.**

Один лист печатается, следующий готовится.

**5. GUI.**

Текущий кадр на экране, следующий рисуется в памяти.

### Circular (round-robin) buffer

Развитие — **N буферов** в кольце.

```
[B0][B1][B2][B3][B4][B5]
 ↑              ↑
 head           tail
```

- Производитель добавляет в head.
- Потребитель читает из tail.
- При полном буфере — блокировка производителя.

Применение:
- Linux NIC ring buffers.
- USB ring buffers.
- DMA ring buffers.
- ALSA audio ring buffers.

### Size of block

Разные устройства имеют разные «естественные» размеры:
- Диск: 512 байт или 4 КБ.
- TCP: 1500 байт (MSS).
- USB: 64-1024 байт (зависит от типа).

Device-independent layer предоставляет **единый** размер блока пользователю (обычно размер страницы памяти, 4 КБ).

### Распределение/освобождение устройств

Некоторые устройства **эксклюзивны** (принтер, привод):
- Только один процесс одновременно.
- `open()` ставит в очередь.

Device-independent уровень управляет:
- Очередью запросов.
- Распределением устройств.
- Освобождением при `close()`.

### Сообщения об ошибках

Стандартные коды (`errno`):
- `EACCES`, `EBUSY`, `ENODEV`, `EIO`, `ENOSPC`, `EPIPE`.

Драйвер возвращает код, device-independent уровень транслирует в errno пользователю.

### Резюме

- Device-independent layer: общий код для всех устройств.
- Функции: именование, защита, буферизация, размер блока, очереди, ошибки.
- Буферизация — снижение overhead, сглаживание скоростей.
- Двойная буферизация: параллельная работа пользователя и устройства.
- Кольцевые буферы — для нескольких уровней буферизации.

## 110. Нижняя и верхняя половина обработчика прерываний. Запрет прерываний. Что делает верхняя половина прерывания для сетевого драйвера.
*Источник: robert_lav.md, Глава 7 «Прерывания и их обработка»*

### Контекст и цели

См. вопрос 75 для общей концепции верхней/нижней половины. Здесь — фокус на Linux-реализацию и сетевые драйверы.

### Зачем разделение

Из robert_lav.md (Глава 7):
> «Два выдвинутых выше требования о том, что обработчик прерывания должен выполняться как можно быстрее и при этом делать много работы, являются противоречивыми. В связи с этим процесс обработки прерываний разбивается на две части, или половины».

**Конфликт:**
- Прерывания должны обрабатываться быстро (другие прерывания ждут).
- Но обработка сама может быть длинной.

**Решение:** разбить на:
- **Верхнюю половину** — быстрая часть.
- **Нижнюю половину** — длинная, выполняется позже.

### Верхняя половина (top half)

Из robert_lav.md:
> «Обработчик прерывания относится к верхней половине (top half). Он запускается сразу после получения прерывания и выполняет только ту работу, которая критична к задержкам по времени, такую, как отправка подтверждения о получении прерывания или сброс аппаратного устройства».

Задачи:
- Минимальная обработка.
- ACK устройства.
- Сохранение состояния для нижней половины.
- Запуск нижней половины.

### Нижняя половина (bottom half)

> «Работа, которую можно выполнить позже, откладывается до запуска нижней (или основной) половины (bottom half). Обработка нижней половины начинается позже, в более удобное время, когда все прерывания разрешены».

В Linux есть несколько механизмов нижних половин:
- **Softirqs** — основа, специально для производительных задач.
- **Tasklets** — обёртка над softirq, проще в использовании.
- **Workqueues** — выполняются в потоках ядра, могут блокироваться.

### Сетевой драйвер: верхняя половина

Из robert_lav.md:
> «Когда сетевая плата получает прибывшие из сети пакеты, она должна уведомить ядро о том, что доступны новые данные. Причем это необходимо сделать как можно скорее».

Действия верхней половины:

**1. Прочитать статус.**

```c
u32 status = readl(regs + STATUS_REG);
```

**2. Подтвердить прерывание.**

```c
writel(status, regs + STATUS_REG);   // ACK
```

Без ACK устройство будет повторно генерировать прерывание.

**3. Скопировать пакеты из буфера устройства в память.**

> «Ядро должно очень быстро скопировать сетевой пакет в основную память компьютера, поскольку буфер данных сетевой платы имеет небольшой фиксированный размер (по сравнению с размером основной памяти компьютера). Любая задержка с копированием пакета может привести к ситуации переполнения буфера».

С DMA это уже сделано:
- Устройство пишет прямо в RAM.
- Драйвер только узнаёт, в какие буферы.

**4. Сделать sk_buff.**

`sk_buff` — структура для пакета в Linux. Драйвер создаёт sk_buff из RAW-данных.

**5. Запустить нижнюю половину.**

```c
netif_rx(skb);
// или (новый API)
napi_schedule(&napi);
```

Это запланирует обработку в softirq.

### Сетевой драйвер: нижняя половина

В **NET_RX_SOFTIRQ**:

1. Разбор Ethernet-заголовка.
2. Извлечение типа протокола (IP, ARP, ...).
3. Передача в соответствующий обработчик.
4. IP-обработчик: проверка, маршрутизация.
5. TCP/UDP-обработчик: проверка checksum, постановка в socket buffer.
6. Уведомление приложения (через socket fd).

### Регистрация обработчика

```c
int request_irq(unsigned int irq,
                irq_handler_t handler,
                unsigned long flags,
                const char *name,
                void *dev);
```

Параметры:
- `irq` — номер прерывания.
- `handler` — функция-обработчик.
- `flags` — флаги.
- `name` — имя (в /proc/interrupts).
- `dev` — указатель, передаётся в обработчик.

### Прототип обработчика

```c
typedef irqreturn_t (*irq_handler_t)(int, void *);

irqreturn_t my_isr(int irq, void *dev_id);
```

Возврат:
- `IRQ_HANDLED` — обработали.
- `IRQ_NONE` — не наш (для shared IRQ).
- `IRQ_WAKE_THREAD` — разбудить threaded handler.

### Флаги

Из robert_lav.md:

**`IRQF_DISABLED`** — все прерывания запрещены во время выполнения.

> «В большинстве обработчиков прерываний данный флаг не устанавливается, поскольку запрещать все прерывания — это плохая идея. Данный флаг используется для критичных по времени обработчиков прерываний».

**`IRQF_SAMPLE_RANDOM`** — использовать тайминги прерываний в /dev/random.

> «Этот флаг нельзя устанавливать, если устройство генерирует прерывания в предсказуемые моменты времени (как, например, системный таймер) или на устройство может повлиять внешний злоумышленник (как, например, сетевое устройство)».

**`IRQF_TIMER`** — таймерное прерывание.

**`IRQF_SHARED`** — линия IRQ может разделяться.

> «Этот флаг указывает, что данный номер IRQ может совместно использоваться несколькими обработчиками прерываний. Его нужно использовать при регистрации обработчика прерываний, если последний использует одну и ту же линию IRQ с другими обработчиками прерываний».

### Запрет прерываний

В современных Linux обработчик прерывания обычно выполняется с **запрещёнными прерываниями той же линии**. Других прерываний — разрешены.

Это связано с тем, что:
- Без `IRQF_DISABLED` другие прерывания могут вытеснять.
- Это улучшает реакцию системы.
- Но требует тщательной синхронизации.

### Освобождение

```c
void free_irq(unsigned int irq, void *dev_id);
```

При выгрузке модуля.

### Резюме

- Верхняя половина: быстрая, ACK, копирование, запуск bottom half.
- Нижняя половина: длинная обработка протоколов.
- Сетевой драйвер: ACK + DMA-данные → sk_buff → запуск softirq.
- Флаги: SHARED для shared IRQ, DISABLED для критичных.

## 111. Передача данных в прерывании. Определение процесса и его виртуального адресного пространства. Копирование данных в виртуальное адресное пространство процесса.
*Источник: общие принципы Linux*

### Проблема

Прерывание происходит **в произвольный момент**. Текущий процесс на CPU — это не обязательно тот, кому нужны данные.

Сценарий:
- Процесс P1 открыл сокет, ждёт данных (заблокирован в read).
- Процесс P2 сейчас работает.
- Приходит пакет для P1.
- Обработчик прерывания вызывается **в контексте P2** (или вне всякого контекста процесса).

### Где сейчас процесс P1

P1 спит:
- Состояние: TASK_INTERRUPTIBLE.
- В очереди ожидания.
- АП P1 не загружено (CR3 ≠ P1).

### Где процесс P2

P2 работает:
- Состояние: TASK_RUNNING.
- АП P2 загружено.

### Можно ли в обработчике писать в АП P1?

**Нет:**
- АП P1 не загружено (CR3 указывает на P2).
- Доступ по адресам P1 даст другую память или page fault.
- Даже если бы было загружено, обработчик не должен «лезть» в чужое АП.

### Решение: буфер в ядре

Драйвер хранит **буфер в ядре**, связанный с открытым файлом/сокетом.

Структура:
```c
struct mydev {
    char buffer[BUFSIZE];      // данные
    size_t data_size;          // сколько есть
    wait_queue_head_t wait;    // кто ждёт
};
```

### Что делает обработчик прерывания

```c
irqreturn_t my_isr(int irq, void *dev_id)
{
    struct mydev *dev = dev_id;

    // 1. Подтвердить прерывание
    writel(IRQ_ACK, dev->regs + REG_IRQ);

    // 2. Скопировать данные в буфер ядра
    int len = readl(dev->regs + REG_LEN);
    memcpy(dev->buffer, dev->dma_buffer, len);
    dev->data_size = len;

    // 3. Разбудить ожидающих
    wake_up_interruptible(&dev->wait);

    return IRQ_HANDLED;
}
```

### Что делает процесс P1, когда проснётся

P1 спал в `read`:
```c
ssize_t my_read(struct file *filp, char __user *user_buf,
                size_t count, loff_t *ppos)
{
    struct mydev *dev = filp->private_data;

    // Подождать данные
    wait_event_interruptible(dev->wait, dev->data_size > 0);

    // ТЕПЕРЬ мы в контексте P1
    // АП P1 загружено

    size_t to_copy = min(count, dev->data_size);
    if (copy_to_user(user_buf, dev->buffer, to_copy))
        return -EFAULT;

    dev->data_size -= to_copy;
    return to_copy;
}
```

### Почему copy_to_user работает

В `my_read` мы выполняемся **в контексте процесса P1**:
- `current` = P1.
- CR3 указывает на АП P1.
- `user_buf` — валидный указатель в АП P1.
- `copy_to_user` копирует из адреса ядра в адрес пользователя.

### Виртуальное АП процесса

При запуске процесса:
- Ядро создаёт `mm_struct` — описание АП.
- Таблицы страниц настраиваются.

При переключении контекста:
- CR3 (на x86) загружается значением из `mm->pgd`.
- Теперь CPU видит АП этого процесса.

### Определение «текущего процесса»

В контексте процесса:
```c
struct task_struct *current;   // макрос
```

Возвращает указатель на текущий task_struct.

В контексте прерывания `current` есть, но это **тот процесс, который был на CPU перед прерыванием**, не «владелец» обработки. Часто это просто случайный процесс.

### Доступ к виртуальной памяти процесса в обработчике

**Запрещено напрямую.** Причины:
- `current` в обработчике может быть не тот, кому нужны данные.
- Даже если тот — `copy_to_user` может вызвать page fault, который в IRQ-контексте не обработать.

**Правильный паттерн:**
1. В обработчике — копировать в буфер ядра.
2. В контексте процесса (когда он проснётся) — `copy_to_user`.

### DMA — устройство пишет прямо в RAM

С DMA устройство пишет в физическую память:
- Буфер выделен через `dma_alloc_coherent`.
- Устройство получает физический адрес.
- Устройство пишет.

Обработчик прерывания не копирует — данные уже в RAM. Только:
- Узнать, что DMA закончил.
- Разбудить ожидающие процессы.

### Zero-copy с mmap

Современная техника:
- DMA-буфер отображается в АП пользователя через `mmap`.
- Пользователь читает прямо из памяти, куда писало устройство.
- **Нет копирования.**

```c
// Драйвер реализует mmap
int my_mmap(struct file *filp, struct vm_area_struct *vma)
{
    struct mydev *dev = filp->private_data;
    return dma_mmap_coherent(&dev->pdev->dev, vma, dev->cpu_addr,
                             dev->dma_handle, size);
}
```

Пользователь:
```c
void *p = mmap(NULL, size, PROT_READ, MAP_SHARED, fd, 0);
// читать p напрямую
```

### Резюме

- В обработчике прерывания нельзя писать прямо в АП произвольного процесса.
- Решение: буфер в ядре, копирование пользователем через `copy_to_user`.
- DMA пишет прямо в RAM — экономия CPU.
- Zero-copy через mmap — минимальные накладные расходы.

## 112. Прерывание. Блокировка вызвавшего процесса, возврат из прерывания. «Пробуждение» пользовательского процесса.
*Источник: общие принципы Linux*

### Полный цикл блокирующего read

Объединение тем из вопросов 88, 111. Полный жизненный цикл.

### Стадии

```
1. P1: вызов read(fd, buf, count)
                ↓
2. Kernel: sys_read → my_read
                ↓
3. my_read: данных нет → wait_event_interruptible
                ↓
4. P1: TASK_INTERRUPTIBLE, в очереди, schedule()
                ↓
5. Kernel: планировщик выбирает P2
                ↓
6. P2 работает...
                ↓
7. Аппаратное прерывание (например, пакет)
                ↓
8. Обработчик: копирование в буфер ядра + wake_up
                ↓
9. P1 → TASK_RUNNING, в runqueue
                ↓
10. После IRQ: проверка need_resched
                ↓
11. Если P1 приоритетнее P2 → планировщик переключает
                ↓
12. P1 продолжает с того места в schedule()
                ↓
13. wait_event возвращает (условие true)
                ↓
14. copy_to_user
                ↓
15. return count → пользователю
```

### Шаг 1-3: пользователь и драйвер

```c
// Пользователь
ssize_t n = read(fd, buf, 4096);

// Ядро: sys_read → drivers
ssize_t my_read(struct file *filp, char __user *buf, size_t count, loff_t *ppos)
{
    struct mydev *dev = filp->private_data;

    if (dev->data_size == 0) {
        // Нет данных, нужно ждать
        if (wait_event_interruptible(dev->wait, dev->data_size > 0))
            return -ERESTARTSYS;
    }
    ...
}
```

### Шаг 4: блокировка процесса

`wait_event_interruptible` внутри:

```c
DEFINE_WAIT(wait);
for (;;) {
    prepare_to_wait(&wq, &wait, TASK_INTERRUPTIBLE);
    if (condition) break;
    if (signal_pending(current)) {
        finish_wait(&wq, &wait);
        return -ERESTARTSYS;
    }
    schedule();   // ← P1 засыпает
}
finish_wait(&wq, &wait);
```

`schedule()`:
- Видит, что текущий — не TASK_RUNNING.
- Удаляет из runqueue.
- Выбирает другой.

### Шаг 5-6: P2 работает

CPU делает другие задачи. P1 спит.

### Шаг 7: прерывание

Аппаратное прерывание от устройства:
- Текущий контекст (P2) сохраняется.
- Запускается обработчик.

### Шаг 8: обработчик и wake_up

```c
irqreturn_t my_isr(int irq, void *dev_id)
{
    struct mydev *dev = dev_id;

    // ACK
    writel(IRQ_ACK, dev->regs + REG_IRQ);

    // Копировать данные
    int len = readl(dev->regs + REG_LEN);
    memcpy(dev->buffer, dev->dma_buffer, len);
    dev->data_size = len;

    // Разбудить
    wake_up_interruptible(&dev->wait);

    return IRQ_HANDLED;
}
```

`wake_up_interruptible`:
- Найти процессы в очереди в состоянии TASK_INTERRUPTIBLE.
- Перевести в TASK_RUNNING.
- Добавить в runqueue.
- Если разбуженный приоритетнее текущего → set `need_resched`.

### Шаг 9-10: возврат из IRQ

После обработчика:
- Восстановление контекста (если возвращаемся в тот же процесс).
- Проверка `need_resched`.

Если `need_resched`:
- Перед возвратом в пользователя — `schedule()`.
- Может переключиться на P1.

### Шаг 11: планировщик выбирает P1

`schedule()` смотрит в runqueue, выбирает самый приоритетный (или CFS-наименьший vruntime).

Если P1 приоритетнее P2 (или P2 не RUNNING) → переключение.

### Шаг 12: P1 продолжает

P1 был в `schedule()` внутри цикла `wait_event_interruptible`. После переключения:
- Контекст P1 восстанавливается.
- Возврат из `schedule()`.
- Цикл проверяет условие — теперь true.
- Выход из цикла, `finish_wait`.

### Шаг 13-14: read продолжается

```c
// После wait_event_interruptible
size_t to_copy = min(count, dev->data_size);
if (copy_to_user(user_buf, dev->buffer, to_copy))
    return -EFAULT;

dev->data_size -= to_copy;
return to_copy;
```

`copy_to_user` работает потому, что мы в контексте P1.

### Шаг 15: возврат пользователю

- `my_read` возвращает `to_copy`.
- `sys_read` возвращает в библиотеку.
- `read()` в пользователе возвращает count.

### Диаграмма

```
P1     │read │           wait               │     │copy_to_user│ret
─────────┼─────┼──────────────────────────────┼─────┼──────────────┼──
P2                  │run    │                                  │
─────────┼─────┼─────┼──────┼──────────────┼─────┼──────────────┼──
IRQ                            │handler│                              
─────────┼─────┼─────┼──────┼─┬─────┼──────┼─────┼──────────────┼──
                                ↓ wake_up
                                
                                  планировщик ↓
                                  переключает на P1
```

### Возврат после сигнала

Если до прихода данных P1 получил сигнал (Ctrl+C):
- `wait_event_interruptible` возвращает `-ERESTARTSYS`.
- Драйвер возвращает `-ERESTARTSYS`.
- `sys_read`:
  - Если `SA_RESTART` установлен и нет специальных условий → рестарт syscall.
  - Иначе → пользователю возвращается -1, errno = EINTR.

### Возврат после таймаута

Если использовать `wait_event_interruptible_timeout`:
- При таймауте возвращает 0.
- Драйвер: решает что вернуть.
  - Может вернуть `-ETIMEDOUT`.
  - Может вернуть 0 (EOF) — пользователь поймёт, что данных нет.
  - Может попробовать снова (в зависимости от семантики).

### Прозрачность для пользователя

С точки зрения пользователя:
- Вызвал `read()`.
- Получил данные (или ошибку).
- Не знает, спал ли процесс, переключался ли контекст, было ли прерывание.

Это **абстракция**, которую предоставляет ОС.

### Резюме

- Блокирующий read: процесс засыпает → планировщик переключает → прерывание → wake_up → процесс просыпается → copy_to_user → возврат.
- wake_up в обработчике прерывания запускает цепочку пробуждения.
- need_resched может вызвать немедленное переключение.
- Прозрачно для пользователя.
- При сигнале — возврат `-ERESTARTSYS`, возможно рестарт.
