## 51. Критическая секция. Race Condition. Пример.
*Источник: tanenbaum.md, Глава 2.3.1 «Состязательная ситуация»*

### Что такое race condition

**Race condition (состязательная ситуация, гонка)** — ошибка, при которой результат работы программы зависит от **точного порядка** выполнения параллельных потоков/процессов. Корректность становится делом случая.

Из rtos.md (6.1):
> «Если свойства надёжности или жизнеспособности для механизма синхронизации не выполняются (в зависимости от чередования потоков нарушаются соглашения, которые определены для задачи), такой механизм называется содержащим гонки или условия гонок (race condition). То есть успешность выполнения алгоритма зависит от того, успел ли какой-то поток сделать свою работу вовремя».

### Почему это серьёзная проблема

- **Невоспроизводимость:** проявляется случайно, зависит от тайминга.
- **Незаметность в тестах:** на компьютере разработчика часто не возникает.
- **Тяжёлая отладка:** при попытке использовать отладчик меняется тайминг, и проблема исчезает (Heisenbug).
- **Коварность:** программа может работать годами «правильно», пока однажды не выдаст странный результат.

### Классический пример: спулер печати (Tanenbaum 2.3.1)

Два процесса А и Б одновременно хотят поставить файлы в очередь печати.

Структура: массив `spool_dir[]` имён файлов, переменная `in` — следующий свободный индекс.

Код «поставить в очередь»:
```c
int next_free_slot = in;              // (1) читаем in
spool_dir[next_free_slot] = filename;  // (2) пишем имя
in = next_free_slot + 1;               // (3) обновляем in
```

Если `in == 7`, должно стать `in == 8` после одной постановки.

**Сценарий race:**

1. Процесс А выполняет (1): `next_free_slot = 7`.
2. **Прерывание таймера.** Планировщик переключает на Б.
3. Процесс Б выполняет (1): тоже `next_free_slot = 7`.
4. Б выполняет (2): записывает имя своего файла в `spool_dir[7]`.
5. Б выполняет (3): `in = 8`.
6. Б завершает свою работу.
7. Планировщик возвращает CPU процессу А.
8. А продолжает с (2): записывает имя своего файла в `spool_dir[7]` — **затирает** имя Б!
9. А выполняет (3): `in = 8`.

Результат: `in == 8`, корректное значение. Но в `spool_dir[7]` имя А, имя Б **исчезло**. Пользователь Б никогда не получит распечатку.

Из tanenbaum.md:
> «Пользователь Б будет годами бродить вокруг принтера, тоскливо надеясь получить распечатку, которой не будет никогда».

### Простой пример: счётчик

```c
volatile int counter = 0;

void increment() {
    counter++;       // race!
}
```

Что такое `counter++` на уровне машинных инструкций:
```asm
mov eax, [counter]   ; (1) load
inc eax              ; (2) increment
mov [counter], eax   ; (3) store
```

Если два потока делают по 1000 инкрементов, ожидаемо `counter = 2000`. Реально может быть 1000-2000.

**Сценарий с потерей:**
- Поток 1: (1) eax = 0.
- **Переключение.**
- Поток 2: (1) eax = 0, (2) eax = 1, (3) counter = 1.
- Поток 1: (2) eax = 1, (3) counter = 1.

Два инкремента, но counter увеличился на 1.

### Условия для возникновения

1. **Разделяемые данные** между потоками/процессами.
2. **Хотя бы один пишет.**
3. **Операция не атомарна** (несколько шагов).
4. **Чередование выполнения** — на SMP это всегда, на однопроцессорной — при прерываниях.

Race condition **не возникает** для:
- Read-only данных.
- Атомарных операций.
- Локальных переменных потока.
- Данных под взаимным исключением.

### Виды race conditions

**1. Read-modify-write race.** Классическая, как в примерах выше.

**2. TOCTOU (Time Of Check To Time Of Use).** Проверка состояния и действие на основе проверки разделены во времени.

```c
if (file_exists(path))    // проверка
    open(path);            // действие
```

Между проверкой и действием файл может быть удалён или подменён (security exploit).

**3. Lost wakeup.** Сигнал приходит до того, как поток дошёл до `wait` → теряется (вопрос 49).

**4. Double-free.** Два потока освобождают одну и ту же память.

### Решения

**1. Критическая секция через мьютекс.**

```c
pthread_mutex_t lock = PTHREAD_MUTEX_INITIALIZER;

void increment() {
    pthread_mutex_lock(&lock);
    counter++;
    pthread_mutex_unlock(&lock);
}
```

**2. Атомарные операции** (вопрос 56).

```c
atomic_int counter = 0;
atomic_fetch_add(&counter, 1);
```

**3. Семафоры, спинлоки, условные переменные.**

**4. Lock-free структуры данных** (вопрос 70).

**5. Иммутабельность.** Если данные не меняются — race condition невозможен.

### Отладочные инструменты

- **ThreadSanitizer (TSan)** в GCC/Clang — детектирует race conditions в runtime.
- **Helgrind** (Valgrind) — анализатор гонок.
- **Lockdep** в ядре Linux — анализ deadlocks.
- **RaceTesting** — стресс-тестирование с искусственными задержками.

### Из tanenbaum.md
> «Отладка программ, в которых присутствует состязательная ситуация, особой радости не доставляет. Результаты большинства прогонов могут быть вполне приемлемыми, но до поры до времени, пока не наступит тот самый редкий случай, когда произойдёт нечто таинственное и необъяснимое. К сожалению, с ростом параллелизма из-за всё большего количества ядер состязательные ситуации встречаются всё чаще».

### Резюме

- Race condition — баг, чьё проявление зависит от тайминга.
- Возникает при разделяемых данных без взаимного исключения.
- Невоспроизводимость делает отладку сложной.
- Решение — мьютексы, атомарные операции, lock-free структуры.

## 52. Критическая секция. TSL. XCHG. Пример.
*Источник: rtos.md, Глава 2.2.3 «Атомарность в многопроцессорных системах»*

### Зачем нужна аппаратная поддержка

Программные алгоритмы (Петерсон, Декер — вопрос 49) для взаимного исключения имеют недостатки:
- Не масштабируются.
- Требуют барьеров памяти.
- Сложно реализовать.

Современные процессоры предоставляют **атомарные RMW-операции** (Read-Modify-Write) — специальные инструкции, выполняющие чтение + модификацию + запись как **одно неделимое действие**.

Из rtos.md (2.2.3):
> «Атомарные операции — специальные инструкции, которые выполняют некоторое составное действие таким образом, что это действие не может быть прерывано и никакой другой процессор (или обработчик прерывания на текущем процессоре) не может увидеть промежуточный результат».

### TSL (Test and Set Lock)

Простейшая RMW-инструкция. Атомарно:
1. Читает значение из ячейки памяти.
2. Устанавливает значение в 1.
3. Возвращает прочитанное.

```c
// Источник: rtos.md, раздел 2.2.3
int atomic_test_and_set(volatile int* ptr) {
    int old_value = *ptr;
    *ptr = 1;
    return old_value;
}
```

Семантика: «зарезервируй ячейку, скажи мне, кто был раньше».

### Использование TSL для блокировки

```c
volatile int lock = 0;     // 0 = свободно, 1 = занято

void enter_cs() {
    while (test_and_set(&lock) == 1) ;
    // если получили 0 — мы первые, входим
}

void exit_cs() {
    lock = 0;
}
```

Разбор:
- `lock = 0` (свободно): `test_and_set` возвращает 0 (был свободен), ставит 1 (занято). Выход из цикла.
- `lock = 1` (занято): `test_and_set` возвращает 1, ставит 1 (без эффекта). Цикл.

### XCHG (Exchange) / SWAP

Атомарно обменивает значение в регистре со значением в памяти.

```c
// Источник: rtos.md, раздел 2.2.3
int atomic_swap(volatile int* ptr, int new_value) {
    int old_value = *ptr;
    *ptr = new_value;
    return old_value;
}
```

На x86: инструкция `XCHG reg, [mem]` **автоматически атомарна** — даже без префикса `LOCK`.

Это историческое исключение x86. Все остальные RMW-операции на x86 требуют `LOCK CMPXCHG`, `LOCK XADD` и т.д.

### Использование XCHG для блокировки

```c
// Источник: rtos.md, реализация спинлока через SWAP
void spinlock_acquire(volatile int* lock) {
    while (atomic_swap(lock, 1) == 1) ;
}

void spinlock_release(volatile int* lock) {
    *lock = 0;
}
```

### Реализация TSL на x86

```asm
; Простой spinlock через xchg
acquire_lock:
    mov eax, 1            ; новое значение = 1 (занято)
spin:
    xchg eax, [lock]      ; атомарно обменять
    test eax, eax          ; eax — старое значение lock
    jnz spin               ; если был 1 — занят, повторить
    ret                    ; иначе lock наш

release_lock:
    mov dword [lock], 0    ; просто записать 0
    ret
```

### Аппаратная реализация

Как процессор гарантирует атомарность?

**Старый способ — блокировка шины.**

Из rtos.md:
> «Самый простой путь обеспечения атомарности — блокировка доступа к памяти для остальных процессоров на все время выполнения операции. Ранние процессоры использовали для этой цели блокировку системной шины, что позволяло предотвратить доступ к данным не только со стороны других процессоров, но и со стороны устройств через механизм прямого доступа к памяти».

Префикс `LOCK` на x86 блокировал системную шину — никто не мог обращаться к памяти.

Проблема: с ростом числа CPU шина становится узким местом.

**Современный способ — протокол когерентности кэша (MESI и др.).**

Из rtos.md:
> «Через сеть межсоединений процессоры передают друг другу информацию о доступах в память, что позволяет им своевременно обновлять и поддерживать в актуальном состоянии кэш-память… Для выполнения атомарной операции процессор вначале добивается эксклюзивного владения данными, а затем выполняет над своим кэшем RMW-операцию».

Процессор:
1. Получает строку кэша в состоянии **Modified/Exclusive**.
2. Другие CPU должны отдать свои копии (`Invalidate`).
3. Выполняет RMW локально.
4. Другие CPU при следующем обращении должны заново загрузить.

### Достоинства

- **Атомарно** — гарантировано железом.
- **Работает на SMP.**
- **Простая идея** и реализация.
- Поддерживается во всех современных архитектурах.

### Недостатки

**1. Активное ожидание (spin)** — тратит CPU.

**2. Cache thrashing.** На SMP несколько CPU постоянно борются за линию кэша. Каждая попытка XCHG требует получения эксклюзивного доступа → инвалидация у других → они снова загружают → передача владения... Это очень неэффективно.

### Оптимизация: test-test-and-set (TTAS)

```c
void spinlock_acquire(volatile int* lock) {
    do {
        while (*lock == 1) ;            // просто читать (дёшево, локально)
    } while (atomic_swap(lock, 1) == 1);  // когда выглядит свободным — пробовать
}
```

Преимущество: пока lock занят, мы только **читаем** локальную копию кэша — не вызываем когерентности. Только когда видим, что lock свободен, делаем дорогую atomic операцию.

### Оптимизация: pause/yield

На x86 — инструкция `PAUSE` в spin-loop:
- Сообщает CPU, что мы в spin-loop.
- Замедляет выполнение, уменьшая cache traffic.
- Помогает hyperthreading-собрату работать.

На ARM — `YIELD`.

```asm
spin:
    cmp [lock], 0
    je try_acquire
    pause            ; CPU hint
    jmp spin
```

### Применение

- **Реализация спинлоков** в ядрах ОС.
- **Простые блокировки** в драйверах.
- **Низкоуровневая синхронизация** между CPU.
- **Основа** для futex (быстрых мьютексов пользовательского пространства).

### Резюме

- TSL и XCHG — атомарные RMW-инструкции, основа простейшего взаимного исключения.
- Реализация через когерентность кэша на современных процессорах.
- Используются для построения спинлоков.
- Имеют недостатки: активное ожидание, cache thrashing.

## 53. Критическая секция. Проблема консенсуса. Пример.
*Источник: rtos.md, Глава 2.2.4 «Неблокирующие алгоритмы»*

### От критической секции к консенсусу

Когда несколько потоков хотят попасть в критическую секцию, нужно решить: **кто пойдёт первым**. Все потоки должны прийти к одному и тому же выводу — иначе двое окажутся в секции одновременно и поломают общие данные. Получается, что вся проблема защиты критической секции в многопоточной системе сводится к более фундаментальному вопросу — **умеют ли потоки договариваться об одном значении**.

Эта общая задача называется **проблемой консенсуса**. Её изучение объясняет, какие именно процессорные инструкции **в принципе позволяют** организовать согласованную работу потоков, а какие — нет. Это не вопрос мастерства программиста, а вопрос математических свойств примитивов.

### Формальное определение

> «Консенсус — это алгоритм, который содержит единственный метод `decide`. Консенсус работает так: каждый процессор указывает в качестве аргумента некоторое уникальное число (например, свой номер) и получает результат».

```c
int decide(int my_value);
```

Несколько потоков **одновременно** вызывают `decide`, каждый передаёт своё значение. Контракт функции — три свойства, все обязательны:

**1. Валидность.**
> «Он должен быть равен одному из аргументов, который передан одним из процессоров».

Возвращаемое значение — это `my_value` одного из вызвавших потоков. Не среднее, не сумма, не выдуманное.

**2. Согласованность.**
> «Все процессоры, вызвавшие метод, должны получить один и тот же результат».

Никаких «разных мнений» — все согласны на одном числе.

**3. Завершимость.** Каждый вызов завершается за конечное число шагов (без блокировок — это требование wait-free).

### Конкретный пример

Три потока одновременно вызывают `decide`:

| Поток | Передал | Получил |
|---|---|---|
| A | 10 | ? |
| B | 20 | ? |
| C | 30 | ? |

**Допустимые исходы:**
- Все трое получили `10` (победил A).
- Все трое получили `20` (победил B).
- Все трое получили `30` (победил C).

Кто именно «победит», заранее неизвестно — это решает гонка.

**Недопустимо:**
- A получил `10`, B получил `20` (нарушена согласованность).
- Все получили `42` (нарушена валидность — никто такого не предлагал).

### Число консенсуса

> «Определяется также число, называемое числом консенсуса (consensus number). Это максимальное количество процессоров, для которого решаема задача консенсуса, используя данный набор атомарных инструкций».

По-простому: **сколько потоков примитив способен честно рассудить**. Чем больше число — тем «мощнее» примитив.

| Примитив | Число консенсуса |
|---|---|
| Обычные чтение и запись | 1 |
| Test-and-set | 2 |
| Swap (XCHG) | 2 |
| Fetch-and-add | 2 |
| Бинарный mutex | 2 |
| Стек | 2 |
| Очередь без `peek` | 2 |
| CAS | ∞ |
| LL/SC | ∞ |

Разберём каждую строку.

### Чтение/запись — число консенсуса 1

«Договорится может только один поток сам с собой». Для двух уже нельзя. Попробуем наивно:

```c
int shared = -1;

int decide(int my_value) {
    if (shared == -1)         // ① читаем
        shared = my_value;    // ② записываем
    return shared;
}
```

Гонка:
- Поток A читает `shared`: видит `-1`. Проходит `if`.
- Поток B читает `shared`: видит `-1`. Проходит `if`.
- Поток A пишет `10`. Возвращает `10`.
- Поток B пишет `20`. Возвращает `20`.

Разные ответы → согласованности нет.

Между чтением и записью могут вмешаться другие. Без **атомарной** «прочитать-и-записать-одной-инструкцией» — никак.

### Test-and-set и Swap — число консенсуса 2

Для двух потоков работает. Пример со SWAP:

```c
int shared = 0;   // 0 = «ещё никто не выиграл»

int decide(int my_value) {
    int prev = swap(&shared, my_value);
    if (prev == 0) return my_value;   // я был первый
    else           return prev;        // кто-то был раньше меня
}
```

SWAP атомарно подменяет значение и **возвращает то, что было**. Первый поток обнаруживает `0` — понимает, что он первый, возвращает своё. Второй обнаруживает значение первого и возвращает его. Оба вернули одинаково.

**Для трёх потоков SWAP уже не работает.** Это формально доказывается через анализ возможных состояний (так называемая «бивалентность» начальных конфигураций), и означает: какой бы алгоритм вы ни придумали на основе SWAP, для трёх потоков всегда найдётся такая последовательность шагов, при которой свойства консенсуса нарушатся.

### CAS — число консенсуса ∞

Compare-and-Swap способен рассудить **любое** число потоков:

```c
int shared = -1;   // ничьё значение

int decide(int my_value) {
    cas(&shared, -1, my_value);   // победит только первый
    return shared;                 // все увидят значение победителя
}
```

`cas(&shared, -1, my_value)` атомарно делает: «если в `shared` сейчас `-1` — замени на `my_value`, иначе ничего не делай».

- Первый успевший делает CAS успешно, теперь в `shared` его значение.
- Все остальные пытаются — CAS не срабатывает (там уже не `-1`), значение не меняется.
- При `return shared` все читают одно и то же.

### Теорема Herlihy (1991)

> «Любой wait-free алгоритм может быть реализован тогда и только тогда, когда существует wait-free решение так называемой проблемы консенсуса».

Эта **фундаментальная теорема** Морриса Херлихи говорит: число консенсуса примитива = «потолок параллелизма», для которого можно построить wait-free алгоритм.

Следствия:
- **CAS — универсальный примитив.** На нём можно построить любую wait-free структуру данных для любого числа потоков.
- **На SWAP/Test-and-Set нельзя построить** wait-free стек, очередь, и т. п. для 3+ потоков. Это математически невозможно.
- Если процессор предоставляет только обычные чтение/запись и SWAP, никакая «изобретательность» программиста не позволит написать корректный wait-free алгоритм для произвольного числа потоков.

### Практическое следствие — почему CAS есть на всех процессорах

Из-за теоремы Herlihy производители процессоров **обязаны** предоставлять CAS или его эквивалент, иначе их процессор не сможет поддерживать современные lock-free структуры данных:

| Архитектура | Примитив |
|---|---|
| x86 / x86-64 | `CMPXCHG`, `CMPXCHG8B`, `CMPXCHG16B` |
| ARM (v7+) | LL/SC: `LDREX` / `STREX` |
| ARMv8.1+ | дополнительно атомарные RMW (`CAS`, `LDADD`, ...) |
| POWER | LL/SC: `lwarx` / `stwcx` |
| RISC-V | LL/SC: `lr.w` / `sc.w` + расширение atomics |

### Аналогия для понимания

Сравним «инструменты для голосования»:

- **Записка на столе** (обычная переменная) — двое могут писать одновременно, путаница. Число консенсуса 1.
- **Поднятая рука** (Test-and-Set, SWAP) — двое подняли руки, видно, кто раньше. Для троих уже надёжно не рассудить. Число 2.
- **Электронная голосовалка** (CAS — одна заявка на одного, остальные видят «занято») — хоть миллион участников, система определит победителя. Число ∞.

### Пример: построение lock-free очереди

С использованием CAS можно построить **lock-free очередь** (алгоритм Michael & Scott, 1996):

```c
struct node { value_t val; struct node *next; };
struct node *head, *tail;

void enqueue(value_t v) {
    struct node *new = malloc(sizeof(struct node));
    new->val = v;
    new->next = NULL;

    while (1) {
        struct node *t = tail;
        struct node *next = t->next;
        if (t == tail) {
            if (next == NULL) {
                if (CAS(&t->next, NULL, new) == NULL) {
                    CAS(&tail, t, new);
                    break;
                }
            } else {
                CAS(&tail, t, next);  // помочь продвинуть tail
            }
        }
    }
}
```

Эта очередь — wait-free для производителей и потребителей, не использует мьютексов.

### Связь с распределёнными системами

Проблема консенсуса в распределённых системах — расширение задачи:
- Узлы вместо процессоров.
- Сеть может терять сообщения, узлы могут падать.
- Решения: **Paxos** (Lamport), **Raft** (Ongaro), **PBFT** (для византийских отказов).

В кратком виде — то же: договориться о значении при наличии параллельных предложений.

### Резюме

- Проблема консенсуса — формальная задача договорённости.
- Число консенсуса примитива — максимальное N процессоров, для которых он решает задачу.
- CAS и LL/SC имеют число консенсуса ∞ — универсальные примитивы.
- Они позволяют строить любые wait-free структуры данных.
- Это теоретическое обоснование выбора CAS как основного примитива атомарности.

## 54. Критическая секция. CAS. Двойной CAS. Пример.
*Источник: rtos.md, Глава 2.2.4 «Неблокирующие алгоритмы»*

### Зачем нужен CAS

TSL и XCHG (вопрос 52) — атомарные операции, но имеют ограничения:
- Число консенсуса 2 — не позволяет строить любые wait-free структуры.
- Не позволяют атомарно проверять и обновлять значение.

**CAS (Compare-And-Swap)** решает эти проблемы. Это **универсальный** примитив для построения lock-free структур.

### Определение CAS

Из rtos.md (2.2.4):
> «CAS принимает три аргумента, указатель на ячейку памяти, ожидаемое значение и новое значение. Инструкция атомарно проверяет значение по указателю, если оно равно ожидаемому значению, ячейка устанавливается в новое значение, в противном случае значение не меняется».

```c
// Источник: rtos.md, раздел 2.2.4
int CAS(volatile int* ptr, int comparand, int new_value) {
    int old_value = *ptr;
    if (old_value == comparand) {
        *ptr = new_value;
    }
    return old_value;
}
```

Семантика: «если ячейка содержит ожидаемое значение, замени; в любом случае верни старое».

### Как использовать возвращаемое значение

```c
int prev = CAS(&x, expected, new_val);
if (prev == expected) {
    // CAS успешен — мы заменили значение
} else {
    // CAS неудачен — значение было не таким
    // prev содержит фактическое значение. Используем его
}
```

### Реализация в архитектурах

**x86:** инструкция `CMPXCHG` с префиксом `LOCK`:
```asm
lock cmpxchg [mem], reg
; неявно сравнивает [mem] с EAX
; если равны: [mem] = reg, ZF=1
; иначе: EAX = [mem], ZF=0
```

**ARMv7 и старше:** через LL/SC (`LDREX`/`STREX`).
**ARMv8.1+:** прямая инструкция `CAS`/`CASP`.
**POWER:** `LWARX`/`STWCX`.

### CAS-loop — типичный паттерн

Из rtos.md:
> «Типичный шаблон использования CAS включает в себя цикл, называемый также CAS-loop, в котором производятся попытки атомарно обновить значение».

```c
// Реализация fetch_and_add через CAS
int my_fetch_and_add(int *ptr, int delta) {
    int old_value, result;
    do {
        old_value = *ptr;
        result = CAS(ptr, old_value, old_value + delta);
    } while (result != old_value);
    return old_value;
}
```

Алгоритм:
1. Прочитать текущее значение.
2. Попробовать сделать CAS с этим значением как ожидаемым.
3. Если кто-то изменил между шагами 1 и 2 — CAS неуспешен, повторить.

### Почему CAS-loop работает

Из rtos.md:
> «Если CAS завершился неудачей, это значит что у кого-то другого процессора он завершился успешно, то есть тот процессор совершил прогресс, а значит, вся система в целом совершила прогресс».

Это **lock-free** свойство: даже если конкретный поток не делает прогресса (его CAS не проходит), **система** делает прогресс (другой поток успел).

Технически это не wait-free (наш поток может бесконечно проигрывать), но lock-free гарантирует, что система не зависает.

### Пример: lock-free стек

```c
typedef struct node {
    int value;
    struct node *next;
} node_t;

node_t *head = NULL;

void push(int val) {
    node_t *new = malloc(sizeof(node_t));
    new->value = val;

    node_t *old_head;
    do {
        old_head = head;
        new->next = old_head;
    } while (CAS(&head, old_head, new) != old_head);
}

int pop() {
    node_t *old_head, *new_head;
    do {
        old_head = head;
        if (!old_head) return -1;       // пусто
        new_head = old_head->next;
    } while (CAS(&head, old_head, new_head) != old_head);

    int val = old_head->value;
    free(old_head);    // ВНИМАНИЕ: ABA-проблема!
    return val;
}
```

### Проблема ABA

Из rtos.md:
> «Тем не менее, у CAS есть и недостатки. Один из них — так называемая проблема ABA. Довольно часто в качестве аргумента CAS используется указатель, в этом случае возможна ситуация, когда численно указатель остался тот же, но указывает уже на другой объект (такое могло случиться, если изначальный объект был удалён а память освобождена, затем она повторно выделена для нового объекта). В таком случае, CAS может быть успешным в тех случаях, когда это нежелательно».

### Конкретный сценарий ABA

Пусть в стеке узлы A → B → C.

1. Поток 1: `pop`. Читает `head = A`, `new_head = A->next = B`. Готов сделать CAS.
2. **Переключение.**
3. Поток 2: `pop` → возвращает A, освобождает память A.
4. Поток 2: `pop` → возвращает B, освобождает B.
5. Поток 2: `push(X)` — `malloc` возвращает **тот же адрес** A (так бывает!).
6. Стек: A(новый X) → C.
7. **Переключение обратно на поток 1.**
8. Поток 1: CAS(&head, A, B). А — текущее значение, CAS успешен!
9. Стек становится: B(освобождён!) → ... → катастрофа.

Указатель A численно тот же, но это **другой** объект.

### Двойной CAS (DCAS)

Из rtos.md:
> «Для того, чтобы обойти это препятствие, некоторые процессоры, например intel x86, поддерживают двойной CAS. Его отличие от описанного выше в том, что аргумент может быть вдвое большей длины. Это позволяет, в случае использования указателя в качестве операнда, снабдить его дополнительной информацией, тегом или счётчиком, который находится в соседней ячейке памяти».

На x86: **CMPXCHG8B** (8 байт, для 32-битного режима) и **CMPXCHG16B** (16 байт, для 64-битного).

### Использование DCAS для решения ABA

```c
struct tagged_pointer {
    void *ptr;
    uint64_t counter;
};

// При каждом обновлении: counter++
// Counter = монотонно растущий — никогда не повторится

bool dcas(struct tagged_pointer *target,
          struct tagged_pointer expected,
          struct tagged_pointer new_val);

void push(int val) {
    node_t *new = malloc(sizeof(node_t));
    new->value = val;

    struct tagged_pointer old, fresh;
    do {
        old = head;
        new->next = old.ptr;
        fresh.ptr = new;
        fresh.counter = old.counter + 1;
    } while (!dcas(&head, old, fresh));
}
```

Даже если ptr возвращается к старому значению, counter уже другой → DCAS не пройдёт → повторяем.

### Альтернативы решения ABA

**1. LL/SC (вопрос 55)** — автоматически решает ABA на уровне железа.

**2. Hazard pointers.** Поток объявляет, какие указатели он использует. Другие не освобождают, пока кто-то ссылается.

**3. RCU (Read-Copy-Update)** — отложенное удаление (вопрос 70).

**4. Версионирование.** Как в DCAS, но программно.

### Применение CAS

**1. Lock-free структуры данных:**
- Стеки, очереди, списки, хэш-таблицы.

**2. Атомарные счётчики:**
```c
atomic_fetch_add(&counter, 1);  // обычно через CAS-loop
```

**3. Futex (Fast Userspace Mutex)** в Linux:
- Реализация мьютексов в user space.
- CAS для определения состояния.
- Системный вызов только при контеншене.

**4. Garbage Collector:** lock-free marking.

**5. Базы данных:** оптимистическая блокировка (optimistic concurrency control).

### Сравнение с другими RMW

| Параметр | TSL/XCHG | CAS | DCAS |
|---|---|---|---|
| Число консенсуса | 2 | ∞ | ∞ |
| Размер | word | word | 2× word |
| Решает ABA | Нет | Нет | Да |
| Поддержка | Все | Большинство | x86 |
| Универсальность | Низкая | Высокая | Высокая |

### Резюме

- CAS — универсальная атомарная операция: «обменять, если совпало».
- Используется в CAS-loop для построения lock-free алгоритмов.
- Имеет проблему ABA при работе с указателями.
- DCAS (двойной CAS) добавляет тег для устранения ABA.
- Основа для большинства lock-free структур данных и пользовательских мьютексов.

## 55. Критическая секция. LL. SC. Пример.
*Источник: rtos.md, Глава 2.2.4 «Неблокирующие алгоритмы»*

### Альтернатива CAS

На некоторых архитектурах (ARM, MIPS, POWER, RISC-V) вместо CAS реализована пара инструкций — **LL** и **SC**.

> «Хотя двойной CAS и решает проблему ABA, он не очень удобен в использовании, поэтому другие процессоры, такие как, например ARM или MIPS, реализуют другой подход, называемый LL/SC».

### Зачем такая пара инструкций

Главная задача атомарных операций — «безопасно изменить значение, основываясь на текущем»: прочитал → посчитал → записал, только если за это время никто не вмешался.

CAS решает её, сравнивая **значения**: «запиши, если значение совпадает с моим `expected`». Минус — проблема ABA: значение могло побывать чужим и вернуться к нашему, и CAS этого не заметит (он сравнивает числа, а не следит за событиями).

LL/SC подходит с другой стороны: процессор **аппаратно следит за самой ячейкой памяти**. Не «значение совпало» — а «никто не трогал ячейку». Если кто-то её хоть раз потрогал (другое ядро, обработчик прерывания, переключение контекста) — мы об этом узнаем, неважно, какое значение там сейчас. Это строже, чем у CAS, и ABA-проблема исчезает сама собой.

### LL — Load Linked (Load Exclusive)

«Прочитай значение и **повесь датчик движения** на эту ячейку». С момента LL процессор внутри себя помечает: «слежу за адресом X». Любое обращение к этому адресу со стороны других ядер (или, на некоторых процессорах, даже со стороны прерывания) сбрасывает флаг слежения.

```c
// Источник: rtos.md
int load_link(volatile int* ptr) {
    int old_val = *ptr;
    start_monitoring(ptr);   // начать отслеживать
    return old_val;
}
```

### SC — Store Conditional

«Запиши значение, **но только если датчик не сработал**» — то есть только если с момента последнего LL никто не трогал ячейку.

- Датчик цел → запись проходит, возвращается `true` (успех).
- Датчик сорван → запись **не происходит**, возвращается `false` (нужен повтор).

```c
// Источник: rtos.md
bool store_conditional(volatile int* ptr, int new_value) {
    if (no_accesses_to_monitored_memory()) {
        *ptr = new_value;
        return true;
    }
    return false;
}
```

### Аналогия — сейф с пломбой

**CAS** = сейф с кодовым замком. Если код всё ещё тот, что я помню — открываю, кладу новые деньги. Но пока я отвернулся, кто-то мог открыть сейф той же комбинацией, забрать деньги, положить такие же бумажки и закрыть. Я этого не замечу. Это и есть **ABA**.

**LL/SC** = сейф с одноразовой пломбой. Я делаю LL — на сейф вешают пломбу. Я считаю, сколько хочу положить. Делаю SC — охранник проверяет пломбу:
- Цела → открывает, кладёт мои деньги.
- Сорвана → отказывает, неважно, что внутри.

Не сравниваются «купюры», сравнивается «трогали ли сейф». Это сильнее, и поэтому LL/SC автоматически защищает от ABA.

### Типичный паттерн

Аналог CAS-loop:

```c
// Источник: rtos.md
do {
    int old_val = load_link(ptr);
    // вычислить новое значение
    int new_val = old_val + delta;
} while (!store_conditional(ptr, new_val));
```

Если кто-то другой модифицировал значение между LL и SC, SC не пройдёт, и мы повторяем.

### Реализация в архитектурах

| Архитектура | LL | SC |
|---|---|---|
| ARM | `LDREX` (LDREXB, LDREXH, LDREXD) | `STREX` (STREXB, ...) |
| MIPS | `LL` | `SC` |
| POWER | `LWARX` | `STWCX.` |
| RISC-V | `LR.W` (Load Reserved) | `SC.W` (Store Conditional) |

### Пример на ARM

Атомарный инкремент:

```asm
atomic_inc:
    ; r0 — указатель
retry:
    ldrex   r1, [r0]        ; LL — загрузить с маркировкой
    add     r1, r1, #1      ; r1 += 1
    strex   r2, r1, [r0]    ; SC — попытка записи; r2 = 0 если успех
    cmp     r2, #0
    bne     retry           ; повторить при неудаче
    bx      lr
```

### Реализация CAS через LL/SC

Из rtos.md (ARM):
```asm
atomic_cas:
    ; r0 — указатель, r1 — ожидаемое, r2 — новое
retry:
    ldrex   r3, [r0]        ; LL
    cmp     r3, r1
    bne     fail            ; не совпадает с ожидаемым
    strex   r4, r2, [r0]    ; SC
    cmp     r4, #0
    bne     retry           ; SC неудачен — повторить
    mov     r0, r1          ; возврат: успех (возвращаем ожидаемое)
    bx      lr
fail:
    mov     r0, r3          ; возврат: текущее значение
    bx      lr
```

### Преимущества LL/SC

**1. Автоматическое решение проблемы ABA.**

Из rtos.md:
> «В отличие от CAS, SC-операция закончится неудачей даже в том случае, если значение, для которого осуществляется мониторинг, было заменено на численно такое же, поскольку отслеживается именно доступ к ячейке со стороны других процессоров».

Если другой CPU обращался к ячейке (даже записал и вернул то же значение), SC не пройдёт.

**2. Простая реализация в железе.** Не нужна сложная логика сравнения, как у CAS.

**3. Универсальность.** Между LL и SC можно выполнить **произвольный код**, не только сравнение. Это позволяет реализовать любые RMW.

**4. Любой размер.** Можно делать LL/SC на байтах, словах, double-words.

### Недостатки LL/SC

**1. Ограниченное окно.** Между LL и SC должно быть мало инструкций. Если в этот промежуток произошло прерывание, переключение контекста, страничное прерывание — SC может неудаться.

Из-за этого:
- Нельзя делать системные вызовы между LL и SC.
- Нельзя обращаться к памяти, которая может вызвать page fault.
- Нельзя долгие вычисления.

**2. Spurious failures.** SC может неудаться по причинам, не связанным с другими CPU:
- Cache line eviction.
- Прерывание.
- TLB shootdown.

Поэтому всегда нужен цикл retry.

**3. Сложно для составных операций.** Например, для одновременного обновления двух связанных значений.

### LL/SC vs CAS

| Параметр | LL/SC | CAS |
|---|---|---|
| Проблема ABA | Решена | Не решена |
| Окно операции | Малое | Большое |
| Произвольный код между | Да | Нет |
| Поддержка | ARM, MIPS, POWER, RISC-V | x86 |
| Spurious failures | Да | Нет |

### Современное состояние

**ARMv8.1+** добавил атомарные RMW-инструкции (`LDADD`, `LDSET`, `CAS`...) — теперь поддерживается и CAS, и LL/SC.

**RISC-V** разделяет реализации:
- Базовое расширение `A` — LL/SC + некоторые атомики.
- Опционально — расширение для большего числа атомиков.

### Применение

LL/SC применяется там же, где CAS:
- Атомарные счётчики.
- Lock-free структуры.
- Реализация мьютексов и спинлоков.

В Linux kernel есть макросы, которые на ARM реализованы через LDREX/STREX, на x86 — через CMPXCHG:

```c
atomic_inc(&counter);
atomic_dec_and_test(&refcount);
cmpxchg(&ptr, old, new);
```

### Резюме

- LL/SC — пара инструкций для атомарных RMW операций.
- LL загружает с маркировкой, SC записывает условно.
- Автоматически решает проблему ABA.
- Между LL и SC должен быть короткий код.
- Применяется в ARM, MIPS, POWER, RISC-V как альтернатива CAS.

## 56. Неделимые операции. Причины возникновения. atomic_t.
*Источник: rtos.md, Глава 2.1.6 «Атомарность»; robert_lav.md, Глава 10 «Неделимые операции»*

### Что это

**Неделимая (атомарная) операция** — операция, которую другие потоки видят либо полностью завершённой, либо ещё не начавшейся. Никакого «полусостояния» снаружи не существует.

> «Возможность выполнять составные операции, которые будут выглядеть как единое действие, очень важна для согласованной и непротиворечивой работы прикладных программ».

### Почему привычные операции не атомарны

`counter++` выглядит как одно действие, но на уровне процессора это **три** инструкции:

1. Прочитать `counter` из памяти в регистр.
2. Увеличить регистр на 1.
3. Записать регистр обратно в `counter`.

Между любыми двумя шагами в работу может вмешаться кто-то ещё.

**Пример проблемы.** В `counter` лежит `5`. Поток A прочитал `5`. Поток B прочитал `5`. Поток A записал `6`. Поток B записал `6`. Оба «инкрементировали», а счётчик увеличился на `1` вместо `2` — потерянное обновление.

Это называется **race condition** (гонка).

### Откуда берётся такая гонка

> «Прерывания могут происходить в произвольных точках в коде, при этом выполняемая программа и обработчик прерывания могут пользоваться общими данными в памяти».

Три основных источника:

1. **Прерывания.** Прерывание может разрезать поток в любой точке — например, между «прочитал» и «записал».
2. **Многоядерность (SMP).** На многоядерной машине другие ядра выполняются **физически параллельно**. Двое могут одновременно читать и писать ту же ячейку.
3. **DMA.** Устройства с прямым доступом к памяти пишут в неё параллельно с CPU, без участия процессора. Запрет прерываний от этого не спасает.

К этому ещё прибавляются оптимизации процессора и компилятора (переупорядочивание инструкций), но они решаются барьерами памяти, см. вопрос 61.

### Откуда берётся атомарность

Сам по себе процессор не может «волшебным образом» сделать любую операцию атомарной — он предоставляет **специальные инструкции**, спроектированные так, чтобы выполняться неделимо:

| Инструкция | Что делает |
|---|---|
| TSL / test-and-set | Установить бит и вернуть старое значение. |
| XCHG / swap | Атомарно обменять значение с регистром. |
| Fetch-and-add | Атомарно прибавить и вернуть старое. |
| CAS / compare-and-swap | Атомарная подмена при совпадении значений. |
| LL/SC | Чтение с «пломбой» + условная запись. |

Внутри железа атомарность обеспечивается либо блокировкой шины памяти (старый способ — префикс `LOCK` на x86), либо протоколом когерентности кэшей (современный — MESI, MOESI):

> «Более продвинутые системы используют для реализации атомарных операций протокол когерентности кэш-памяти… Для выполнения атомарной операции процессор вначале добивается эксклюзивного владения данными, а затем выполняет над своим кэшем RMW-операцию».

### atomic_t в Linux

Вызывать ассемблерные инструкции напрямую неудобно и непереносимо. Ядро Linux заворачивает их в удобный API над специальным типом:

```c
// Источник: robert_lav.md, Глава 10
typedef struct {
    volatile int counter;
} atomic_t;
```

Это просто **обёртка над `int`**, спрятанная в структуру. С ней работают только через функции `atomic_*`:

```c
atomic_t v = ATOMIC_INIT(0);

atomic_inc(&v);              // v++ атомарно
atomic_dec(&v);              // v-- атомарно
atomic_add(2, &v);           // v += 2
int cur = atomic_read(&v);   // прочитать значение
atomic_set(&v, 10);          // v = 10
```

Есть ещё «составные» операции, которые делают изменение и проверку за одну атомарную инструкцию:

```c
atomic_dec_and_test(&v);     // v--; вернуть (v == 0)?
atomic_inc_and_test(&v);     // v++; вернуть (v == 0)?
atomic_add_negative(2, &v);  // v += 2; вернуть (v < 0)?
```

Их важно понимать: они выполняют **два действия атомарно вместе** (изменение и проверку результата). Если бы это были два отдельных вызова — другой поток мог бы вклиниться посередине.

### Зачем нужен отдельный тип, а не обычный int

> «В функции, выполняющие неделимые операции, можно передать только аргументы типа `atomic_t`».

Три причины:

**1. Защита от ошибок программиста.** Компилятор не даст случайно сделать `v.counter++` мимо API — это будет ошибка. Если бы счётчик был обычным `int`, программист легко забыл бы про атомарность и написал бы `counter++`, что в многопоточном коде сломалось бы.

**2. Защита от оптимизаций компилятора.**
> «Использование типа `atomic_t` позволяет гарантировать, что компилятор для повышения эффективности не будет оптимизировать операции обращения к таким переменным».

`volatile` внутри запрещает кешировать значение в регистре. Без этого компилятор мог бы решить, что `counter` в цикле не меняется, и читать его только один раз.

**3. Переносимость.**
> «Использование типа `atomic_t` позволяет не учитывать различия в реализации для конкретной аппаратной платформы».

Один и тот же код `atomic_inc(&v)` на x86 превратится в `LOCK INC`, на ARM — в цикл с `LDREX/STREX`. Программист этого не видит.

### Типичное применение — счётчик ссылок

Классическая задача: объектом могут владеть несколько частей кода. Когда последний владелец отпускает объект — нужно освободить память.

```c
struct my_obj {
    atomic_t refcount;
    /* ... данные ... */
};

void get(struct my_obj *o) {
    atomic_inc(&o->refcount);
}

void put(struct my_obj *o) {
    if (atomic_dec_and_test(&o->refcount))
        free(o);
}
```

Почему атомарность здесь критична:

Если бы `put` делал не одной операцией, а двумя — сначала декремент, потом проверка на ноль — два потока могли бы одновременно сделать `--refcount` (получив 1 → 0 → -1), оба увидеть `0` и оба вызвать `free`. Двойное освобождение → крах. `atomic_dec_and_test` склеивает «уменьшить» и «проверить» в одно неделимое действие, и только **один** поток получит ответ «да, я последний».

### Когда atomic_t не подходит

**1. Нужно изменять две переменные согласованно.** `atomic_t` работает только с одной ячейкой. Если надо одновременно обновить две — нужен мьютекс или CAS на составной структуре (тегированный указатель и т. п.).

**2. Нужна синхронизация, а не просто атомарность.** Если один поток пишет данные, а другой должен их дождаться — `atomic_t` не подходит. Нужны мьютексы, семафоры или condition variables.

**3. Нужен порядок относительно других обращений.** Атомарность инкремента не означает, что другие CPU увидят его в правильном порядке относительно других записей. Для упорядочивания нужны **барьеры памяти** (см. вопрос 61):

```c
data = 42;
smp_wmb();                // ← барьер: убедиться, что data записано раньше
atomic_set(&ready, 1);
```

### Расширения

В современных ядрах помимо `atomic_t` (32 бита) есть `atomic_long_t` и `atomic64_t` для длинных целых. C11 ввёл стандартные `<stdatomic.h>` — `atomic_int`, `atomic_fetch_add` и т. п. Идея одинаковая, отличается только API.

## 57. Спин-блокировки. Применение. Особенности.
*Источник: rtos.md, Глава 6.2.3 «Использование спинлоков»; robert_lav.md, Глава 10*

### Что такое спинлок

**Спинлок (spinlock)** — блокировка, при которой поток, не сумевший её захватить, **активно ждёт** в цикле, постоянно проверяя состояние.

В отличие от мьютекса/семафора, которые **усыпляют** поток, спинлок просто **крутится** (spin = вращаться).

### Реализация

```c
// Источник: rtos.md, через atomic_swap
typedef int spinlock_t;

void spinlock_init(spinlock_t *lock) { *lock = 0; }

void spinlock_acquire(spinlock_t *lock) {
    while (atomic_swap(lock, 1) == 1) ;
}

void spinlock_release(spinlock_t *lock) {
    *lock = 0;
}
```

### Когда применять — обоснование

Из robert_lav.md (Глава 10):
> «Спин-блокировки являются очень простым и быстрым средством блокировки. Выполнение постоянных проверок в цикле является оптимальным, когда блокировки захватываются на очень короткое время и код не может переходить в состояние ожидания (например, в обработчиках прерываний)».

**Случай 1: короткие критические секции.**

Сравним два сценария:

Сценарий A — спинлок, 10 итераций ожидания:
- Активное ожидание: ~50 нс.

Сценарий B — мьютекс, переключение контекста:
- Засыпание + пробуждение: ~2 мкс.

Если CS короткая (например, увеличить счётчик), активное ожидание выгоднее.

**Случай 2: контекст, где нельзя усыпиться.**

Обработчики прерываний не могут переключать контекст — нет связи с конкретным процессом. Семафор/мьютекс там работать не может, только спинлок.

### Когда НЕ применять

**1. Длинные критические секции.**

Если в CS делается дорогая операция (I/O, ожидание, аллокация), спинлок плохой — другие CPU крутятся впустую секунды.

**2. Однопроцессорная система (без preempt).**

На однопроцессорной системе без вытеснения если один поток держит спинлок — никто не может его освободить, пока он не отдаст CPU. Активное ожидание бесконечное.

В Linux на однопроцессорной системе `spin_lock` становится **просто `preempt_disable`** — спинлок не нужен, нужно только запретить вытеснение.

### Особенности и ограничения

**1. Невытесняемость (atomic context).**

Из rtos.md:
> «Из-за особенностей реализации, на использование спинлоков накладывается множество ограничений, таких как возможность их захвата только в регионах, где запрещено вытеснение».

**Почему?** Представьте, что планировщик переключил поток, удерживающий спинлок, на другой поток того же приоритета на том же CPU. Если новый поток тоже хочет тот же спинлок — активное ожидание навсегда (первый поток не получит CPU обратно, чтобы освободить).

Решение: **`spin_lock` неявно делает `preempt_disable()`**.

**2. Защита от прерываний.**

Если спинлок может захватываться **и из обработчика прерывания, и из обычного кода**, нужно дополнительно отключать прерывания.

Пример deadlock:
1. Поток в обычном коде делает `spin_lock(&L)`.
2. Приходит прерывание, обработчик пытается сделать `spin_lock(&L)`.
3. Обработчик ждёт освобождения L.
4. Поток ждёт окончания обработчика прерывания, чтобы продолжить.
5. Deadlock.

Решение:
```c
unsigned long flags;
spin_lock_irqsave(&lock, flags);    // запретить IRQ + захватить
// CS
spin_unlock_irqrestore(&lock, flags);
```

`spin_lock_irqsave` атомарно:
- Сохраняет состояние прерываний в `flags`.
- Отключает прерывания.
- Захватывает спинлок.

**3. Активное ожидание.**

Из rtos.md:
> «Поскольку спинлок предполагает активное ожидание в виде цикла, с ними связано также много проблем с энергопотреблением».

На батарейных устройствах активное ожидание разряжает аккумулятор.

**4. Cache thrashing.**

На SMP все ожидающие CPU обращаются к одной линии кэша, постоянно вызывая когерентность. Решения:
- **TTAS (Test-Test-And-Set)** — сначала просто читать, и только когда выглядит свободным, делать атомарную попытку.
- **`PAUSE`/`YIELD`** — инструкции-подсказки CPU.
- **Ticket spinlock** — каждый ожидающий читает свою позицию.
- **MCS lock** — у каждого CPU собственная локация для ожидания.

### Спинлоки с очередью (Ticket Spinlock)

Проблема обычного спинлока: нет гарантии FIFO. Один CPU может постоянно выигрывать гонку → starvation.

Из rtos.md (6.6 про секвенсоры):
> «Аналогичные идеи используются при реализации некоторых разновидностей спинлоков. В этом случае спинлок представляет собой два значения, одно используется как секвенсор, а второе — как счётчик событий».

Структура:
```c
struct ticket_lock {
    atomic_t next_ticket;
    atomic_t now_serving;
};

void acquire(struct ticket_lock *l) {
    int my = atomic_fetch_add(&l->next_ticket, 1);
    while (atomic_read(&l->now_serving) != my) ;
}

void release(struct ticket_lock *l) {
    atomic_inc(&l->now_serving);
}
```

Аналогия — талоны в банке: получил номер 7, ждёшь, пока вызовут №7.

**Преимущества:**
- FIFO порядок.
- Нет голодания.
- Простая реализация.

**Недостаток:** все CPU крутятся на `now_serving` — cache thrashing.

### Варианты спинлоков в Linux

```c
spinlock_t lock;
spin_lock_init(&lock);

// Базовый
spin_lock(&lock);
spin_unlock(&lock);

// С защитой от прерываний
spin_lock_irqsave(&lock, flags);
spin_unlock_irqrestore(&lock, flags);

// С защитой от softirq
spin_lock_bh(&lock);
spin_unlock_bh(&lock);

// Только если получится
if (spin_trylock(&lock)) {
    // ...
    spin_unlock(&lock);
}
```

### Сравнение spinlock vs semaphore

Из robert_lav.md, после описания семафоров:
> «При удержании семафора процесс может переходить в состояние ожидания… При захвате семафора нельзя удерживать спин-блокировку, поскольку процесс может переходить в состояние ожидания до освобождения семафора, а при удержании спин-блокировки в состояние ожидания переходить нельзя».

| Параметр | Spinlock | Semaphore/Mutex |
|---|---|---|
| Ожидание | Активное (CPU) | Пассивное (sleep) |
| Накладные расходы захвата | Низкие | Высокие |
| Подходит для короткой CS | Да | Нет (overhead) |
| Подходит для длинной CS | Нет | Да |
| Допустимо в обработчиках прерываний | Да | Нет |
| Может удерживать при sleep | Нет | Да |

### Применение в Linux

- **Защита короткоживущих ресурсов** в ядре: связные списки, статистика.
- **Структуры драйверов.**
- **Реализация других примитивов** (мьютексов в ядре).

### Резюме

- Спинлок — блокировка с активным ожиданием.
- Хорош для коротких CS и контекстов без переключения (прерывания).
- Плох для длинных CS, разряжает батарею, может вызывать starvation.
- В Linux неявно отключает вытеснение.
- Защита от IRQ через `spin_lock_irqsave`.
- Ticket spinlock и MCS-lock — оптимизации с FIFO и без cache thrashing.

## 58. Критическая секция с применением приоритетов. Инверсия приоритетов. Наследование приоритетов.
*Источник: rtos.md, Глава 6.7 «Инверсия приоритета», 6.8.1 «Наследование приоритета», 6.8.2 «Потолок приоритета»*

### Использование приоритетов

Из rtos.md (6.2.4):
> «Если используется известная заранее политика планирования потоков, например FIFO, синхронизация между потоками может быть реализована с помощью этой политики. Запрет прерываний и прочие подобные методы, используются для того, чтобы предотвратить вытеснение потока во время выполнения критической секции, но те же самые гарантии могут быть получены путём временного повышения приоритета потока».

Простой подход: на время CS поднять приоритет потока до максимума → никто не вытеснит.

```c
// Условный псевдокод
int old_priority = get_my_priority();
set_my_priority(MAX_PRIORITY);
// CS
set_my_priority(old_priority);
```

**Преимущества:**
- Простая идея.
- В отличие от запрета прерываний, время реакции не страдает.
- Время реакции системы не зависит от длины CS.

**Недостатки:**
- Не работает на SMP (другие CPU не знают).
- Зависимость от политики планирования (плохая переносимость).

### Проблема: инверсия приоритетов

Когда мьютексы используются между потоками разных приоритетов, возникает тонкая проблема.

### Сценарий

Три потока: H (high), M (medium), L (low). H и L имеют общий ресурс под мьютексом. M не использует ресурс.

1. **L работает** (других готовых нет).
2. L входит в CS, захватывает мьютекс.
3. **H становится готов**, вытесняет L. (L сохранён, держит мьютекс.)
4. H пытается захватить мьютекс → блокируется (L держит).
5. **L снова работает** — единственный готовый.
6. **M становится готов** (например, истёк его сон).
7. M приоритетнее L → M вытесняет L.
8. **M работает долго** (не нуждается в мьютексе).
9. L всё это время ждёт. H ждёт L.

**Парадокс:** H фактически ждёт M, хотя M не имеет к мьютексу никакого отношения.

### Почему «неограниченная» инверсия

Из rtos.md (6.7):
> «Высокоприоритетный поток получается, по сути, вытеснен среднеприоритетным потоком, потому что тот вытеснил третий поток, который удерживает нужные высокоприоритетному ресурсы. Такая нежелательная ситуация называется неограниченной инверсией приоритета (unbounded priority inversion). Неограниченной она называется потому, что время, на которое среднеприоритетный поток вытесняет низкоприоритетный потенциально не ограничено».

Время ожидания H зависит не от длины CS (что было бы предсказуемо), а от того, как долго M будет работать (что неограниченно).

### Реальный пример: Mars Pathfinder (1997)

NASA отправило марсоход Pathfinder. На VxWorks RTOS работали три задачи:
- Высокоприоритетная: bus management (управление шиной).
- Средняя: communications (связь).
- Низкоприоритетная: meteorological data (метеоданные).

Высокая и низкая задачи делили шину через мьютекс. Возникала инверсия — после того, как средняя задача сильно нагружала систему, высокая опаздывала с управлением шиной → watchdog timer срабатывал → марсоход перезагружался.

Спасли удалённо: загрузили патч, который включил **наследование приоритетов** на мьютексе.

### Решение 1: наследование приоритетов (priority inheritance)

Идея: когда H пытается захватить мьютекс, удерживаемый L, **временно повысить** приоритет L до приоритета H.

Из rtos.md (6.8.1):
> «При попытке подождать мьютекс, поскольку его владелец известен, можно сравнить приоритет владельца и приоритет текущего потока, захватывающего мьютекс. Если приоритет ожидающего больше, чем приоритет владельца, приоритет последнего асинхронно повышается. То есть повышение приоритета владельца вызывает тот факт, что кто-то с большим приоритетом попытался захватить мьютекс».

После повышения:
1. H ждёт мьютекс, удерживаемый L.
2. L получает приоритет H.
3. M не может вытеснить L (теперь L приоритетнее M).
4. L быстро выходит из CS, отдаёт мьютекс.
5. Приоритет L возвращается к исходному.
6. H захватывает мьютекс, продолжает.

### Сложности реализации

**1. Цепочки наследования.**

Из rtos.md (6.8.1):
> «Большинство ОС реализуют наследование приоритета только на одном уровне, то есть от ожидателя мьютекса к его владельцу, но оказывается, что этого бывает недостаточно. Поток может владеть сразу несколькими мьютексами, поэтому инверсия приоритета может возникнуть опосредованно, через цепочку владельцев: когда кто-то пытается ждать мьютекса, владелец которого сам ждёт некоторого другого мьютекса, владельцем которого является третий поток».

Полная защита требует **прохода по цепочке** мьютексов: T1 ждёт мьютекс у T2, T2 ждёт у T3, T3 ждёт у T4...

Приоритеты T2, T3, T4 нужно повысить до T1.

**2. Глобальная синхронизация.**

Из rtos.md:
> «Гарантированное выполнение наследования приоритетов с учётом цепочек требует глобальной блокировки ядра, которая чревата большими проблемами с производительностью и масштабируемостью».

Чтобы корректно обновить приоритеты по цепочке, нужно атомарно «заморозить» состояние всех мьютексов и потоков → ухудшение масштабируемости на SMP.

**3. Таймауты, отмена ожидания.**

Поток может прервать ожидание мьютекса (по таймауту). Нужно корректно обработать понижение приоритета владельца.

### Реализация в Linux

В POSIX:
```c
pthread_mutexattr_t attr;
pthread_mutexattr_init(&attr);
pthread_mutexattr_setprotocol(&attr, PTHREAD_PRIO_INHERIT);
pthread_mutex_init(&mutex, &attr);
```

В ядре Linux: RT-mutex с автоматическим priority inheritance.

### Решение 2: потолок приоритета (priority ceiling)

Идея: с мьютексом сопоставляется **потолок** — максимальный приоритет любого потока, который может его захватить.

Из rtos.md (6.8.2):
> «С мьютексом сопоставляется приоритет, называемый потолком, по аналогии с потоком (указывается при создании). Все потоки, захватывающие этот мьютекс, получают повышение приоритета до потолка. При этом не требуется сложной логики внутри реализации мьютекса, алгоритм работает за фиксированное время и обеспечивает необходимую защиту от инверсии приоритета».

При захвате мьютекса приоритет потока **автоматически** повышается до потолка. После освобождения — возвращается к исходному.

### Пример

Мьютекс M с потолком 50. Потоки L (приоритет 10), M (приоритет 30), H (приоритет 80).

1. L захватывает M, его приоритет становится 50.
2. M не может вытеснить L (приоритет L = 50 > 30).
3. L работает, выходит из CS, приоритет = 10.

Если H захочет мьютекс, его приоритет уже 80 > 50 (потолок). Но потолок должен быть выбран ≥ максимума всех потоков, использующих мьютекс.

### Immediate Priority Ceiling Protocol (IPCP)

Из rtos.md:
> «Данный протокол называется протоколом с непосредственным потолком приоритета (immediate priority ceiling protocol, IPCP). Существует ещё оригинальный протокол потолка приоритета (Original priority ceiling protocol, OPCP), но он довольно сложен в реализации и практически не применяется на практике».

IPCP — упрощённая версия:
- Поток получает потолок **сразу** при захвате (вне зависимости от того, есть ли ожидающие).
- Простая и быстрая реализация.

OPCP — оригинальный (Sha, 1990):
- Сложнее.
- Гарантирует отсутствие deadlock.

### Применяемость

| Параметр | Priority Inheritance | Priority Ceiling (IPCP) |
|---|---|---|
| Момент повышения | При попытке захвата H | При захвате мьютекса |
| Заранее знать потолок | Нет | Да |
| Сложность реализации | Высокая | Низкая |
| Накладные расходы | Высокие | Низкие |
| Гарантия отсутствия deadlock | Нет | Да (OPCP) |
| Применение | Linux pthread | RTOS (FX-RTOS, VxWorks) |

### Реализация в FX-RTOS (rtos.md)

```c
typedef struct _fx_mutex_t {
    fx_sync_waitable_t wait_queue;
    fx_thread_t* owner;
    fx_sched_params_t ceiling_params;     // параметры потолка
    fx_sched_params_t original_params;    // исходные параметры потока
} fx_mutex_t;
```

При захвате:
1. Записать `original_params = текущие параметры потока`.
2. Применить `ceiling_params` к потоку.

При освобождении:
1. Восстановить `original_params`.

Важное ограничение IPCP — порядок освобождения мьютексов:
> «Мьютексы с потолком приоритета чувствительны к порядку, в котором мьютекс освобождается. Нарушение этого порядка чревато неправильной установкой приоритета потока после завершения критической секции».

Мьютексы должны освобождаться в обратном порядке захвата (LIFO).

### Применение

- **Mars Pathfinder** — после исправления использовал priority inheritance.
- **AUTOSAR** — автомобильные системы.
- **Медицинская техника.**
- **Промышленная автоматика.**

### Резюме

- Инверсия приоритетов — проблема, когда H ждёт L, а M (среднеприоритетный) надолго отнимает CPU у L.
- Решения:
  - **Priority inheritance** — повышение приоритета владельца до приоритета ожидающего.
  - **Priority ceiling** — потолок, до которого повышается приоритет при захвате.
- Inheritance динамическое, ceiling статическое.
- Реальные катастрофы (Mars Pathfinder) демонстрируют важность.

## 59. Семафоры. Типы семафоров. Семафоры для чтения-записи.
*Источник: rtos.md, Главы 6.4 «Семафоры», 6.5 «Разновидности семафоров»; robert_lav.md, Глава 10 «Семафоры»*

### Что это, на пальцах

**Семафор — это просто счётчик «свободных мест».** Поток хочет войти → проверяет «есть место?». Если есть — занимает и проходит (счётчик −1). Если нет — засыпает в очереди. Когда другой поток выходит и возвращает место (счётчик +1), один из ждущих просыпается и заходит.

Всё. Дальше пойдут детали и применения, но **базовая идея — именно эта**.

> «Подлинно универсальным объектом синхронизации, в том смысле, что он может использоваться как для реализации взаимного исключения, так и для уведомлений, является предложенный в 1965 году семафор. Название происходит от железнодорожного семафора, который обеспечивает нахождение только одного поезда на данном участке пути».

### Аналогия — парковка

Парковка на N мест. На въезде табло «свободных мест: N» — это значение семафора.

- Подъехал → табло уменьшается, шлагбаум поднимается.
- На табло `0` → стоишь в очереди у въезда.
- Кто-то уехал → табло увеличивается, шлагбаум впускает следующего из очереди.

Семафор работает ровно так. Только вместо машин — потоки.

### Две операции

| Операция | Что делает | По-человечески |
|---|---|---|
| `wait()` / `P()` / `down()` | Уменьшить счётчик на 1. Если стал бы отрицательным — заблокировать поток, пока не появится место. | «Хочу войти» |
| `post()` / `V()` / `up()` | Увеличить счётчик на 1. Если в очереди есть спящие — разбудить одного. | «Выхожу, освобождаю место» |

Названия `P` и `V` — от Дейкстры: голландские *Proberen* («тестировать») и *Verhogen* («увеличивать»). В коде Linux используются `down` и `up`, в POSIX — `sem_wait` и `sem_post`.

### Применение 1: ограничить число потоков в критической секции

Инициализируем семафор начальным значением N — это «разрешённое число одновременных входов». Если `N = 1` — это **взаимное исключение** (один внутри, остальные ждут). Если `N = 10` — пускаем до 10 потоков (например, пул из 10 коннектов к БД).

```c
sem_t db_pool;
sem_init(&db_pool, /*между процессами*/ 0, /*начальное*/ 10);

void query() {
    sem_wait(&db_pool);     // ждать свободный коннект
    /* работаем с БД */
    sem_post(&db_pool);     // вернуть коннект в пул
}
```

Если 11-й поток придёт, когда все 10 заняты — он заснёт. Как только кто-то закончит и сделает `sem_post`, его разбудят.

### Применение 2: уведомление о событии

Семафор начинает с `0` — «событий ещё не было». Тот, кто формирует событие, делает `sem_post()`. Тот, кто ждёт — `sem_wait()` (если событий нет, засыпает).

```c
sem_t event;
sem_init(&event, 0, 0);

// Прерывание от железа:
void ISR() {
    sem_post(&event);       // «случилось!»
}

// Поток-обработчик:
void worker() {
    for (;;) {
        sem_wait(&event);   // спит, пока кто-нибудь не подаст сигнал
        handle();
    }
}
```

**Ключевое преимущество над простым флагом-битом:** счётчик помнит, **сколько** раз сигналили. Если событий было три, а поток только начал работать — он три раза подряд пройдёт `sem_wait` без блокировки. У флага-бита такого не бывает: либо стоит, либо нет.

> «Если поток пробуждается без ошибок, значит, значение счётчика семафора стало равно нулю и любой другой поток, попытавшийся войти в критическую секцию, защищаемую семафором, будет заблокирован».

### Семафор vs мьютекс — главное отличие

С первого взгляда бинарный семафор и мьютекс выглядят одинаково: оба пускают одного, остальных тормозят. Но у мьютекса есть **владелец** — освободить мьютекс может только тот поток, который его захватил. У семафора владельца нет — `post()` может вызвать кто угодно.

Поэтому семафор подходит для уведомлений (один поток ждёт, другой посылает сигнал), а мьютекс — нет: «другому» нечего «освобождать», он мьютекс не захватывал.

| Параметр | Семафор | Мьютекс |
|---|---|---|
| Владелец | Нет | Есть |
| Кто может освободить | Любой | Только владелец |
| Для уведомлений | Да | Нет |
| Защита от инверсии приоритетов | Нет | Часто есть |
| Рекурсивный захват | Нет | Можно |

### Типы семафоров

**Счётный семафор.** Счётчик ≥ 0, произвольное начальное значение. Самый общий случай. Используется для пулов ресурсов, для задачи «producer-consumer».

**Бинарный семафор.** Счётчик ограничен {0, 1}. Простой как лампочка: горит или нет. Удобен для взаимного исключения и одиночных сигналов.

> «Широкое распространение имеет также немного урезанный вариант семафора, счётчик которого ограничен только двумя значениями: 0 и 1. Такой семафор называется бинарным и является усовершенствованной версией события».

Главное отличие бинарного от обычного: если делается `post()`, а счётчик уже `1` — попытка пропадает. У счётного она увеличила бы значение до `2` и сохранилась бы.

> «Бинарный семафор, так же как и обычный, является универсальным примитивом, это означает, что на основе бинарных семафоров можно реализовать обычные и наоборот».

То есть оба типа эквивалентны по выразительности — выбор больше про удобство и эффективность.

**RW-семафор** — читателей-писателей, см. ниже.

### Политика очереди

Когда `post()` будит ждущих — кого именно? Есть две основные стратегии:

- **По приоритету** — будит самого приоритетного. Лучше для жёсткого реального времени, но низкоприоритетные могут голодать.
- **FIFO** — будит того, кто встал в очередь первым. Справедливее и дешевле в реализации.

Многие RTOS (включая FX-RTOS) позволяют выбрать политику при создании.

### Семафоры для чтения-записи (RW)

Бывает, что данные часто читают и редко пишут — конфигурация системы, маршрутные таблицы, кэш DNS. Заставлять читателей ждать друг друга бессмысленно: они не конфликтуют, ничего не меняют. Конфликтует только запись.

RW-семафор учитывает это и допускает **три состояния**:
- Никто не держит.
- Держат **сколько угодно читателей** одновременно.
- Держит **ровно один писатель**, читателей в это время нет.

> «Все семафоры для чтения-записи являются мьютексами, т.е. их счётчик использования равен единице. Несмотря на это, они предоставляют монопольный доступ только для записывающих, а не считывающих данные потоков. Блокировку по чтению может одновременно удерживать любое количество потоков, при условии, что при этом нет ни одного потока записи. И наоборот, только один поток может удерживать блокировку по записи, если при этом нет ни одного потока, захватившего эту же блокировку по чтению».

### API в Linux

```c
static DECLARE_RWSEM(my_rwsem);

down_read(&my_rwsem);     // вход для чтения (можно параллельно с другими читателями)
/* читаем данные */
up_read(&my_rwsem);

down_write(&my_rwsem);    // вход для записи (эксклюзивно)
/* меняем данные */
up_write(&my_rwsem);
```

### Когда использовать RW-семафор

> «Семафоры для чтения-записи, так же как и спин-блокировки аналогичного типа, должны использоваться, только если есть чёткое разделение участков кода, которые выполняют чтение и которые осуществляют запись. Использование механизмов блокировок по чтению-записи приводит к дополнительным накладным расходам».

**Подходит:** читателей **много**, пишут **редко**, и читатели не должны мешать друг другу. Параллельное чтение даёт настоящий выигрыш в пропускной способности.

**Не подходит:** чтение и запись примерно поровну, или одновременных читателей мало — тогда обычный мьютекс будет проще и быстрее.

### Особенность Linux: читатели приоритетнее

> «В операционной системе Linux семафоры для чтения-записи имеют особенность: блокировка по чтению всегда имеет большее преимущество по сравнению с блокировкой по записи».

Если хоть один читатель удерживает блокировку, новые читатели проходят, даже если уже стоит в очереди писатель. Это даёт лучшую пропускную способность на чтении, но писатель может **голодать** — если поток читателей не иссякает, ему так и не достанется блокировка.

В других системах политика может быть другой (writer-preference, fair FIFO), это компромисс между throughput и справедливостью.

### downgrade_write

Атомарно превращает write-lock в read-lock. Полезный приём:

```c
down_write(&sem);          // захватили эксклюзивно, потому что собирались писать
/* записали изменения */
downgrade_write(&sem);     // теперь читаем — пусть другие читатели тоже подключатся
/* читаем */
up_read(&sem);
```

Альтернатива была бы `up_write` + `down_read`, но между ними мог бы вклиниться чужой писатель и испортить картину. `downgrade_write` это исключает.

### Сравнение с другими примитивами

| Параметр | Семафор | Мьютекс | Спинлок |
|---|---|---|---|
| Владелец | Нет | Есть | – |
| Sleep при блокировке | Да | Да | Нет (крутится) |
| Для уведомлений | Да | Нет | Нет |
| Для взаимного исключения | Да | Да | Да |
| Можно из обработчика прерывания | Нет (sleep) | Нет | Да |
| Накладные расходы | Средние | Средние | Низкие |
| Защита от инверсии приоритетов | Нет | Часто есть | Нет |

## 60. Секвенсоры.
*Источник: rtos.md, Глава 6.6 «Счетчики событий и секвенсоры»; robert_lav.md, Глава 10 (seqlock)*

### Что такое секвенсор

Из rtos.md (6.6):
> «Более интересным объектом является секвенсор. Он содержит единственный метод, который, по сути, сводится к атомарному инкременту».

```c
int32_t ticket(sequencer_t* object);
```

Каждый вызов возвращает **уникальный возрастающий номер**. Подобно автомату выдачи талонов.

### Связанный примитив — счётчик событий

Секвенсор обычно используется в паре со **счётчиком событий**:

```c
int advance(event_count_t* object);
int read(event_count_t* object, int32_t* value);
int await(event_count_t* object, int32_t value);
```

- **`advance`** — атомарный инкремент счётчика. Будит всех ожидающих этого значения.
- **`read`** — текущее значение.
- **`await(N)`** — ждать, пока счётчик не станет ≥ N.

### Применение — электронная очередь

Аналогия — талоны в банке:
1. Каждый клиент берёт талон с номером.
2. Обслуживается тот, чей номер показан на табло.
3. После обслуживания номер на табло увеличивается.

```c
sequencer_t seq;          // выдача номеров
event_count_t ec;          // текущий обслуживаемый

void access_resource() {
    int my_ticket = ticket(&seq);    // взять номер
    await(&ec, my_ticket);            // дождаться своей очереди
    // работа с ресурсом
    advance(&ec);                     // позвать следующего
}
```

### Свойства

Из rtos.md:
> «По сравнению с семафором, секвенсор обеспечивает последовательность, то есть потоки получат доступ к разделяемому ресурсу именно в той последовательности, в которой они его запрашивали».

**FIFO гарантировано** — потоки обслуживаются в порядке прибытия.

### Возможность оценки ожидания

Из rtos.md:
> «По аналогии с цифровой очередью, вытянув номер, указывающий, что впереди находятся 50 человек, можно пойти заниматься другими делами, вернувшись к очереди позже. Если потоку есть чем заняться помимо ожидания, он может, определив, с помощью своего "талона" и функции read, примерное количество потоков перед ним, заняться другими делами, вернувшись к ожиданию позже».

Поток может посмотреть, сколько перед ним:
```c
int my_ticket = ticket(&seq);
int current;
read(&ec, &current);
int waiting_before = my_ticket - current;
if (waiting_before > 10) {
    do_other_work();  // занят чем-то другим
}
await(&ec, my_ticket);
```

### Преимущества

- **Справедливость (FIFO).**
- **Нет голодания.**
- **Возможность оценки времени ожидания.**

### Недостатки

Из rtos.md:
> «Это одновременно является минусом, потому что, если по каким-либо причинам, поток, очередь которого подошла, не может выполняться, все остальные потоки будут ждать, даже если они могли бы получить доступ к ресурсу и закончить свою работу».

Один заблокированный поток держит всю очередь.

### Счётчики событий

Из rtos.md:
> «Счётчик событий, это, как следует из названия, счётчик, с которыми сопоставлены три основных метода: advance, read, await. В отличие от семафора, счётчик событий никогда не уменьшается. Потоки могут ожидать определённого значения счётчика».

В отличие от семафора, счётчик растёт **без декремента**. Это может казаться странным, но даёт интересные возможности — каждое значение счётчика как «время» в системе.

Из rtos.md:
> «Счётчики событий сложнее с точки зрения реализации: счётчик никогда не уменьшается, поэтому, рано или поздно, он переполняется и логика использования примитива должна это учитывать. По причине не очень широкой поддержки в операционных системах, сложностей в реализации, а также в связи с более сложным прикладным интерфейсом, счётчики событий и секвенсоры не завоевали большой популярности в прикладном программировании».

### Применение в спинлоках — Ticket Spinlock

Хотя секвенсоры мало используются в прикладном программировании, их идеи живут в реализации **ticket spinlock**.

Из rtos.md:
> «В этом случае спинлок представляет собой два значения, одно используется как секвенсор, а второе — как счётчик событий. Всякий раз, при попытке подождать спинлок, ожидающий поток атомарно увеличивает секвенсор и получает свой номер. Ожидание происходит до тех пор, пока счётчик событий меньше полученного значения "талона". Освобождение спинлока происходит путём увеличения счётчика событий».

```c
struct ticket_lock {
    atomic_t next_ticket;   // секвенсор
    atomic_t now_serving;   // счётчик событий
};

void acquire(struct ticket_lock *l) {
    int my = atomic_fetch_add(&l->next_ticket, 1);
    while (atomic_read(&l->now_serving) != my)
        cpu_relax();
}

void release(struct ticket_lock *l) {
    atomic_inc(&l->now_serving);
}
```

**Преимущества ticket spinlock:**
- FIFO порядок.
- Нет голодания.
- Простая реализация.

**Недостаток:**
- Все CPU крутятся на `now_serving` → cache thrashing на больших SMP.

Решение — **MCS-lock**: каждый поток крутится на своей переменной.

### Seqlock в Linux

Из robert_lav.md (Глава 10):
> «Спин-блокировка для последовательностей (sequence lock) предназначена для поддержки чтения и записи общих данных».

```c
// Источник: robert_lav.md, Глава 10
seqlock_t mr_seq_lock = DEFINE_SEQLOCK(mr_seq_lock);

// Писатель
write_seqlock(&mr_seq_lock);
// модифицировать данные
write_sequnlock(&mr_seq_lock);

// Читатель
unsigned int seq;
do {
    seq = read_seqbegin(&mr_seq_lock);
    // прочитать данные
} while (read_seqretry(&mr_seq_lock, seq));
```

### Принцип seqlock

- Внутри seqlock — счётчик последовательности (sequence counter).
- Писатель **увеличивает** счётчик при входе и выходе из CS. Между ними счётчик нечётный.
- Читатель:
  - Читает счётчик до начала.
  - Если нечётный — писатель работает, повторить.
  - Читает данные.
  - Читает счётчик снова.
  - Если изменился — данные могли быть некорректны, повторить.

### Преимущества seqlock

- **Очень дёшево для читателей** — никаких атомарных операций, только чтение.
- **Хорошая масштабируемость** на read-heavy данных.
- Подходит для часто читаемых данных.

### Недостатки seqlock

- **Голодание читателей** при частых записях.
- **Не подходит для сложных структур** — данные читаются в потенциально несогласованном состоянии, нужны меры предосторожности.
- Писатели **не блокируются друг другом** автоматически — нужен отдельный мьютекс.

### Применение seqlock в Linux

- **`xtime_lock`** — для системного времени. Читается часто (`gettimeofday`), пишется редко (тик таймера).
- **Маршрутные таблицы.**
- **Конфигурации с редкими изменениями.**

```c
// Псевдо-пример из ядра
unsigned long get_jiffies_64(void) {
    unsigned int seq;
    unsigned long ret;
    do {
        seq = read_seqbegin(&jiffies_lock);
        ret = jiffies_64;
    } while (read_seqretry(&jiffies_lock, seq));
    return ret;
}
```

### Сравнение

| Примитив | FIFO | Cache thrashing | Применение |
|---|---|---|---|
| Базовый спинлок | Нет | Да | Короткие CS |
| Ticket spinlock | Да | Да | FIFO важно |
| MCS-lock | Да | Минимально | Большие SMP |
| Seqlock | – | Минимально | Read-mostly данные |

### Резюме

- Секвенсор — атомарный счётчик, выдающий уникальные возрастающие номера.
- В паре со счётчиком событий реализует FIFO очередь ожидания.
- Прикладное применение редкое.
- Идеи используются в **ticket spinlock** — FIFO-вариант обычного спинлока.
- В Linux похожая концепция — **seqlock** для read-mostly данных с очень дешёвыми читателями.
