## 51. Критическая секция. Race Condition. Пример.
*Источник: tanenbaum.md, Глава 2.3.1 «Состязательная ситуация»; rtos.md, Глава 6.1*

**Race condition (условие гонки, состязательная ситуация)** — ошибка, когда результат выполнения программы зависит от **порядка** выполнения потоков/процессов. Корректность результата — дело случая.

Из rtos.md (6.1):
> «Успешность выполнения алгоритма зависит от того, успел ли какой-то поток сделать свою работу вовремя. Поскольку у пользователя есть ограниченные возможности влияния на чередование потоков, получается, что всё зависит от воли случая, что, разумеется, недопустимо».

### Классический пример (Tanenbaum, спулер печати)

Два процесса А и Б одновременно хотят поставить файл в очередь печати. Очередь имеет переменные `in` (следующая свободная позиция).

```c
int next_free_slot = in;        // (1) читаем
spool_dir[next_free_slot] = filename;  // (2) пишем имя
in = next_free_slot + 1;         // (3) обновляем
```

Сценарий:
1. А выполняет (1): `next_free_slot = 7`.
2. Прерывание → переключение на Б.
3. Б выполняет (1): `next_free_slot = 7`.
4. Б выполняет (2): записывает свой файл в 7.
5. Б выполняет (3): `in = 8`.
6. Возврат к А, продолжает с (2).
7. А выполняет (2): **затирает** имя файла Б своим именем.
8. А выполняет (3): `in = 8`.

Результат: файл Б потерян. Очередь выглядит корректно, но Б никогда не дождётся печати.

### Простой пример счётчика

```c
int counter = 0;

void increment() {
    counter++;       // на самом деле: load, +1, store
}
```

Два потока выполняют `increment()` 1000 раз каждый. Ожидаемо: counter = 2000.

На практике:
- Поток 1: загружает counter (0).
- **Переключение.**
- Поток 2: загружает counter (0), увеличивает (1), сохраняет (1).
- Возврат к 1: увеличивает (1), сохраняет (1).

Одно увеличение **потеряно**. Counter может быть от 1000 до 2000.

### Причины race condition

1. **Неатомарность операций.** `counter++` — три инструкции.
2. **Переключение в неподходящий момент.**
3. **Параллельное выполнение на SMP.**
4. **Переупорядочивание** инструкций компилятором/CPU.

### Решения

- **Критическая секция** — взаимное исключение через мьютекс, семафор, спинлок.
- **Атомарные операции** (вопрос 56) — для простых случаев.
- **Lock-free структуры данных** (вопрос 70).

### Особенности

- Race condition **трудно воспроизвести** — зависит от тайминга.
- В тестах может не проявляться.
- Проявляется случайно в продакшене.
- Отладочные инструменты: ThreadSanitizer (TSan), Valgrind/Helgrind.

## 52. Критическая секция. TSL. XCHG. Пример.
*Источник: rtos.md, Глава 2.2.3 «Атомарность в многопроцессорных системах»; tanenbaum.md, Глава 2.3.3*

### Зачем нужна аппаратная поддержка

Программные алгоритмы (Петерсон, Декер) для взаимного исключения сложны и не масштабируются. Современные процессоры предоставляют **атомарные RMW-операции** (Read-Modify-Write).

Из rtos.md (2.2.3):
> «Большинство таких операций включают в себя 3 стадии, чтение, модификацию и запись нового значения. Из-за этого все операции, имеющие такую семантику называются RMW-операциями».

### TSL (Test and Set Lock)

Из rtos.md:
> «Инструкция TEST_AND_SET, которая является несколько частным случаем операции SWAP».

```c
// Источник: rtos.md, раздел 2.2.3
int atomic_test_and_set(volatile int* ptr) {
    int old_value = *ptr;
    *ptr = 1;
    return old_value;
}
```

Атомарно: читает значение, устанавливает в 1, возвращает старое значение.

### Использование TSL для блокировки

```c
volatile int lock = 0;

void enter_cs() {
    while (test_and_set(&lock) == 1) ;  // активное ожидание
}

void exit_cs() {
    lock = 0;
}
```

Если `lock` был 0 → стал 1, возвращает 0 → выход из цикла, в CS.
Если `lock` уже 1 → остаётся 1, возвращает 1 → цикл.

### XCHG (Exchange / SWAP)

Атомарно обменивает значение в регистре со значением в памяти.

```c
// Источник: rtos.md, раздел 2.2.3
int atomic_swap(volatile int* ptr, int new_value) {
    int old_value = *ptr;
    *ptr = new_value;
    return old_value;
}
```

На x86: инструкция `XCHG reg, [mem]` автоматически атомарна (lock prefix не нужен).

### Использование XCHG для блокировки

```c
// Источник: rtos.md, реализация спинлока через SWAP
void spinlock_acquire(volatile int* lock) {
    while (atomic_swap(lock, 1) == 1) ;
}

void spinlock_release(volatile int* lock) {
    *lock = 0;
}
```

Принцип тот же: атомарно установить в "занято" и проверить, было ли "свободно".

### Реализация на x86

```asm
; lock — адрес переменной блокировки
acquire_lock:
    mov eax, 1
spin:
    xchg eax, [lock]    ; атомарно обменять
    test eax, eax
    jnz spin            ; если был 1 — снова попробовать
    ret

release_lock:
    mov dword [lock], 0
    ret
```

### Достоинства

- **Атомарно**.
- **Работает на SMP.**
- **Простая реализация.**
- Поддержка во всех современных архитектурах.

### Недостатки

- **Активное ожидание** (spin) — тратит CPU.
- **Cache thrashing** на SMP: все CPU пытаются модифицировать одну линию кэша.

Оптимизация — **test-test-and-set**:
```c
while (lock == 1 || test_and_set(&lock) == 1) ;
```
Сначала просто читать (дёшево), и только если выглядит свободным — пытаться захватить (дорого).

### Применение

- Реализация **спинлоков** в ядрах ОС.
- **Аппаратная блокировка** в драйверах устройств.
- Основа для более сложных примитивов синхронизации.

## 53. Критическая секция. Проблема консенсуса. Пример.
*Источник: rtos.md, Глава 2.2.4 «Неблокирующие алгоритмы»*

### От критической секции к консенсусу

Что значит «зайти в критическую секцию»? Это договориться между всеми желающими, **кто пойдёт первым**. По сути, потоки должны прийти к одному и тому же выводу: «сейчас работает поток X». Без такой договорённости двое окажутся в критической секции одновременно — и поломают общие данные.

Эта задача в общем виде называется **проблемой консенсуса**: несколько потоков должны прийти к одному решению. Изучение консенсуса показывает, **какие именно процессорные инструкции в принципе позволяют это сделать**, а какие — нет.

### Формальная постановка

> «Консенсус — это алгоритм, который содержит единственный метод `decide`. Консенсус работает так: каждый процессор указывает в качестве аргумента некоторое уникальное число (например, свой номер) и получает результат».

```c
int decide(int my_value);
```

Контракт функции:
1. **Валидность.** Возвращённое значение — это `my_value` **одного из** вызывающих потоков. Не выдуманное.
2. **Согласованность.** **Все** потоки получают **одно и то же** значение.
3. **Завершимость.** Каждый вызов завершается за конечное число шагов (для wait-free — без блокировок).

### Пример

Три потока одновременно вызвали `decide`:

| Поток | Передал | Получил |
|---|---|---|
| A | 10 | ? |
| B | 20 | ? |
| C | 30 | ? |

Допустимые исходы — все трое получают `10`, или все трое `20`, или все трое `30`. Кто из них «победил» — заранее неизвестно, это решает гонка.

Недопустимо: A получил `10`, а B получил `20` (разные ответы → согласованности нет). Недопустимо: все получили `42` (никто такого не предлагал → валидности нет).

### Число консенсуса

> «Определяется также число, называемое числом консенсуса (consensus number). Это максимальное количество процессоров, для которого решаема задача консенсуса, используя данный набор атомарных инструкций».

То есть: **сколько потоков примитив способен честно рассудить**.

| Примитив | Число консенсуса |
|---|---|
| Обычные чтение и запись | 1 |
| Test-and-set, Swap (XCHG), Fetch-and-add | 2 |
| CAS, LL/SC | ∞ |

### Почему обычная переменная даёт 1

Двух потоков уже не рассудить. Попробуем наивно:

```c
int shared = -1;
int decide(int my_value) {
    if (shared == -1)         // ① проверка
        shared = my_value;    // ② запись
    return shared;
}
```

Двое одновременно прочитали `-1`, оба прошли `if`, оба записали свои значения — последний затёр первого, но первый уже мог вернуть своё значение. Разные ответы → согласованность нарушена. Без атомарного «прочитать и записать одной операцией» договориться невозможно.

### Почему SWAP даёт 2

Для двух потоков работает:

```c
int decide(int my_value) {
    int prev = swap(&shared, my_value);
    if (prev == 0) return my_value;   // я был первый — мой ответ
    else           return prev;        // кто-то был раньше — его ответ
}
```

Поток, обнаруживший `0`, понимает: «я первый, мой ответ — правильный». Второй обнаруживает значение первого и возвращает его. Все вернули одинаково.

Для **трёх** потоков такой трюк не работает — доказывается, что никаким алгоритмом на SWAP трое потоков не согласуются.

### Почему CAS даёт ∞

Compare-and-Swap способен рассудить любое число потоков:

```c
int shared = -1;
int decide(int my_value) {
    cas(&shared, -1, my_value);   // победит только первый
    return shared;                 // все увидят значение победителя
}
```

`cas(&shared, -1, my_value)` атомарно меняет `-1` на `my_value` **только если** там сейчас `-1`. Первый успевший установил своё значение. Все последующие CAS не сработают (там уже не `-1`), но при `return shared` они прочитают то же самое значение победителя. Все потоки возвращают одно и то же.

### Главный практический вывод — теорема Herlihy

> «Любой wait-free алгоритм может быть реализован тогда и только тогда, когда существует wait-free решение проблемы консенсуса».

Поскольку CAS имеет число консенсуса ∞, **на нём можно построить любую wait-free структуру данных** (очереди, списки, хеш-таблицы без блокировок). Без CAS — нельзя, и это математический факт, а не вопрос мастерства программиста.

Поэтому **на всех современных процессорах есть CAS** (x86: `CMPXCHG`, ARM/RISC-V: `LL/SC`). Без него многоядерное программирование без замков было бы принципиально невозможно для произвольного числа потоков.

### Аналогия

Сравним инструменты для голосования:

- **Записка** (обычная переменная) — двое могут писать одновременно, путаница. Число консенсуса 1.
- **Поднятая рука** (TAS / SWAP) — двое поднимают руки, видно, кто раньше. Для троих уже не получится надёжно. Число 2.
- **Электронная голосовалка** (CAS — одна заявка от одного, остальные видят «занято») — хоть миллион участников, система рассудит. Число ∞.

## 54. Критическая секция. CAS. Двойной CAS. Пример.
*Источник: rtos.md, Глава 2.2.4 «Неблокирующие алгоритмы»*

### CAS — Compare-And-Swap

Из rtos.md (2.2.4):
> «Простейшая инструкция, обладающая бесконечным числом консенсуса, то есть достаточно универсальная для реализации любого wait-free алгоритма, называется сравнение с обменом (compare and swap, CAS)».

Принимает 3 аргумента: указатель, ожидаемое значение, новое значение.

```c
// Источник: rtos.md, раздел 2.2.4
int CAS(volatile int* ptr, int comparand, int new_value) {
    int old_value = *ptr;
    if (old_value == comparand) {
        *ptr = new_value;
    }
    return old_value;
}
```

Атомарно: если значение по указателю равно ожидаемому → заменить на новое. Иначе — не менять. Вернуть старое значение.

**Успех:** возвращённое значение == comparand.
**Неудача:** возвращённое != comparand (значит, кто-то изменил).

### Реализация в архитектурах

- **x86:** `CMPXCHG`.
- **ARM:** через LL/SC (вопрос 55).
- **POWER:** через LL/SC.

### CAS-loop — типичный паттерн

```c
// Источник: rtos.md, fetch-and-add через CAS
int old_value, result;
do {
    old_value = *ptr;
    result = CAS(ptr, old_value, old_value + N);
} while (result != old_value);
```

Алгоритм:
1. Прочитать текущее значение.
2. Попытаться CAS с этим значением как ожидаемым.
3. Если успех → готово. Если неудача → значение менялось, повторить.

### Реализация других операций через CAS

```c
// fetch_and_add
int my_fetch_and_add(int *p, int delta) {
    int old;
    do {
        old = *p;
    } while (CAS(p, old, old + delta) != old);
    return old;
}

// max
int atomic_max(int *p, int v) {
    int old;
    do {
        old = *p;
        if (v <= old) return old;
    } while (CAS(p, old, v) != old);
    return old;
}
```

### Проблема ABA

Из rtos.md:
> «У CAS есть и недостатки. Один из них — так называемая проблема ABA. Довольно часто в качестве аргумента CAS используется указатель, в этом случае возможна ситуация, когда численно указатель остался тот же, но указывает уже на другой объект».

Сценарий:
1. Поток 1 читает указатель: `ptr = A`.
2. Поток 1 готовится сделать CAS.
3. Поток 2: освобождает A, выделяет новый объект B, освобождает B, выделяет A снова (другие данные!).
4. Поток 1 делает CAS — успех (значение A такое же), но **объект другой**!

### Двойной CAS (DCAS, DWCAS)

Из rtos.md:
> «Некоторые процессоры, например intel x86, поддерживают двойной CAS. Его отличие от описанного выше в том, что аргумент может быть вдвое большей длины. Это позволяет, в случае использования указателя в качестве операнда, снабдить его дополнительной информацией, тегом или счётчиком, который находится в соседней ячейке памяти, а CAS, таким образом, обновляет сразу оба значения».

На x86-64: инструкция `CMPXCHG16B` — атомарно сравнивает и заменяет 128 бит.

```c
struct tagged_pointer {
    void *ptr;
    uint64_t tag;       // счётчик
};

bool DCAS(tagged_pointer *target, tagged_pointer expected, tagged_pointer new_val);
```

При каждой модификации **tag увеличивается**. Если кто-то «вернул» указатель к старому значению, tag другой → CAS не пройдёт.

### Применение CAS

- **Lock-free структуры данных:** очереди, стеки, списки.
- **Атомарные счётчики** (`atomic_inc`, `atomic_dec`).
- **Реализация мьютексов** в пользовательском пространстве (futex).
- **RCU** (Read-Copy-Update).

### Пример: lock-free стек

```c
typedef struct node { int val; struct node *next; } node_t;
node_t *head;

void push(int val) {
    node_t *n = malloc(sizeof(node_t));
    n->val = val;
    node_t *old_head;
    do {
        old_head = head;
        n->next = old_head;
    } while (CAS(&head, old_head, n) != old_head);
}

int pop() {
    node_t *old_head, *new_head;
    do {
        old_head = head;
        if (!old_head) return -1;
        new_head = old_head->next;
    } while (CAS(&head, old_head, new_head) != old_head);
    int val = old_head->val;
    free(old_head);  // ВНИМАНИЕ: ABA-проблема!
    return val;
}
```

В реальности `free` здесь опасен из-за ABA. Решения: DCAS или RCU/hazard pointers.

## 55. Критическая секция. LL. SC. Пример.
*Источник: rtos.md, Глава 2.2.4 «Неблокирующие алгоритмы»*

### Альтернатива CAS — LL/SC

> «Хотя двойной CAS и решает проблему ABA, он не очень удобен в использовании, поэтому другие процессоры, такие как, например ARM или MIPS, реализуют другой подход, называемый LL/SC. Процессор поддерживает вместо одного CAS пару инструкций, называемых Load Link (LL) и Store Conditional (SC)».

### Зачем такая пара

Главная задача атомарных операций — безопасно изменить значение по схеме «прочитал → подумал → записал, если никто не вмешался». CAS решает её, сравнивая **значения**: «запиши, если в ячейке всё ещё то, что я ожидал». Отсюда проблема ABA: значение могло побывать чужим и вернуться к нашему, и CAS этого не заметит.

LL/SC подходит с другой стороны: процессор **аппаратно следит за самой ячейкой памяти**. Если кто угодно (другое ядро, прерывание, переключение контекста) обратится к этой ячейке — мы об этом узнаем, неважно, какое значение там сейчас стоит. Получается контракт «запиши, **только если никто не трогал ячейку**», а не «значение совпало». Это сильнее, и ABA здесь в принципе не возникает.

### LL — Load Link

«Прочитай значение и **повесь датчик** на эту ячейку». С момента LL процессор внутри себя помечает: «слежу за адресом X». Если кто угодно обратится по этому адресу — флаг слежения сбрасывается.

```c
// Источник: rtos.md
int load_link(volatile int* ptr) {
    int old_val = *ptr;
    start_monitoring(ptr);
    return old_val;
}
```

### SC — Store Conditional

«Запиши значение, **но только если датчик не сработал**» — то есть только если с момента LL никто не трогал ячейку. Если кто-то трогал — SC сообщает «не получилось», и мы должны попробовать заново.

```c
// Источник: rtos.md
bool store_conditional(volatile int* ptr, int new_value) {
    if (no_accesses_to_monitored_memory()) {
        *ptr = new_value;
        return true;
    }
    return false;
}
```

### Аналогия — сейф с пломбой

**CAS** = сейф с кодовым замком. Если код не изменился — открываешь, забираешь. Но пока ты не смотрел, кто-то мог открыть сейф, забрать деньги, положить такие же бумажки и закрыть — ты этого не заметишь (это и есть ABA).

**LL/SC** = сейф с одноразовой пломбой. После LL на сейф повесили пломбу. Делаешь SC — охранник проверяет: цела пломба или сорвана. Если сорвана — отказ, неважно, что внутри. Если цела — пишем.

### Использование

Типичный цикл (rtos.md):

```c
do {
    int old_val = load_link(ptr);
    // ... модификация значения ...
    int new_val = old_val + ...;
} while (!store_conditional(ptr, new_val));
```

### Реализация в архитектурах

- **ARM:** `LDREX` (Load Exclusive) / `STREX` (Store Exclusive).
- **MIPS:** `LL` / `SC`.
- **POWER:** `LWARX` / `STWCX`.
- **RISC-V:** `LR` / `SC`.

Пример на ARM:

```asm
retry:
    LDREX   r1, [r0]       ; LL — загрузить с маркировкой
    ADD     r1, r1, #1     ; модификация
    STREX   r2, r1, [r0]   ; SC — попытка записи
    CMP     r2, #0         ; r2 = 0 если успех
    BNE     retry          ; повторить при неудаче
```

### Отличие от CAS

Из rtos.md:
> «В отличие от CAS, SC-операция закончится неудачей даже в том случае, если значение, для которого осуществляется мониторинг, было заменено на численно такое же, поскольку отслеживается именно доступ к ячейке со стороны других процессоров».

**LL/SC автоматически решает проблему ABA** — отслеживается само обращение к ячейке, а не значение.

### Преимущества LL/SC

- **Нет проблемы ABA.**
- **Простой интерфейс** для процессора.
- **Гибкость** — можно делать произвольную модификацию между LL и SC.

### Недостатки LL/SC

- **Limited monitoring window.** Между LL и SC должен быть короткий код (некоторые архитектуры не разрешают прерывания, системные вызовы между ними).
- **Spurious failures.** SC может неудаться по причинам, не связанным с модификацией (например, переключение контекста).
- **Реализация CAS через LL/SC сложнее.**

### Пример реализации CAS через LL/SC (rtos.md, ARM)

```asm
; Источник: rtos.md, реализация atomic_cas для ARM
atomic_cas:
    ; R0 — указатель, R1 — ожидаемое, R2 — новое
retry:
    ldrex   r3, [r0]        ; LL
    cmp     r3, r1
    bne     fail
    strex   r4, r2, [r0]    ; SC
    cmp     r4, #0
    bne     retry
    mov     r0, r1          ; возврат: успех
    bx      lr
fail:
    mov     r0, r3          ; возврат: текущее значение
    bx      lr
```

### Применение

- Та же область, что CAS: lock-free структуры, атомарные счётчики, мьютексы.
- В архитектурах ARM, RISC-V, POWER — основной примитив атомарности.

## 56. Неделимые операции. Причины возникновения. atomic_t.
*Источник: rtos.md, Глава 2.1.6 «Атомарность»; robert_lav.md, Глава 10 «Неделимые операции»*

### Что это

**Неделимая (атомарная) операция** — операция, которую другие потоки видят либо полностью завершённой, либо ещё не начавшейся. Промежуточного состояния «снаружи» не существует.

> «Возможность выполнять составные операции, которые будут выглядеть как единое действие, очень важна для согласованной и непротиворечивой работы прикладных программ».

### Почему привычные операции не атомарны

`counter++` на уровне процессора — это **три** инструкции: загрузить значение в регистр, увеличить регистр, записать обратно. Между ними другой поток может вклиниться и тоже сделать `counter++`. В результате два «инкремента» могут увеличить счётчик не на 2, а на 1 — потерянное обновление.

### Откуда берётся проблема

1. **Прерывания** могут прервать поток в любой точке, в том числе между шагами «прочитал» и «записал».
2. **Многоядерность (SMP)** — другие ядра работают параллельно и могут изменить ячейку одновременно с нами.
3. **DMA** — устройства тоже пишут в память без участия CPU.

### Откуда берётся атомарность

Процессор предоставляет **специальные инструкции**, которые аппаратно склеивают «прочитать-изменить-записать» в одно неделимое действие: TSL, XCHG, fetch-and-add, CAS, LL/SC. На уровне железа это обеспечивается либо блокировкой шины (старый способ), либо протоколом когерентности кэшей (современный: MESI и др.).

### atomic_t в Linux

Чтобы не вызывать ассемблер вручную, ядро Linux обёртывает атомарные операции в типобезопасный API над специальным типом:

```c
// Источник: robert_lav.md, Глава 10
typedef struct {
    volatile int counter;
} atomic_t;
```

`atomic_t` — это **обёртка** над int, скрывающая внутреннее значение. С ней можно работать только через функции `atomic_*`, которые гарантированно используют атомарные инструкции:

```c
atomic_t v = ATOMIC_INIT(0);

atomic_inc(&v);            // v++ атомарно
atomic_dec(&v);            // v-- атомарно
atomic_add(2, &v);         // v += 2 атомарно
int cur = atomic_read(&v); // прочитать
```

### Зачем отдельный тип, а не обычный int

> «В функции, выполняющие неделимые операции, можно передать только аргументы типа `atomic_t`. Это гарантирует, что неделимые операции будут выполняться только с данными этого специального типа».

Три причины:

1. **Защита от ошибок.** Компилятор не даст случайно увеличить `atomic_t` обычным `v.counter++` мимо API. Если использовать просто `int`, программист легко забудет про атомарность.
2. **Защита от оптимизаций компилятора.** `volatile` внутри запрещает кешировать значение в регистре — каждое обращение действительно идёт в память.
3. **Переносимость.** Один и тот же код работает на x86, ARM, RISC-V — реализация спрятана за функциями.

### Типичное применение — счётчик ссылок

```c
struct my_obj {
    atomic_t refcount;
};

void get(struct my_obj *o) {
    atomic_inc(&o->refcount);
}

void put(struct my_obj *o) {
    if (atomic_dec_and_test(&o->refcount))   // атомарно: v--; вернуть (v == 0)?
        free(o);
}
```

Здесь `atomic_dec_and_test` за одну неделимую операцию уменьшает счётчик и проверяет, не стал ли он нулём. Без этого два потока могли бы одновременно решить «я последний» и дважды освободить объект.

### Когда не подходит

- **Только одна ячейка.** Если нужно атомарно изменить **две** переменные согласованно — `atomic_t` не поможет, нужен мьютекс или CAS на составной структуре.
- **Атомарность ≠ упорядоченность.** То, что инкремент атомарен, не значит, что другие CPU увидят его в правильном порядке относительно других записей. Для порядка нужны барьеры памяти.

## 57. Спин-блокировки. Применение. Особенности.
*Источник: rtos.md, Глава 6.2.3 «Использование спинлоков»; robert_lav.md, Глава 10*

### Что такое спинлок

**Спинлок (spinlock)** — блокировка, при которой поток, не сумевший её захватить, **активно ждёт** в цикле, постоянно проверяя состояние.

Из rtos.md (6.2.3):
> «Спинлоки являются мощным средством синхронизации и с оговорками могут использоваться во встроенных приложениях, так как вносимые ими накладные расходы невелики».

### Реализация

```c
// Источник: rtos.md, через atomic_swap
void spinlock_acquire(volatile int *lock) {
    while (atomic_swap(lock, 1) == 1) ;   // активное ожидание
}

void spinlock_release(volatile int *lock) {
    *lock = 0;
}
```

В Linux:
```c
spin_lock(&lock);
// критическая секция
spin_unlock(&lock);
```

### Когда применять

**1. Короткие критические секции** — пара десятков инструкций.

**2. Обработчики прерываний** — там нельзя блокироваться (нет переключения контекста).

**3. Когда блокировка освобождается быстро** — активное ожидание дешевле, чем переключение.

Из robert_lav.md:
> «Спин-блокировки являются очень простым и быстрым средством блокировки. Выполнение постоянных проверок в цикле является оптимальным, когда блокировки захватываются на очень короткое время и код не может переходить в состояние ожидания (например, в обработчиках прерываний)».

### Когда НЕ применять

**1. Длинные критические секции.** Поток-ожидающий тратит CPU впустую.

**2. Однопроцессорная система.** Если на одном CPU спинлок занят — никто не освободит, пока ожидающий не отдаст CPU. Поэтому в Linux на однопроцессорных конфигурациях `spin_lock` становится `preempt_disable`.

### Особенности и ограничения

**1. Невытесняемость.** Из rtos.md:
> «Из-за особенностей реализации, на использование спинлоков накладывается множество ограничений, таких как возможность их захвата только в регионах, где запрещено вытеснение».

Если бы планировщик переключил поток, удерживающий спинлок, на другой поток на том же CPU, и тот тоже захотел бы этот спинлок → активное ожидание без надежды на освобождение.

Поэтому **`spin_lock` неявно делает `preempt_disable()`**.

**2. Защита от прерываний.** Если спинлок может быть захвачен из обработчика прерывания, нужен **`spin_lock_irqsave`**:

```c
unsigned long flags;
spin_lock_irqsave(&lock, flags);
// CS
spin_unlock_irqrestore(&lock, flags);
```

Иначе deadlock: поток держит спинлок, приходит прерывание, обработчик пытается взять тот же спинлок → ждёт вечно.

**3. Активное ожидание.** Из rtos.md:
> «Поскольку спинлок предполагает активное ожидание в виде цикла, с ними связано также много проблем с энергопотреблением».

В батарейных устройствах spinlock увеличивает разрядку.

**4. Cache thrashing.** На SMP несколько CPU делают `swap` на одну линию кэша → постоянная инвалидация. Современные процессоры имеют инструкцию `PAUSE` (x86) или `YIELD` (ARM) для оптимизации spin-loop.

### Спинлоки с очередью (ticket spinlock)

Простой спинлок не гарантирует FIFO — может возникнуть starvation (один CPU постоянно «выигрывает»). Решение — ticket spinlock:

```c
struct ticket_lock {
    atomic_t next_ticket;
    atomic_t now_serving;
};

void acquire(struct ticket_lock *l) {
    int my_ticket = atomic_fetch_add(&l->next_ticket, 1);
    while (atomic_read(&l->now_serving) != my_ticket) ;
}

void release(struct ticket_lock *l) {
    atomic_inc(&l->now_serving);
}
```

Аналогия — электронная очередь в банке: каждый получает номер, обслуживание в порядке номеров.

### Применение в Linux

- **Защита короткоживущих ресурсов** в ядре.
- **Связные списки**, статистика.
- **Структуры драйверов.**
- **Реализация других примитивов.**

### Варианты спинлоков

| Тип | Описание |
|---|---|
| `spinlock_t` | Базовый |
| `rwlock_t` | Чтение/запись |
| `seqlock_t` | Для readeros — оптимистичная блокировка |
| `arch_spinlock_t` | Аппаратно-специфичная |

### Сравнение со семафором

| Параметр | Spinlock | Semaphore |
|---|---|---|
| Ожидание | Активное (CPU) | Пассивное (sleep) |
| Накладные расходы | Низкие | Высокие |
| Допустимость в IRQ | Да | Нет |
| Подходит для длинных CS | Нет | Да |
| Подходит для коротких CS | Да | Нет |

## 58. Критическая секция с применением приоритетов. Инверсия приоритетов. Наследование приоритетов.
*Источник: rtos.md, Глава 6.7 «Инверсия приоритета», 6.8.1 «Наследование приоритета», 6.8.2 «Потолок приоритета»*

### Использование приоритетов для синхронизации

Из rtos.md (6.2.4):
> «Если используется известная заранее политика планирования потоков, например FIFO, синхронизация между потоками может быть реализована с помощью этой политики».

Например, повысить приоритет потока на время критической секции — это исключит вытеснение.

### Проблема: инверсия приоритета

Классический сценарий с 3 потоками: H (high), M (medium), L (low).

1. L работает, входит в критическую секцию (захватывает мьютекс).
2. H становится готов, вытесняет L. Пытается захватить мьютекс → блокируется (его держит L).
3. L снова работает (пытается завершить CS).
4. **Активируется M.** M приоритетнее L, вытесняет его.
5. M долго работает (он не нуждается в мьютексе).
6. L всё это время ждёт. И H ждёт L.

**Парадокс:** высокоприоритетный H фактически ждёт среднеприоритетного M, хотя M напрямую с H никак не связан.

Из rtos.md (6.7):
> «Время ожидания потока 3, показанное как T зависит теперь не только от длины критической секции в потоке 1, но и от времени вытеснения этого потока третьими потоками, что и составляет суть проблемы инверсии приоритета. Такая нежелательная ситуация называется неограниченной инверсией приоритета (unbounded priority inversion)».

«Неограниченная» — потому что время ожидания H не имеет верхней границы (зависит от M, который может работать долго).

### Реальный пример: Mars Pathfinder (1997)

Марсоход терял связь из-за инверсии приоритетов в RTOS VxWorks. Спасли удалённой переконфигурацией с включением наследования приоритетов.

### Решение 1: наследование приоритета (priority inheritance)

Идея: когда H пытается захватить мьютекс, удерживаемый L, **временно повысить** приоритет L до приоритета H.

Из rtos.md (6.8.1):
> «При попытке подождать мьютекс, поскольку его владелец известен, можно сравнить приоритет владельца и приоритет текущего потока, захватывающего мьютекс. Если приоритет ожидающего больше, чем приоритет владельца, приоритет последнего асинхронно повышается».

Теперь:
1. H ждёт мьютекс, удерживаемый L.
2. Приоритет L поднимается до H.
3. M не может вытеснить L (теперь L приоритетнее).
4. L быстро заканчивает CS, освобождает мьютекс.
5. Приоритет L возвращается к исходному.
6. H захватывает мьютекс, продолжает.

### Цепочки наследования

Проблема: L может ждать другой мьютекс у L2, который ждёт у L3... Нужно повысить приоритет **по цепочке**.

Из rtos.md:
> «Большинство ОС реализуют наследование приоритета только на одном уровне, то есть от ожидателя мьютекса к его владельцу, но оказывается, что этого бывает недостаточно».

Полное наследование по цепочке требует глобальной блокировки → проблемы производительности.

### Сложности реализации

- Поток может владеть несколькими мьютексами.
- Приоритет потока сам может меняться.
- Таймауты на ожидание мьютекса.
- Синхронизация изменения приоритета.

В Linux: `PTHREAD_PRIO_INHERIT` атрибут мьютекса.

### Решение 2: потолок приоритета (priority ceiling)

Из rtos.md (6.8.2):
> «С мьютексом сопоставляется приоритет, называемый потолком, по аналогии с потоком (указывается при создании). Все потоки, захватывающие этот мьютекс, получают повышение приоритета до потолка».

При создании мьютекса указывается **потолок** — максимальный приоритет любого потока, который может его захватить.

Когда поток захватывает мьютекс — его приоритет **автоматически** поднимается до потолка. После освобождения — возвращается к исходному.

**Преимущества:**
- Простая реализация.
- Фиксированное время.
- Гарантированно исключает инверсию.

**Недостатки:**
- Нужно знать потолок заранее.
- Может приводить к ненужным повышениям приоритета.
- Чувствителен к порядку освобождения мьютексов.

В FX-RTOS и многих RTOS реализован **Immediate Priority Ceiling Protocol (IPCP)**.

### Различия

| Параметр | Priority Inheritance | Priority Ceiling |
|---|---|---|
| Момент повышения | При попытке захвата H | При захвате мьютекса |
| Заранее известный потолок | Нет | Да |
| Сложность реализации | Высокая | Низкая |
| Накладные расходы | Высокие | Низкие |
| Поведение | Динамическое | Статическое |
| Применение | Linux pthread, RTOS | RTOS (FX-RTOS, VxWorks) |

### Применение

- **Авиация и космос** (Mars Pathfinder теперь использует priority inheritance).
- **Автомобильные ECU** (AUTOSAR).
- **Медицинская техника.**
- **Промышленная автоматика.**

## 59. Семафоры. Типы семафоров. Семафоры для чтения-записи.
*Источник: rtos.md, Глава 6.4 «Семафоры», 6.5 «Разновидности семафоров»; robert_lav.md, Глава 10*

### Что это, на пальцах

**Семафор — это счётчик «свободных мест».** Поток, который хочет войти, проверяет: «есть свободное место?» Если есть — занимает его (счётчик уменьшается на 1) и проходит. Если нет — засыпает и ждёт, пока кто-нибудь не освободит место. Когда поток уходит, он возвращает место обратно (счётчик +1), и спящие просыпаются.

> «Подлинно универсальным объектом синхронизации, в том смысле, что он может использоваться как для реализации взаимного исключения, так и для уведомлений, является предложенный в 1965 году семафор. Название происходит от железнодорожного семафора, который обеспечивает нахождение только одного поезда на данном участке пути».

### Аналогия — парковка с шлагбаумом

Представь парковку на 5 мест. На въезде висит табло — «свободных мест: N». Это и есть семафор.

- Машина подъехала → табло уменьшается, шлагбаум поднимается. Если на табло `0` — машина встаёт в очередь у въезда.
- Машина уехала → табло увеличивается, шлагбаум впускает следующего из очереди.

`N` — это **значение семафора**. Все правила сводятся к этой картинке.

### Две операции

У семафора всего две операции:

| Операция | Что делает | По-человечески |
|---|---|---|
| `wait()` / `P()` / `down()` | Уменьшить счётчик на 1. Если стал бы отрицательным — спать, пока не появится место. | «Хочу зайти» |
| `post()` / `V()` / `up()` | Увеличить счётчик на 1. Если кто-то спит в очереди — разбудить одного. | «Выхожу, освобождаю место» |

Имена `P` и `V` — от Дейкстры: голландские *Proberen* («тестировать») и *Verhogen* («увеличивать»).

### Два типичных применения

**1. Ограничить число потоков в критической секции.** Инициализируем семафор начальным значением — это «количество разрешённых одновременных входов».

```c
sem_t sem;
sem_init(&sem, /*начальное значение*/ 1);   // только один за раз

sem_wait(&sem);
// критическая секция
sem_post(&sem);
```

Если начальное значение `1` — это **взаимное исключение** (один внутри, остальные ждут). Если `10` — пускаем до 10 потоков одновременно (например, лимит коннектов к БД).

**2. Уведомление о событии.** Семафор начинает с `0` — «событий пока нет». Тот, кто формирует событие, делает `sem_post()`, тот, кто ждёт — `sem_wait()`.

```c
sem_t event;
sem_init(&event, 0);

// Прерывание:
void ISR() {
    sem_post(&event);     // «событие случилось»
}

// Поток:
void worker() {
    for (;;) {
        sem_wait(&event); // спим, пока кто-нибудь не сигналит
        handle();
    }
}
```

**Ключевое преимущество над простой «сигнальной переменной»:** счётчик не теряет повторных сигналов. Если событий было три, а поток только начал ждать — он три раза подряд пройдёт `sem_wait` без блокировки.

### Типы семафоров

**Счётный семафор** — счётчик ≥ 0, произвольное значение. Используется для пулов ресурсов (10 коннектов, 5 буферов), для producer-consumer.

**Бинарный семафор** — счётчик ограничен {0, 1}. Простая «лампа»: горит/не горит. Удобен для взаимного исключения и одиночных уведомлений.

> «Бинарный семафор, так же как и обычный, является универсальным примитивом, это означает, что на основе бинарных семафоров можно реализовать обычные и наоборот».

**RW-семафор** (читателей-писателей) — особый случай, см. ниже.

### Семафор vs мьютекс — главное отличие

Мьютекс выглядит как бинарный семафор (есть/занят), но у него есть **владелец**: освободить мьютекс может только тот поток, который его захватил. У семафора владельца нет — `post()` может сделать кто угодно. Поэтому семафор подходит для уведомлений (одно прерывание, другой поток ждёт — разные сущности), а мьютекс — нет.

| Параметр | Семафор | Мьютекс |
|---|---|---|
| Владелец | Нет | Есть |
| Кто может освободить | Любой | Только владелец |
| Для уведомлений | Да | Нет |
| Защита от инверсии приоритетов | Нет | Часто есть |
| Рекурсивный захват | Нет | Можно |

### Политика очереди

Когда `sem_post()` будит ждущих — кого выбирать? Многие RTOS позволяют выбрать политику:
- **По приоритету** — разбудить самого приоритетного. Лучше для жёсткого реального времени, но низкоприоритетные могут голодать.
- **FIFO** — разбудить первого вставшего в очередь. Справедливее и дешевле в реализации.

### RW-семафоры (для чтения и записи)

Бывает, что данные часто читают и редко изменяют (конфигурация, кэш DNS, маршрутные таблицы). Заставлять читателей ждать друг друга бессмысленно — они не конфликтуют. Конфликтует только запись.

**RW-семафор** допускает три состояния:
- Никто не держит.
- Держат **сколько угодно читателей** одновременно.
- Держит **ровно один писатель**, читателей нет.

> «Блокировку по чтению может одновременно удерживать любое количество потоков, при условии, что при этом нет ни одного потока записи. И наоборот, только один поток может удерживать блокировку по записи, если при этом нет ни одного потока, захватившего эту же блокировку по чтению».

```c
down_read(&sem);    // вход для чтения
// читаем
up_read(&sem);

down_write(&sem);   // вход для записи (эксклюзивно)
// пишем
up_write(&sem);
```

**Когда применять.** Только если читателей много, а пишут редко. Если читают и пишут примерно поровну — обычный семафор/мьютекс будет быстрее.

**Особенность Linux:** читатели приоритетнее. Если хоть один читатель удерживает блокировку, новые читатели проходят, даже если уже ждёт писатель. Это даёт лучший throughput на чтении, но писатель может голодать.

**`downgrade_write`** — атомарно превращает write-lock в read-lock. Полезно, когда захватили на запись, что-то изменили и хотим оставить блокировку для чтения, не отпуская и не пуская промежуточного писателя.

## 60. Секвенсоры.
*Источник: rtos.md, Глава 6.6 «Счетчики событий и секвенсоры»; robert_lav.md, Глава 10 (seqlock)*

### Что такое секвенсор

Из rtos.md (6.6):
> «Более интересным объектом является секвенсор. Он содержит единственный метод, который, по сути, сводится к атомарному инкременту».

```c
int32_t ticket(sequencer_t* object);
```

Каждый вызов возвращает **уникальный возрастающий номер** ("талон").

### Применение — электронная очередь

Аналогия — талоны в банке:
1. Каждый клиент берёт талон с номером.
2. Обслуживается тот, чей номер равен текущему.
3. После обслуживания номер увеличивается.

```c
sequencer_t seq;
event_count_t ec;

void access_resource(int my_data) {
    int my_ticket = ticket(&seq);    // взять номер
    await(&ec, my_ticket);            // дождаться своей очереди
    // работа с ресурсом
    advance(&ec);                     // следующий
}
```

### Свойства

Из rtos.md:
> «По сравнению с семафором, секвенсор обеспечивает последовательность, то есть потоки получат доступ к разделяемому ресурсу именно в той последовательности, в которой они его запрашивали».

**Преимущества:**
- **Строгая FIFO** дисциплина.
- **Справедливость.**
- **Возможность оценки** ожидаемого времени:
> «Если потоку есть чем заняться помимо ожидания, он может, определив с помощью своего "талона" и функции read, примерное количество потоков перед ним, заняться другими делами, вернувшись к ожиданию позже».

**Недостатки:**

> «Если по каким-либо причинам, поток, очередь которого подошла, не может выполняться, все остальные потоки будут ждать, даже если они могли бы получить доступ к ресурсу и закончить свою работу».

То есть **один заблокированный держит всю очередь**.

### Счётчики событий (event count)

Парный с секвенсором примитив. Из rtos.md:

```c
int advance(event_count_t* object);
int read(event_count_t* object, int32_t* value);
int await(int32_t value);
```

- **`advance`** — инкремент счётчика, разбудить всех ожидающих этого значения.
- **`read`** — прочитать текущее значение.
- **`await(N)`** — ждать, пока счётчик не достигнет N.

### Применение в спинлоках с очередью

Из rtos.md:
> «Хотя в области прикладного программирования секвенсоры и счётчики событий не получили распространения, аналогичные идеи используются при реализации некоторых разновидностей спинлоков».

**Ticket spinlock** — комбинация секвенсора и счётчика событий:

```c
struct ticket_lock {
    atomic_t next_ticket;     // секвенсор
    atomic_t now_serving;     // счётчик событий
};

void acquire(struct ticket_lock *l) {
    int my = atomic_fetch_add(&l->next_ticket, 1);
    while (atomic_read(&l->now_serving) != my) ;
}

void release(struct ticket_lock *l) {
    atomic_inc(&l->now_serving);
}
```

**Преимущества ticket spinlock:**
- FIFO порядок.
- Нет голодания.
- Простая реализация.

**Недостаток:** все CPU крутятся на одной переменной `now_serving` → cache thrashing. Решение — MCS-locks (по очереди на отдельных переменных).

### Seqlock в Linux

Похожая идея — **seqlock** (`seqlock_t`):

```c
// Источник: robert_lav.md, Глава 10
seqlock_t mr_seq_lock = DEFINE_SEQLOCK(mr_seq_lock);

// Писатель
write_seqlock(&mr_seq_lock);
// модификация
write_sequnlock(&mr_seq_lock);

// Читатель
do {
    seq = read_seqbegin(&mr_seq_lock);
    // чтение
} while (read_seqretry(&mr_seq_lock, seq));
```

Принцип:
- Писатель увеличивает счётчик при входе и выходе из CS.
- Чтение нечётного счётчика → запись идёт, повторить.
- Между чтениями счётчик не должен меняться → данные согласованы.

Преимущества:
- Очень дешёвое **чтение** (без блокировок).
- Хорошо для read-mostly данных (системное время, конфигурация).

Недостатки:
- **Голодание читателей** при частых записях.
- Сложность для сложных структур.

Применение в Linux: `xtime_lock` для системного времени.

### Сравнение

| Примитив | FIFO | Cache thrashing | Применение |
|---|---|---|---|
| Базовый спинлок | Нет | Да | Короткие CS |
| Ticket spinlock | Да | Да | FIFO-критично |
| MCS-lock | Да | Нет | Большие SMP |
| Seqlock | – | Минимально (читатели) | Read-mostly |

### Резюме

Секвенсоры/счётчики событий не получили распространения в прикладном программировании, но идеи живут в реализации синхронизационных примитивов ядра.
