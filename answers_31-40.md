## 31. Обработчик вызова системных функций. Таблица символов. Системные вызовы.
*Источник: robert_lav.md, Глава 5 «Системные функции», разделы «Взаимодействие с ядром», «Обработчик вызова системных функций»*

**Системный вызов (syscall)** — единственная законная точка входа из пользовательского режима в ядро (кроме исключений и прерываний). Пользовательские программы не могут вызывать функции ядра напрямую, потому что код ядра находится в защищённой области памяти.

**Механизм передачи управления:**
1. Пользовательская программа выполняет **программное прерывание** (на x86 — `int 0x80`, или `syscall`/`sysenter`).
2. Процессор переключается в режим ядра.
3. Запускается обработчик исключения с вектором 128 — функция **`system_call()`** (определена на ассемблере в `entry_64.S`).

**Номер системной функции.** Каждой системной функции присвоен уникальный номер. Программа помещает его в регистр **`rax`** (x86-64) или `eax` (x86-32) перед вызовом. Функция `system_call()` считывает этот номер.

**Таблица символов (sys_call_table).** Список всех зарегистрированных системных функций хранится в таблице `sys_call_table`. Каждому номеру соответствует адрес реальной функции. Формат зависит от платформы:

```
// Источник: robert_lav.md, Глава 5
ENTRY(sys_call_table)
    .long sys_restart_syscall    /* 0 */
    .long sys_exit
    .long sys_fork
    .long sys_read
    .long sys_write
    .long sys_open               /* 5 */
    ...
```

Вызов нужной функции из `system_call()`:
```asm
call *sys_call_table(,%rax,8)
```
(размер записи — 8 байт на 64-битах; на x86-32 — 4 байта).

**Передача параметров.** Через регистры процессора: на x86-32 — `ebx`, `ecx`, `edx`, `esi`, `edi` (до 5 параметров). Если параметров больше — в регистр загружается адрес блока параметров. Возвращаемое значение — в `eax`.

**Проверка корректности номера.** `system_call()` проверяет, что номер меньше `NR_syscalls`. Иначе возвращается `-ENOSYS`.

**Имена функций в ядре.** По соглашению — `sys_имя()`. Например, системный вызов `bar()` реализуется как `sys_bar()`.

**Назначение системных вызовов:**
1. Абстракция между оборудованием и приложениями (например, `read()` работает одинаково независимо от типа диска).
2. Безопасность и стабильность системы — посредник между приложением и ресурсами.
3. Реализация многозадачности и виртуальной памяти — единый общий слой.

В Linux для x86 ~335 системных функций. В Windows — более 1000.

## 32. Реализация системных функций. Проверка параметров.
*Источник: robert_lav.md, Глава 5 «Системные функции», раздел «Проверка параметров»*

**Принцип:** «не доверять параметрам из пользовательского пространства». Любая проверка некорректных значений может привести к нарушению стабильности и безопасности всей системы.

**Виды проверок:**

1. **Семантическая корректность.** Например, для вызовов ввода-вывода — корректность файлового дескриптора. Для вызовов управления процессами — допустимость PID.

2. **Проверка прав.** Прикладные программы не должны получать доступ к ресурсам, к которым у них нет прав. Раньше использовалась `suser()` (проверка root). Сейчас — система «возможностей» (capabilities) через `capable(CAP_*)`:

```c
// Источник: robert_lav.md, Глава 5, sys_reboot
if (!capable(CAP_SYS_BOOT))
    return -EPERM;
```

3. **Проверка указателей.** Самая важная. Указатель из пользователя должен:
   - Указывать на область памяти, относящуюся к пространству пользователя (нельзя позволить ядру читать свои данные).
   - Указывать на память, принадлежащую адресному пространству процесса (нельзя позволить читать чужие процессы).
   - Не нарушать прав доступа (чтение, запись, выполнение).

Из robert_lav.md:
> «Представьте себе, что произойдёт, если в ядро беспрепятственно будет передан указатель на область памяти, к которой прикладная программа не имеет доступа по чтению! В результате обманным путём она может заставить ядро скопировать данные из этого участка памяти, который принадлежит другому процессу».

Для проверок предусмотрены **`copy_to_user()`** и **`copy_from_user()`** (см. вопрос 33).

**Пример sys_reboot()** иллюстрирует все три типа проверок:
- Проверка прав: `capable(CAP_SYS_BOOT)`.
- Проверка «магических» значений: `magic1`, `magic2`.
- Проверка команды: только разрешённые `cmd`.
- Безопасное копирование строки из пользователя: `strncpy_from_user()`.

**Возврат при ошибке.** Стандартно — отрицательное значение (`-EFAULT`, `-EPERM`, `-EINVAL` и т.д.). Библиотека C преобразует это в установку `errno` и возврат -1 пользователю.

## 33. Функции copy_from_user() и copy_to_user()
*Источник: robert_lav.md, Глава 5 «Системные функции», раздел «Проверка параметров»*

**Назначение.** Безопасно копировать данные между пространством ядра и пространством пользователя с автоматической проверкой указателей.

**`copy_to_user(to, from, n)`** — копирует `n` байт из ядра в пользовательское пространство.
- `to` — указатель в пользователе.
- `from` — указатель в ядре.
- `n` — длина в байтах.

**`copy_from_user(to, from, n)`** — копирует `n` байт из пользователя в ядро.
- `to` — указатель в ядре.
- `from` — указатель в пользователе.
- `n` — длина в байтах.

**Возврат:** при ошибке — число байт, которые **не удалось** скопировать. При успехе — 0.

**Пример (silly_copy из robert_lav.md):**

```c
// Источник: robert_lav.md, Глава 5
SYSCALL_DEFINE3(silly_copy,
    unsigned long *, src,
    unsigned long *, dst,
    unsigned long len)
{
    unsigned long buf;

    if (copy_from_user(&buf, src, len))
        return -EFAULT;

    if (copy_to_user(dst, &buf, len))
        return -EFAULT;

    return len;
}
```

**Особенности:**

1. **Могут блокировать процесс.** Если страница пользователя выгружена на диск, обращение вызовет page fault → процесс ждёт загрузки. Поэтому эти функции **нельзя** использовать в контексте прерывания.

2. **Внутри проверяют:**
   - Что адрес `from`/`to` в пользователе находится в допустимом диапазоне.
   - Что страницы доступны (или могут быть загружены).

3. **При ошибке** (например, попытка чтения из недопустимого адреса) возвращают ненулевое значение. Системная функция должна вернуть `-EFAULT`.

**Прочие функции:**
- `get_user(x, ptr)` / `put_user(x, ptr)` — для одного значения (быстрее).
- `strncpy_from_user(dst, src, n)` — для строк.
- `clear_user(ptr, n)` — обнуление в пользователе.

## 34. Контекст системной функции. system_call(). Завершающие этапы регистрации системной функции.
*Источник: robert_lav.md, Глава 5 «Системные функции», разделы «Контекст системной функции», «Завершающие этапы регистрации»*

### Контекст системной функции

При выполнении системной функции **ядро работает в контексте процесса**. В переменной `current` хранится указатель на дескриптор задачи, вызвавшей функцию.

В контексте процесса ядро:
- **Может переходить в состояние ожидания** (вызывать `schedule()`, блокироваться на семафорах). Это упрощает программирование — можно использовать большую часть возможностей ядра.
- **Является полностью вытесняемым** — другой процесс может вытеснить текущий, даже если он в ядре.

Из robert_lav.md:
> «Поскольку в новой задаче также может выполняться та же самая системная функция, необходимо тщательно проверить, чтобы системные функции были реентерабельными (повторновходимыми)».

Это означает, что системные функции должны работать корректно при одновременных вызовах из разных потоков. Требуется синхронизация через спин-блокировки, семафоры, мьютексы.

### Функция system_call()

Точка входа в обработку системного вызова. Реализована на ассемблере в `entry_64.S`. Действия:
1. Сохраняет регистры в стек ядра.
2. Проверяет корректность номера: если `номер >= NR_syscalls` → `-ENOSYS`.
3. Вызывает функцию по таблице: `call *sys_call_table(,%rax,8)`.
4. Восстанавливает регистры.
5. Возвращает управление в пользовательский режим через `iret` / `sysret`.

После возврата из системной функции управление вновь попадает в `system_call()`, которая переключается в пространство пользователя.

### Завершающие этапы регистрации

Чтобы добавить новую системную функцию в ядро:

**1. Добавить функцию в `sys_call_table`** (в `entry.S` для x86):
```
.long sys_foo
```
Положение в таблице = номер функции.

**2. Определить номер в `<asm/unistd.h>`:**
```c
#define __NR_foo 338
```

**3. Скомпилировать функцию** в составе образа ядра (обычно в `kernel/sys.c`).

**Особенности:**
- После назначения номера его **нельзя менять** (нарушит существующие программы).
- При удалении функции номер не переиспользуется — используется заглушка `sys_ni_syscall()`, возвращающая `-ENOSYS`.
- Номера могут различаться между архитектурами — это часть **ABI** платформы.

Из robert_lav.md:
> «В системе Linux предусмотрена так называемая "не реализованная" функция `sys_ni_syscall()`, которая просто возвращает значение `-ENOSYS` — код ошибки, соответствующий некорректной системной функции. Эта функция служит для "затыкания дыр" в случае такого редкого события, как удаление системной функции».

## 35. Доступ к системным функциям из пользовательских приложений.
*Источник: robert_lav.md, Глава 5 «Системные функции», раздел «API, POSIX и библиотека C»*

Пользовательские приложения **не вызывают системные функции напрямую**. Они используют API (Application Programming Interface), который реализован в библиотеке C (glibc).

**Цепочка вызовов:**
```
Приложение     →    Библиотека C    →    Ядро
printf()       →    printf()        →    write()
                    write() wrapper      sys_write()
```

Из robert_lav.md (рис. 5.1): вызов `printf()` → функция `printf()` в библиотеке C → функция `write()` в библиотеке C → системная функция `write()`.

**Этапы вызова системной функции:**

1. Приложение вызывает функцию из библиотеки C (например, `write(fd, buf, n)`).
2. Библиотека C — обёртка — помещает параметры в нужные регистры:
   - Номер syscall в `rax`/`eax`.
   - Параметры в `ebx`, `ecx`, `edx`, `esi`, `edi`.
3. Выполняется инструкция перехода в ядро: `int 0x80` (старый способ), `syscall`, или `sysenter`.
4. Процессор переключается в режим ядра, запускает `system_call()`.
5. `system_call()` по номеру находит `sys_write()` и вызывает её.
6. Функция выполняется, возвращает значение в `eax`.
7. Возврат в пользователя.
8. Обёртка библиотеки проверяет значение. Если ошибка — устанавливает `errno`, возвращает -1.

**Стандарты:**
- **POSIX** — набор стандартов IEEE для переносимости Unix-подобных ОС.
- **SUSv3** — Single UNIX Specification версии 3.

Linux стремится к соответствию POSIX и SUSv3.

**Зачем библиотека C посередине:**
- **Переносимость:** на разных платформах разные интерфейсы системных вызовов, но API библиотеки C везде одинаков.
- **Удобство:** `printf()` форматирует строку перед вызовом `write()`. Приложение не работает с регистрами.
- **Оптимизация:** часть функций (`getpid()`, `gettimeofday()`) может выполняться без системного вызова через vDSO.
- **Безопасность:** библиотека C может проверить аргументы до вызова ядра.

**Реализация в пользовательском пространстве:**
- Для редко используемых syscall'ов можно вызывать `syscall(номер, аргументы...)` напрямую.
- Для частых — есть библиотечные обёртки.

## 36. Реализация доступа к системным объектам. Количество обращений. Квоты. Особенности хранения указателей.
*Источник: robert_lav.md, Глава 5 «Системные функции»; общие принципы Linux*

**Системные объекты** — внутренние структуры ядра, доступные через системные вызовы: файловые дескрипторы, сокеты, разделяемые сегменты памяти, мьютексы, семафоры, очереди сообщений.

**Принцип:** пользователь не может работать с указателями на объекты ядра напрямую — они находятся в защищённой памяти.

**Решение — идентификаторы (handles).**
- Пользователь получает целочисленный **идентификатор** (например, `int fd = open(...)`).
- Ядро хранит таблицу: id → указатель на структуру в ядре.
- Все операции пользователя — через id; ядро по id находит указатель.

**Пример: файловые дескрипторы.**
- Каждый процесс имеет таблицу открытых файлов в `task_struct->files`.
- Дескриптор — индекс в этой таблице.
- `read(5, ...)` — ядро берёт `files->fd[5]` → получает указатель на `struct file`.

**Количество обращений (reference counting).** Чтобы объект не был удалён, пока кто-то его использует:
- В структуре объекта есть счётчик `count`.
- `open()` увеличивает счётчик.
- `close()` уменьшает.
- При `count == 0` объект удаляется.
- Например, при `fork()` дочерний процесс наследует файлы → `count++`.

```c
struct file {
    atomic_t f_count;       // счётчик
    // ...
};
```

**Квоты.** Ограничение ресурсов на пользователя или процесс:
- `RLIMIT_NOFILE` — максимум открытых файлов.
- `RLIMIT_STACK` — размер стека.
- `RLIMIT_AS` — размер виртуального АП.
- `RLIMIT_NPROC` — число процессов пользователя.

Проверяются при выделении ресурсов. Возврат `-EMFILE`, `-EAGAIN` и т.д. при превышении.

**Особенности хранения указателей.**

1. **Указатели ядра нельзя передавать пользователю.** Это раскроет адреса структур ядра — угроза безопасности.

2. **Указатели пользователя нельзя использовать в ядре без проверки.** Их обрабатывают `copy_from_user`/`copy_to_user`.

3. **Объекты «привязаны» к процессу.** Файловый дескриптор 5 в одном процессе и в другом — разные объекты.

4. **При `exec()` некоторые объекты сбрасываются.** Например, файлы с флагом `FD_CLOEXEC` закрываются. Память — полностью заменяется.

5. **Объекты переживают процесс при `fork()`.** Открытые файлы наследуются (через увеличение счётчика).

## 37. Завершение процесса при обработке системной функции. Блокирующие вызовы из режима пользователя.
*Источник: robert_lav.md, Глава 3 «Управление процессами», раздел «Завершение процесса»; rtos.md, Глава 7.7.8 «Блокирующие вызовы из режима пользователя»*

### Завершение процесса

Процесс может завершиться:
- **Добровольно:** через `exit()` или возврат из `main()` (компилятор добавляет вызов `exit()`).
- **Принудительно:** при получении необрабатываемого сигнала или исключения.

Основная работа — в **`do_exit()`** (`kernel/exit.c`):
1. Устанавливается флаг `PF_EXITING` в `task_struct->flags`.
2. Удаляются все таймеры ядра (`del_timer_sync()`).
3. Запись информации об учёте ресурсов.
4. **`exit_mm()`** — освобождение `mm_struct` (адресного пространства). Если другие процессы не используют — уничтожается.
5. **`exit_sem()`** — выход из очередей ожидания семафоров IPC.
6. **`exit_files()`** и **`exit_fs()`** — уменьшают счётчики ссылок на файлы и информацию о ФС.
7. Уведомление родителя сигналом `SIGCHLD`.
8. Перевод в **состояние zombie** — ресурсы освобождены, но дескриптор сохраняется.
9. После вызова родителем `wait()`/`waitpid()` дескриптор окончательно удаляется.

### Завершение в середине системного вызова

Если процесс получил сигнал на завершение во время выполнения системной функции:
- Если он находится в **`TASK_INTERRUPTIBLE`** — сигнал разбудит его, вернёт `-EINTR`.
- Если в **`TASK_UNINTERRUPTIBLE`** — сигнал не получит, должен дождаться завершения операции.
- После возврата из ядра поток обработает сигнал.

### Блокирующие вызовы из режима пользователя

Многие системные вызовы могут заблокировать процесс надолго:
- `read()` ждёт данных.
- `wait()` ждёт завершения дочернего процесса.
- `select()`/`poll()` ждут готовности дескрипторов.
- `sem_wait()` ждёт семафор.

**Реализация в ядре:** процесс помещается в очередь ожидания, состояние меняется на `TASK_INTERRUPTIBLE`/`UNINTERRUPTIBLE`, вызывается `schedule()`.

Из rtos.md (7.7.8):
> Блокирующие вызовы должны быть прерываемыми — при поступлении сигнала или запросе на завершение поток должен корректно выйти из ожидания.

**Особенности:**
- Если процесс ждёт в `TASK_UNINTERRUPTIBLE` (например, физический I/O) — его нельзя убить даже `SIGKILL`. Видно в `ps` как статус `D`.
- Большинство ожиданий — `TASK_INTERRUPTIBLE`, что позволяет завершить процесс.

**Тонкость с прерываниями сигналом.** Если `read()` прервался сигналом и вернул `-EINTR`, приложение должно повторить вызов:
```c
while ((n = read(fd, buf, size)) == -1 && errno == EINTR)
    ;
```

В современных Unix есть флаг `SA_RESTART` — заставляет ядро автоматически перезапустить прерванный вызов.

## 38. Мультипрограммный режим. Процессы, ориентированные на ввод/вывод и на вычисления. Приоритет процессов. Квант времени.
*Источник: tanenbaum.md, Глава 2.4 «Планирование»*

### Мультипрограммный режим

**Мультипрограммный режим** — одновременное выполнение нескольких программ путём разделения времени CPU. Без него процессор простаивает, пока программа ждёт ввода-вывода. С ним, пока одна программа ждёт, другая работает.

Эффект: общая загрузка процессора растёт. При смешанной нагрузке (часть процессов жмут CPU, часть — диск) загрузка может приближаться к 100%.

### Процессы CPU-bound и I/O-bound

Из tanenbaum.md (2.4.1):
> «Некоторые процессы… проводят основную часть своего времени за вычислениями, а другие… основную часть своего времени ожидают завершения операций ввода-вывода. Первые процессы называются процессами, ограниченными скоростью вычислений, а вторые — процессами, ограниченными скоростью работы устройств ввода-вывода».

**CPU-bound (ограниченные скоростью вычислений):**
- Длинные периоды активности CPU.
- Нечастые периоды ожидания I/O.
- Примеры: научные расчёты, рендеринг, сжатие, шифрование.

**I/O-bound (ограниченные скоростью I/O):**
- Короткие периоды активности CPU.
- Частые периоды ожидания I/O.
- Примеры: текстовый редактор, веб-сервер, копирование файлов.

**Ключевой фактор** — продолжительность пика активности CPU между ожиданиями, а не суммарное время I/O.

С ускорением CPU всё больше процессов становятся I/O-bound — диски развиваются медленнее.

### Приоритет процессов

Назначение разной важности разным процессам. Из tanenbaum.md (2.4.3):
> «Основная идея проста: каждому процессу присваивается значение приоритетности и запускается тот процесс, который находится в состоянии готовности и имеет наивысший приоритет».

В Linux:
- **`nice`** — от -20 (наивысший) до 19 (низший). По умолчанию 0.
- **Real-time приоритеты** — 1..99 (выше любых обычных).
- **Static priority** — задаётся вручную.
- **Dynamic priority** — корректируется ядром по поведению процесса (I/O-bound получают повышение).

**Старение (aging).** Чтобы низкоприоритетные не голодали, их приоритет со временем повышается.

### Квант времени

**Квант** — отрезок времени, на который процесс получает CPU при round-robin планировании. По истечении кванта — вытеснение.

Выбор кванта:
- **Слишком короткий** — много переключений, накладные расходы (Tanenbaum: при кванте 4 мс и переключении 1 мс — 20% времени теряется).
- **Слишком длинный** — плохая отзывчивость (50 процессов × 100 мс = 5 с ожидания для последнего).
- **Разумный** — 20–50 мс.

В Linux квант динамический, зависит от приоритета:
- Высокий приоритет → длинный квант.
- Низкий приоритет → короткий квант.

В CFS (Completely Fair Scheduler) квант рассчитывается как доля от «целевой задержки» (обычно 6–20 мс).

## 39. IPC. Shared Memory, сигналы.
*Источник: tanenbaum.md, Глава 2.3 «Взаимодействие процессов»; rtos.md, Глава 3.3.3, 5.7*

### IPC — Interprocess Communication

Из tanenbaum.md (2.3):
> «Довольно часто процессам необходимо взаимодействовать с другими процессами. Например, в канале оболочки выходные данные одного процесса могут передаваться другому процессу».

Поскольку процессы имеют **изолированные адресные пространства**, для обмена данными нужны специальные механизмы. Это и есть **межпроцессное взаимодействие (IPC)**.

**Три основные задачи IPC:**
1. Передача информации между процессами.
2. Обеспечение совместной работы без помех (синхронизация).
3. Определение правильной последовательности при взаимозависимостях.

### Виды IPC в Unix/Linux

- **Pipes (каналы)** — однонаправленный поток между родительским и дочерним процессами.
- **Named pipes (FIFO)** — каналы с именем в файловой системе.
- **Shared memory** — общая область памяти.
- **Message queues** — очереди сообщений.
- **Semaphores** — семафоры для синхронизации.
- **Sockets** — сетевые сокеты (в том числе локальные Unix-сокеты).
- **Signals** — асинхронные уведомления.

### Shared Memory

**Разделяемая память (shared memory)** — самая быстрая форма IPC. Один и тот же физический регион памяти отображается в виртуальные АП нескольких процессов.

**Преимущество:** после установки нет накладных расходов на копирование данных через ядро.

**Недостаток:** требует ручной синхронизации — параллельный доступ может вызвать race condition.

**Реализация в Linux:**

**POSIX shared memory:**
```c
int fd = shm_open("/myshm", O_CREAT | O_RDWR, 0666);
ftruncate(fd, 4096);
void *ptr = mmap(NULL, 4096, PROT_READ | PROT_WRITE, MAP_SHARED, fd, 0);
```

**System V shared memory:**
```c
int shmid = shmget(KEY, 4096, IPC_CREAT | 0666);
void *ptr = shmat(shmid, NULL, 0);
```

Внутри ядра — общие физические страницы, отображённые в разные процессы через таблицы страниц.

**Применение:**
- Высокопроизводительные серверы баз данных.
- Кэши, разделяемые между процессами.
- IPC в gaming (например, между движком и рендером).

### Сигналы

**Сигнал** — асинхронное уведомление процесса. Аналог программного прерывания, направленного на процесс.

Из tanenbaum.md:
> «Системный вызов `kill` позволяет пользователям и пользовательским процессам посылать сигналы».

**Использование:**
- Сообщить процессу о событии (`SIGUSR1`, `SIGUSR2`).
- Запросить завершение (`SIGTERM`, `SIGKILL`).
- Уведомить об ошибке (`SIGSEGV`, `SIGFPE`).
- Управление потоком (`SIGSTOP`, `SIGCONT`).

**Отправка:** `kill(pid, signal)`.

**Обработка:** процесс регистрирует обработчик через `signal()` или `sigaction()`. По умолчанию — реакция системы (часто завершение).

**Особые сигналы:**
- **`SIGKILL`** и **`SIGSTOP`** — нельзя перехватить, замаскировать или игнорировать.
- **`SIGCHLD`** — посылается родителю при завершении дочернего.
- **`SIGSEGV`** — при нарушении доступа к памяти.

**Реализация (rtos.md, 5.7).** В простейшем случае — битовая маска ожидающих сигналов. В моменты выхода из системных вызовов и обработчиков прерываний маска анализируется, и при наличии сигнала — модифицируется фрейм возврата так, чтобы поток выполнил обработчик сигнала.

**Очереди сигналов.** Битовая маска теряет повторные сигналы. В современных POSIX сигналы хранятся в очереди — `sigqueue()`.

**Сигналы и многопоточность.** В многопоточном процессе сигнал может быть направлен одному потоку (через `pthread_kill`) или всему процессу. Каждый поток имеет свою маску.

## 40. Синхронизация через разделяемые объекты. Асинхронный обмен сообщений.
*Источник: rtos.md, Глава 6.3 «События», Глава 6.4 «Обмен сообщениями»; общие принципы*

### Синхронизация через разделяемые объекты

В отличие от **прямого уведомления** (отправитель явно указывает, кому шлёт), при синхронизации через **разделяемые объекты** есть отдельная сущность — объект синхронизации, с которой работают и отправитель, и получатель. Они не знают друг о друге, знают только об объекте.

**Примеры разделяемых объектов:**
- **Семафоры** — счётчик ресурсов.
- **Мьютексы** — взаимное исключение.
- **События** (event flags) — флаги, на которые ждут.
- **Условные переменные.**
- **Очереди сообщений.**

**Преимущества:**
- Слабая связанность отправителя и получателя.
- Можно иметь несколько отправителей и получателей.
- Объекты могут существовать в системе независимо от потоков.

### Асинхронный обмен сообщений

Отправитель и получатель **не синхронизированы**: отправитель посылает сообщение и продолжает работу, не дожидаясь, пока получатель его прочитает. Сообщение хранится в **буфере** (очереди).

**Структура:**

```
Отправитель → [сообщение]    Очередь   [сообщение] → Получатель
              ──→ ←────  | M1 | M2 | M3 | ←──── ←──
```

**Свойства:**
- Очередь имеет конечный размер.
- Если очередь полна — отправитель либо ждёт, либо получает ошибку, либо вытесняет старое сообщение.
- Если очередь пуста — получатель ждёт нового сообщения.

**Преимущества:**
- Отправитель не блокируется (если очередь не полна).
- Сглаживает пики нагрузки — буфер поглощает всплески.
- Отправитель и получатель могут работать с разной скоростью.

**Недостатки:**
- Требуется память на очередь.
- Задержка: сообщение проводит время в очереди.
- При полной очереди приходится отбрасывать сообщения или блокировать отправителя.

### Реализация очереди сообщений

```c
typedef struct {
    void *buffer[N];      // массив указателей на сообщения
    int head, tail;       // индексы
    int count;            // число элементов
    semaphore_t empty;    // семафор «есть место»
    semaphore_t full;     // семафор «есть сообщение»
    mutex_t lock;         // защита от параллельного доступа
} msg_queue_t;

void send(msg_queue_t *q, void *msg) {
    sem_wait(&q->empty);   // ждать, пока появится место
    mutex_lock(&q->lock);
    q->buffer[q->head] = msg;
    q->head = (q->head + 1) % N;
    q->count++;
    mutex_unlock(&q->lock);
    sem_post(&q->full);    // уведомить, что есть сообщение
}

void *recv(msg_queue_t *q) {
    sem_wait(&q->full);    // ждать сообщение
    mutex_lock(&q->lock);
    void *msg = q->buffer[q->tail];
    q->tail = (q->tail + 1) % N;
    q->count--;
    mutex_unlock(&q->lock);
    sem_post(&q->empty);   // освободить место
    return msg;
}
```

### Применение асинхронного обмена

- **Веб-серверы:** входящие запросы попадают в очередь, обрабатываются рабочими потоками.
- **Логирование:** потоки кладут записи в очередь, фоновый поток пишет их в файл.
- **Игровые движки:** события (нажатия, сообщения сети) в очереди, основной цикл их обрабатывает.
- **Микросервисы:** RabbitMQ, Kafka — асинхронные очереди между сервисами.
- **GUI:** очередь событий пользователя.

### Сравнение с синхронным обменом

| Параметр | Асинхронный | Синхронный (рандеву) |
|---|---|---|
| Буфер | Есть | Нет |
| Блокировка отправителя | Только при полной очереди | Всегда до приёма |
| Связанность | Слабая | Сильная |
| Использование памяти | Больше | Меньше |
| Поведение под нагрузкой | Сглаживание | Естественное ограничение |
