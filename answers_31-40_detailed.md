## 31. Обработчик вызова системных функций. Таблица символов. Системные вызовы.
*Источник: robert_lav.md, Глава 5 «Системные функции», разделы «Взаимодействие с ядром», «Обработчик вызова системных функций»*

### Зачем нужны системные вызовы

Представим ситуацию: пользовательская программа хочет прочитать файл. Чтение файла требует:
- Обращения к контроллеру диска через привилегированные инструкции ввода-вывода.
- Доступа к таблицам файловой системы в памяти ядра.
- Возможной активации DMA, прерываний.

Всё это **запрещено** в режиме пользователя. Если бы пользовательский код мог обращаться к коду ядра напрямую (например, `call kernel_function`), он мог бы перейти в любое место памяти ядра, прочитать любые данные, обойти проверки доступа. Безопасности и стабильности системы не было бы.

Из robert_lav.md:
> «В операционной системе Linux вызовы системных функций являются единственным средством, благодаря которому пользовательские программы могут связываться с ядром; они являются единственной законной точкой входа в ядро, помимо исключений и прерываний».

Системный вызов — это **единственная контролируемая** точка входа в ядро из пользователя.

### Три цели системных вызовов

Из robert_lav.md (Глава 5):

**1. Абстракция между оборудованием и пользовательскими приложениями.** Например, чтение из файла одинаково работает на любом диске или с любой файловой системой.

**2. Безопасность и стабильность.** Ядро решает, разрешён ли доступ к ресурсу, по правам пользователя.

**3. Виртуализация процессов.** Единый общий слой позволяет реализовать многозадачность и виртуальную память. Без этого слоя было бы невозможно изолировать процессы.

### Механизм системного вызова

Пользовательская программа **не может прямо вызвать функцию ядра**. Нужен механизм, который:
- Атомарно переключит процессор в режим ядра.
- Передаст управление в заранее зарегистрированный обработчик.
- Сохранит контекст пользователя.
- Передаст параметры вызова.

Решение — **программное прерывание**. Программа выполняет специальную инструкцию, процессор возбуждает исключительную ситуацию, переключается в режим ядра, запускает обработчик исключения.

### Конкретные инструкции

**На x86 — `int 0x80`.** Прерывание номер 128, специально зарезервированное для системных вызовов в Linux. Обработчик в таблице прерываний с этим вектором — функция **`system_call()`** (на ассемблере, в `entry_64.S`).

```asm
; Пример: чтение из файла на x86-32
mov  eax, 3          ; номер sys_read
mov  ebx, fd
mov  ecx, buffer
mov  edx, size
int  0x80            ; программное прерывание → ядро
; eax содержит результат после возврата
```

**На x86 — `sysenter` (32-битный) и `syscall` (64-битный).** Более быстрые инструкции, разработанные специально для системных вызовов:
- Не используют таблицу прерываний.
- Адрес обработчика — в специальном MSR-регистре.
- Меньше сохраняемого состояния → быстрее.

Из robert_lav.md:
> «В современных процессорах x86 была введена специальная команда, называемая `sysenter`. Она позволяет ускорить процесс перехода в привилегированный режим и, как следствие, вызов системной функции, причём использует для этого более подходящий способ, а не команду вызова прерывания `int`».

**На ARM — `SVC` (Supervisor Call).** Аналог в ARM-архитектуре.

### Функция system_call()

После переключения в режим ядра управление получает функция `system_call()`. Её задача — определить, **какую** системную функцию вызвал пользователь, и передать ей управление.

Действия:
1. Сохранить регистры пользователя.
2. Прочитать номер системной функции из `rax`/`eax`.
3. Проверить, что номер корректный (< `NR_syscalls`).
4. Найти адрес функции в таблице.
5. Вызвать функцию с параметрами.
6. После возврата — восстановить регистры, вернуться в пользовательский режим.

### Таблица системных вызовов (sys_call_table)

Список всех зарегистрированных системных функций хранится в массиве **`sys_call_table`**. Каждой записи соответствует один номер системного вызова; значение — адрес функции-обработчика.

```c
// Источник: robert_lav.md, Глава 5
ENTRY(sys_call_table)
    .long sys_restart_syscall    /* 0 */
    .long sys_exit
    .long sys_fork
    .long sys_read
    .long sys_write
    .long sys_open               /* 5 */
    ...
    .long sys_eventfd2
    .long sys_epoll_create1
    .long sys_dup3               /* 330 */
    .long sys_pipe2
    .long sys_inotify_init1
    ...
```

Размер записи — на 32-битных архитектурах 4 байта (`.long`), на 64-битных — 8 байт.

Вызов функции по таблице:
```asm
call *sys_call_table(,%rax,8)
```

Это «callq [sys_call_table + rax * 8]» — взять значение по адресу таблица + номер*8, перейти на него.

### Номера системных вызовов

В Linux на x86 ~335 системных функций. Каждой назначен уникальный номер. Из robert_lav.md:
> «Номер системной функции является важным атрибутом. После того как он будет назначен системной функции, его уже нельзя изменять, поскольку это нарушит работу уже скомпилированных прикладных программ».

**Правила:**
- Номер назначается один раз и не меняется.
- При удалении функции номер не переиспользуется.
- Вместо удалённой функции в таблицу помещается **`sys_ni_syscall()`** — заглушка, возвращающая `-ENOSYS`.

Номера определены в `<asm/unistd.h>`:
```c
#define __NR_restart_syscall 0
#define __NR_exit            1
#define __NR_fork            2
#define __NR_read            3
#define __NR_write           4
#define __NR_open            5
...
```

### Передача параметров

После номера системной функции нужно передать её параметры. Параметры помещаются в **регистры процессора** перед вызовом `int 0x80`:

- **x86-32:** `eax` (номер), `ebx`, `ecx`, `edx`, `esi`, `edi`, `ebp` (до 6 параметров).
- **x86-64:** `rax` (номер), `rdi`, `rsi`, `rdx`, `r10`, `r8`, `r9` (до 6 параметров).

Если параметров больше 6 — в один регистр помещается указатель на блок параметров.

Возвращаемое значение — в `eax` (на x86-32) или `rax` (на x86-64). Знаковое: отрицательное значение — код ошибки (`-EFAULT`, `-EINVAL` и т.д.), неотрицательное — успех.

### Имена функций в ядре

По соглашению все обработчики системных вызовов в ядре имеют префикс `sys_`:
- `read()` → `sys_read()`
- `write()` → `sys_write()`
- `open()` → `sys_open()`

Они объявляются с модификатором **`asmlinkage`**, говорящим компилятору, что аргументы — в стеке (а не в регистрах, как обычно вызывают C-функции):

```c
asmlinkage long sys_getpid(void);
```

Все системные функции возвращают `long` (для совместимости 32- и 64-битных систем).

В современном ядре используется макрос `SYSCALL_DEFINEN`:

```c
// Источник: robert_lav.md, Глава 5
SYSCALL_DEFINE0(getpid)
{
    return task_tgid_vnr(current);
}
```

Это разворачивается в:
```c
asmlinkage long sys_getpid(void) {
    return task_tgid_vnr(current);
}
```

### Полная картина вызова read()

1. Приложение вызывает `read(fd, buf, n)`.
2. Это вызов библиотеки C, в которой определена функция-обёртка `read()`.
3. Обёртка:
   ```asm
   mov rax, 0       ; __NR_read (на x86-64)
   mov rdi, fd
   mov rsi, buf
   mov rdx, n
   syscall          ; → переключение в ядро
   ```
4. Процессор переключается в режим ядра, выполняется `system_call()`.
5. Из `rax` берётся 0 → `sys_call_table[0]` = `sys_read`.
6. Вызывается `sys_read(fd, buf, n)`.
7. `sys_read` находит `struct file *` по `fd`, вызывает метод `f_op->read`, который физически читает данные.
8. Возврат значения в `rax`.
9. Возврат из ядра через `sysret`.
10. Библиотека C проверяет: если `rax < 0` → `errno = -rax`, return -1; иначе return `rax`.

### Резюме

- Системный вызов — единственная контролируемая точка входа в ядро из пользователя.
- Механизм: программное прерывание (`int 0x80`) или специальная инструкция (`syscall`).
- `system_call()` — функция-обработчик; читает номер из регистра, ищет в `sys_call_table`, вызывает функцию.
- Таблица `sys_call_table` сопоставляет номер с адресом функции.
- Параметры — в регистрах; возврат — тоже в регистре.
- В ядре имена функций — `sys_*`.

## 32. Реализация системных функций. Проверка параметров.
*Источник: robert_lav.md, Глава 5 «Системные функции», раздел «Проверка параметров»*

### Главный принцип

Ядро **никогда не доверяет** данным, поступающим из пользователя. Параметр от пользователя — это входные данные с «потенциально враждебной» стороны.

Из robert_lav.md:
> «Поскольку системные функции выполняются в пространстве ядра, передача им некорректных значений может нарушить стабильность и безопасность работы системы в целом».

Поэтому все параметры проходят строгую проверку.

### Категории проверок

**1. Семантическая корректность параметров.**

Каждый параметр должен иметь смысл:
- В вызовах ввода-вывода — корректный файловый дескриптор (должен быть открыт).
- В вызовах управления процессами — допустимый PID (процесс существует).
- В вызовах работы с памятью — корректный размер, выравнивание.

Пример:
```c
SYSCALL_DEFINE3(read, unsigned int, fd, char __user *, buf, size_t, count)
{
    struct file *f = fget(fd);
    if (!f)
        return -EBADF;        // нет такого файла
    if (!(f->f_mode & FMODE_READ))
        return -EBADF;        // файл не открыт для чтения
    ...
}
```

**2. Проверка прав доступа.**

Прикладные программы не должны получать доступ к ресурсам, к которым у них нет прав.

В старых версиях Linux использовалась функция `suser()` — проверка, является ли вызывающий пользователь root. Теперь — система **возможностей (capabilities)**:

```c
// Источник: robert_lav.md, Глава 5, фрагмент sys_reboot
SYSCALL_DEFINE4(reboot, ...)
{
    if (!capable(CAP_SYS_BOOT))
        return -EPERM;
    ...
}
```

`capable(флаг)` возвращает true, если у вызывающего есть указанная привилегия. Привилегии разделены тонко:
- `CAP_SYS_BOOT` — перезагрузка.
- `CAP_NET_ADMIN` — управление сетью.
- `CAP_SYS_NICE` — изменение приоритетов.
- `CAP_DAC_OVERRIDE` — обход проверок доступа к файлам.
- и десятки других.

Это позволяет дать программе только нужные права, не давая полного root. Например, веб-сервер может иметь `CAP_NET_BIND_SERVICE` (привязка к привилегированным портам), но не иметь права читать чужие файлы.

**3. Проверка указателей** — самая критическая.

Из robert_lav.md:
> «Самой важной является проверка корректности указателей, переданных прикладной программой. Представьте себе, что произойдёт, если в ядро беспрепятственно будет передан указатель на область памяти, к которой прикладная программа не имеет доступа по чтению! В результате обманным путём она может заставить ядро скопировать данные из этого участка памяти, который принадлежит другому процессу или вовсе не подлежит чтению».

Прежде чем читать или писать по указателю от пользователя, ядро **должно проверить**:

- Указатель в **пользовательской** части адресного пространства (не в ядерной).
- Память принадлежит **этому** процессу (а не другому).
- Соответствие правам доступа (чтение для `copy_from_user`, запись для `copy_to_user`).

Без этих проверок пользователь мог бы:
- Передать указатель на память ядра в `read()` → ядро прочитало бы свои внутренние данные и отдало пользователю.
- Передать указатель на чужой процесс — то же самое.
- Передать указатель на read-only страницу в `write()` и заставить ядро её затереть.

Для этого предусмотрены **`copy_to_user()`** и **`copy_from_user()`** (вопрос 33).

### Пример полной проверки — sys_reboot()

Из robert_lav.md:

```c
SYSCALL_DEFINE4(reboot,
    int, magic1,
    int, magic2,
    unsigned int, cmd,
    void __user *, arg)
{
    char buffer[256];

    // 1. Проверка прав
    if (!capable(CAP_SYS_BOOT))
        return -EPERM;

    // 2. Проверка магических чисел — защита от случайного вызова
    if (magic1 != LINUX_REBOOT_MAGIC1 ||
        (magic2 != LINUX_REBOOT_MAGIC2 &&
         magic2 != LINUX_REBOOT_MAGIC2A &&
         magic2 != LINUX_REBOOT_MAGIC2B &&
         magic2 != LINUX_REBOOT_MAGIC2C))
        return -EINVAL;

    // 3. Проверка команды и выполнение
    if ((cmd == LINUX_REBOOT_CMD_POWER_OFF) && !pm_power_off)
        cmd = LINUX_REBOOT_CMD_HALT;

    lock_kernel();
    switch (cmd) {
        case LINUX_REBOOT_CMD_RESTART:
            kernel_restart(NULL);
            break;
        ...
        case LINUX_REBOOT_CMD_RESTART2:
            // 4. Безопасное копирование строки от пользователя
            if (strncpy_from_user(&buffer[0], arg, sizeof(buffer) - 1) < 0) {
                unlock_kernel();
                return -EFAULT;
            }
            buffer[sizeof(buffer) - 1] = '\0';
            kernel_restart(buffer);
            break;
        default:
            unlock_kernel();
            return -EINVAL;
    }
    unlock_kernel();
    return 0;
}
```

Видим:
- Проверка прав через `capable()`.
- Проверка магических чисел (исторически — для защиты от случайного нажатия).
- Проверка команды.
- Безопасное копирование строки от пользователя через `strncpy_from_user()`.
- Тщательная обработка всех ошибочных случаев.

### Возврат при ошибке

При ошибке системная функция должна вернуть **отрицательное** значение — код ошибки. Стандартные коды в `<errno.h>`:
- `EFAULT` — некорректный указатель.
- `EINVAL` — некорректный параметр.
- `EPERM` — нет прав.
- `ENOMEM` — нехватка памяти.
- `EBADF` — некорректный файловый дескриптор.
- `ENOSYS` — функция не реализована.
- `EAGAIN` — попробуйте позже.

Библиотека C преобразует отрицательное возвращаемое значение в установку `errno` и возврат -1.

### Резюме

- Ядро не доверяет параметрам из пользователя.
- Проверяется: семантика, права доступа, корректность указателей.
- Указатели проверяются через `copy_to_user`/`copy_from_user`.
- Права — через `capable(CAP_*)`.
- При ошибке — отрицательный код возврата.

## 33. Функции copy_from_user() и copy_to_user()
*Источник: robert_lav.md, Глава 5 «Системные функции», раздел «Проверка параметров»*

### Зачем эти функции

Системный вызов часто получает или передаёт данные пользователю по указателю. Например:
- `read(fd, buf, n)` — записать `n` байт в `buf` (буфер пользователя).
- `write(fd, buf, n)` — прочитать `n` байт из `buf` пользователя.
- `getsockname(sock, addr, len)` — записать адрес сокета по `addr`.

Прямое использование указателя пользователя в ядре опасно:
1. Пользователь мог передать указатель в ядерную область → ядро отдало бы свои данные.
2. Пользователь мог передать указатель на чужой процесс → ядро отдало бы данные чужого.
3. Указатель может вести в выгруженную страницу → нужна подкачка с диска.

Функции `copy_from_user()` и `copy_to_user()` решают все эти проблемы.

### Сигнатуры

```c
unsigned long copy_from_user(void *to, const void __user *from, unsigned long n);
unsigned long copy_to_user(void __user *to, const void *from, unsigned long n);
```

**`copy_from_user(to, from, n)`** — копирует `n` байт из пользователя в ядро.
- `to` — указатель в ядре (куда копировать).
- `from` — указатель в пользователе (откуда копировать).
- `n` — размер в байтах.

**`copy_to_user(to, from, n)`** — копирует `n` байт из ядра в пользователя.
- `to` — указатель в пользователе (куда копировать).
- `from` — указатель в ядре (откуда копировать).
- `n` — размер в байтах.

Аннотация **`__user`** — это макрос, который для компиляции — пустой, но **sparse** (статический анализатор кода ядра) проверяет, что user-указатели не используются напрямую.

### Возвращаемое значение

Обе функции возвращают **число байт, которые НЕ удалось скопировать**:
- `0` — успех.
- `>0` — частичное копирование (или вообще не удалось).

```c
if (copy_from_user(&buf, src, len))
    return -EFAULT;
```

### Полный пример

Из robert_lav.md (silly_copy):

```c
// Источник: robert_lav.md, Глава 5
/*
* silly_copy — крайне бесполезная системная функция, которая копирует
* 'len' байтов из 'src' в 'dst', с промежуточным копированием в ядро.
*/
SYSCALL_DEFINE3(silly_copy,
    unsigned long *, src,
    unsigned long *, dst,
    unsigned long len)
{
    unsigned long buf;

    // Копируем из пользователя в ядро
    if (copy_from_user(&buf, src, len))
        return -EFAULT;

    // Копируем из ядра в пользователя
    if (copy_to_user(dst, &buf, len))
        return -EFAULT;

    return len;
}
```

Это иллюстративный пример (на практике можно было бы написать `silly_copy` как один `memcpy_in_user`, но идея — показать обе функции).

### Что происходит внутри

Когда вызывается `copy_from_user(dst, src, n)`:

1. **Проверка диапазона.** `src` должен быть в пользовательской части АП (на x86-64 — ниже определённого порога). Иначе сразу `-EFAULT`.

2. **Попытка прочитать.** Ядро читает байты из `src`. Каждый доступ проходит через MMU.

3. **Если страница не отображена** (например, выгружена на диск или была обработана через demand paging) — генерируется page fault.

4. **Обработчик page fault** в ядре:
   - Если страница часть АП процесса — загружает её с диска. Это блокирующая операция.
   - Если нет — возвращает ошибку.

5. **Если страница загружена** — обращение продолжается, байт читается.

6. **При успехе всех байт** — возврат 0. Если хоть один байт не удалось — возврат числа недоступных байт.

### Важное: блокировка

Из robert_lav.md:
> «В процессе работы обеих функций, `copy_to_user()` и `copy_from_user()`, выполнение процесса может быть приостановлено. Так происходит, например, если какая-либо из страниц, содержащая пользовательские данные, отсутствует в физической памяти компьютера, поскольку ранее была вытеснена на диск».

**Следствие:** эти функции **нельзя** использовать:
- В обработчиках прерываний.
- В контексте, где запрещён переход в состояние ожидания.
- При удерживаемом спинлоке.

Для таких случаев есть атомарные версии (`__copy_from_user_inatomic`), которые не блокируют, но могут вернуть «не получилось».

### Другие функции работы с памятью пользователя

**Для одного значения:**
```c
get_user(x, ptr);    // прочитать ptr в x (один элемент)
put_user(x, ptr);    // записать x в ptr
```

Быстрее, чем `copy_*_user`, потому что компилятор знает тип:

```c
int val;
if (get_user(val, (int __user *)arg))
    return -EFAULT;
```

**Для строк:**
```c
long strncpy_from_user(char *dst, const char __user *src, long count);
```

Копирует строку с нуль-терминатором, безопасная (не выйдет за границы буфера).

**Для обнуления:**
```c
unsigned long clear_user(void __user *ptr, unsigned long n);
```

Записывает `n` нулей в пользовательскую память.

**Проверка возможности доступа без копирования:**
```c
int access_ok(int type, const void __user *addr, unsigned long size);
```

Проверяет, что регион доступен (но не копирует). Раньше требовалось вызывать перед `copy_*_user`, сейчас они проверяют сами.

### Резюме

- `copy_from_user`/`copy_to_user` — безопасный способ обмена данными между ядром и пользователем.
- Проверяют корректность указателей.
- Возвращают число некопированных байт (0 — успех).
- Могут блокировать процесс — нельзя в прерываниях/спинлоках.
- Альтернативы: `get_user/put_user` для одного значения, `strncpy_from_user` для строк.

## 34. Контекст системной функции. system_call(). Завершающие этапы регистрации системной функции.
*Источник: robert_lav.md, Глава 5 «Системные функции», разделы «Контекст системной функции», «Завершающие этапы регистрации»*

### Контекст процесса

Когда выполняется системный вызов, **ядро работает «от имени» вызвавшего процесса**. Этот режим работы называется **контекстом процесса**.

Признаки контекста процесса:
- Макрос **`current`** корректно работает — указывает на `task_struct` вызвавшего процесса.
- Можно обращаться к данным процесса (открытые файлы, АП, права).
- Можно блокироваться — вызывать `schedule()`, ждать семафор, делать I/O.
- Можно использовать `copy_from_user`/`copy_to_user` — пользовательский АП доступен.

Сравните с **контекстом прерывания** (когда выполняется обработчик прерывания):
- `current` имеет неопределённое значение — обработчик не связан с конкретным процессом.
- Нельзя блокироваться.
- Нельзя обращаться к памяти пользователя напрямую.

### Возможность ожидания

Из robert_lav.md:
> «В контексте процесса ядро может переходить в состояние ожидания (например, если при вызове системной функции выполнение процесса будет приостановлено или будет явно вызвана функция `schedule()`), а также является полностью вытесняемым».

Это важное свойство:
- Можно реализовать **блокирующие** системные вызовы (`read` ждёт данных).
- Можно использовать **mutex_lock**, **semaphore_down** — функции, которые могут ждать.
- Можно вызывать функции аллокации памяти с флагом `GFP_KERNEL` (могут блокировать ради подкачки).

### Полная вытесняемость

Из robert_lav.md:
> «Тот факт, что контекст процесса является вытесняемым, означает, что, как и программа пользователя, текущая задача, выполняющая системную функцию, может быть вытеснена другой задачей».

Это означает:
- Пока выполняется `sys_read()`, придёт прерывание, разблокирует более приоритетный процесс — `sys_read` будет приостановлен.
- Нужна **синхронизация** доступа к общим данным.
- Системные функции должны быть **реентерабельными** — корректно работать при одновременном выполнении из разных потоков.

### Реентерабельность

**Реентерабельная функция** — может быть вызвана повторно (в том числе из другого потока или прерывания), пока первый вызов ещё не завершён, и работать корректно.

Требования:
- Не использовать глобальные переменные без синхронизации.
- Не использовать статические локальные переменные.
- Использовать спин-блокировки/мьютексы для общих данных.
- Параметры — в стеке.

В Linux большинство системных функций реентерабельные.

### Функция system_call()

Это **точка входа** в ядро при системном вызове. Реализована на ассемблере в `entry_64.S` (для x86-64). Структура:

```asm
ENTRY(system_call)
    ; 1. Сохранить регистры пользователя в стек ядра
    SAVE_ARGS
    ; 2. Проверить номер
    cmp rax, NR_syscalls
    jae bad_syscall
    ; 3. Вызвать функцию по таблице
    call *sys_call_table(,%rax,8)
    ; 4. Сохранить результат в фрейме
    mov [rsp + RAX], rax
    ; 5. Восстановить регистры
    RESTORE_ARGS
    ; 6. Возврат в пользовательский режим
    sysret
bad_syscall:
    mov rax, -ENOSYS
    jmp out
END(system_call)
```

(Это упрощённая версия; реальный код учитывает много нюансов: tracing, signals, multi-arch.)

### Завершающие этапы регистрации системной функции

После реализации системной функции её нужно **зарегистрировать**. Шаги:

**Шаг 1: Добавить в `sys_call_table`.**

Открываем `arch/x86/entry/syscalls/syscall_64.tbl` (или `entry.S` в старых ядрах). Файл выглядит так:

```
0   common  read         sys_read
1   common  write        sys_write
2   common  open         sys_open
3   common  close        sys_close
...
337 common  recvmmsg     sys_recvmmsg
```

Добавляем в конец:
```
338 common  foo          sys_foo
```

Каждая аппаратная платформа имеет свою таблицу — номер может отличаться. Это часть **ABI** (Application Binary Interface) платформы.

**Шаг 2: Определить номер в заголовочном файле.**

В `<asm/unistd.h>` для платформы x86:
```c
#define __NR_foo 338
```

Это нужно, чтобы пользовательские программы могли использовать константу:
```c
syscall(__NR_foo, arg1, arg2);
```

**Шаг 3: Реализовать функцию.**

Обычно в `kernel/sys.c` или в подходящем подкаталоге:

```c
// kernel/foo.c
#include <linux/syscalls.h>

SYSCALL_DEFINE2(foo, int, arg1, char __user *, arg2)
{
    char buf[256];

    if (arg1 < 0)
        return -EINVAL;

    if (copy_from_user(buf, arg2, sizeof(buf)))
        return -EFAULT;

    // ... делаем работу ...
    return 0;
}
```

**Шаг 4: Скомпилировать в ядро.**

Системные функции **не могут** быть модулями — они должны быть в основном образе ядра. Это потому, что `sys_call_table` — постоянная структура.

Добавить в Makefile подходящего каталога:
```
obj-y += foo.o
```

Пересобрать ядро и загрузить.

### Особенности номеров

- **Номер нельзя менять** после первой публикации — это разрушит существующие бинарники, скомпилированные с этим номером.
- **При удалении функции** номер не переиспользуется. На его место в таблицу помещается `sys_ni_syscall`:
  ```c
  asmlinkage long sys_ni_syscall(void) {
      return -ENOSYS;
  }
  ```

Из robert_lav.md:
> «В системе Linux предусмотрена так называемая "не реализованная" функция `sys_ni_syscall()`, которая просто возвращает значение `-ENOSYS` — код ошибки, соответствующий некорректной системной функции».

- **Номера для разных архитектур могут отличаться.** Например, `__NR_read` на x86 — 0, на ARM — 3.

### Архитектурно-зависимая часть

Часть системных вызовов специфичны для конкретной архитектуры (например, для x86 есть `iopl`, `ioperm`). Они не присутствуют на других платформах.

Из robert_lav.md:
> «Для аппаратной платформы x86 существует около 335 системных функций (для каждой аппаратной платформы разрешается определять свои уникальные системные функции)».

### Резюме

- Контекст процесса: `current` доступен, можно блокироваться, можно работать с памятью пользователя.
- Системные функции должны быть реентерабельными.
- `system_call()` — точка входа в ядро при syscall.
- Регистрация: добавить в `sys_call_table`, объявить номер в `<asm/unistd.h>`, скомпилировать в составе ядра.
- Номера нельзя менять; удалённые функции заменяются заглушкой `sys_ni_syscall`.

## 35. Доступ к системным функциям из пользовательских приложений.
*Источник: robert_lav.md, Глава 5 «Системные функции», раздел «API, POSIX и библиотека C»*

### Пользователь не вызывает syscall напрямую

Из robert_lav.md:
> «Для прикладного программиста системные функции не имеют особого значения, поскольку всё, с чем он работает, — это интерфейс API».

Пользовательский код **обычно не вызывает** системные функции напрямую. Между приложением и ядром есть **библиотека C (libc, glibc)**, которая:
- Предоставляет удобные API.
- Реализует обёртки над системными вызовами.
- Дополняет их полезной логикой (буферизация, кэширование).

### Слои абстракции

```
┌─────────────────────────┐
│   Приложение            │  printf(), fopen(), malloc(), pthread_*
├─────────────────────────┤
│   Библиотека C (glibc)  │  Конвертация: fopen() → open(); printf() → write()
├─────────────────────────┤
│   Системные вызовы      │  sys_open(), sys_write(), sys_brk()
├─────────────────────────┤
│   Ядро                  │  Реализация
└─────────────────────────┘
```

### Пример: printf("Hello\n")

Из robert_lav.md (рис. 5.1): «Взаимосвязь между приложениями, библиотекой C и ядром на примере вызова функции `printf()`».

Что происходит:

1. Приложение вызывает `printf("Hello\n")`.
2. Библиотечная `printf()`:
   - Форматирует строку.
   - Помещает результат в свой буфер.
   - Если буфер полон или встречается `\n` — вызывает `write()`.
3. Библиотечная `write(1, buf, n)`:
   - Подготавливает регистры:
     - `rax = 1` (`__NR_write` на x86-64).
     - `rdi = 1` (fd = stdout).
     - `rsi = buf`.
     - `rdx = n`.
   - Выполняет `syscall` (или `int 0x80`).
4. Процессор атомарно:
   - Переключается в режим ядра.
   - Сохраняет адрес возврата.
   - Переходит на обработчик `system_call()`.
5. `system_call()`:
   - Сохраняет регистры в стек ядра.
   - Проверяет `rax < NR_syscalls`.
   - Вызывает `sys_call_table[1]` = `sys_write`.
6. `sys_write(1, buf, n)`:
   - Находит `struct file *` по fd=1.
   - Проверяет права (можно ли писать).
   - Вызывает `f_op->write(...)` — конкретный метод для типа устройства (терминал, файл, сокет).
7. Метод пишет данные.
8. Возврат значения через все уровни до приложения.

### Стандарты API

**POSIX (Portable Operating System Interface).** Набор стандартов IEEE, цель — создать переносимый стандарт Unix-подобной ОС.

Из robert_lav.md:
> «Стандарт POSIX является хорошим примером взаимосвязи между интерфейсами API и системными функциями. В большинстве Unix-подобных ОС функции интерфейса API, определённые в стандарте POSIX, имеют очень тесную связь с системными функциями».

POSIX определяет:
- Функции — `open`, `read`, `write`, `fork`, `exec`, `wait`, `pthread_*`.
- Типы — `pid_t`, `mode_t`, `off_t`.
- Заголовки — `<unistd.h>`, `<fcntl.h>`, `<sys/stat.h>`.
- Сигналы — `SIGINT`, `SIGTERM`, etc.

Linux стремится к соответствию POSIX и SUSv3 (Single UNIX Specification v3).

**SUSv3** — расширение POSIX с дополнительными требованиями.

### Зачем нужна библиотека C посередине

**1. Переносимость.** Один и тот же исходный код компилируется на Linux, BSD, macOS — потому что API библиотеки везде одинаковый, хотя системные вызовы разные.

**2. Удобство.** `printf("%d", x)` гораздо удобнее, чем форматирование вручную и вызов `write`.

**3. Буферизация.** `printf` / `fread` буферизуют данные — меньше системных вызовов.

**4. Дополнительная функциональность.**
- `malloc` управляет кучей через `brk` / `mmap` (системные вызовы), но скрывает детали.
- `fopen` управляет файловыми потоками поверх `open`.

**5. Совместимость через стандарты.** Программа, написанная по POSIX, работает на любой POSIX-совместимой ОС.

**6. Оптимизации.**

Некоторые «системные вызовы» на самом деле могут быть выполнены **без перехода в ядро**:

- **vDSO (virtual Dynamic Shared Object).** Маленькая разделяемая библиотека, отображённая ядром в каждый процесс. Содержит реализации часто вызываемых функций вроде `gettimeofday()`, `clock_gettime()`.
- Ядро обновляет данные (текущее время) в специальной странице, доступной пользователю для чтения. Приложение читает напрямую — без системного вызова.

### Прямой вызов системного вызова

Иногда нужно вызвать системный вызов напрямую, минуя библиотеку C:
- Для очень редкого syscall, не покрытого библиотекой.
- Для системного программирования низкого уровня.
- Для отладки.

Способ — функция `syscall()` из `<sys/syscall.h>`:

```c
#include <sys/syscall.h>
#include <unistd.h>

long result = syscall(__NR_gettid);   // системный вызов без обёртки
```

Или прямой ассемблер:

```c
long my_write(int fd, const void *buf, size_t n) {
    long ret;
    asm volatile (
        "syscall"
        : "=a"(ret)
        : "0"(__NR_write), "D"(fd), "S"(buf), "d"(n)
        : "rcx", "r11", "memory"
    );
    return ret;
}
```

### Как библиотека C обрабатывает ошибки

При системном вызове:
- Положительное / нулевое возвращаемое значение — успех.
- Отрицательное — код ошибки.

Библиотека C преобразует это в стандартный Unix-стиль:

```c
ssize_t write(int fd, const void *buf, size_t count) {
    ssize_t ret = syscall(__NR_write, fd, buf, count);
    if (ret < 0) {
        errno = -ret;
        return -1;
    }
    return ret;
}
```

Приложение проверяет:
```c
ssize_t n = write(fd, buf, count);
if (n < 0) {
    perror("write");    // выводит errno в виде строки
}
```

### Резюме

- Приложения работают с API библиотеки C, а не с системными вызовами напрямую.
- API стандартизованы (POSIX, SUSv3) — переносимость.
- Библиотека C делает обёртки: помещает аргументы в регистры, выполняет `syscall`, обрабатывает ошибки.
- Часть «системных вызовов» оптимизирована через vDSO — выполняется без перехода в ядро.
- Прямой вызов через `syscall(номер, ...)`.

## 36. Реализация доступа к системным объектам. Количество обращений. Квоты. Особенности хранения указателей.
*Источник: robert_lav.md, Глава 5; общие принципы Linux*

### Что такое системные объекты

**Системные объекты** — структуры данных в ядре, представляющие ресурсы, которые могут быть использованы пользовательскими программами:
- Открытые файлы (`struct file`).
- Сокеты (`struct socket`).
- Процессы (`struct task_struct`).
- Разделяемые сегменты памяти.
- IPC-объекты (семафоры, очереди сообщений).
- Сигнальные обработчики.

Эти объекты хранятся **в памяти ядра** — защищённой от пользователя.

### Проблема: как пользователь работает с этими объектами

Если объект — структура в ядре, пользователь не может с ней работать напрямую:
- Указатель на ядерную память — недоступен пользователю.
- Даже если бы был доступен — пользователь мог бы повредить.

**Решение — идентификаторы (handles).**

Пользователь получает **целое число** — идентификатор объекта. Ядро поддерживает таблицу `id → указатель на структуру`. Все операции пользователя — через id; ядро находит структуру.

### Пример: файловые дескрипторы

```c
int fd = open("/etc/passwd", O_RDONLY);
// fd = 3, скажем
read(fd, buf, 100);
close(fd);
```

Что происходит:
1. `open()` создаёт `struct file` в ядре.
2. Помещает указатель на неё в `current->files->fd[3]`.
3. Возвращает 3.
4. `read(3, buf, 100)` — ядро смотрит в `current->files->fd[3]`, находит указатель.
5. Вызывает методы `struct file`.

```c
struct task_struct {
    // ...
    struct files_struct *files;
    // ...
};

struct files_struct {
    struct file **fd_array;   // массив указателей
    int max_fds;
    // ...
};
```

Файловые дескрипторы 0, 1, 2 — стандартный stdin, stdout, stderr. Они открыты автоматически при запуске процесса.

### Изоляция таблиц объектов

**Каждый процесс имеет свою таблицу** `files`. Дескриптор 5 в одном процессе и в другом — **разные** объекты. Это часть изоляции процессов.

При `fork()` таблица **копируется** (или разделяется, в зависимости от флагов `clone()`).
При `exec()` — таблица сохраняется (если не помечен `FD_CLOEXEC`).

### Количество обращений (Reference Counting)

Объект ядра может быть **доступен из нескольких мест**:
- `dup()` дублирует файловый дескриптор → два дескриптора указывают на один `struct file`.
- `fork()` копирует таблицу → ребёнок имеет тот же `struct file`.
- Один файл может быть открыт несколькими процессами одновременно.

Чтобы объект не был удалён, пока кто-то его использует, в нём хранится **счётчик ссылок** (reference count):

```c
struct file {
    atomic_long_t f_count;    // счётчик ссылок
    // ...
};
```

Правила:
- При создании счётчик = 1.
- При появлении новой ссылки — `f_count++`.
- При удалении ссылки — `f_count--`.
- При `f_count == 0` — объект освобождается.

Для атомарного инкремента/декремента используется `atomic_t`:

```c
struct file *fget(unsigned int fd) {
    struct file *f = current->files->fd_array[fd];
    atomic_long_inc(&f->f_count);
    return f;
}

void fput(struct file *f) {
    if (atomic_long_dec_and_test(&f->f_count))
        __fput(f);     // освобождение, если стало 0
}
```

Идиома `dec_and_test` гарантирует атомарность: декремент и проверка на 0 — одна операция.

### Когда счётчик увеличивается

- **`open()`** — создаёт новый `struct file` с count=1.
- **`dup(fd)`** — создаёт ещё один дескриптор, указывающий на тот же `file`. `count++`.
- **`fork()`** — таблица копируется, все `file` получают `count++`.
- **`fget(fd)`** в коде ядра — временная ссылка на `file`. `count++`.

### Когда уменьшается

- **`close(fd)`** — удаление дескриптора. `count--`.
- **`exit()`** — закрытие всех дескрипторов.
- **`fput(f)`** в коде ядра — освобождение временной ссылки.

### Квоты

**Квоты** — ограничения на потребление ресурсов. Защищают систему от:
- Случайного исчерпания ресурсов одной программой.
- Злонамеренных атак (DoS).
- Утечек в баггованных программах.

В Linux установлены через `setrlimit()`:

| Лимит | Что ограничивает |
|---|---|
| `RLIMIT_AS` | Размер виртуального АП |
| `RLIMIT_CORE` | Размер core-файла |
| `RLIMIT_CPU` | Время CPU |
| `RLIMIT_DATA` | Размер сегмента данных |
| `RLIMIT_FSIZE` | Максимальный размер файла |
| `RLIMIT_NOFILE` | Число открытых файлов |
| `RLIMIT_NPROC` | Число процессов пользователя |
| `RLIMIT_STACK` | Размер стека |
| `RLIMIT_MEMLOCK` | Закреплённая в RAM память |
| `RLIMIT_RSS` | Размер резидентной памяти |

У каждого лимита есть **soft** и **hard** значения:
- Программа может уменьшать soft до hard.
- Только root может увеличивать hard.

Превышение лимита → возврат `-EAGAIN`, `-EMFILE`, или сигнал (например, `SIGXCPU` при превышении `RLIMIT_CPU`).

### Дисковые квоты

Отдельная система — **`quota`** — ограничивает размер занимаемого пространства и число файлов на пользователя:
- `usrquota` — на пользователя.
- `grpquota` — на группу.
- `prjquota` — на проект.

Управляются через `quotactl()` и утилиты `quota`, `edquota`.

### Особенности хранения указателей

**1. Указатели ядра не передаются пользователю.**

Адреса структур ядра — секретны. Если бы пользователь их знал, он мог бы:
- Использовать в эксплойтах для обхода ASLR ядра.
- Изучать раскладку памяти ядра.

Поэтому ядро возвращает пользователю только **идентификаторы**, никогда — указатели.

**2. Указатели пользователя проверяются.**

Любой указатель из пользователя — потенциально враждебный. Используется `copy_from_user`/`copy_to_user` (вопрос 33).

**3. Объекты живут, пока кто-то их использует.**

Reference counting гарантирует это. Даже если процесс закрыл файл, объект может оставаться (если ещё дублирован в другом процессе через `fork`).

**4. Локальность.**

Объекты «привязаны» к процессу. Изменение `current->files->fd[5]` влияет только на текущий процесс.

**5. Безопасное удаление.**

Когда объект освобождается, **все** ссылки на него уже должны быть удалены. Иначе — use-after-free.

В Linux есть **RCU (Read-Copy-Update)** для безопасного удаления объектов, на которые могут быть выпускаемые ссылки (см. вопрос 70).

### Резюме

- Системные объекты — структуры в ядре, доступные пользователю через идентификаторы.
- Каждый процесс имеет свою таблицу объектов (например, `files`).
- Reference counting защищает от преждевременного удаления.
- Квоты (`rlimit`, дисковые) ограничивают потребление.
- Указатели ядра не показываются пользователю; указатели пользователя проверяются.

## 37. Завершение процесса при обработке системной функции. Блокирующие вызовы из режима пользователя.
*Источник: robert_lav.md, Глава 3 «Завершение процесса»; rtos.md, Глава 7.7.8*

### Завершение процесса — обзор

Процесс может завершиться несколькими способами:

**Добровольное:**
- Вызов `exit(code)`.
- Возврат из `main()` (компилятор C автоматически добавляет вызов `exit()`).

**Принудительное:**
- Получение неперехватываемого сигнала (`SIGKILL`).
- Получение сигнала с обработчиком по умолчанию = завершение (`SIGTERM`, `SIGSEGV`).
- Возникновение неперехватываемого исключения (необработанный SIGFPE и т.д.).

В любом случае основную работу делает функция **`do_exit()`** в `kernel/exit.c`.

### Что делает do_exit()

Из robert_lav.md:

1. **`task_struct->flags |= PF_EXITING`** — отметка, что процесс завершается.

2. **`del_timer_sync()`** — удаление всех таймеров ядра, связанных с процессом. Гарантирует, что после возврата нет ожидающих таймеров.

3. **Учёт ресурсов** (BSD process accounting, если включен). `acct_update_integrals()` записывает статистику.

4. **`exit_mm()`** — освобождение `mm_struct` (адресного пространства). Если другие процессы не используют (счётчик ссылок = 0) — `mm_struct` уничтожается.

5. **`exit_sem()`** — удаление из очередей ожидания семафоров IPC.

6. **`exit_files()`** и **`exit_fs()`** — уменьшение счётчиков ссылок на открытые файлы и информацию о файловой системе. Файлы с `f_count == 0` закрываются.

7. **`exit_notify()`** — уведомление родителя сигналом `SIGCHLD`.

8. Перевод процесса в **`TASK_ZOMBIE`** (состояние zombie).

### Состояние zombie

Процесс **завершён**, но его дескриптор `task_struct` остаётся в памяти, пока родитель не вызовет `wait()`. Зачем?

Родителю может быть нужен **статус выхода** ребёнка:
- Код возврата (`exit(0)` vs `exit(1)`).
- Сигнал, от которого ребёнок умер.
- Статистика использования ресурсов.

`wait()` / `waitpid()` / `waitid()`:
- Получают статус через указатель пользователя.
- Удаляют дескриптор завершённого процесса.

Если родитель не вызывает `wait`, у zombie накапливаются — может быть утечка PID.

**Если родитель умер раньше** — процесс наследует `init` (PID 1) или подсистема `subreaper`. Init постоянно вызывает `wait` для всех своих усыновлённых детей — поэтому zombie там не задерживаются.

### Завершение в середине системного вызова

Что, если пользователю пришёл сигнал `SIGTERM` (просьба завершиться) во время выполнения `read()`, который ждёт данных с диска?

Зависит от состояния:

**1. `TASK_INTERRUPTIBLE` (большинство ожиданий).**
- Сигнал сразу разбудит процесс.
- Системный вызов вернёт `-EINTR` (interrupted).
- Возврат из ядра в пользователя.
- Перед возвратом — обработка сигнала (вызов обработчика или default-действие).
- Если default = завершение → `do_exit()`.

**2. `TASK_UNINTERRUPTIBLE` (критический I/O).**
- Сигнал откладывается до окончания ожидания.
- После пробуждения сигнал обрабатывается.
- Такие процессы видны в `ps` как статус `D` — «неубиваемые».

Поэтому процесс, ждущий диска в `D`-состоянии, нельзя завершить даже `SIGKILL`. Это редкая ситуация, но возможная при сбоях оборудования.

### Блокирующие вызовы из режима пользователя

**Блокирующий системный вызов** — может надолго остановить процесс, ожидая событие. Примеры:

| Вызов | Что ждёт |
|---|---|
| `read()` на не-неблокирующем fd | Данные |
| `write()` (если буфер полон) | Место в буфере |
| `wait()` | Завершение ребёнка |
| `select()`/`poll()`/`epoll_wait()` | Готовность дескрипторов |
| `sem_wait()` (POSIX) | Семафор |
| `pthread_mutex_lock()` | Мьютекс |
| `accept()` | Подключение клиента |
| `connect()` | Подключение к серверу |
| `sleep()`/`nanosleep()` | Тайм-аут |

### Реализация в ядре

Типичная схема (на примере `read`):

```c
ssize_t my_read(struct file *f, char __user *buf, size_t count, loff_t *pos)
{
    DEFINE_WAIT(wait);
    ssize_t ret;

    while (нет_данных) {
        if (f->f_flags & O_NONBLOCK)
            return -EAGAIN;

        // Подготовка к засыпанию
        add_wait_queue(&data_ready_queue, &wait);
        prepare_to_wait(&data_ready_queue, &wait, TASK_INTERRUPTIBLE);

        if (signal_pending(current)) {
            finish_wait(&data_ready_queue, &wait);
            return -EINTR;     // сигнал!
        }

        schedule();    // спать

        finish_wait(&data_ready_queue, &wait);
    }

    // данные есть — копировать
    if (copy_to_user(buf, kernel_buf, count))
        return -EFAULT;

    return count;
}
```

Когда придут данные, обработчик прерывания вызовет `wake_up(&data_ready_queue)`. Процесс проснётся, проверит условие, скопирует данные и вернётся.

### Неблокирующий режим

Файловые дескрипторы могут быть открыты с флагом **`O_NONBLOCK`**. В этом режиме:
- `read()` без данных → возврат `-EAGAIN` (или `-EWOULDBLOCK`).
- `write()` без места → возврат `-EAGAIN`.
- `accept()` без клиента → возврат `-EAGAIN`.

Программа должна сама проверять, готов ли дескриптор. Обычно используется в комбинации с `select`/`poll`/`epoll`.

### Прерывание сигналом и SA_RESTART

Если блокирующий `read()` прерван сигналом, он возвращает `-EINTR`. Это раздражает приложения — нужно вручную перезапускать:

```c
ssize_t n;
while ((n = read(fd, buf, size)) == -1 && errno == EINTR)
    ;    // повтор
```

В современных Unix есть флаг **`SA_RESTART`** для обработчика сигнала:

```c
struct sigaction sa;
sa.sa_handler = my_handler;
sa.sa_flags = SA_RESTART;
sigaction(SIGUSR1, &sa, NULL);
```

С этим флагом ядро **автоматически перезапускает** прерванный сигналом системный вызов. Но не все вызовы перезапускаются — некоторые (например, `select`) всё равно вернут `-EINTR`.

### Из rtos.md (7.7.8)

> «Блокирующие вызовы должны быть прерываемыми — при поступлении сигнала или запросе на завершение поток должен корректно выйти из ожидания».

В ОСРВ это особенно важно — длинные блокирующие вызовы должны иметь возможность быть прерванными ради высокоприоритетных задач.

### Резюме

- Завершение процесса — `do_exit()`: освобождение ресурсов, перевод в zombie, ожидание `wait()` от родителя.
- В середине системного вызова сигнал может разбудить процесс в `TASK_INTERRUPTIBLE` (возврат `-EINTR`); процессы в `TASK_UNINTERRUPTIBLE` — нет.
- Блокирующие вызовы (`read`, `wait`, `select`) ставят процесс в очередь ожидания, вызывают `schedule()`.
- Можно открыть fd с `O_NONBLOCK` для неблокирующего режима.
- `SA_RESTART` автоматически перезапускает прерванные вызовы.

## 38. Мультипрограммный режим. Процессы, ориентированные на ввод/вывод и на вычисления. Приоритет процессов. Квант времени.
*Источник: tanenbaum.md, Глава 2.4 «Планирование»*

### Зачем мультипрограммный режим

Представим систему без мультипрограммного режима: одна программа использует CPU полностью. Если эта программа делает `read()` и ждёт данных с диска, CPU **простаивает** — десятки миллисекунд. Это плохо:
- Современные CPU выполняют миллиарды операций в секунду — потеря огромная.
- При смешанной нагрузке (часть процессов I/O, часть CPU) общая пропускная способность мала.

**Мультипрограммный режим** — одновременное хранение в памяти и выполнение нескольких программ путём разделения CPU. Когда одна ждёт I/O, другая использует CPU.

### Загрузка CPU

При мультипрограммировании:
- 1 процесс делает I/O 80% времени → CPU простаивает 80%.
- 5 процессов, каждый делает I/O 80% → CPU простаивает (0.8)⁵ ≈ 33%.
- 10 процессов → (0.8)¹⁰ ≈ 11%.

Эта формула — упрощение (предполагает независимость), но показывает: чем больше процессов в памяти, тем лучше загрузка.

### CPU-bound и I/O-bound процессы

Из tanenbaum.md (2.4.1):
> «Первые процессы называются процессами, ограниченными скоростью вычислений, а вторые — процессами, ограниченными скоростью работы устройств ввода-вывода. Процессы, ограниченные скоростью вычислений, обычно имеют продолжительные пики вычислительной активности и, соответственно, нечастые периоды ожидания ввода-вывода, а процессы, ограниченные скоростью работы устройств ввода-вывода, имеют короткие периоды активности центрального процессора и, соответственно, довольно частые периоды ожидания ввода-вывода».

**CPU-bound (compute-bound):**
- Долго работают на CPU.
- Редко делают I/O.
- Примеры: научные расчёты (FFT, симуляции), рендеринг 3D, сжатие данных, шифрование, компиляция кода.

**I/O-bound:**
- Короткие пики CPU.
- Частые операции I/O.
- Примеры: редактор текста, файловый менеджер, веб-сервер, СУБД (часто), копирование файлов.

### Важная тонкость

Из tanenbaum.md:
> «Ключевым фактором здесь является период пиковой активности центрального процессора, а не продолжительность активности устройств ввода-вывода. Процессы, ограниченные скоростью работы устройств ввода-вывода, считаются таковыми только потому, что не занимаются продолжительными вычислениями в промежутках между запросами ввода-вывода».

То есть классификация — по **длительности CPU-burst** (пика активности CPU между блокировками), а не по объёму I/O.

### Тенденция: всё больше процессов становятся I/O-bound

Из tanenbaum.md:
> «Стоит заметить, что чем быстрее становятся центральные процессоры, тем больше процессы ограничиваются скоростью работы устройств ввода-вывода. Это связано с более быстрым совершенствованием центральных процессоров по сравнению с совершенствованием дисковых устройств».

CPU удваивает производительность каждые ~2 года (закон Мура), HDD — намного медленнее. Программы, которые в 90-х были CPU-bound, сейчас I/O-bound.

С появлением SSD и NVMe ситуация частично меняется, но общий тренд сохраняется.

### Балансировка нагрузки в планировщике

Планировщик заинтересован в том, чтобы **CPU и I/O работали параллельно**. Поэтому:
- I/O-bound процессы получают **повышенный приоритет** — когда они становятся готовы, их быстро запускают, чтобы они сразу делали следующий I/O-запрос.
- CPU-bound процессы — наоборот, понижают приоритет — они и так не блокируются, могут подождать.

Стратегия Linux CFS использует именно этот принцип через `vruntime`.

### Приоритет процессов

Приоритет — числовое значение, определяющее «важность» процесса. Из tanenbaum.md (2.4.3):
> «Каждому процессу присваивается значение приоритетности и запускается тот процесс, который находится в состоянии готовности и имеет наивысший приоритет».

В Linux:

**1. `nice` value (-20..19).**
- -20 — наивысший.
- 0 — по умолчанию.
- 19 — низший.

Меняется командой `nice` или `renice`. Только root может понижать nice (повышать приоритет).

**2. Real-time priority (1..99).**
- Применяется к процессам с политикой SCHED_FIFO или SCHED_RR.
- Выше любого обычного процесса.

**3. Динамический приоритет.**
- Корректируется ядром.
- I/O-bound получают повышение.
- CPU-bound — понижение.

В CFS используется концепция **vruntime** — виртуальное время процесса. Запускается процесс с наименьшим vruntime. Приоритет влияет на скорость роста vruntime: высокоприоритетные растут медленнее → чаще запускаются.

### Старение (aging)

Чтобы низкоприоритетные процессы не голодали:
- Со временем их приоритет повышается.
- Когда такой процесс получит CPU и поработает — приоритет возвращается.

Это гарантирует прогресс всех процессов.

### Приоритеты в реальном времени

Linux поддерживает несколько политик планирования:

| Политика | Описание |
|---|---|
| **SCHED_OTHER** (CFS) | Обычная, для большинства |
| **SCHED_FIFO** | RT, без вытеснения внутри приоритета |
| **SCHED_RR** | RT, round-robin внутри приоритета |
| **SCHED_DEADLINE** | EDF |
| **SCHED_BATCH** | Для CPU-bound, нечувствительных к задержке |
| **SCHED_IDLE** | Очень низкий приоритет |

### Квант времени (time slice)

**Квант** — максимальное время, на которое CPU выделяется процессу при round-robin между равноприоритетными. По истечении кванта — вытеснение, поставновка в конец очереди.

Зачем нужен:
- Без него один процесс может захватить CPU надолго → плохая отзывчивость.
- Слишком короткий → много переключений, много накладных расходов.
- Слишком длинный → плохая отзывчивость в интерактивных системах.

### Выбор размера кванта

Из tanenbaum.md (2.4.3):
> «Предположим, что переключение процесса, или переключение контекста… занимает 1 мс, включая переключение карт памяти, сброс на диск и перезагрузку кэша и т. д. Также предположим, что значение кванта времени установлено на 4 мс. При таких параметрах настройки после 4 мс полезной работы центральному процессору придётся затратить (то есть потерять) 1 мс на переключение процесса. Таким образом, 20% процессорного времени будет выброшено на административные издержки».

При увеличении кванта:
- 100 мс кванта → потеря 1%.
- Но если в очереди 50 процессов и каждый берёт свой полный квант → последний ждёт 5 секунд. Слишком долго для интерактивной работы.

Tanenbaum рекомендует: **20-50 мс** — разумный компромисс.

### Динамический квант в Linux

В современном Linux (CFS) квант не фиксирован, а вычисляется из:
- **Целевой задержки** (sched_latency_ns, обычно 6-20 мс) — время, за которое все готовые процессы должны получить CPU хотя бы раз.
- **Числа готовых процессов** — `slice = sched_latency_ns / nr_running`.
- **Приоритета процесса** — высокий приоритет получает больший slice.

Например, при 4 готовых процессах с одинаковым приоритетом и `sched_latency_ns = 20 мс`:
- Каждый получает 5 мс.

При 100 готовых процессах вычисленный slice стал бы 0.2 мс — это меньше минимального gangrene quantum. Используется `sched_min_granularity_ns` (обычно 0.75 мс) как нижний предел.

### Резюме

- Мультипрограммный режим — основа эффективного использования CPU.
- CPU-bound vs I/O-bound — классификация по длине CPU-burst.
- I/O-bound получают повышение приоритета (для параллелизма с I/O).
- Приоритет в Linux: nice (-20..19), RT (1..99), динамический.
- Квант — максимум времени для round-robin; разумный размер 20-50 мс.
- В CFS квант динамический, зависит от числа готовых и приоритета.

## 39. IPC. Shared Memory, сигналы.
*Источник: tanenbaum.md, Глава 2.3 «Взаимодействие процессов»; rtos.md, Глава 5.7*

### Зачем нужно IPC

Каждый процесс изолирован: своё АП, свои файлы, свои сигналы. Если процессы должны **взаимодействовать**, нужны специальные механизмы.

Из tanenbaum.md (2.3):
> «Довольно часто процессам необходимо взаимодействовать с другими процессами. Например, в канале оболочки выходные данные одного процесса могут передаваться другому процессу, и так далее вниз по цепочке. Поэтому возникает необходимость во взаимодействии процессов, и желательно по хорошо продуманной структуре».

### Три задачи IPC

Tanenbaum выделяет три задачи:

**1. Передача информации.** Как один процесс может передать данные другому?

**2. Синхронизация.** Как обеспечить совместную работу без помех — два процесса не должны одновременно изменять одни данные.

**3. Упорядочивание.** Если процесс А производит данные, а Б их потребляет, Б должен ждать готовности данных.

### Виды IPC в Linux

**Передача данных:**
- **Pipe** (канал) — однонаправленный поток между связанными процессами.
- **Named pipe (FIFO)** — канал с именем в файловой системе.
- **Socket** — двунаправленная связь, в том числе локально через Unix domain sockets.
- **Message queue** — очередь сообщений (System V и POSIX).
- **Shared memory** — общая область памяти.

**Синхронизация:**
- **Semaphore** — счётчик с примитивами P/V.
- **Mutex** (через POSIX threads или futex).
- **Условные переменные.**

**Сигналы:**
- **Signals** — асинхронные уведомления.

**Уведомления и события:**
- **eventfd** — счётчик с возможностью ждать.
- **signalfd** — получение сигналов как файлового дескриптора.
- **inotify** — уведомления о событиях файловой системы.

### Shared Memory — общее адресное пространство

**Разделяемая память** — самый быстрый способ обмена данными между процессами. Один физический регион памяти отображается в виртуальные АП нескольких процессов.

```
Процесс A           Физическая память           Процесс B
ВАП                                              ВАП
┌──────┐                                         ┌──────┐
│ A.code│                                         │ B.code│
├──────┤            ┌──────────┐                 ├──────┤
│ A.data│  ────→    │ Shared   │  ←────          │ B.data│
├──────┤            │ memory   │                 ├──────┤
│ shm  │  ←────     └──────────┘                 │ shm  │
└──────┘                                         └──────┘
```

Оба процесса видят одну и ту же память. Запись из A — мгновенно видна B.

### Преимущества shared memory

- **Самая быстрая форма IPC.** После установки нет копирования через ядро — прямое обращение к памяти.
- **Большие объёмы данных.** Можно делить десятки гигабайт.
- **Эффективно для частых обменов.**

### Недостатки

- **Нужна ручная синхронизация.** Параллельный доступ → race condition. Обычно дополняется семафорами или мьютексами в самой разделяемой памяти.
- **Сложнее переносить:** работа с конкретными адресами.
- **Сложнее отлаживать:** ошибки одного процесса могут «отравить» данные другого.

### POSIX shared memory

```c
#include <sys/mman.h>
#include <fcntl.h>

// Создание и открытие
int fd = shm_open("/myshm", O_CREAT | O_RDWR, 0666);
ftruncate(fd, 4096);    // размер 4 КБ

// Отображение в АП
void *ptr = mmap(NULL, 4096, PROT_READ | PROT_WRITE,
                 MAP_SHARED, fd, 0);

// Работа
*((int*)ptr) = 42;

// Освобождение
munmap(ptr, 4096);
close(fd);
shm_unlink("/myshm");
```

`shm_open` создаёт «файл» в `/dev/shm/` — tmpfs-файловой системе, живущей в RAM. `mmap` отображает его в АП.

### System V shared memory

Более старый API:

```c
#include <sys/ipc.h>
#include <sys/shm.h>

// Создание
key_t key = ftok("/some/path", 'A');
int shmid = shmget(key, 4096, IPC_CREAT | 0666);

// Присоединение
void *ptr = shmat(shmid, NULL, 0);

// Работа
*((int*)ptr) = 42;

// Отсоединение
shmdt(ptr);

// Удаление (для последнего использующего)
shmctl(shmid, IPC_RMID, NULL);
```

### Применение shared memory

- **СУБД** (PostgreSQL, MySQL) — буферный пул, разделяемый между процессами.
- **Веб-серверы** — общий кэш ответов.
- **Игры** — между движком и рендером.
- **Высокопроизводительные вычисления** — большие массивы данных.
- **Inter-process video processing** — кадры в shared memory.

### Сигналы

**Сигнал** — асинхронное уведомление, посылаемое процессу. Аналог программного прерывания на уровне процесса.

Из tanenbaum.md:
> «Системный вызов `kill` позволяет пользователям и пользовательским процессам посылать сигналы. Если процесс готов принять определённый сигнал, то при его поступлении запускается обработчик сигнала».

### Виды сигналов

| Сигнал | Назначение | По умолчанию |
|---|---|---|
| `SIGHUP` | Закрытие терминала | Завершение |
| `SIGINT` | Ctrl+C | Завершение |
| `SIGQUIT` | Ctrl+\ | Завершение + core dump |
| `SIGILL` | Недопустимая инструкция | core dump |
| `SIGABRT` | abort() | core dump |
| `SIGFPE` | Деление на 0 / переполнение | core dump |
| `SIGKILL` | Уничтожение | Завершение (нельзя перехватить!) |
| `SIGSEGV` | Нарушение доступа к памяти | core dump |
| `SIGPIPE` | Запись в закрытый pipe | Завершение |
| `SIGALRM` | Будильник | Завершение |
| `SIGTERM` | Вежливая просьба | Завершение |
| `SIGCHLD` | Завершение ребёнка | Игнорирование |
| `SIGCONT` | Продолжить | Игнорирование |
| `SIGSTOP` | Остановить | Остановка (нельзя перехватить!) |
| `SIGTSTP` | Ctrl+Z | Остановка |
| `SIGUSR1`, `SIGUSR2` | Пользовательские | Завершение |

`SIGKILL` и `SIGSTOP` — **особые**: их нельзя перехватить, замаскировать или игнорировать. Это гарантирует, что система всегда может остановить или убить процесс.

### Обработка сигналов

Процесс регистрирует обработчик через `signal()` или `sigaction()`:

```c
#include <signal.h>

void my_handler(int sig) {
    printf("Получен сигнал %d\n", sig);
}

int main() {
    signal(SIGINT, my_handler);
    // или
    struct sigaction sa = { .sa_handler = my_handler, .sa_flags = SA_RESTART };
    sigaction(SIGINT, &sa, NULL);

    while (1) pause();
    return 0;
}
```

При Ctrl+C запустится `my_handler` вместо завершения.

### Реализация сигналов (rtos.md, 5.7)

В простейшем случае:
- **Битовая маска** ожидающих сигналов в `task_struct`.
- При отправке `kill(pid, SIGUSR1)` — соответствующий бит устанавливается.
- В моменты выхода из системных вызовов и обработчиков прерываний маска анализируется.
- Если есть ожидающий сигнал — **доставка**: модификация фрейма прерывания так, чтобы возврат в пользователя пошёл не на исходный код, а на обработчик сигнала.

Из rtos.md:
> «Доставка может быть произведена с помощью модификации фрейма из обработчика программного прерывания… Со временем, правда, обнаружился такой недостаток, что в случае использования битовой маски повторные сигналы теряются, поэтому в последующих редакциях стандарта POSIX были введены очереди сигналов».

**Очереди сигналов** (real-time signals, `SIGRTMIN`..`SIGRTMAX`):
- Сигналы представлены объектами в очереди.
- Множественные сигналы не теряются.
- Можно передавать данные с сигналом (`sigqueue()`).

### Сигналы и многопоточность

В многопоточном процессе:
- Сигнал может быть направлен **процессу** (`kill(pid, sig)`) — ядро выбирает один поток для доставки.
- Сигнал может быть направлен **конкретному потоку** (`pthread_kill(tid, sig)`).
- Каждый поток имеет свою **маску** заблокированных сигналов.
- Обработчики сигналов — общие для всего процесса.

### Особенности использования

**Что МОЖНО делать в обработчике сигнала:**
- Установить флаг (volatile sig_atomic_t).
- Вызвать **async-signal-safe** функции из узкого списка (`write`, `signal`, `_exit`).

**Что НЕЛЬЗЯ:**
- `malloc`, `printf`, `strtok` и большинство стандартных функций — они не реентерабельны.
- Брать мьютексы — может быть deadlock.
- Долго работать — другие сигналы заблокированы.

Современный паттерн — `signalfd()`: получение сигналов через файловый дескриптор, обработка в обычном цикле событий:

```c
int sfd = signalfd(-1, &mask, 0);
struct signalfd_siginfo info;
read(sfd, &info, sizeof(info));
// info.ssi_signo — номер сигнала
```

### Резюме

- IPC — механизмы взаимодействия между изолированными процессами.
- Shared memory — самая быстрая форма; требует ручной синхронизации.
- POSIX (`shm_open`/`mmap`) и System V (`shmget`/`shmat`).
- Сигналы — асинхронные уведомления, аналог прерываний для процессов.
- Обработчик регистрируется через `sigaction`.
- `SIGKILL`/`SIGSTOP` не перехватываются — гарантия управляемости.
- Очереди сигналов в POSIX RT исключают потерю повторных сигналов.

## 40. Синхронизация через разделяемые объекты. Асинхронный обмен сообщений.
*Источник: rtos.md, Глава 6 «События»; общие принципы*

### Прямое уведомление vs разделяемые объекты

**Прямое уведомление:** отправитель явно указывает, кому шлёт. Например, `pthread_kill(thread_id, signal)` — конкретному потоку.
- Простое.
- Жёсткая связанность: отправитель должен знать получателя.

**Через разделяемый объект:** есть «третья сущность» — объект синхронизации. Отправитель и получатель работают с ним, не зная друг о друге.
- Слабая связанность.
- Гибкость: несколько отправителей, несколько получателей.
- Возможность анонимности.

### Примеры разделяемых объектов

| Объект | Что хранит | Кто работает |
|---|---|---|
| **Семафор** | Счётчик | Производители (V), потребители (P) |
| **Мьютекс** | Состояние lock/unlock | Захватчики и освобождающие |
| **Событие (event flag)** | Флаг(и) | Сигнализирующие, ждущие |
| **Условная переменная** | (нет состояния) | Сигнализирующие, ждущие |
| **Очередь сообщений** | Очередь сообщений | Отправители (put), получатели (get) |
| **Канал (pipe)** | Буфер байт | Писатели, читатели |

### Преимущества модели «через объект»

**1. Слабая связанность.** Отправитель и получатель не зависят друг от друга. Можно менять одну сторону без изменения другой.

**2. Гибкость количества участников.** На одном объекте могут работать произвольное число отправителей и получателей.

**3. Возможность буферизации.** Объект может хранить данные между моментами отправки и получения.

**4. Анонимность.** Производитель не знает, кто потребитель, и наоборот. Полезно для модулей, разрабатываемых независимо.

**5. Естественная очерёдность.** Если несколько потоков ждут одно событие, очередь определяет, кто получит первым.

### Асинхронный обмен сообщений

**Асинхронный обмен** — отправитель посылает сообщение и **продолжает работу**, не дожидаясь, пока получатель его прочитает. Сообщение хранится в **очереди**.

```
                  Очередь
Producer  →  [M1] [M2] [M3]  →  Consumer
            ↓                  ↑
            обмен сообщений    приём
```

### Свойства асинхронной очереди

- **Емкость:** конечный максимум сообщений.
- **Поведение при полной очереди:**
  - Блокировка отправителя (он будет ждать места).
  - Возврат ошибки (отправитель должен обработать).
  - Отбрасывание старых сообщений (lossy queue).
  - Отбрасывание нового сообщения.
- **Поведение при пустой очереди:**
  - Блокировка получателя (он ждёт сообщения).
  - Возврат «нет сообщения».
- **Порядок:** обычно FIFO, но возможно с приоритетами.

### Производитель-потребитель

Классическая задача (Tanenbaum 2.5):
- **Производитель** генерирует данные и кладёт в очередь.
- **Потребитель** забирает из очереди и обрабатывает.

Без синхронизации возникают проблемы:
- Производитель пишет в полную очередь → перезапись.
- Потребитель читает из пустой → мусор.
- Параллельный доступ → коррупция структуры.

Решается двумя семафорами и мьютексом — см. вопрос 66.

### Реализация очереди сообщений

```c
typedef struct {
    void *buffer[N];      // массив указателей на сообщения
    int head, tail;       // индексы (циклический буфер)
    int count;            // текущее число элементов
    semaphore_t empty;    // initial = N (число пустых ячеек)
    semaphore_t full;     // initial = 0 (число занятых)
    mutex_t lock;         // защита от параллельного доступа
} msg_queue_t;

void send(msg_queue_t *q, void *msg) {
    sem_wait(&q->empty);      // подождать место (P)
    mutex_lock(&q->lock);

    q->buffer[q->head] = msg;
    q->head = (q->head + 1) % N;
    q->count++;

    mutex_unlock(&q->lock);
    sem_post(&q->full);       // уведомить о новом сообщении (V)
}

void *recv(msg_queue_t *q) {
    sem_wait(&q->full);       // подождать сообщение (P)
    mutex_lock(&q->lock);

    void *msg = q->buffer[q->tail];
    q->tail = (q->tail + 1) % N;
    q->count--;

    mutex_unlock(&q->lock);
    sem_post(&q->empty);      // освободить место (V)
    return msg;
}
```

Здесь:
- `empty` гарантирует, что есть место (отправитель не пишет в полную).
- `full` гарантирует, что есть сообщение (получатель не читает из пустой).
- `lock` защищает от параллельного изменения индексов несколькими отправителями или получателями.

### POSIX message queues

```c
#include <mqueue.h>

// Создание
struct mq_attr attr = { .mq_maxmsg = 10, .mq_msgsize = 1024 };
mqd_t mq = mq_open("/myq", O_CREAT | O_RDWR, 0666, &attr);

// Отправка
mq_send(mq, "Hello", 5, 0);

// Получение
char buf[1024];
mq_receive(mq, buf, sizeof(buf), NULL);

mq_close(mq);
mq_unlink("/myq");
```

### System V message queues

```c
#include <sys/msg.h>

struct my_msg { long mtype; char mtext[128]; };

int msqid = msgget(KEY, IPC_CREAT | 0666);

struct my_msg m = { .mtype = 1, .mtext = "Hello" };
msgsnd(msqid, &m, sizeof(m.mtext), 0);

msgrcv(msqid, &m, sizeof(m.mtext), 1, 0);

msgctl(msqid, IPC_RMID, NULL);
```

### Применение асинхронного обмена

**1. Веб-серверы.** Входящие HTTP-запросы → очередь → пул рабочих потоков. Запросы не теряются при пиках нагрузки.

**2. Логирование.** Потоки складывают записи в очередь, фоновый поток пишет в файл. Основная работа не блокируется I/O.

**3. Игровые движки.** События пользователя (ввод, сеть) → очередь → главный цикл обрабатывает по одному.

**4. Микросервисы.** RabbitMQ, Kafka — асинхронные очереди между сервисами. Сервис-отправитель не зависит от доступности сервиса-получателя.

**5. GUI.** Очередь событий: клик пользователя, перерисовка, таймер — всё в очереди, цикл обработки ECP вызывает обработчики.

**6. Producer-consumer паттерн** в многопоточных приложениях.

### Сравнение с другими формами IPC

| Метод | Скорость | Связанность | Буфер | Подходит для |
|---|---|---|---|---|
| Shared memory | Самая высокая | Высокая | Можно реализовать | Большие данные, частые обмены |
| Pipe | Средняя | Жёсткая (родственные процессы) | Есть (4 КБ) | Поток данных |
| Socket | Средняя | Слабая | Есть | Сеть, локально |
| Message queue (асинх.) | Средняя | Слабая | Явный | Сообщения |
| Rendezvous (синх.) | Низкая | Слабая | Нет | Жёсткая синхронизация |

### Резюме

- Через разделяемые объекты — слабая связанность, гибкость, буферизация.
- Асинхронный обмен — отправитель не блокируется (если место есть), сообщение в очереди.
- Реализация через два семафора (`empty`, `full`) + мьютекс.
- POSIX и System V — стандартные API.
- Применение: веб-серверы, логирование, игры, микросервисы, GUI.
