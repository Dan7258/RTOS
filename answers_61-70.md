## 61. Мьютексы. Повторный захват мьютекса.
*Источник: rtos.md, Глава 6.8 «Мьютекс»; robert_lav.md, Глава 10 «Мьютексы»*

**Мьютекс (mutex, mutual exclusion)** — примитив синхронизации для взаимного исключения. Работает как бинарный семафор, но имеет важное отличие — **владельца**.

Из rtos.md (6.8):
> «Работает практически так же, как и бинарный семафор и используется для задач взаимного исключения, но есть и одно важное отличие от семафора — у мьютекса есть "владелец", тот поток, который находится в критической секции».

### Интерфейс

```c
// Источник: rtos.md, раздел 6.8
int mutex_timedacquire(mutex_t* mutex, uint32_t timeout);
int mutex_release(mutex_t* mutex);
```

### Отличия от семафора

1. **Освобождать может только владелец.** Семафор может «отпустить» кто угодно — мьютекс только тот, кто захватил.
2. **Защита от инверсии приоритетов.** Известный владелец позволяет реализовать priority inheritance / ceiling.
3. **Только для взаимного исключения.** Не для нотификаций.
4. **Нельзя использовать из обработчиков прерываний.** У них нет потока-владельца.

### Повторный захват мьютекса

**Проблема:** что, если поток, уже владеющий мьютексом, попытается захватить его снова?

Возникает в сложных системах, где функции с CS вызывают друг друга.

Из rtos.md:
> «В случае с семафором, если он используется для взаимного исключения, поток может, находясь внутри критической секции, вновь попытаться его подождать, что, разумеется, приведёт к взаимоблокировке».

С мьютексом ситуация особая — никто не может освободить его, кроме владельца, который сам заблокирован.

### Два подхода

**1. Нерекурсивный (ошибка).**

Возврат ошибки при повторном захвате тем же потоком. Приложение получает возможность обнаружить проблему.

**2. Рекурсивный.**

Из rtos.md:
> «Второй вариант увеличивает внутренний счётчик вложения мьютекса, таким образом, его освобождение произойдёт тогда, когда поток-владелец вызовет операцию освобождения столько раз, сколько раз была вызвана операция захвата».

```c
mutex_lock(&m);    // counter = 1
mutex_lock(&m);    // counter = 2 (тот же поток)
// ... вложенная CS ...
mutex_unlock(&m);  // counter = 1
mutex_unlock(&m);  // counter = 0 — действительно освобождён
```

### Преимущества и недостатки рекурсивных мьютексов

**Плюсы:** удобство для рекурсивных вызовов, не нужно специально следить за вложением.

**Минусы:**
- Скрывают плохой дизайн.
- Сложнее реализация.
- Сложнее отладка race conditions.

В Linux **обычные мьютексы НЕ рекурсивные**. В POSIX можно создать рекурсивный через атрибут `PTHREAD_MUTEX_RECURSIVE`.

### Завершение потока с мьютексом

Из rtos.md:
> «Если поток явно не освободит мьютекс, значит никакой другой поток не сможет его захватить и система окажется в заблокированном состоянии».

Подходы:
- Хранить список захваченных мьютексов в TCB и освобождать при завершении.
- Оставлять мьютекс заблокированным (безопаснее для данных).

### Применение

- Защита данных в многопоточных программах.
- Защита структур ядра в Linux (`mutex_lock` / `mutex_unlock`).
- Альтернатива семафорам со счётчиком 1.

### Ограничения мьютексов в Linux

Из robert_lav.md (Глава 10):
- В один момент только одна задача удерживает.
- Освободить может только захвативший.
- Нельзя рекурсивно (в Linux ядре).
- Процесс не может завершиться без освобождения.
- Нельзя в обработчиках прерываний.

## 62. Мониторы. Отличие мьютекса от семафора. Отличие мониторов от семафоров и мьютексов.
*Источник: rtos.md, Глава 6.9 «Мониторы»; tanenbaum.md*

### Отличие мьютекса от семафора

| Параметр | Мьютекс | Семафор |
|---|---|---|
| Владелец | Есть | Нет |
| Освобождает | Только владелец | Кто угодно |
| Применение | Только взаимное исключение | Взаимное исключение + уведомления |
| Защита от приоритетной инверсии | Часто да | Нет |
| Использование в IRQ | Нет | Нет |
| Рекурсивность | Возможна | Нет |
| Накладные расходы | Больше | Меньше |
| Стандарт | POSIX | POSIX, SysV |

Из robert_lav.md:
> «Всегда используйте мьютексы, а не семафоры, если перечисленные выше ограничения для мьютексов вам подходят. При написании нового кода семафоры используются только в очень специфических (часто низкоуровневых) модулях».

### Что такое монитор

Предложен Hoare/Brinch Hansen в 1974. Из rtos.md (6.9):
> «Монитором считается не просто объект, наподобие семафора, а совокупность следующих сущностей: набор разделяемых данных и функций, при выполнении которых обеспечивается взаимное исключение, а также один или более особых объектов, называемых условными переменными. То есть, в терминах ООП, монитор это класс, для методов которого обеспечивается взаимное исключение».

Монитор = **класс** + **автоматическое взаимное исключение** для всех его методов + **условные переменные** для условной синхронизации.

### Условные переменные

```c
int cond_timedwait(cond_t* cond, mutex_t* mutex, uint32_t timeout);
int cond_signal(cond_t* cond);
int cond_broadcast(cond_t* cond);
```

Из rtos.md:
> «Это отличие заключается в том, что она, во-первых, должна использоваться только в критической секции, во-вторых — ожидание условной переменной приводит к тому что поток помещается в очередь, затем разблокируется мьютекс и только после этого поток блокируется».

### Принцип работы

1. Поток входит в критическую секцию (захватывает мьютекс).
2. Если условие не выполнено, вызывает `cond_wait(&cv, &mutex)`:
   - Атомарно: помещает себя в очередь ожидания + освобождает мьютекс.
   - Блокируется.
3. Другой поток в CS изменяет состояние и вызывает `cond_signal(&cv)`.
4. Поток просыпается, заново захватывает мьютекс, продолжает.

### Различия

| Параметр | Семафор | Мьютекс | Монитор |
|---|---|---|---|
| Внутреннее состояние | Счётчик | Флаг + владелец | Нет (состояние в коде) |
| Универсальность | Высокая | Только mutex | Высокая |
| Поддержка языка | Не нужна | Не нужна | Желательна |
| Используется в | Любые языки | Любые языки | Java, C# |

### Пример реализации семафора через монитор

```c
// Источник: rtos.md, раздел 6.9
cond_t condition_variable;
mutex_t mutex;
uint32_t semaphore;

// sem_wait через монитор
mutex_timedacquire(&mutex, ...);
do {
    if (semaphore > 0) {
        --semaphore;
        break;
    } else {
        cond_timedwait(&condition_variable, &mutex, ...);
    }
} while (1);
mutex_release(&mutex);

// sem_post через монитор
mutex_timedacquire(&mutex);
++semaphore;
cond_signal(&condition_variable);
mutex_release(&mutex);
```

### Достоинства монитора

Из rtos.md:
> «Монитор это как бы "вывернутый наизнанку" примитив, состояние которого определяется программой, а не заключено внутри объекта в качестве его состояния».

- **Состояние определяется задачей**, а не фиксировано примитивом.
- **Универсальность** — можно реализовать любой механизм синхронизации.
- **Удобство** — встроено в Java, C#.

### Поддержка в языках

В Java каждый объект имеет встроенный мьютекс:
```java
synchronized void method() {
    // CS
    object.wait();      // условная переменная
    // ...
    object.notify();
}
```

В C#:
```csharp
lock(obj) {
    Monitor.Wait(obj);
    Monitor.Pulse(obj);
}
```

В C мониторы реализуются вручную через мьютекс + condvar.

### broadcast

Из rtos.md:
> «Некоторые реализации условной переменной содержат также функцию broadcast, вместе с signal, которая вызывает пробуждение всех ожидающих потоков, а не только одного».

Полезно, когда не любой ожидающий поток может продолжить (например, ждут разной памяти).

## 63. Условные переменные. Самоблокировка. Пример возникновения.
*Источник: rtos.md, Глава 6.9; общие принципы синхронизации*

### Что такое условная переменная

**Условная переменная (CV, condition variable)** — это объект, позволяющий потоку «уйти спать» внутри критической секции, пока другой поток не изменит общее состояние. Используется **всегда в паре с мьютексом**.

Зачем она нужна: ты зашёл под мьютексом, увидел, что продолжать нельзя (буфер пуст, ресурс занят и т.п.) — нужно дождаться, пока кто-то это исправит. Просто крутиться в цикле нельзя: ты держишь мьютекс, никто не зайдёт и не изменит. Просто отпустить мьютекс и заснуть — можно пропустить сигнал между «отпустить» и «заснуть».

CV решает это **одной атомарной операцией**: отпустить мьютекс и заснуть в одном неделимом действии.

### Три операции

| Операция | Что делает |
|---|---|
| `cond_wait(cv, mutex)` | Атомарно: отпускает `mutex` + усыпляет поток. Когда разбудят — снова захватывает `mutex`. |
| `cond_signal(cv)` | Будит **одного** ждущего на `cv`. |
| `cond_broadcast(cv)` | Будит **всех** ждущих на `cv`. |

### Базовый паттерн

```c
// Сторона, ожидающая условие:
mutex_lock(&mtx);
while (!condition)                  // ← while, не if!
    cond_wait(&cv, &mtx);
/* условие выполнено, работаем */
mutex_unlock(&mtx);

// Сторона, меняющая условие:
mutex_lock(&mtx);
condition = true;
cond_signal(&cv);
mutex_unlock(&mtx);
```

Почему `while` обязательно: между моментом, когда поток получил сигнал, и моментом, когда он реально захватил мьютекс и продолжил, **состояние могло снова измениться**. Например, другой поток успел вклиниться и опять «потратил» ресурс. Перепроверяем.

### Главное про CV — три ключевые особенности

**1. CV не помнят сигналы.** Если `signal` пришёл, когда никто не ждёт — он улетит в никуда. У семафора есть счётчик, у CV нет. Поэтому состояние всегда хранится в общей переменной (`condition`), а CV — только механизм пробуждения.

**2. CV работают только в паре с мьютексом.** Без него проверка условия и засыпание не были бы атомарны, и сигнал терялся бы между ними.

**3. Возможны spurious wakeup** — пробуждение без причины. POSIX это разрешает (бывает из-за внутренних оптимизаций, прихода UNIX-сигналов и т.п.). Поэтому условие проверяем в `while`.

### Самоблокировка (self-deadlock) — главная тема вопроса

**Самоблокировка** возникает, когда поток ждёт условия, которое **никто не может выполнить**. Поток засыпает на `cond_wait` и не просыпается никогда. Это не классический deadlock с двумя мьютексами — это «зависание на пустом месте», и его сложнее заметить.

### Пример 1 — ждать сообщений, когда отправителей нет

```c
mutex_lock(&mtx);
while (queue_is_empty())
    cond_wait(&not_empty, &mtx);    // ждём, что кто-то положит
item = queue_pop();
mutex_unlock(&mtx);
```

Сам по себе код правильный. Но если в системе **нет производителей** (например, тот, кто должен был класть, упал с ошибкой, или забыли его запустить, или он уже завершился) — потребитель будет ждать **вечно**. Никто никогда не вызовет `cond_signal(&not_empty)`.

Это самоблокировка через **отсутствие условия наступления события**.

### Пример 2 — циклическая зависимость

```c
// Поток A:                         // Поток B:
mutex_lock(&mtx);                   mutex_lock(&mtx);
while (!event_b)                    while (!event_a)
    cond_wait(&cv_a, &mtx);             cond_wait(&cv_b, &mtx);
event_a = true;                     event_b = true;
cond_signal(&cv_b);                 cond_signal(&cv_a);
mutex_unlock(&mtx);                 mutex_unlock(&mtx);
```

Поток A ждёт `event_b`. Поток B ждёт `event_a`. Каждый мог бы сигналить «свой» событие, но **сначала** должен дождаться «чужого». Циклическая зависимость — оба ждут навсегда.

### Пример 3 — поток ждёт самого себя

```c
void some_function() {
    mutex_lock(&mtx);
    while (!data_ready)
        cond_wait(&data_cv, &mtx);
    /* ... */
    prepare_data();              // именно это должно было поставить data_ready
    data_ready = true;
    mutex_unlock(&mtx);
}
```

В системе только один поток, и условие `data_ready` устанавливает он сам — но **после** ожидания. Поток засыпает в `cond_wait` и никогда не доходит до `prepare_data`. Классический «ждёт сигнала от самого себя».

### Как избежать самоблокировки

1. **Перед `cond_wait` убедиться, что есть кто-то, кто потенциально может сигналить.** Если нет производителей — не ждать, а возвращать ошибку или признак «больше не будет».
2. **Использовать таймауты:** `cond_timedwait` вместо бесконечного `cond_wait`. Если за разумное время не дождались — продолжать как-то иначе.
3. **«Завершающий сигнал».** При остановке подсистемы — установить флаг «всё, больше работы не будет» и сделать `cond_broadcast`. Ждущие проснутся, проверят флаг и выйдут.
4. **Аккуратно проектировать «кто кого сигналит»** — нарисовать граф: «поток X меняет условие Y, ждущие на Z просыпаются». Если в графе нашёлся узел без входящих стрелок — там самоблокировка.

### Связанные грабли — потерянное пробуждение

Если делать сигнализацию **без мьютекса** или не использовать `while` — сигнал может прийти между проверкой условия и `cond_wait`, и поток заснёт навсегда (lost wakeup):

```c
// ПЛОХО — сигнализация без мьютекса:
condition = true;
cond_signal(&cv);   // может прийти, пока другой между if и wait
```

Поэтому **и проверка, и сигнал должны быть под одним мьютексом** — это закрывает гонку.

## 64. Отключение мультипрограммного режима работы ядра.
*Источник: robert_lav.md, Глава 10 «Отключение мультипрограммного режима работы ядра»*

### Что такое «мультипрограммный режим работы ядра»

**Мультипрограммный режим (вытесняемое ядро, preemptible kernel)** — это свойство ядра, при котором **поток, выполняющийся в режиме ядра, может быть в любой момент прерван планировщиком и заменён другим потоком**. То есть «вытеснение» работает не только для пользовательского кода, но и внутри самого ядра.

Раньше (до Linux 2.6) ядро было **невытесняемым**: пока поток выполнял системный вызов или работал в ядерном коде, планировщик не мог его прервать — нужно было дождаться возврата в пользовательский режим. Это упрощало написание ядра (внутри ядра не нужны были многие блокировки), но ухудшало отзывчивость: высокоприоритетный поток мог ждать, пока низкоприоритетный закончит долгий системный вызов.

С версии 2.6 Linux ввёл **вытесняемое ядро**. Теперь:

> «Поскольку ядро системы Linux является вытесняемым, выполнение некоторого процесса в ядре может быть приостановлено в произвольный момент времени и управление передано более высокоприоритетному процессу».

Это улучшило отзывчивость, но создало проблему: код ядра должен быть готов к тому, что его прервут в произвольной точке. Если в эту точку он работает с общими данными — будет гонка. Поэтому появилась необходимость **временно отключать** этот режим в коротких критических участках.

### Не путать с другими видами защиты

| Уровень | Что блокирует |
|---|---|
| `preempt_disable()` | **Только вытеснение** на текущем CPU. Прерывания работают, другие ядра работают. |
| `local_irq_disable()` | Прерывания на текущем CPU. Вытеснение тоже становится невозможным (некому переключать). |
| `spin_lock()` | + защита от других CPU |

«Отключение мультипрограммного режима» — это самый лёгкий уровень: запрещаем планировщику переключить нас на другой поток, но прерывания продолжают работать.

### Когда нужно отключать вытеснение

**1. Работа с данными процессора (per-CPU data).**

Если каждый CPU имеет свои данные — блокировка не нужна. Но если в середине работы мы будем вытеснены, и другой поток на том же CPU начнёт работать с теми же данными — будет конфликт.

Из robert_lav.md:
> «Чаще всего такая ситуация возникает из-за доступа к данным, привязанным к определенному процессору (per-processor data). Если каждый процессор обращается только к своим уникальным данным, то защищать их с помощью блокировок нет никакой необходимости. Дело в том, что в подобных случаях к этим данным может обратиться только один процессор. Однако если никакая из блокировок не удерживается, ядро является вытесняемым».

**2. Внутри спинлоков** (неявно).

**3. Перед операциями, которые должны атомарно завершиться.**

### Функции

```c
preempt_disable();    // отключить вытеснение
// CS
preempt_enable();     // включить
preempt_enable_no_resched();  // включить, но не планировать
preempt_count();      // текущая глубина запрета
```

Внутри — счётчик `preempt_count` в `thread_info`:
- При `disable` инкремент.
- При `enable` декремент.
- Планировщик проверяет: если ≠ 0 → не переключать.

### Пример

```c
int cpu_var;

// Без preempt_disable — race condition:
//   1. CPU 0: читает cpu_var[0]
//   2. Вытеснение
//   3. Новый поток на CPU 0: пишет в cpu_var[0]
//   4. Старый поток возвращается, использует устаревшее значение

preempt_disable();
int my_cpu = smp_processor_id();
cpu_var[my_cpu] += 1;
preempt_enable();
```

### BKL (Big Kernel Lock) — историческое

Из robert_lav.md:
> «Большая блокировка ядра (Big Kernel Lock, или BKL) — это глобальная спин-блокировка, которая была создана специально для того, чтобы облегчить переход от первоначальной реализации SMP в операционной системе Linux к мелкоструктурным блокировкам».

Свойства BKL:
- Глобальная — одна на всё ядро.
- Рекурсивная — можно захватывать несколько раз.
- Автоматически освобождается при переходе в sleep.
- Только в контексте процесса.

```c
lock_kernel();
// весь код ядра защищён
unlock_kernel();
```

История: ядро 2.0 разрешало только одну задачу в ядре одновременно. Переход к параллельному ядру (2.2+) сделали через BKL, постепенно заменяя на мелкоструктурные блокировки. В новых ядрах BKL **удалён**.

### preempt_disable vs spin_lock

| Действие | preempt_disable | spin_lock |
|---|---|---|
| Защита от вытеснения на нашем CPU | Да | Да (неявно) |
| Защита от других CPU | Нет | Да |
| Защита от прерываний | Нет | Нет |
| Накладные расходы | Минимум | Низкие |
| Use case | Per-CPU data | Общие данные |

### Применение

- **Per-CPU переменные:** `get_cpu()` неявно отключает вытеснение, `put_cpu()` включает.
- **RCU read-side critical sections.**
- **Внутри обработчиков прерываний** (неявно).
- **Перед операциями, требующими `current` оставаться неизменным.**

## 65. Порядок выполнения операций и барьеры.
*Источник: rtos.md, Глава 2.2.5 «Модель памяти»*

### Проблема порядка операций

Современные процессоры применяют оптимизации, нарушающие очевидный порядок:
- Store buffer (буфер записи).
- Write-back кэш.
- Спекулятивное исполнение.
- Переупорядочивание инструкций компилятором и CPU.

Из rtos.md (2.2.5):
> «С точки зрения самого процессора, он поддерживает последовательность обращений, то есть при попытке чтения значения, записанного во write buffer, оно будет прочитано оттуда, а не из кэша. То есть обеспечивается выполнение алгоритма таким образом, что все оптимизации происходят прозрачно. Но другие процессоры в системе не имеют доступа к буферам сохранения, а значит, видят последовательность обращений такой, какой она появляется на внешней шине».

### Классический пример

```c
// Поток-производитель:
global_struct = malloc(...);
init_struct(global_struct);
ready_flag = 1;

// Поток-потребитель:
if (ready_flag) {
    use_struct(global_struct);   // данные ещё не записаны!
}
```

CPU может переупорядочить операции:
- Записать `ready_flag = 1` **до** инициализации структуры.
- Другие CPU увидят `ready_flag = 1`, но `global_struct` ещё не готов.

### Модели памяти

- **Sequentially consistent (последовательная согласованность):** все обращения в том порядке, как в программе. Идеал, но медленно.
- **Total Store Order (TSO):** x86. Записи могут переупорядочиваться с чтениями, но не друг с другом.
- **Relaxed:** ARM, POWER. Любые переупорядочивания возможны.

### Барьеры памяти

Специальные инструкции, **запрещающие переупорядочивание** в определённой точке.

Из rtos.md:
> «Для избежания таких ситуаций должны использоваться барьеры памяти — специальные инструкции, которые вводят ограничения на то, как и какие оптимизации процессор может применять в данном участке кода».

### x86

- **`LFENCE`** — завершение всех чтений из памяти.
- **`SFENCE`** — завершение всех записей.
- **`MFENCE`** — завершение всех обращений.

### ARM

- **`DMB`** — Data Memory Barrier.
- **`DSB`** — Data Synchronization Barrier.
- **`ISB`** — Instruction Synchronization Barrier.

### Использование в коде

```c
// Источник: rtos.md, раздел 2.2.5
global_struct = malloc(...);
init_struct(global_struct);
mem_barrier();      // барьер!
ready_flag = 1;

// Другая сторона:
flag = ready_flag;
mem_barrier();      // барьер!
if (flag) {
    use_struct(global_struct);
}
```

### В Linux

```c
mb();           // полный барьер
rmb();          // read barrier
wmb();          // write barrier
smp_mb();       // SMP-only
smp_rmb();
smp_wmb();

barrier();      // только компилятор (asm volatile)
```

### Применение

- **Передача данных** между потоками.
- **Реализация мьютексов, спинлоков, condvar.**
- **Lock-free алгоритмы.**
- **MMIO-устройства** — отображённые на память регистры требуют строгого порядка.

### Особо для встраиваемых устройств

Из rtos.md:
> «Особую важность этот вопрос имеет при использовании отображённых на память устройств, которые требуют определённой последовательности доступа к своим регистрам».

Регистры устройств чувствительны к порядку — нужны барьеры или отключение кэширования.

## 66. Задача производителей и потребителей. Пример на семафорах. Объяснить почему две переменные.
*Источник: tanenbaum.md, Глава 2.3.4-2.3.5*

### Постановка

> «Два процесса используют общий буфер фиксированного размера. Один из них, производитель, помещает информацию в буфер, а другой, потребитель, извлекает её оттуда».

Буфер на N мест. Производитель кладёт элементы, потребитель забирает. Правила:
1. Буфер **полон** → производитель ждёт, пока появится место.
2. Буфер **пуст** → потребитель ждёт, пока появятся данные.
3. Производитель и потребитель **не лезут в буфер одновременно** — иначе гонка.

### Аналогия — конвейерная лента на N деталей

Сборщик кладёт детали на ленту, упаковщик снимает. Лента полна → сборщик стоит. Лента пуста → упаковщик стоит. Двое не тянутся к одному месту одновременно.

### Решение использует три примитива

| Примитив | Зачем |
|---|---|
| **mutex** | Чтобы только один поток за раз работал с буфером (защита от гонки) |
| **empty** (семафор) | Счётчик свободных мест, начинает с N. Производитель ждёт на нём. |
| **full** (семафор) | Счётчик занятых мест, начинает с 0. Потребитель ждёт на нём. |

Здесь семафоры — не блокировки, а **счётчики того, сколько чего есть**. `empty` — сколько свободных мест осталось. `full` — сколько готовых данных лежит.

### Код (Tanenbaum, листинг 2.6)

```c
#define N 100
semaphore mutex = 1;    // взаимное исключение
semaphore empty = N;    // пустые ячейки
semaphore full  = 0;    // заполненные ячейки

void producer(void) {
    while (TRUE) {
        item = produce_item();
        down(&empty);       // 1. жду свободное место
        down(&mutex);       // 2. захожу в буфер эксклюзивно
        insert_item(item);
        up(&mutex);         // 3. выхожу
        up(&full);          // 4. сигналю: появилось готовое
    }
}

void consumer(void) {
    while (TRUE) {
        down(&full);        // 1. жду готовых данных
        down(&mutex);       // 2. захожу в буфер эксклюзивно
        item = remove_item();
        up(&mutex);         // 3. выхожу
        up(&empty);         // 4. сигналю: освободилось место
        consume_item(item);
    }
}
```

### Как это работает по шагам

Начало: `empty = 100`, `full = 0`, буфер пуст.

- **Производитель кладёт:** `down(&empty)` → `empty = 99`. Зашёл под мьютексом, положил, вышел. `up(&full)` → `full = 1`. Если потребитель спит — будит.
- **Буфер пуст, пришёл потребитель:** `down(&full)` → счётчик `0`, идти некуда → засыпает.
- **Производитель кладёт, потребитель спит:** `up(&full)` → `full = 1`, потребитель просыпается.
- **Буфер полон (100 элементов):** `empty = 0`, `full = 100`. Производитель приходит → `down(&empty)` засыпает, пока потребитель не освободит.

Получается **симметричная пара**: производитель ждёт на `empty`, потребитель — на `full`. Каждый сам будит другого после своих действий.

### Зачем две переменные

Главный вопрос вопроса — почему `empty` и `full` отдельно, не один счётчик?

Потому что **производителю и потребителю нужны противоположные условия**:
- Производитель проходит, когда есть **свободные** места.
- Потребитель проходит, когда есть **занятые** места.

Один счётчик не может означать сразу два разных события. `down` всегда «жду, пока что-то не уменьшится», а ждать им нужно **разные вещи**: производителю — наличие свободных, потребителю — наличие занятых.

Геометрически: представь, как двигать ползунок от 0 до N. Производитель «ждёт ползунка слева от N», потребитель «ждёт ползунка справа от 0». Эти два условия проверяются разными счётчиками: `empty` ползёт от N к 0 (по числу свободных мест), `full` — от 0 к N (по числу занятых). В любой момент они связаны инвариантом `empty + full = N`.

### Зачем mutex отдельно

Семафоры считают «сколько мест есть», но **не защищают сам буфер** от одновременной модификации двумя производителями (или двумя потребителями). `mutex` нужен именно для этого — чтобы внутри ветки «положить в буфер» / «забрать из буфера» был только один поток.

Если в системе **по одному** производителю и потребителю — теоретически `mutex` можно убрать (классический lock-free кольцевой буфер). Но как только их больше, без mutex'а двое влезут одновременно и испортят структуру.

### Знаменитая ловушка — порядок захвата

**Сначала семафор ожидания ресурса, потом mutex.** Если поменять местами:

```c
// ОШИБКА:
down(&mutex);
down(&empty);     // буфер полон → засыпаю под мьютексом!
```

1. Производитель захватил mutex.
2. Буфер полон → производитель засыпает на `down(&empty)`, **держа mutex**.
3. Потребитель приходит, делает `down(&full)` — проходит, но `down(&mutex)` — заблокирован.
4. Никто не может ничего сделать — **дедлок**.

> «Одна малозаметная ошибка — и всё будет остановлено. Это напоминает программирование на ассемблере, но здесь ситуация ещё хуже».

### Применение в реальности

Producer-consumer — фундаментальный паттерн:
- Веб-серверы: запросы в очередь, обрабатываются воркерами.
- Логирование: writer кладёт строки, отдельный поток сбрасывает на диск.
- Сетевые стеки: пакеты от драйвера к обработчику протокола.
- Pipe в Unix.
- Channels в Go.

## 67. Задача писателей и читателей. Пример на семафорах.
*Источник: tanenbaum.md, Глава 2.5.2*

### Постановка

> «Другая общеизвестная задача касается читателей и писателей (Courtois et al., 1971). В ней моделируется доступ к базе данных. Представим, к примеру, систему бронирования авиабилетов, в которой есть множество соревнующихся процессов, желающих обратиться к ней для чтения и записи. Вполне допустимо наличие нескольких процессов, одновременно считывающих информацию из базы данных, но если один процесс обновляет базу данных (проводит операцию записи), никакой другой процесс не может получить доступ к базе данных даже для чтения информации».

Правила доступа к общим данным:
1. **Сколько угодно читателей** одновременно — можно (они не мешают друг другу).
2. **Один писатель** — можно, но в это время никаких других читателей или писателей.
3. **Читатель + писатель одновременно** — нельзя.

Это отличается от обычного мьютекса, который пускает только одного независимо от типа операции. Здесь смысл — **читателям не нужна эксклюзивность, потому что они ничего не меняют**.

### Идея решения

«Хитрость» в том, что читатели **сами** управляют блокировкой БД:
- Первый читатель, который заходит, захватывает БД (закрывает её от писателей).
- Все последующие читатели проходят без блокировки — БД уже их.
- Последний выходящий читатель отпускает БД.
- Писатель — просто захватывает БД эксклюзивно.

Для этого нужен счётчик активных читателей и защита самого счётчика от гонок.

### Решение (Tanenbaum, листинг 2.15)

```c
semaphore mutex = 1;    // защита счётчика rc
semaphore db    = 1;    // эксклюзивный доступ к БД
int rc = 0;             // сколько читателей сейчас читает

void reader(void) {
    while (TRUE) {
        down(&mutex);                    // вход в счётчик
        rc++;
        if (rc == 1) down(&db);          // первый читатель — захватить БД
        up(&mutex);                      // выход из счётчика

        read_data_base();                // ← читаем (могут одновременно)

        down(&mutex);                    // вход в счётчик
        rc--;
        if (rc == 0) up(&db);            // последний читатель — отпустить
        up(&mutex);

        use_data_read();
    }
}

void writer(void) {
    while (TRUE) {
        think_up_data();
        down(&db);                       // эксклюзивный доступ
        write_data_base();
        up(&db);
    }
}
```

### Зачем здесь два семафора

- **`mutex`** защищает только **счётчик `rc`** — крошечную критическую секцию вокруг инкремента/декремента. Без него двое читателей могли бы одновременно сделать `rc++` и потерять одно значение.
- **`db`** — это собственно блокировка БД, разделяемая между читателями и писателями.

Два разных назначения — две разных блокировки.

### Как это работает по шагам

**Сценарий 1: один читатель.**
- `mutex` → `rc = 1` → захватил `db` → `mutex` отпустил.
- Читает.
- `mutex` → `rc = 0` → отпустил `db` → `mutex` отпустил.

**Сценарий 2: три читателя по очереди заходят.**
- Читатель 1: `rc = 1`, захватил `db`.
- Читатель 2: `rc = 2`, `db` уже наш, **не делает `down(&db)`** (важно — иначе сам себя заблокировал бы).
- Читатель 3: `rc = 3`.
- Все читают одновременно.
- Поочерёдно выходят, последний (когда `rc == 0`) отпускает `db`.

**Сценарий 3: читатель и пришедший писатель.**
- Читатель уже читает, держит `db`.
- Писатель: `down(&db)` → блокируется.
- Читатель выходит, `rc = 0`, `up(&db)` → писатель просыпается, начинает писать.

### Проблема: голодание писателей

Из tanenbaum.md:
> «Допустим, что какой-то читатель уже использует базу данных и тут появляется ещё один читатель. Поскольку одновременная работа двух читателей разрешена, второй читатель допускается к базе данных. По мере появления к ней могут быть допущены и другие дополнительные читатели. Теперь допустим, что появился писатель. Он может не получить доступа к базе данных, поскольку писатели должны иметь к ней исключительный доступ, поэтому писатель приостанавливает свою работу… Если новый читатель будет прибывать, скажем, каждые 2 с и каждый читатель затратит на свою работу по 5 с, писатель доступа никогда не дождётся».

### Решение голодания: приоритет писателям

Из tanenbaum.md:
> «Чтобы предотвратить такую ситуацию, программу можно слегка изменить: когда писатель находится в состоянии ожидания, то вновь прибывающий читатель не получает немедленного доступа, а приостанавливает свою работу и встаёт в очередь за писателем. При этом писатель должен дождаться окончания работы тех читателей, которые были активны при его прибытии, и не должен пережидать тех читателей, которые прибывают после него».

Цена: **меньше параллелизма**.

### Применение

- **Реляционные БД** (Read locks vs Write locks).
- **In-memory кэши.**
- **Маршрутные таблицы.**
- **POSIX `pthread_rwlock_t`.**
- **Linux `rw_semaphore`.**

В Linux: read locks приоритетнее → возможно голодание писателей.

## 68. Задача обедающих философов. Пример на семафорах.
*Источник: tanenbaum.md, Глава 2.5.1*

### Постановка

> «Дейкстра сформулировал, а затем решил проблему синхронизации, названную им задачей обедающих философов. Пять философов сидят за круглым столом, и у каждого из них есть тарелка спагетти. Эти спагетти настолько скользкие, что есть их можно только двумя вилками. Между каждыми двумя тарелками лежит одна вилка».

Каждый философ в цикле: думает → проголодался → берёт **обе** вилки (левую и правую) → ест → кладёт вилки → снова думает. Соседние философы делят одну вилку — её одновременно может держать только один.

**Задача:** написать программу так, чтобы все философы могли есть, никто не висел в дедлоке и никто не голодал бесконечно.

Эта задача классическая, потому что иллюстрирует **проблему захвата нескольких ресурсов одновременно** — частая ситуация в реальных системах (две блокировки, два устройства, два узла сети).

### Ошибочное решение №1 — «возьми левую, потом правую»

```c
void philosopher(int i) {
    while (TRUE) {
        think();
        take_fork(i);              // левая
        take_fork((i+1) % N);      // правая
        eat();
        put_fork(i);
        put_fork((i+1) % N);
    }
}
```

**Проблема — дедлок.** Если все 5 философов одновременно проголодались и взяли левую вилку, теперь у каждого в руке одна вилка. Каждый ждёт правую — но правая для него = левая для соседа, она уже занята. Все висят вечно.

### Ошибочное решение №2 — «положить левую, если правая занята»

«Взять левую → попробовать правую → если занята, положить левую, подождать, повторить».

**Проблема — livelock.** Если все философы синхронно начали, все возьмут левую, увидят, что правая занята, синхронно положат, синхронно повторят. И так бесконечно — никто не дедлок, но и никто не ест. Это «живой дедлок».

### Слишком грубое решение — один мьютекс на всех

```c
semaphore mutex = 1;

void philosopher(int i) {
    think();
    down(&mutex);
    /* взять обе вилки и поесть */
    up(&mutex);
}
```

**Корректно**, но за столом одновременно ест **только один**, хотя за круглым столом из пяти физически могут есть до двух непротивостоящих философов (например, 1 и 3, или 2 и 4). Параллелизм потерян.

### Правильное решение (Tanenbaum, листинг 2.14)

Идея: вместо «взять вилку → взять вилку», использовать **разрешение на еду**. Философ заявляет «я хочу есть», система проверяет, не едят ли соседи, и если нет — разрешает.

```c
#define N 5
#define LEFT  (i + N - 1) % N
#define RIGHT (i + 1) % N

enum { THINKING, HUNGRY, EATING };

int       state[N];
semaphore mutex = 1;     // защита массива state
semaphore s[N];          // личный «звонок» каждого философа (нач. = 0)

void philosopher(int i) {
    while (TRUE) {
        think();
        take_forks(i);   // блокируется, пока не получит обе
        eat();
        put_forks(i);    // освобождает соседей
    }
}

void take_forks(int i) {
    down(&mutex);
    state[i] = HUNGRY;
    test(i);             // попытаться разрешить себе еду
    up(&mutex);
    down(&s[i]);         // если test не разрешил — спим на своём семафоре
}

void put_forks(int i) {
    down(&mutex);
    state[i] = THINKING;
    test(LEFT);          // может, теперь сосед-слева может поесть
    test(RIGHT);         // и сосед-справа
    up(&mutex);
}

void test(int i) {
    if (state[i] == HUNGRY &&
        state[LEFT]  != EATING &&
        state[RIGHT] != EATING) {
        state[i] = EATING;
        up(&s[i]);       // разрешаем философу i есть
    }
}
```

### Что здесь происходит

Три ключевых объекта:
- **`state[i]`** — текущее состояние философа: думает / голоден / ест.
- **`mutex`** — защищает массив состояний от гонок при чтении/изменении.
- **`s[i]`** — личный семафор философа. Начальное значение `0`. Если ему разрешили есть — кто-то делает `up(&s[i])`, и он просыпается.

**Когда философ хочет есть** (`take_forks`):
1. Под `mutex` помечает себя как `HUNGRY`.
2. Вызывает `test(i)` — проверяет, можно ли разрешить себе есть.
   - Если соседи не едят — переводит себя в `EATING` и делает `up(&s[i])` (заранее «звонит в свой звонок»).
   - Если кто-то из соседей ест — ничего не делает, философ остаётся `HUNGRY`.
3. Отпускает `mutex`.
4. Делает `down(&s[i])`. Если `test` уже сделал `up` — проходит мгновенно. Если нет — засыпает.

**Когда философ заканчивает есть** (`put_forks`):
1. Под `mutex` помечает себя как `THINKING`.
2. Вызывает `test(LEFT)` и `test(RIGHT)` — теперь его соседи могут получить разрешение, если они голодны и их **других** соседей тоже нет за едой.

Получается «волна разрешений», передаваемая через освобождение `s[]`. Дедлок невозможен — никто не «держит вилки и ждёт». Голодание формально возможно в определённых сценариях, но в практике не возникает при равномерных задержках.

### Применение

- **Илюстрация проблем синхронизации** с несколькими ресурсами.
- **Тест для примитивов синхронизации** — каждая новая абстракция «обязана» решать эту задачу.
- **Идея применима** к распределённым системам, БД, сетям.

## 69. RW-блокировки. Блокировки памяти.
*Источник: robert_lav.md, Глава 10 «Спин-блокировки по чтению-записи», «Семафоры для чтения-записи»; tanenbaum.md, mlock*

### RW-блокировки

**RW-lock (reader-writer lock)** — разделение на читателей и писателей:
- N читателей одновременно — да.
- 1 писатель — да.
- Читатель + писатель — нет.

### Типы в Linux

**Спин-блокировки RW:**
```c
rwlock_t mr_rwlock = RW_LOCK_UNLOCKED;

read_lock(&mr_rwlock);
// чтение
read_unlock(&mr_rwlock);

write_lock(&mr_rwlock);
// запись
write_unlock(&mr_rwlock);
```

**Семафоры RW (`rw_semaphore`):**
```c
DECLARE_RWSEM(my_rwsem);

down_read(&my_rwsem);
// чтение
up_read(&my_rwsem);

down_write(&my_rwsem);
// запись
up_write(&my_rwsem);
```

### Применение

- **Read-mostly данные:** конфигурация, маршрутные таблицы.
- **Кэши с редким обновлением.**

### Особенность Linux

Читатели приоритетнее писателей → возможно голодание писателей.

### Когда не использовать

- Чтения и записи примерно одинаково часто.
- Малое число одновременных читателей.
- Накладные расходы выше выгоды.

### Блокировки памяти (memory locking)

**mlock / munlock** — закрепить страницы в физической памяти, **запретить свопинг**.

```c
#include <sys/mman.h>

mlock(addr, len);     // закрепить регион
munlock(addr, len);
mlockall(MCL_CURRENT | MCL_FUTURE);   // закрепить всё АП
```

### Применение

**1. Системы реального времени.** Page fault от подкачки — миллисекунды задержки → нарушение времени реакции.

**2. Криптография.** Не дать ключам попасть на диск (потенциальная утечка через swap).

**3. Производительность.** Гарантировать, что данные в RAM.

### Ограничения

- **Привилегированная операция** (нужно `CAP_IPC_LOCK`).
- **`RLIMIT_MEMLOCK`** — лимит закреплённой памяти.
- **Влияет на систему** — меньше памяти для свопинга.

### Page-level vs region-level

- `mlock` — конкретные страницы.
- `mlockall` — всё текущее и будущее АП процесса.

## 70. Неблокирующие операции. RCU. Пример.
*Источник: rtos.md, Глава 2.2.4 «Неблокирующие алгоритмы»; robert_lav.md (RCU)*

### Неблокирующие алгоритмы

Из rtos.md (2.2.4):
> «Хотя спин-блокировки позволяют создавать критические секции, очевидно, такое решение страдает не очень хорошей масштабируемостью. Чем больше в алгоритме критических секций, тем больше другие процессоры будут тратить времени на ожидание. Начиная с какого-то момента, блокировка станет бутылочным горлышком, поэтому особый интерес представляют алгоритмы, которые не используют такие блокировки, а пользуются только атомарными инструкциями».

### Иерархия неблокирующих алгоритмов

**Obstruction-free** — слабейшая гарантия. Процессор закончит выполнение за конечное время, **если** остальные процессоры не обращаются к разделяемым данным.

**Lock-free** — конечное количество процессоров закончат за конечное время. **Хотя бы один CPU** делает прогресс в любой момент.

**Wait-free** — самая строгая. **Каждый CPU** делает прогресс за фиксированное время.

```
Wait-free ⊂ Lock-free ⊂ Obstruction-free
```

### Преимущества

- **Нет deadlock** — нет блокировок.
- **Нет priority inversion.**
- **Лучшая масштабируемость** на SMP.
- **Безопасны в обработчиках прерываний.**

### Недостатки

- **Сложность.** Lock-free структуры разрабатываются как исследовательская работа.
- **Проблема ABA** (вопрос 54).
- **Требуют барьеров памяти.**
- **Сложно отлаживать.**

### Примеры lock-free структур

- **Lock-free очередь** (Michael & Scott, 1996).
- **Lock-free стек.**
- **Lock-free hash table** (Cliff Click).
- **Lock-free linked list** (Harris, 2001).

### RCU — Read-Copy-Update

**RCU** — техника синхронизации для read-mostly данных. Разработана для Linux ядра.

**Идея:** читатели **не блокируются** вообще. Писатели создают **новую копию** данных, обновляют указатель атомарно, удаляют старую копию **после** того, как все читатели закончили.

### Использование RCU в Linux

```c
// Читатель
rcu_read_lock();
ptr = rcu_dereference(global_ptr);
use(ptr->data);
rcu_read_unlock();

// Писатель
new = kmalloc(sizeof(*new), GFP_KERNEL);
memcpy(new, old, sizeof(*new));
new->data = new_value;
rcu_assign_pointer(global_ptr, new);
synchronize_rcu();   // подождать всех текущих читателей
kfree(old);
```

### Принцип

1. Читатели только читают через `rcu_dereference` — это **обычная загрузка** с барьером.
2. Писатель копирует данные, обновляет копию.
3. **Атомарная замена** указателя `rcu_assign_pointer`.
4. Старые читатели всё ещё видят старую версию (через сохранённый указатель).
5. Новые читатели видят новую версию.
6. **Период благодати (grace period)** — ждать, пока все текущие читатели закончат `rcu_read_lock`.
7. Только потом удалить старую версию.

### Преимущества RCU

- **Читатели бесплатно** — нет блокировок, барьеров (кроме легких).
- **Отличная масштабируемость** для read-mostly.
- **Никогда не блокирует читателей.**
- **Безопасно из обработчиков прерываний.**

### Недостатки RCU

- **Сложнее использовать.**
- **Большие задержки удаления** старых версий.
- **Дополнительная память** на копии.
- **Нужен grace period.**

### Применение в Linux

- **Маршрутные таблицы.**
- **Списки задач.**
- **Файловая система:** dentry cache.
- **VFS.**
- **Сетевые подсистемы.**

### Пример: обновление маршрутной таблицы

```c
// Чтение — миллионы раз в секунду
struct route *find_route(u32 addr) {
    struct route *r;
    rcu_read_lock();
    r = rcu_dereference(route_table);
    while (r && r->addr != addr)
        r = rcu_dereference(r->next);
    rcu_read_unlock();
    return r;   // ВНИМАНИЕ: после rcu_read_unlock использовать нельзя!
}

// Обновление — редко
void add_route(u32 addr, ...) {
    struct route *new = kmalloc(...);
    new->addr = addr;
    new->next = route_table;
    rcu_assign_pointer(route_table, new);
    // никакого synchronize_rcu — только при удалении
}

void delete_route(u32 addr) {
    struct route *r, *prev = NULL;
    // ... найти узел ...
    rcu_assign_pointer(prev->next, r->next);
    synchronize_rcu();
    kfree(r);
}
```

### Резюме

- Неблокирующие алгоритмы — без блокировок, на атомарных операциях.
- RCU — оптимизирован для read-mostly: читатели не блокируются, писатели используют copy-on-write + grace period.
- Применяется массово в ядре Linux.
- Сложно реализовать, но даёт огромный выигрыш в производительности и масштабируемости.
