## 71. Реализация и виртуализация потоков. Разделяемое ядро.
*Источник: rtos.md, Глава 3.3.5 «Архитектуры ОС», Глава 5.8.4 «Реализация потоков»*

### Реализация потоков

**Поток** = последовательность инструкций + контекст (регистры) + стек. Каждый поток имеет свою структуру TCB (Thread Control Block), хранящую:
- Идентификатор.
- Состояние (READY, BLOCKED, RUNNING).
- Приоритет.
- Контекст (сохраняется при вытеснении).
- Указатель стека.
- Принадлежность к процессу.

Реализация включает:
- Планировщик (выбор потока).
- Переключение контекста (сохранение/восстановление регистров).
- Очереди ожидания.
- Механизмы синхронизации.

### Виртуализация потоков

Потоки **виртуализируются** — каждому даётся иллюзия владения процессором. ОС создаёт «виртуальный CPU» через переключение контекста.

Из rtos.md (3.3.2): «Поток можно также определить как виртуальный процессор, который имеет сохраняемый в памяти контекст».

### Разделяемое ядро — libOS

Из rtos.md (3.3.5):
> «ОСРВ представляет собой библиотеку, статически компонуемую с приложением и использующуюся для реализации потоков и прочих сервисов, так называемую libOS… В этом случае граница между ОС и приложением практически стирается».

**libOS архитектура:**
- ОСРВ — статическая библиотека, линкуется с приложением.
- Нет разделения на пользователя и ядро.
- Прямой доступ к аппаратуре.
- Размер: единицы–десятки КБ.

**Преимущества:**
- Очень быстро (нет переключения режимов).
- Минимум памяти.
- Простая разработка.

**Недостатки:**
- Нет изоляции.
- Ошибка приложения валит ОС.

### Архитектуры ядер

| Архитектура | Сервисы | Производительность | Надёжность |
|---|---|---|---|
| **libOS** | Минимум, статически | Очень высокая | Низкая |
| **Монолитное ядро** | Всё в ядре | Высокая | Средняя |
| **Микроядро** | Минимум в ядре, остальное — процессы | Низкая (IPC) | Высокая |
| **Экзоядро** | Только распределение ресурсов | Высокая | Высокая |
| **Гибридное** | Критичное в ядре, остальное процессы | Хорошая | Хорошая |

### Разделяемое ядро между процессами

В системе с MMU ядро находится в **верхней половине** виртуального АП каждого процесса. Это:
- Общий код ядра во всех процессах.
- Системные вызовы — без смены адресного пространства.
- Изоляция через права доступа MMU.

### Применение

- **libOS:** FreeRTOS, Zephyr, FX-RTOS для микроконтроллеров.
- **Монолитное:** Linux, Windows NT (по сути).
- **Микроядро:** QNX, L4, MINIX 3.
- **Экзоядро:** MIT Exokernel (research).

## 72. Прерывания и их обработка. Аппаратные и программные прерывания. Контроллер прерываний. Обработчики прерываний.
*Источник: rtos.md, Глава 2.1.4 «Прерывания»*

### Зачем нужны прерывания

**Полинг** — периодическая проверка состояния устройств — расходует CPU, плохая реакция.

Из rtos.md (2.1.4):
> «В идеале, для достижения наилучшей скорости реакции, хотелось бы проверять состояние устройств после каждой выполненной инструкции. Поэтому решение заключается в том, чтобы встроить логику проверки состояния устройств непосредственно в процессор и сделать ее частью цикла выборки и исполнения команд».

**Прерывание** — аппаратный механизм, заставляющий процессор **прервать** текущее выполнение и перейти к обработчику.

### Типы прерываний

**Аппаратные:**
- Внешние, от устройств (диск, таймер, сеть, клавиатура).
- Асинхронные — приходят в произвольный момент.
- Идентифицируются вектором.

**Программные:**
- Инициируются самой программой через специальную инструкцию (`INT`, `SVC`, `syscall`).
- Используются для системных вызовов, перехода в режим ядра.
- Синхронные (заранее запрограммированы).

**Исключения (exceptions):**
- Связаны с конкретной инструкцией (деление на 0, page fault).
- Синхронные.

### Контроллер прерываний

Из rtos.md:
> «Контроллер прерываний — это интерфейсное устройство для сопряжения внешних устройств с разными процессорами».

Устройства не подключаются к процессору напрямую — они подключаются к **контроллеру прерываний** (NVIC на ARM Cortex-M, APIC на x86, GIC на ARM Cortex-A).

Функции контроллера:
- Принимает запросы от устройств.
- Приоритизирует.
- Сообщает процессору **вектор**.
- Маскирует прерывания.

### Обработчики прерываний

**Обработчик прерывания (ISR — Interrupt Service Routine)** — функция, выполняющаяся в ответ на прерывание.

Из rtos.md:
> «При наличии активного запроса прерывания, процессор производит действия, эквивалентные вызову подпрограммы: помещает адрес возврата в стек и переходит к выполнению функции-обработчика прерывания, адрес которой предварительно должен быть сообщен процессору».

Действия процессора при прерывании:
1. Завершить текущую инструкцию.
2. Сохранить адрес возврата (обычно в стек).
3. Сохранить регистр состояния.
4. Перейти на адрес обработчика.

Обработчик должен:
1. Сохранить регистры, которые будет использовать.
2. Опросить устройство, выполнить нужные действия.
3. Очистить флаг запроса прерывания.
4. Восстановить регистры.
5. Вернуть управление (специальной инструкцией `IRET`/`RFE`).

### Особенности обработчиков

- Прерывают **произвольное место** программы.
- Не имеют связи с конкретным процессом.
- Должны быть короткими (см. вопрос 75 о split-half).
- Не могут блокироваться (нет планировщика для них).
- Должны использовать только специальные примитивы (атомарные операции, спинлоки).

### Энергосбережение

Из rtos.md:
> «Прерывания дают возможность реализовать в процессоре различные режимы энергосбережения. Например, система команд может содержать специальную инструкцию останова, которая приостанавливает выборку и исполнение инструкций процессором до получения прерывания».

`WFI`/`HLT` — процессор спит до прерывания.

## 73. Прерывания и их обработка. Вектор прерываний, таблица векторов прерываний. Приоритеты прерываний.
*Источник: rtos.md, Глава 2.1.4*

### Вектор прерываний

Из rtos.md:
> «Вектор — атрибут источника прерывания, который позволяет различать запросы от разных устройств. Разным векторам соответствуют разные функции-обработчики прерывания».

Каждое устройство имеет уникальный **номер вектора**. По нему процессор находит нужный обработчик.

### Таблица векторов прерываний (IVT/IDT)

**Подход 1: таблица векторов в памяти.**

Из rtos.md:
> «Первый из них предполагает существование в памяти так называемой таблицы векторов прерываний (interrupt vector table, IVT). Она находится либо в фиксированном месте в памяти, о котором знает процессор, либо у последнего есть специальный регистр, который указывает на таблицу».

- Массив указателей на функции-обработчики.
- Индексируется вектором.
- На x86 — IDT (Interrupt Descriptor Table), указатель в регистре IDTR.
- На ARM — таблица векторов в начале памяти (или адрес настраивается через VTOR).

Пример (x86):
```
IDT[0]   — Division Error
IDT[1]   — Debug
IDT[14]  — Page Fault
IDT[32+] — Аппаратные прерывания
IDT[128] — Системные вызовы Linux (INT 0x80)
```

**Подход 2: единый обработчик с диспетчеризацией.**

Из rtos.md:
> «Использование одного вектора на все аппаратные прерывания плюс возможность узнать текущий вектор уже внутри обработчика путём чтения либо регистров процессора, либо регистров контроллера прерываний».

Удобно для расширения и инфраструктурного каркаса.

### Приоритеты прерываний

Из rtos.md:
> «Несколько источников вполне могут установить запрос прерывания одновременно. Возникающую при этом неопределенность: какой обработчик должен быть вызван, разрешают с помощью приоритета прерываний».

Каждому вектору назначается **приоритет**:
- Статически (зашит в железо).
- Динамически (программно настраиваемый).

### Вложенные обработчики

Из rtos.md:
> «Высокоприоритетные прерывания могут прерывать низкоприоритетные».

Реализация:
- Контроллер сравнивает приоритет нового запроса с текущим.
- Если новый выше → вытесняет текущий обработчик.
- При выходе — возврат в прерванный обработчик.

### Где реализуются приоритеты

- **В процессоре** (через регистр маски IPL).
- **В контроллере прерываний** (NVIC, APIC).
- Часто комбинация.

### Маска прерываний

Текущий «уровень маски» определяет, какие прерывания может принять CPU:
- Прерывания с приоритетом ≤ маски — игнорируются (откладываются).
- Прерывания с приоритетом > маски — обрабатываются.

Изменение маски — основа критических секций при работе с прерываниями.

### Пример: ARM Cortex-M / NVIC

- 256 векторов прерываний.
- 8–256 уровней приоритета (зависит от модели).
- Программируется через NVIC.
- Поддерживает вложение «из коробки».

## 74. Прерывания и их обработка. Запуск обработчика прерываний. Блокировка обработчика прерываний. Запрет прерываний.
*Источник: rtos.md, Глава 2.1.4, 2.1.6 «Атомарность»; Глава 4.4.6 «Уровень маски прерываний»*

### Запуск обработчика

Процесс при возникновении прерывания:
1. **Окончание текущей инструкции.** Прерывания обычно проверяются между инструкциями.
2. **Анализ маски.** Если приоритет ≤ маски, прерывание откладывается.
3. **Сохранение контекста процессора.** На x86: автоматически в стек (CS, IP, EFLAGS). На ARM Cortex-M: автоматически (R0-R3, R12, LR, PC, xPSR).
4. **Переключение на стек прерываний** (если есть).
5. **Получение вектора.**
6. **Поиск в таблице векторов.**
7. **Переход на обработчик.**
8. **Обработка.**
9. **Возврат через специальную инструкцию** (`IRET` x86, автомагический возврат на ARM Cortex-M).

### Запрет прерываний

Из rtos.md (2.1.6):
> «Нужен способ приостановить получение новых прерываний во время работы с разделяемыми данными из приложения. Для реализации такой возможности, процессор поддерживает специальные инструкции, с помощью которых можно разрешать или запрещать получение прерываний».

```c
// Источник: rtos.md, раздел 2.1.6
disable_interrupts();
<код критической секции>
enable_interrupts();
```

### Инструкции запрета

- **x86:** `CLI` (Clear Interrupt Flag) / `STI` (Set).
- **ARM Cortex-M:** `CPSID i` / `CPSIE i`, или установка PRIMASK.
- **ARM Cortex-A:** установка бита I в CPSR.

### Что блокируется

При полном запрете:
- Все маскируемые прерывания не вызывают обработчиков.
- Запросы накапливаются в контроллере.
- При разрешении — сразу обрабатываются (в порядке приоритета).

**NMI (Non-Maskable Interrupt)** — особое прерывание, которое нельзя замаскировать. Используется для критических событий (например, ECC ошибка памяти).

### Блокировка отдельных обработчиков (маска приоритета)

Из rtos.md (4.4.6):
> «Уровень маски прерываний» — текущий приоритет в процессоре. Прерывания с приоритетом ниже маски не вызывают обработчик».

Это **тоньше**, чем полный запрет:
- Высокоприоритетные продолжают работать.
- Низкоприоритетные ждут.
- Полезно для критических секций между обработчиками разного приоритета.

### Вложение и счётчик

Если процессор поддерживает вложение, нужен счётчик глубины:
- При входе в обработчик: `intr_nesting++`.
- При выходе: `intr_nesting--`.
- Если стал 0 → можно вызывать планировщик.

Из rtos.md (2.1.6):
> «Может потребоваться вести счётчик вложения прерываний, который увеличивался бы всякий раз при входе в обработчик, и уменьшался бы при выходе. Процессор ничего об этом не знает».

### Атомарность входа в обработчик

Из rtos.md:
> «Атомарность передачи управления обработчику обычно поддерживается аппаратно».

Но между входом и инкрементом счётчика возможна гонка — может прийти ещё одно прерывание. Решения:
- Аппаратный запрет прерываний при входе в обработчик (x86).
- Запрет прерываний того же приоритета (ARM Cortex-M).

### Запрет в критических секциях

Запрет прерываний — самый простой способ синхронизации в однопроцессорной системе (см. вопрос 47).

Недостатки:
- Не работает на SMP.
- Грубо — блокирует всё.
- Влияет на время реакции.

В RT-системах длинные критические секции с запрещёнными прерываниями недопустимы.

## 75. Прерывания и их обработка. Исключения, их отличия от прерываний. Первичные и вторичные обработчики (верхняя и нижняя половина). Переключение контекста.
*Источник: rtos.md, Глава 2.1.5 «Исключения», Глава 4.4.3 «Переключение контекста», Глава 5.5.2 «Отложенные процедуры»*

### Исключения vs прерывания

Из rtos.md (2.1.5):
> «Исключения или исключительные ситуации. Если сравнивать их с внешними аппаратными прерываниями, то основное отличие заключается в том, что они являются синхронными: инструкция, выполнение которой вызывает исключение, будет вызывать это исключение всегда (с теми же параметрами), тогда как прерывания возникают независимо от действий процессора».

| Параметр | Прерывания | Исключения |
|---|---|---|
| Источник | Внешнее устройство | Конкретная инструкция |
| Тип | Асинхронные | Синхронные |
| Повторяемость | Зависит от устройства | Та же инструкция → то же исключение |
| Назначение | Реакция на события | Ошибки, спец. ситуации |
| Контроллер | Используется | Не используется |

### Примеры исключений

- **Деление на 0.**
- **Page fault** (обращение к выгруженной странице).
- **General protection fault** (нарушение прав).
- **Недопустимая инструкция.**
- **Trap** (точка останова).
- **Системный вызов** (программное прерывание).

### Использование исключений

Из rtos.md:
> «В большинстве же случаев, в отличие от прерываний, возникновение исключений является ошибкой и сигнализирует о проблемах и невозможности продолжения работы прикладной программы, а действия обработчика сводятся к тому, чтобы завершить выполнение ошибочного приложения».

Но есть и «нормальные» исключения:
- **Page fault** — для подкачки.
- **Copy-on-Write** — копирование страницы при записи.
- **Системные вызовы.**

### Верхняя и нижняя половина

Из rtos.md (5.5.2):

**Верхняя половина (top half) / первичный обработчик** — выполняется немедленно при прерывании. Должна быть **короткой**:
- Подтвердить устройство.
- Минимальная обработка.
- Запланировать «долгую» работу.

**Нижняя половина (bottom half) / вторичный обработчик** — отложенная работа:
- Выполняется в более удобном контексте (поток, DPC).
- Может занять больше времени.
- Может быть прервана другими прерываниями.

### Зачем разделение

- **Время реакции:** короткие обработчики позволяют другим прерываниям приходить быстро.
- **Возможности:** в верхней половине нельзя ждать; в нижней — можно.
- **Распределение нагрузки:** долгая обработка не блокирует прерывания.

### Реализация в Linux

- **Tasklets:** softirq-based, выполняются вскоре после возврата из прерывания.
- **Softirqs:** более общая инфраструктура.
- **Workqueues:** выполняются в потоках ядра, могут блокироваться.

### Переключение контекста через прерывания

Из rtos.md (4.4.3):

В обработчике прерывания может выясниться, что нужно переключить контекст (более приоритетный поток стал готов). Но прерывания могут быть **вложенными** — переключать контекст внутри обработчика опасно.

**Решение:** счётчик вложения. «Кто сделал последний декремент и достиг нуля, тот вызывает планировщик».

Альтернатива — программное прерывание DISPATCH с минимальным приоритетом. Обработчики аппаратных прерываний запрашивают DISPATCH, который срабатывает после всех вложенных аппаратных.

```c
// rtos.md, раздел 4.8
void interrupt_handler(void) {
    if (get_current_vector() == SIGNAL_VECTOR) {
        thread_ready = true;
        request_software_interrupt(DISPATCH);
    }
}
```

## 76. Обработка прерываний и исключений в пользовательском процессе.
*Источник: rtos.md, Глава 7.8.1, 7.8.2*

### Сценарий

Пользовательский процесс выполняет код. Происходит прерывание (от диска) или исключение (page fault).

Действия:
1. **Аппаратный переход в режим ядра.** Сохранение PSW, PC, переключение на стек ядра потока.
2. **Сохранение контекста пользователя** в стек ядра.
3. **Запуск обработчика** в режиме ядра.
4. **Обработка.**
5. **Возможное переключение контекста** (если разблокирован более приоритетный поток).
6. **Возврат** в пользователя через `IRET`/`ERET`.

### Особенности прерывания пользовательского кода

- **Переключение стека:** обязательно, аппаратно (см. вопрос 30).
- **Сохранение контекста:** пользовательские регистры → стек ядра потока (НЕ стек прерываний, чтобы потом можно было переключить контекст).
- **Невидимость для пользователя:** возврат в ту же инструкцию.

### Исключения пользователя

При исключении (page fault, illegal instruction):
- **Page fault** — может быть «нормальной»: ОС подгрузит страницу, вернёт управление.
- **Реальная ошибка** (SIGSEGV) — пользовательский процесс получает сигнал.
- **Системный вызов** — намеренно вызванное «исключение».

### Сигналы

Сигналы — асинхронные уведомления процесса (см. вопрос 39). Доставка:
1. При возврате из ядра в пользователя проверяется ожидающий сигнал.
2. Если есть и не маскирован — модифицируется фрейм возврата: PC указывает на обработчик сигнала.
3. Сохраняется оригинальное состояние для последующего восстановления.
4. Возврат в пользователя → выполняется обработчик сигнала.
5. По окончании обработчика — `sigreturn`, восстановление состояния.

### Многопоточность

- Сигнал, направленный процессу, доставляется **одному** из потоков.
- Сигнал, направленный конкретному потоку (`pthread_kill`) — ему.
- Маска сигналов — у каждого потока своя.

### Прерывание длинных вычислений

Пользовательский CPU-bound поток выполняется → таймерное прерывание → планировщик может вытеснить → даёт CPU другому. Через квант снова возвращается.

## 77. Старший и младший номера устройств. MKDEV. Получение и освобождение номеров устройств.
*Источник: robert_lav.md, Глава 13; Linux Device Drivers (LDD)*

### Номера устройств

В Unix/Linux устройства представляются файлами в `/dev`. У каждого файла устройства — **номер**, идентифицирующий драйвер и конкретное устройство.

Номер состоит из двух частей:
- **Major (старший) номер** — определяет драйвер.
- **Minor (младший) номер** — конкретное устройство, обслуживаемое этим драйвером.

### dev_t

Тип `dev_t` (32-битное, в современных ядрах) хранит оба номера:
- 12 бит — major.
- 20 бит — minor.

### Макросы

```c
#include <linux/kdev_t.h>

dev_t dev;

// Создать dev_t из major и minor
dev = MKDEV(major, minor);

// Извлечь
unsigned int maj = MAJOR(dev);
unsigned int min = MINOR(dev);
```

### Пример

```c
dev_t dev = MKDEV(240, 5);
// 240 — major, 5 — minor
```

В `/dev`:
```
$ ls -l /dev/sda
brw-rw---- 1 root disk 8, 0 ... /dev/sda
$ ls -l /dev/sda1
brw-rw---- 1 root disk 8, 1 ... /dev/sda1
```
major = 8 (диски SCSI), minor = 0 для устройства, 1 для первого раздела.

### Статическое выделение номеров

```c
#include <linux/fs.h>

int register_chrdev_region(dev_t first, unsigned int count, char *name);
```

Зарезервировать диапазон от `first` до `first + count - 1` для драйвера с именем `name`.

Пример:
```c
dev_t dev = MKDEV(240, 0);
register_chrdev_region(dev, 4, "mydriver");
// зарезервировали (240, 0), (240, 1), (240, 2), (240, 3)
```

**Недостаток:** major-номер выбирается «на удачу». Могут быть конфликты с другими драйверами.

### Освобождение номеров

```c
void unregister_chrdev_region(dev_t first, unsigned int count);
```

Освобождает диапазон. Вызывается при выгрузке драйвера.

### Просмотр выделенных major-номеров

```
$ cat /proc/devices
Character devices:
  1 mem
  4 /dev/vc/0
  5 /dev/tty
  ...
Block devices:
  8 sd
  9 md
```

## 78. Старший и младший номера устройств. Динамическое выделение старших номеров. alloc_chrdev_region.
*Источник: robert_lav.md, Глава 13; LDD*

### Проблема статического выделения

При `register_chrdev_region` нужно знать major заранее. Если другой драйвер использует тот же major — конфликт.

### Динамическое выделение

```c
int alloc_chrdev_region(dev_t *dev, unsigned int firstminor,
                        unsigned int count, char *name);
```

Параметры:
- `dev` — выходной, заполнится назначенным dev_t.
- `firstminor` — начальный minor (обычно 0).
- `count` — сколько minor номеров.
- `name` — имя драйвера.

Ядро **само** выберет свободный major-номер.

### Пример использования

```c
dev_t dev;
int result;
int my_major;

result = alloc_chrdev_region(&dev, 0, 4, "mydriver");
if (result < 0) {
    printk("Cannot allocate dev numbers\n");
    return result;
}
my_major = MAJOR(dev);
printk("Got major %d\n", my_major);
```

### Преимущества

- **Нет конфликтов** с другими драйверами.
- **Портируемость** — работает на любой системе.
- **Стандартная практика** для современных драйверов.

### Недостатки

- **Major номер не известен заранее** — пользователь должен прочитать его из `/proc/devices` или через `udev`.

### Узел в /dev

Для создания файла в `/dev`:
```bash
mknod /dev/mydriver c 240 0
# c — character, 240 — major, 0 — minor
```

Или через `udev`:
```c
struct class *my_class;
my_class = class_create(THIS_MODULE, "mydriver");
device_create(my_class, NULL, MKDEV(my_major, 0), NULL, "mydriver");
```

`udev` автоматически создаст `/dev/mydriver`.

### Освобождение

```c
unregister_chrdev_region(dev, 4);
```

## 79. Файловые операции. int (*flush) (struct file *);
*Источник: robert_lav.md, Глава 13 «Объект file_operations»; LDD*

### Структура file_operations

Из robert_lav.md (Глава 13):
```c
struct file_operations {
    struct module *owner;
    loff_t (*llseek) (struct file *, loff_t, int);
    ssize_t (*read) (struct file *, char __user *, size_t, loff_t *);
    ssize_t (*write) (struct file *, const char __user *, size_t, loff_t *);
    int (*ioctl) (struct inode *, struct file *, unsigned int, unsigned long);
    int (*mmap) (struct file *, struct vm_area_struct *);
    int (*open) (struct inode *, struct file *);
    int (*flush) (struct file *, fl_owner_t id);
    int (*release) (struct inode *, struct file *);
    int (*fsync) (struct file *, struct dentry *, int datasync);
    // ... и много других
};
```

Это таблица функций, реализующих файловые операции. Драйвер инициализирует поля своими функциями.

### Функция flush

```c
int (*flush) (struct file *file, fl_owner_t id);
```

**Назначение:** вызывается при **каждом** вызове `close()` пользователем, **прежде** чем будет вызван `release`.

### Когда вызывается

- Каждый раз когда процесс делает `close(fd)`.
- Если файл открыт несколько раз (через `dup`, `fork`) — `flush` вызывается при каждом `close`.
- `release` вызывается **только** при закрытии последней копии.

### Разница flush vs release

| Параметр | flush | release |
|---|---|---|
| Когда | Каждый close | Только последний close |
| Назначение | Сброс буферов | Освобождение ресурсов |
| Параметры | file, fl_owner_t | inode, file |

### Пример

```c
int my_flush(struct file *file, fl_owner_t id) {
    // Сбросить буферы устройства
    flush_device_buffers();
    return 0;
}
```

### Применение

- **Сброс данных** на физическое устройство перед закрытием fd.
- **Очистка состояния**, связанного с открытым fd.
- **Сетевые драйверы:** flush очередей при close.

### Примеры из ядра

В большинстве драйверов `flush` **не используется** — поле NULL. Используется:
- В именованных каналах (FIFO).
- В сетевых файлах.
- В файловых системах с буферизацией.

Чаще всего для драйверов символьных устройств достаточно `release`.

## 80. Структура inode
*Источник: robert_lav.md, Глава 13 «Объект inode»*

### Что такое inode

Из robert_lav.md (Глава 13):
> «Для каждого файла в файловой системе существует представляющий его объект `inode`. Однако сам файловый индекс создается в памяти только в момент доступа к файлу. Это справедливо и для специальных файлов, таких как файлы устройств или конвейеры».

**inode** = индексный узел (index node), описывающий файл независимо от его имени.

### Структура (фрагмент)

```c
// Источник: robert_lav.md, Глава 13
struct inode {
    struct hlist_node       i_hash;          /* список хеш-таблицы */
    struct list_head        i_list;
    struct list_head        i_sb_list;
    struct list_head        i_dentry;       /* связанные dentries */
    unsigned long           i_ino;          /* номер индекса */
    atomic_t                i_count;        /* счетчик ссылок */
    unsigned int            i_nlink;        /* количество жестких ссылок */
    uid_t                   i_uid;          /* владелец */
    gid_t                   i_gid;          /* группа */
    kdev_t                  i_rdev;         /* номер устройства */
    loff_t                  i_size;         /* размер файла */
    struct timespec         i_atime;        /* последний доступ */
    struct timespec         i_mtime;        /* последнее изменение */
    struct timespec         i_ctime;        /* изменение индекса */
    umode_t                 i_mode;         /* права доступа */
    struct inode_operations *i_op;          /* операции с индексом */
    struct file_operations  *i_fop;         /* операции с файлом */
    struct super_block      *i_sb;          /* суперблок */
    union {
        struct pipe_inode_info *i_pipe;     /* для FIFO */
        struct block_device    *i_bdev;     /* блочное устройство */
        struct cdev            *i_cdev;     /* символьное устройство */
    };
    // ... много других полей
};
```

### Основные поля

**Идентификация:**
- `i_ino` — номер индекса (уникален в файловой системе).
- `i_uid`, `i_gid` — владелец.
- `i_mode` — права доступа.

**Размер и времена:**
- `i_size` — размер в байтах.
- `i_atime` (access), `i_mtime` (modification), `i_ctime` (status change).

**Ссылки и состояние:**
- `i_count` — атомарный счётчик использования.
- `i_nlink` — число жёстких ссылок (имён в файловой системе).

**Операции:**
- `i_op` — операции с индексом (создание, удаление файлов, mkdir).
- `i_fop` — операции с открытым файлом (read, write, ioctl).

**Тип-специфичные данные:**
- `i_pipe`, `i_bdev`, `i_cdev` — объединение, заполняется для FIFO, блочного или символьного устройства.

### inode и файл

- **inode** — описывает файл в файловой системе.
- **dentry** — связывает имя с inode.
- **file** — описывает открытый файл (свой указатель позиции у каждого open).

```
имя файла → dentry → inode → данные
                                 ↑
                       file (открытый)
```

### Жёсткие ссылки

Несколько имён (dentry) могут указывать на один inode. `i_nlink` считает их. Когда становится 0 **и** `i_count` = 0 — файл удаляется.

### Операции с inode

Поле `i_op` — таблица функций:
```c
struct inode_operations {
    int (*create) (struct inode *, struct dentry *, int, struct nameidata *);
    struct dentry *(*lookup) (struct inode *, struct dentry *, struct nameidata *);
    int (*link) (struct dentry *, struct inode *, struct dentry *);
    int (*unlink) (struct inode *, struct dentry *);
    int (*mkdir) (struct inode *, struct dentry *, int);
    int (*rmdir) (struct inode *, struct dentry *);
    // ... и т.д.
};
```

### Файловые системы

Файловая система реализует свою версию операций. В дисковой ФС inode хранится в физических блоках, а в RAM кэшируется в `struct inode`.

В файловых системах без диска (tmpfs, procfs) inode существует только в памяти.

### Применение

- **VFS** работает с inode абстрактно — единый интерфейс для разных файловых систем.
- **Драйверы устройств** регистрируют операции через `i_fop` или через `cdev`.
- **Кэш inodes** (icache) ускоряет повторный доступ.
