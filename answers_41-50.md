## 41. Синхронизация через разделяемые объекты. Синхронный обмен сообщений.
*Источник: rtos.md, Глава 8.7 «Особенности реализации синхронного обмена сообщениями»; общие принципы IPC*

**Синхронный обмен сообщениями** — отправитель и получатель встречаются «лицом к лицу». Отправитель **блокируется** до момента приёма получателем; получатель блокируется до момента отправки. Сообщение передаётся напрямую, без промежуточного буфера.

**Также называется рандеву (rendezvous).** Подобно встрече двух человек: оба должны прийти в назначенное место, и только тогда происходит «обмен».

### Свойства

- **Нет буфера.** Сообщение не хранится — передаётся прямо между процессами.
- **Естественная синхронизация.** После завершения `send` отправитель знает, что получатель прочитал.
- **Естественное ограничение нагрузки.** Если получатель не успевает, отправитель ждёт — нет переполнения.
- **Меньше памяти** на инфраструктуру.

### Псевдокод

```c
typedef struct {
    void *msg;
    bool msg_ready;
    bool msg_consumed;
    semaphore_t producer_sem;
    semaphore_t consumer_sem;
} rendezvous_t;

void send(rendezvous_t *r, void *msg) {
    sem_wait(&r->consumer_sem);   // ждать, пока получатель готов
    r->msg = msg;
    r->msg_ready = true;
    sem_post(&r->producer_sem);   // сигнал получателю
    sem_wait(&r->consumer_sem);   // ждать подтверждения
}

void *recv(rendezvous_t *r) {
    sem_post(&r->consumer_sem);   // готов к приёму
    sem_wait(&r->producer_sem);   // ждать сообщения
    void *msg = r->msg;
    sem_post(&r->consumer_sem);   // подтверждение
    return msg;
}
```

### Преимущества

- **Простая семантика:** «отправлено» = «доставлено».
- **Гарантированная доставка** (если не было ошибки).
- **Память не нужна для буферов.**
- **Естественное согласование скоростей** — медленный получатель замедляет отправителя.

### Недостатки

- **Низкая пропускная способность** при асимметричной нагрузке.
- **Чувствительность к задержкам** — отправитель ждёт.
- **Невозможно сглаживать пики** нагрузки.

### Применение

- **CSP (Communicating Sequential Processes)** — модель Хоара. Реализована в **Go** (каналы без буфера), **Erlang**, **Occam**.
- **Ada** — рандеву встроено в язык.
- **POSIX MQ с особыми атрибутами**, **System V IPC**.
- В ОСРВ — между задачами с жёсткими требованиями к синхронизации.

### Сравнение

| Параметр | Асинхронный (вопрос 40) | Синхронный |
|---|---|---|
| Буфер | Есть | Нет |
| Блокировка отправителя | При полном буфере | Всегда |
| Гарантия доставки | Доставлено в очередь | Доставлено получателю |
| Связанность | Слабая | Сильнее (по времени) |
| Применение | Web, очереди задач | CSP, реактивные системы |

## 42. Критическая секция. Условия возникновения взаимоблокировок. Моделирование блокировок. Граф.
*Источник: tanenbaum.md, Глава 6 «Взаимоблокировка», 6.2*

**Критическая секция** — участок кода, в котором происходит работа с разделяемыми данными. Должна выполняться **атомарно** в смысле взаимного исключения: только один процесс/поток одновременно.

**Взаимоблокировка (deadlock)** — ситуация, когда группа процессов навсегда заблокирована, каждый ждёт ресурса, удерживаемого другим.

Из tanenbaum.md (6.2):
> «Взаимоблокировка в группе процессов возникает в том случае, если каждый процесс из этой группы ожидает события, наступление которого зависит исключительно от другого процесса из этой же группы».

### Классический пример (Tanenbaum)

Процессы A и B хотят оба ресурса (принтер и сканер):
- Процесс A берёт принтер, запрашивает сканер.
- Процесс B берёт сканер, запрашивает принтер.
- Оба ждут вечно.

### Условия Коффмана (4 необходимых условия)

Все 4 должны выполняться одновременно:

1. **Взаимное исключение.** Каждый ресурс либо выделен одному процессу, либо доступен.
2. **Удержание и ожидание.** Процессы могут удерживать одни ресурсы и запрашивать новые.
3. **Невыгружаемость.** Ресурсы нельзя отобрать принудительно — только сам процесс может освободить.
4. **Циклическое ожидание.** Существует кольцо процессов, каждый ждёт ресурс соседа.

Если **хоть одно** условие не выполняется — deadlock невозможен.

### Моделирование графом (Holt, 1972)

Из tanenbaum.md (6.2.2):
- **Процессы** — окружности.
- **Ресурсы** — квадраты.
- **Ребро ресурс → процесс** — ресурс выделен процессу.
- **Ребро процесс → ресурс** — процесс ждёт ресурс.

**Цикл в графе = deadlock** (при условии одного экземпляра каждого ресурса).

Пример deadlock:
```
   ┌────┐         ┌────┐
   │ P1 │ ←──── │ R2 │
   └─┬──┘       └──▲─┘
     │             │
     ↓             │
   ┌────┐         ┌──┴─┐
   │ R1 │ ────→ │ P2 │
   └────┘         └────┘
```

P1 удерживает R1, ждёт R2. P2 удерживает R2, ждёт R1. Цикл!

### Применение

Графы используются для:
- **Обнаружения** deadlock (вопрос 43).
- **Уклонения** через предсказание (вопрос 44).
- **Анализа** последовательности запросов.

## 43. Стратегии для борьбы с взаимоблокировками. Страусинный алгоритм. Обнаружение взаимоблокировок с использованием одного типа ресурса/нескольких ресурсов каждого типа.
*Источник: tanenbaum.md, Глава 6.3, 6.4*

### Четыре стратегии

1. **Игнорирование** (страусинный алгоритм).
2. **Обнаружение и восстановление.**
3. **Уклонение** (динамическое распределение).
4. **Предотвращение** (структурное подавление условий).

### Страусинный алгоритм

«Спрячь голову в песок и сделай вид, что проблемы нет.»

Из tanenbaum.md:
> «Математики считают её неприемлемой… Инженеры спрашивают, как часто ожидается возникновение проблемы, как часто система даёт сбой по другим причинам и насколько серьёзны последствия. Если взаимоблокировка возникает в среднем один раз в пять лет, а система раз в неделю сбоит из-за технических отказов… большинство инженеров не захотят платить за избавление от взаимоблокировок существенным снижением производительности».

Применяется в **Linux, Windows, macOS** для большинства ресурсов. Системы рассчитаны на то, что deadlock редок, и в крайнем случае — перезагрузка.

### Обнаружение при одном экземпляре каждого ресурса

Строится **граф ресурсов**. Алгоритм:

1. Для каждого узла N запустить поиск в глубину, используя N как корень.
2. На каждом шаге добавлять текущий узел в список L.
3. **Если узел появился в L дважды** — найден цикл (deadlock).
4. Если поиск закончился без цикла — нет deadlock из этого корня.

Сложность алгоритма O(V·E).

### Обнаружение при нескольких экземплярах

Используются матрицы (rtos.md, 6.4.2):
- **E** (existing) — вектор существующих ресурсов каждого типа.
- **A** (available) — вектор доступных (не выделенных).
- **C** (current allocation) — матрица: `C[i][j]` = сколько ресурса j у процесса i.
- **R** (request) — матрица: `R[i][j]` = сколько ресурса j нужно процессу i.

**Условие согласованности:** для каждого j сумма выделенных + A = E.

**Алгоритм:**
1. Найти **непомеченный** процесс P, у которого `R[P] ≤ A` (можно удовлетворить запросы).
2. Если найден: пометить P, освободить его ресурсы — `A += C[P]`. Перейти к шагу 1.
3. Если не найден — все непомеченные процессы в **состоянии взаимоблокировки**.

Алгоритм пытается «прокрутить» систему вперёд: если есть процесс, чьи запросы можно удовлетворить — он завершится и вернёт свои ресурсы. Если в итоге все процессы помечены — deadlock нет.

### Выход из обнаруженного deadlock

**1. Приоритетное овладение.** Отобрать ресурс у одного процесса, передать другому. Возможно не для всех ресурсов (например, не для принтера во время печати).

**2. Откат (rollback).** Использовать заранее сохранённые контрольные точки процесса. Откатить к точке до получения ресурса.

**3. Уничтожение процесса.** Грубо, но просто. Уничтожить процесс в цикле, освободить его ресурсы. Если не помогло — уничтожить следующий.

Выбор жертвы — по принципам: минимальная потеря работы, легко перезапустить, низкий приоритет.

### Когда проверять

- **Каждый запрос.** Точно, но дорого.
- **Каждые k минут** или по таймеру.
- **При снижении загрузки CPU** — если процессы не работают, возможен deadlock.

## 44. Стратегии для борьбы с взаимоблокировками. Выход из взаимоблокировок. Уклонение от взаимоблокировок. Безопасное и небезопасное состояние. Алгоритм банкира.
*Источник: tanenbaum.md, Глава 6.5*

### Уклонение vs предотвращение

**Уклонение (avoidance):** при каждом запросе ресурса система проверяет, не приведёт ли его выделение к состоянию, из которого нельзя гарантированно выйти. Если приведёт — отложить запрос.

**Предотвращение (prevention):** структурно гарантировать, что одно из условий Коффмана не выполнится — deadlock в принципе невозможен.

Уклонение — динамическое, требует **прогноза** будущих запросов.

### Безопасное и небезопасное состояние

**Безопасное состояние** — существует порядок планирования, при котором каждый процесс может **доработать** до конца, даже если все процессы внезапно запросят максимальное количество ресурсов.

**Небезопасное** — такого порядка нет. **Не значит, что deadlock неизбежен** — но он возможен в худшем сценарии.

### Пример (Tanenbaum, рис. 6.7)

Один тип ресурса, 10 экземпляров. Процессы:
- A удерживает 3, может запросить ещё до 9 (максимум 9 всего).
- B удерживает 2, максимум 4 (нужно ещё 2).
- C удерживает 2, максимум 7 (нужно ещё 5).

Доступно: 10 − 7 = 3.

**Безопасно?** Да:
1. Выделить 2 процессу B → B доработает → освободит 4.
2. Доступно 5. Выделить 5 процессу C → C доработает → освободит 7.
3. Доступно 7. Выделить 6 процессу A → доработает.

Если бы C нужно было 6 (а не 5), безопасности не было бы.

### Алгоритм банкира (Dijkstra, 1965)

Аналогия: банкир, выдающий кредиты клиентам. Банкир знает максимум каждого клиента, имеет ограниченный резерв. Решение по кредиту: проверить, после выдачи остаётся ли система в безопасном состоянии. Если да — выдать. Если нет — отложить.

### Алгоритм банкира для одного ресурса

При каждом запросе:
1. Предположить, что запрос удовлетворён.
2. Проверить, можно ли последовательно завершить все процессы.
3. Если да — удовлетворить запрос. Если нет — отложить.

### Для нескольких типов ресурсов

Те же матрицы E, A, C, R, что в вопросе 43. Алгоритм проверки безопасности:

1. Найти процесс P с `R[P] ≤ A`.
2. Если найден: предположить P завершается. `A += C[P]`. Пометить.
3. Повторять, пока все помечены или не найдено.
4. Если все помечены — безопасное состояние.

При запросе ресурса:
- Симулировать выделение.
- Проверить безопасность.
- Если безопасно — выделить. Иначе — отказать.

### Практическая ценность

Из tanenbaum.md:
> «Хотя алгоритм замечателен в теории, на практике он по существу бесполезен, поскольку нечасто можно определить заранее, каковы будут максимальные потребности процессов в ресурсах. Кроме того, количество процессов не фиксированно… ресурсы, считавшиеся доступными, могут внезапно пропасть».

Поэтому алгоритм банкира **почти не применяется в реальных ОС**. Используются эвристики (например, «дросселирование при заполнении буфера на 70%»).

## 45. Стратегии для борьбы с взаимоблокировками. Выход из взаимоблокировок. Предотвращение взаимоблокировок.
*Источник: tanenbaum.md, Глава 6.6 «Предотвращение взаимоблокировки»*

### Идея предотвращения

Структурно гарантировать, что **одно из четырёх условий Коффмана** никогда не выполнится. Тогда deadlock невозможен.

### Атака условия взаимного исключения

Сделать данные доступными для чтения нескольким одновременно. Для принтера — использовать **спулинг**: процессы пишут в очередь, реальный принтер обслуживает только демон.

Проблема: не все ресурсы виртуализируемы. Записи БД должны блокироваться при записи.

### Атака условия удержания и ожидания

Заставить процесс **сразу запросить все нужные ресурсы**. Либо все получает, либо ждёт.

Проблема:
- Многие процессы не знают заранее, что им понадобится.
- Ресурсы используются неоптимально (плоттер удерживается всё время выполнения, хотя нужен в конце).

Альтернатива: при запросе нового ресурса процесс должен сначала освободить все, что удерживает.

### Атака условия невыгружаемости

Виртуализация: вместо физического принтера — очередь печати. Захваченный «ресурс» можно перенести.

Не все ресурсы виртуализируемы.

### Атака условия циклического ожидания

**Самый практичный подход.** Два варианта:

**1. Правило «одного ресурса».** Процесс может иметь только один ресурс одновременно. Запрос нового → освободить старый. Слишком ограниченно.

**2. Глобальная нумерация ресурсов.** Все ресурсы пронумерованы. Процесс может запрашивать ресурсы **только в порядке возрастания номеров**.

Если процессы могут запрашивать ресурсы только по нумерации, цикл в графе **невозможен**. Доказательство: предположим, есть цикл `P1 → R(i) → P2 → R(j) → P1`. Тогда P1 удерживает R(j) и просит R(i), значит `i > j`. P2 удерживает R(i) и просит R(j), значит `j > i`. Противоречие.

Слегка ослабленный вариант: «ресурс с большим номером можно получить только если все удерживаемые имеют меньшие номера».

### Сравнение

| Стратегия | Удобство | Безопасность | Использование |
|---|---|---|---|
| Игнорирование | + | – | Linux, Windows |
| Обнаружение | – | + (но с потерями) | Базы данных |
| Уклонение | – | + | Редко |
| Предотвращение (нумерация) | средне | + | Системы реального времени, ядра ОС |

## 46. Критические секции и доступ к данным. Отличие в процессах и потоках.
*Источник: rtos.md, Глава 6.1 «Задача синхронизации»; общие принципы*

**Критическая секция** — участок кода, где происходит работа с **разделяемыми данными**. Если в неё могут одновременно войти два потока/процесса — данные могут быть повреждены (race condition).

### Различия для потоков и процессов

| Параметр | Потоки одного процесса | Процессы |
|---|---|---|
| Разделяемая память | По умолчанию (общее АП) | Только через явное создание (shm) |
| Сложность синхронизации | Низкая | Выше |
| Используемые примитивы | pthread_mutex, atomic | shm + sysv/posix sem |
| Производительность синхронизации | Высокая | Ниже (system calls) |
| Защита от ошибок | Нет | Есть (MMU) |

### Потоки

- Все потоки одного процесса видят **одни и те же глобальные переменные, кучу, статические данные**.
- Критическая секция нужна **очень часто** — почти любая работа с общими данными.
- Примитивы: `pthread_mutex_t`, `atomic_int`, `pthread_rwlock_t`, `pthread_cond_t`.
- Реализованы в user space через futex — быстро.

Пример:
```c
pthread_mutex_t lock = PTHREAD_MUTEX_INITIALIZER;
int counter = 0;

void *worker(void *arg) {
    pthread_mutex_lock(&lock);
    counter++;          // критическая секция
    pthread_mutex_unlock(&lock);
}
```

### Процессы

- Каждый процесс имеет **своё** АП. По умолчанию данные **не разделяются**.
- Критическая секция нужна только когда есть **явно созданная** общая память.
- Примитивы должны находиться **в самой общей памяти** или быть доступны через файловую систему.
- Применяются POSIX semaphores (`sem_open`), System V (`semget`).

Пример с разделяемой памятью:
```c
int fd = shm_open("/shm", O_CREAT|O_RDWR, 0666);
ftruncate(fd, sizeof(shared_t));
shared_t *s = mmap(NULL, sizeof(shared_t), PROT_READ|PROT_WRITE, MAP_SHARED, fd, 0);

// Инициализация мьютекса с атрибутом PSHARED:
pthread_mutexattr_t attr;
pthread_mutexattr_init(&attr);
pthread_mutexattr_setpshared(&attr, PTHREAD_PROCESS_SHARED);
pthread_mutex_init(&s->lock, &attr);
```

### Когда о чём не беспокоиться

- В потоках **обязательно** защищать любой доступ к общим данным.
- В процессах достаточно защищать **shm-области** и IPC-объекты.
- Глобальные переменные обычных процессов **не разделяются** — синхронизация не нужна.

### Выбор примитива

- **Потоки одного процесса:** futex-based mutex (быстрый).
- **Процессы:** semaphore в shm, file lock (`flock`).
- **Между сетью:** распределённые алгоритмы (Lamport, Paxos).

## 47. Критическая секция. Запрещение прерываний. Достоинства и недостатки. Пример.
*Источник: rtos.md, Глава 6.2.1 «Запрет прерываний и вытеснения»; tanenbaum.md, Глава 2.3.3*

### Идея

В **однопроцессорной** системе переключение между потоками происходит **только при прерывании** (таймер, I/O). Если запретить прерывания → переключение невозможно → текущий поток выполняется монопольно → нет конкуренции.

### Реализация

```c
disable_interrupts();
// критическая секция
enable_interrupts();
```

В терминах HAL — повышение SPL до SYNC:

```c
// Источник: rtos.md, раздел 6.2.1
spl_t prev_spl = raise_spl(SPL_SYNC);
// ... критическая секция ...
lower_spl(prev_spl);
```

На x86: `cli` / `sti`. На ARM: маска регистра PRIMASK.

### Достоинства

- **Очень быстро.** Одна инструкция.
- **Минимальные накладные расходы.**
- **Простота.**
- **Защищает от всех источников** — потоков, обработчиков прерываний, DPC.

### Недостатки

- **Не работает на SMP.** Запрет на одном CPU не блокирует другие.
- **Грубая гранулярность.** Затрагивает всю систему, а не только участвующие объекты. Из rtos.md: «как система светофоров с двумя положениями для всего города».
- **Влияет на время реакции.** Прерывания не обрабатываются → пропадает реактивность.
- **В РВ-системах опасно.** Если критическая секция долгая → большие задержки реакции.
- **Опасно использовать в пользовательском коде** — обычно требует привилегий ядра.

### Пример: атомарная вставка в связный список

```c
typedef struct node { int val; struct node *next; } node_t;
node_t *head;

void push(int v) {
    node_t *n = malloc(sizeof(node_t));
    n->val = v;

    disable_interrupts();
    n->next = head;
    head = n;
    enable_interrupts();
}
```

Без запрета прерываний обработчик прерывания, делающий аналогичную вставку, мог бы повредить список.

### Применение

- В **простых RTOS** на однопроцессорных системах.
- В **обработчиках прерываний** ядра.
- Для очень коротких критических секций.
- В embedded системах без MMU/SMP.

В Linux x86 макросы: `local_irq_save(flags) / local_irq_restore(flags)`.

## 48. Критическая секция. Запрещение вытеснения. Достоинства и недостатки. Пример.
*Источник: rtos.md, Глава 5.3.6 «Блокирование вытеснения»; общие принципы*

### Идея

В отличие от запрета прерываний — **разрешаем** прерываниям приходить и обрабатываться, но **запрещаем** планировщику переключать поток. Текущий поток выполняется до явного освобождения CPU.

### Реализация

```c
preempt_disable();
// критическая секция
preempt_enable();
```

В rtos.md: `fx_sched_lock()` / `fx_sched_unlock()`. В Linux: `preempt_disable()` / `preempt_enable()`.

Внутри обычно увеличивается счётчик `preempt_count`. Планировщик проверяет: если ≠ 0 → не переключать.

### Достоинства

- **Прерывания продолжают работать.** Время реакции на прерывания не ухудшается.
- **Менее агрессивно**, чем запрет прерываний.
- **Лучше для систем с критическим временем реакции на прерывания.**
- **Простая реализация.**

### Недостатки

- **Не защищает от обработчиков прерываний.** Если данные разделяются с прерыванием — недостаточно.
- **Не работает на SMP.** Другие CPU не знают о запрете вытеснения на нашем.
- **Длинная критическая секция** блокирует переключение → проблемы с откликом.

### Пример

```c
preempt_disable();
// безопасно: планировщик не вытеснит нас
do_some_work();
modify_shared_data();
preempt_enable();
```

Если в `do_some_work()` происходит прерывание — оно нормально обрабатывается, а после возврата мы продолжаем работу.

### Применение

- В Linux — часто **в комбинации со спинлоками**.
- Для защиты от других потоков, но не от прерываний.
- Для коротких операций, где запрет прерываний был бы избыточен.

В Linux: `spin_lock()` неявно делает `preempt_disable()`. `spin_lock_irqsave()` дополнительно отключает прерывания.

## 49. Критическая секция на основе условий и переменных. Проблемы.
*Источник: tanenbaum.md, Глава 2.3.3 (попытки взаимного исключения); rtos.md*

### Программные попытки взаимного исключения

До появления аппаратных примитивов синхронизации делались попытки реализовать взаимное исключение на «голой» памяти. Результат — серия алгоритмов с **тонкими проблемами**.

### Попытка 1: одиночная переменная

```c
volatile int lock = 0;

void enter_cs() {
    while (lock != 0) ;    // активное ожидание
    lock = 1;
}

void exit_cs() {
    lock = 0;
}
```

**Проблема:** между проверкой `while (lock != 0)` и записью `lock = 1` может произойти переключение. Оба потока увидят 0 и оба войдут в критическую секцию.

### Попытка 2: переменная очереди

```c
volatile int turn = 0;

// Поток 0
void enter_cs0() {
    while (turn != 0) ;
    // CS
    turn = 1;
}

// Поток 1
void enter_cs1() {
    while (turn != 1) ;
    // CS
    turn = 0;
}
```

**Проблема:** строгое чередование. Если поток 0 не хочет в CS, поток 1 всё равно ждёт его очереди. Один поток блокирует другой даже вне CS.

### Алгоритм Петерсона (1981)

Корректное решение для двух потоков без аппаратной поддержки:

```c
volatile bool interested[2] = {false, false};
volatile int turn = 0;

void enter_cs(int self) {
    int other = 1 - self;
    interested[self] = true;
    turn = self;
    while (interested[other] && turn == self) ;
}

void exit_cs(int self) {
    interested[self] = false;
}
```

Работает, но:
- **Только для 2 потоков** (есть обобщения, но сложные).
- **Активное ожидание** — расход CPU.
- **Не работает на современных процессорах** без барьеров памяти — компилятор и CPU могут переупорядочить операции.

### Условные переменные

Альтернатива spinning — **блокирующее ожидание** на условии:

```c
pthread_mutex_t mtx;
pthread_cond_t cv;
int condition_met = 0;

void wait_for_condition() {
    pthread_mutex_lock(&mtx);
    while (!condition_met)
        pthread_cond_wait(&cv, &mtx);
    pthread_mutex_unlock(&mtx);
}

void signal_condition() {
    pthread_mutex_lock(&mtx);
    condition_met = 1;
    pthread_cond_signal(&cv);
    pthread_mutex_unlock(&mtx);
}
```

### Проблемы условных переменных

**1. Spurious wakeup (фиктивное пробуждение).** Поток может проснуться без сигнала. Поэтому проверка условия в `while` (а не `if`) обязательна.

**2. Lost wakeup (потерянный сигнал).** Если `signal` придёт до того, как поток дошёл до `wait` — он потеряется. Поэтому проверка условия под мьютексом обязательна.

**3. Thundering herd.** `broadcast` будит всех ожидающих, хотя обычно только один может продолжить. Все проверяют условие, остальные снова засыпают.

**4. Самоблокировка (deadlock).** Если поток держит мьютекс и пытается ждать условие, которое может выполнить только он сам.

**5. Зависимость от мьютекса.** Условная переменная без мьютекса бессмысленна — атомарность «проверить + ждать» обеспечивает мьютекс.

### Главный принцип

**Все программные попытки** взаимного исключения **без аппаратной поддержки** имеют тонкие проблемы (race condition, потерянные пробуждения, переупорядочивание). Современные системы используют **аппаратные примитивы**: TSL, CAS, atomic operations — которые обеспечивают атомарность операций «проверь и измени» (вопрос 52, 54).

## 50. Понятие синхронизации. Уведомления, события. Двухстороннее рандеву.
*Источник: rtos.md, Глава 6.1, 6.2.5 «Синхронизация с прямым уведомлением», 6.3 «События»*

### Понятие синхронизации

Из rtos.md (6.1):
> «Во всех ситуациях, где присутствует некоторый общий для нескольких пользователей ресурс, всегда неизбежно возникает проблема синхронизации доступа к этому ресурсу».

**Синхронизация** делится на два больших класса:

1. **Взаимное исключение (mutual exclusion).** Только один может работать с ресурсом одновременно.
2. **Уведомление (notification).** Один сообщает другому о событии. Получатель ждёт.

### Уведомления и события

**Уведомление** — отправитель сигнализирует получателю. Получатель блокируется до сигнала.

Из rtos.md (6.3):
> «Событие имеет внутреннее состояние в виде бинарного значения, указывающего на то, находится ли объект в активном состоянии или нет, а также 3 основных метода:»

```c
int event_set(event_t *event);              // сделать активным
int event_reset(event_t *event);            // сбросить
int event_timedwait(event_t *event, uint32_t timeout);   // ждать активности
```

Поведение:
- Поток ждёт на неактивном событии → блокируется.
- При `event_set()` все ожидающие пробуждаются.
- Активное состояние сохраняется до `event_reset()`.
- Ожидание активного события не блокирует.

### Пример с обработчиком прерывания (rtos.md)

```c
// Источник: rtos.md, раздел 6.3 «События»
static event_t event;

void ISR(void *arg) {
    /* отключить прерывания от устройства */
    dpc_insert(&dpc, DPC, NULL);
}

void DPC(void *arg) {
    event_set(&event);
}

void Thread(void *arg) {
    do {
        event_timedwait(&event, ...);
        event_reset(&event);
        /* обработка прерывания */
        /* включить прерывания от устройства */
    } while (1);
}
```

### Достоинства событий

- **Простой интерфейс.**
- **Широковещательный (1-к-N).** Все ожидающие просыпаются одновременно.
- **Можно ждать с таймаутом.**

### Недостатки событий

- **Повторные сигналы теряются** (как у битовой маски).
- **Не подходит для взаимного исключения** (нет счётчика).
- **Время `event_set` зависит от числа ожидающих** — не детерминировано.

### Прямое уведомление

Без отдельного объекта синхронизации — через приостановку/возобновление потока:

```c
void ISR(void *arg) {
    dpc_insert(&dpc, DPC, NULL);
}

void DPC(void *arg) {
    thread_resume(&thread);
}

void Thread(void *arg) {
    do {
        sched_lock(...);
        thread_suspend();
        sched_unlock(...);
        /* обработка */
    } while (1);
}
```

**Однонаправленное рандеву (unilateral rendezvous)** — отправитель сигнализирует и продолжает работу, не ожидая ответа.

### Двухстороннее рандеву (bilateral rendezvous)

Из rtos.md:
> «Два события могут использоваться для синхронизации между клиентом и сервером. В этом случае каждый из них использует свой объект для ожидания, и второй — для уведомления другой стороны. Такой тип взаимодействия предусматривает не только уведомление о каком-либо запросе, но и ожидание ответа, поэтому называется двухсторонним рандеву (bilateral rendezvous)».

**Двухстороннее рандеву = запрос + ответ.** Используется в RPC, клиент-сервер.

```c
event_t request, response;
data_t shared_data;

void Client() {
    shared_data = prepare_request();
    event_set(&request);          // отправить запрос
    event_timedwait(&response, ...);  // ждать ответ
    use(shared_data);             // получить ответ
}

void Server() {
    while (1) {
        event_timedwait(&request, ...);  // ждать запрос
        event_reset(&request);
        shared_data = process(shared_data);
        event_set(&response);     // отправить ответ
    }
}
```

### Применение

- **RPC** — клиент посылает запрос, сервер обрабатывает, шлёт ответ.
- **Producer/consumer с подтверждением** — производитель ждёт, что потребитель забрал.
- **Распределённые системы** — request-reply паттерн.

### Сравнение односторонне и двустороннее

| Параметр | Унилатеральное | Билатеральное |
|---|---|---|
| Блокировка отправителя | Нет | Да (ждёт ответа) |
| Поток выполнения | Параллельный | Синхронный |
| Применение | Нотификация о событии | RPC, запрос-ответ |
| Сложность | Простое | Сложнее |

В Linux POSIX message queues, signals — унилатеральное. RPC, gRPC, REST API — билатеральное.
