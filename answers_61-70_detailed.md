## 61. Мьютексы. Повторный захват мьютекса.
*Источник: rtos.md, Глава 6.8 «Мьютекс»; robert_lav.md, Глава 10*

### Зачем нужен мьютекс

Бинарный семафор уже даёт взаимное исключение (см. вопрос 59). Зачем тогда отдельный примитив?

Из rtos.md (6.8):
> «Основной проблемой, вызывающей инверсию приоритета, является тот факт, что в системе нет информации о том, какой именно поток должен сделать инкремент семафора. Если бы эта информация была, можно было бы повысить приоритет тому потоку, который должен сделать `sem_post`, что решило бы проблему. Поэтому со временем был разработан объект называемый мьютексом (mutual exclusion)».

**Мьютекс** = бинарный семафор + **знание о владельце**.

### Интерфейс

```c
// Источник: rtos.md, раздел 6.8
int mutex_timedacquire(mutex_t* mutex, uint32_t timeout);
int mutex_release(mutex_t* mutex);
```

### Ключевые отличия от семафора

**1. Владелец.**

При захвате мьютекса в его структуре сохраняется указатель на захвативший поток. Знание владельца даёт возможности:
- Защита от приоритетной инверсии (вопрос 58).
- Обнаружение deadlock.
- Проверка корректности освобождения.

**2. Освобождать может только владелец.**

`mutex_release` проверяет, что вызывающий поток — владелец. Иначе ошибка. Это **существенное отличие** от семафора, где `post` может вызвать кто угодно.

**3. Только для взаимного исключения.**

Не используется для нотификаций.

Из rtos.md:
> «Ещё одним отличием мьютексов от семафоров является возможность использования мьютекса только для задач взаимного исключения, мьютексы не могут использоваться для уведомления о событиях. Происходит это главным образом потому, что, как правило, во встроенных системах уведомления используются из обработчиков прерываний. Обработчик прерывания не может блокироваться и использовать функции ожидания, а значит, не может быть владельцем мьютекса, которым может быть только поток».

### Проблема повторного захвата

**Сценарий:**

```c
mutex_lock(&m);
function_using_mutex();   // эта функция тоже захватывает m!
mutex_unlock(&m);

void function_using_mutex() {
    mutex_lock(&m);   // deadlock — мы уже владеем!
    // ...
    mutex_unlock(&m);
}
```

Поток уже владеет мьютексом, пытается захватить его снова. Что делать?

С семафором: поток просто заблокируется. Бесконечно, потому что **никто не освободит** — кроме самого заблокированного.

Из rtos.md:
> «В случае с мьютексом такой подход не сработает, если поток уже является его владельцем, то, в том случае, если этот же поток вновь попытается его подождать, никакой другой поток уже не сможет вывести систему из этого состояния, поскольку освободить мьютекс может только его владелец».

### Два подхода к решению

**Вариант 1: вернуть ошибку.**

```c
int mutex_lock(mutex_t *m) {
    if (m->owner == current_thread)
        return -EDEADLK;   // error: я уже владею
    // обычный захват
}
```

Приложение получает шанс обнаружить и исправить ошибку логики.

Из rtos.md:
> «Первый вариант предполагает анализ текущего владельца мьютекса, и, если он совпадает с текущим потоком, из функции возвращается ошибка, мьютекс при этом остается захваченным, а приложение получает возможность обнаружить и обработать такой тип взаимоблокировки».

**Вариант 2: рекурсивный мьютекс.**

```c
int mutex_lock(mutex_t *m) {
    if (m->owner == current_thread) {
        m->count++;       // увеличить счётчик вложения
        return SUCCESS;
    }
    // обычный захват
}

int mutex_unlock(mutex_t *m) {
    if (--m->count > 0)
        return SUCCESS;   // ещё не до конца отпустили
    // действительно освободить
}
```

Из rtos.md:
> «Второй вариант увеличивает внутренний счётчик вложения мьютекса, таким образом, его освобождение произойдёт тогда, когда поток-владелец вызовет операцию освобождения столько раз, сколько раз была вызвана операция захвата. Мьютексы поддерживающие такую возможность называются рекурсивными».

### Пример рекурсивного мьютекса

```c
mutex_lock(&m);     // count = 1
{
    mutex_lock(&m); // count = 2 (тот же поток)
    {
        mutex_lock(&m); // count = 3
        // ...
        mutex_unlock(&m); // count = 2
    }
    mutex_unlock(&m); // count = 1
}
mutex_unlock(&m);   // count = 0 — действительно освобождён
```

### Преимущества рекурсивных мьютексов

- **Удобно** для рекурсивных функций.
- **Не нужно специально отслеживать** вложение.
- Полезно для библиотечного кода, где функции могут вызывать друг друга.

### Недостатки рекурсивных мьютексов

- **Скрывают плохой дизайн.** Если функция случайно блокируется на собственном мьютексе — это симптом архитектурной проблемы.
- **Сложнее реализация.**
- **Сложнее отладка** — race conditions замаскированы.

### В Linux

Обычный `mutex_t` в ядре Linux **НЕ рекурсивный**:

Из robert_lav.md (Глава 10):
> «Повторные захваты и освобождения мьютексов не разрешаются. Это означает, что нельзя повторно захватить тот же самый мьютекс и нельзя освободить уже освобождённый мьютекс».

Ядро Linux придерживается дисциплины: если нужна рекурсия — это плохой дизайн.

В пользовательском пространстве (POSIX) можно создать рекурсивный:
```c
pthread_mutexattr_t attr;
pthread_mutexattr_init(&attr);
pthread_mutexattr_settype(&attr, PTHREAD_MUTEX_RECURSIVE);
pthread_mutex_init(&m, &attr);
```

### Ограничения мьютексов в Linux

Из robert_lav.md:
- В один момент только одна задача владеет.
- Только владелец освобождает.
- Не рекурсивные.
- Процесс не может завершиться без освобождения.
- Нельзя в обработчиках прерываний.
- Только официальный API (нельзя копировать, ручная инициализация).

### Проблема: завершение потока с мьютексом

Если поток умирает, держа мьютекс, **никто не сможет его освободить**.

Из rtos.md:
> «Самый оптимальный вариант — не завершать потоки во время критической секции. Один из возможных подходов: отслеживать мьютексы, которыми владеет поток, как часть состояния потока (список захваченных мьютексов, находящийся в TCB), что даёт возможность их освободить при завершении работы потока. Критическая секция при этом, возможно, остается в неконсистентном состоянии».

В POSIX есть **robust mutexes** — `PTHREAD_MUTEX_ROBUST`. При смерти владельца мьютекс маркируется как `EOWNERDEAD`, и следующий захватывающий получает уведомление.

### Производительность

Из rtos.md:
> «Из-за дополнительной функциональности (по сравнению с семафорами) мьютексы, в целом, работают медленнее в большинстве реализаций, семафор является более легковесным примитивом, однако для задач взаимного исключения предпочтительно использование именно мьютексов».

### Резюме

- Мьютекс = бинарный семафор + владелец.
- Освобождает только владелец.
- Защита от приоритетной инверсии.
- Повторный захват: либо ошибка, либо рекурсивный.
- В Linux ядре — не рекурсивный.

## 62. Мониторы. Отличие мьютекса от семафора. Отличие мониторов от семафоров и мьютексов.
*Источник: rtos.md, Глава 6.9 «Мониторы»*

### Отличие мьютекса от семафора

Подробно — в вопросе 61. Кратко:

| Параметр | Семафор | Мьютекс |
|---|---|---|
| Внутреннее состояние | Счётчик | Флаг + владелец |
| Кто может освободить | Любой | Только владелец |
| Универсальность | Высокая (mutex + сигналы) | Только взаимное исключение |
| Используется в IRQ | Нет (для блокирующего) | Нет |
| Защита от инверсии | Нет | Часто да |
| Семантика | Дейкстра 1965 | Появился позже как уточнение |

### Зачем нужны мониторы

Семафоры — мощный примитив, но имеют **ограничения**:
1. **Состояние внутри** — счётчик. Сложные задачи требуют переформулировки в терминах счётчика, что не всегда естественно.
2. **Сложная композиция.** Например, задача читателей-писателей требует нескольких семафоров.
3. **Риск ошибок.** Несколько семафоров → возможные deadlock при разном порядке.

Из rtos.md (6.9):
> «Можно считать, что проблема примитивов, хранящих внутреннее состояние, является то, что они требуют, чтобы всякая задача могла быть переформулирована таким образом, чтобы состояние задачи отображалось на состояние примитива. В случае, если состояние алгоритма сложное и включает много параметров, доступ к нему должен синхронизироваться с помощью примитивов, обеспечивающих взаимное исключение и тут может возникнуть проблема блокировки внутри критической секции».

**Идея монитора** (Hoare 1974, Brinch Hansen 1973):

Из rtos.md:
> «Более универсальным случаем было бы попробовать реализовать примитив таким образом, чтобы пользователь сам определял состояние, соответствующее задаче, а не пытался придумать, как состояние его задачи отображается на предопределённое поведение такого примитива как семафор».

### Что такое монитор

Из rtos.md:
> «Монитором считается не просто объект, наподобие семафора, а совокупность следующих сущностей: набор разделяемых данных и функций, при выполнении которых обеспечивается взаимное исключение, а также один или более особых объектов, называемых условными переменными. То есть, в терминах ООП, монитор это класс, для методов которого обеспечивается взаимное исключение».

**Монитор =**
- **Класс** с разделяемыми данными.
- **Методы**, выполняющиеся под автоматическим mutual exclusion.
- **Условные переменные** для условной синхронизации.

### Поддержка в языках

Монитор задуман как **встроенный** в язык. Компилятор автоматически вставляет блокировки.

**Java:**
```java
class Counter {
    private int count = 0;

    synchronized void increment() {     // автоматический mutual exclusion
        count++;
    }

    synchronized int get() {
        return count;
    }
}
```

Каждый объект в Java имеет встроенный «монитор» — мьютекс + условная переменная.

**C#:**
```csharp
class Counter {
    private int count = 0;
    private readonly object _lock = new object();

    void Increment() {
        lock(_lock) {
            count++;
        }
    }
}
```

В C поддержки нет — нужно реализовывать через мьютекс + condvar вручную.

### Условные переменные

Ключевой элемент монитора. Из rtos.md:
> «Условная переменная играет роль очереди ожидания, имеющейся в других примитивах, но с одним отличием».

```c
int cond_timedwait(cond_t* cond, mutex_t* mutex, uint32_t timeout);
int cond_signal(cond_t* cond);
int cond_broadcast(cond_t* cond);
```

### Принцип работы

Условная переменная **обязательно** используется в паре с мьютексом.

**`cond_wait(&cv, &mtx)`:**
1. Поток уже владеет mtx.
2. Атомарно: помещает себя в очередь ожидания cv + освобождает mtx.
3. Блокируется.
4. По сигналу: просыпается и **заново захватывает mtx**.

Из rtos.md:
> «Ожидание условной переменной приводит к тому что поток помещается в очередь, затем разблокируется мьютекс и только после этого поток блокируется. За счёт того, что поток начинает ожидание виртуально ДО того, как он освободил мьютекс, возможные уведомления со стороны других потоков не будут пропущены».

### Использование

```c
mutex_lock(&mtx);
while (!condition)
    cond_wait(&cv, &mtx);
// CS с выполненным условием
mutex_unlock(&mtx);
```

```c
mutex_lock(&mtx);
condition = true;
cond_signal(&cv);
mutex_unlock(&mtx);
```

### Пример: реализация семафора через монитор

Из rtos.md:

```c
cond_t cv;
mutex_t mtx;
uint32_t semaphore;

void sem_wait() {
    mutex_lock(&mtx);
    while (semaphore == 0)
        cond_wait(&cv, &mtx);
    --semaphore;
    mutex_unlock(&mtx);
}

void sem_post() {
    mutex_lock(&mtx);
    ++semaphore;
    cond_signal(&cv);
    mutex_unlock(&mtx);
}
```

Здесь:
- **Состояние** (счётчик `semaphore`) — в пользовательской переменной, не в примитиве.
- **Мьютекс** защищает доступ к ней.
- **Условная переменная** — для блокирующего ожидания.

### Достоинства монитора

- **Гибкость:** состояние определяется задачей.
- **Универсальность:** реализуем любой синхронизационный примитив.
- **Удобство:** автоматическое взаимное исключение методов.
- **Структурированность:** инкапсуляция (ООП-стиль).

### Недостатки

- **Требует поддержки языка** для полной красоты.
- **В C мониторы реализуются вручную** — преимущество перед прямыми мьютекс+condvar уменьшается.
- **Двойная блокировка** при сигнализации (придётся отпустить mtx, потом захватить заново при пробуждении).

### Резюме

- Монитор — высокоуровневый примитив: класс + автоматическое взаимное исключение + условные переменные.
- Состояние задачи в пользовательских переменных, а не в примитиве.
- Хорошо поддерживается в Java, C#.
- В C реализуется через mutex + condvar.
- Более гибок, чем семафоры; универсальный примитив.

## 63. Условные переменные. Самоблокировка. Пример возникновения.
*Источник: rtos.md, Глава 6.9; общие принципы синхронизации*

### Основы условных переменных

См. вопрос 62. Здесь — о проблемах.

### Правильный паттерн

```c
mutex_lock(&mtx);
while (!condition)           // ← while, не if!
    cond_wait(&cv, &mtx);
// CS — условие выполнено
mutex_unlock(&mtx);
```

```c
mutex_lock(&mtx);
condition = true;
cond_signal(&cv);            // или cond_broadcast
mutex_unlock(&mtx);
```

### Проблема 1: Spurious wakeup

Поток может проснуться **без сигнала**. POSIX это разрешает.

Причины:
- Внутренние оптимизации pthread.
- Получение сигнала ОС.
- Гонки в реализации.

Решение: **проверка условия в цикле**:

```c
while (!condition) cond_wait(...);   // правильно
if   (!condition) cond_wait(...);     // НЕПРАВИЛЬНО
```

### Проблема 2: Lost wakeup

Если `signal` приходит **до** того, как поток дошёл до `wait` — сигнал потерян.

**Пример ошибки:**

```c
// Поток-ожидающий
if (!condition) {
    cond_wait(&cv, &mtx);    // что если сигнал уже прошёл?
}

// Поток-сигнальный
condition = true;
cond_signal(&cv);
```

**Решение через мьютекс:**

```c
// Поток-ожидающий
mutex_lock(&mtx);
while (!condition)
    cond_wait(&cv, &mtx);    // атомарность: проверка → ожидание
mutex_unlock(&mtx);

// Поток-сигнальный
mutex_lock(&mtx);
condition = true;
cond_signal(&cv);
mutex_unlock(&mtx);
```

Мьютекс гарантирует:
- Поток-ожидающий проверяет condition под мьютексом.
- Сигнальный поток меняет condition под мьютексом.
- `cond_wait` атомарно отпускает мьютекс и засыпает → сигнал не пропадёт.

### Проблема 3: Самоблокировка

**Самоблокировка** — когда поток ждёт условия, которое **никто не может выполнить**, потому что:
- Никто не пишет в данные.
- Только сам поток мог бы это сделать, но он заблокирован.

### Пример 1: Producer-consumer без производителя

```c
void consume() {
    mutex_lock(&mtx);
    while (queue_is_empty())
        cond_wait(&not_empty, &mtx);
    item = queue_pop();
    mutex_unlock(&mtx);
}
```

Если нет производителей, потребитель ждёт **навсегда**. Это самоблокировка.

Решение:
- **Таймаут** (`cond_timedwait`).
- **Контроль:** проверять, есть ли активные производители.
- **Завершающий сигнал** при остановке системы.

### Пример 2: Цикл зависимостей

```c
// Поток A
mutex_lock(&mtx);
while (!event_b)
    cond_wait(&cv_a, &mtx);
// ...
event_a = true;
cond_signal(&cv_b);
mutex_unlock(&mtx);

// Поток B
mutex_lock(&mtx);
while (!event_a)
    cond_wait(&cv_b, &mtx);
// ...
event_b = true;
cond_signal(&cv_a);
mutex_unlock(&mtx);
```

A ждёт event_b. B ждёт event_a. **Циклическая зависимость → оба ждут вечно.**

### Пример 3: Самоблокировка через тот же мьютекс

```c
void some_function() {
    mutex_lock(&mtx);
    // ...
    while (!condition)
        cond_wait(&cv, &mtx);
    // Условие должно быть выполнено самим этим потоком,
    // но он заблокирован!
    do_thing_that_sets_condition();
    mutex_unlock(&mtx);
}
```

Если `condition` устанавливается только в этом же потоке — он застрянет на `cond_wait`.

### Пример 4: Thundering herd при broadcast

```c
// Несколько потоков ждут на cv
mutex_lock(&mtx);
while (!my_specific_condition)
    cond_wait(&cv, &mtx);
mutex_unlock(&mtx);

// Сигнализирующий
mutex_lock(&mtx);
state = updated;
cond_broadcast(&cv);   // разбудит всех
mutex_unlock(&mtx);
```

Все потоки просыпаются, по одному захватывают mtx, проверяют **своё** условие. Если только один может продолжить — остальные снова засыпают. Большие накладные расходы.

Решение: использовать `cond_signal` (будит одного) или разные условные переменные для разных условий.

### Решения от самоблокировки

**1. Таймауты.**
```c
struct timespec timeout = ...;
int ret = cond_timedwait(&cv, &mtx, &timeout);
if (ret == ETIMEDOUT) {
    // ваше действие при таймауте
}
```

**2. Условные таблицы зависимостей.** Перед wait проверять, есть ли «производители» условия. Если нет → не ждать.

**3. Использование флагов «нет больше работы».**

**4. Подробное проектирование** алгоритма — кто что устанавливает, когда сигнализирует.

### Резюме

- Условные переменные требуют тщательного программирования.
- Spurious wakeup → проверка в цикле обязательна.
- Lost wakeup → проверка и signal под мьютексом.
- Самоблокировка возможна при ожидании условия, которое никто не выполнит.
- Решения: таймауты, тщательное проектирование, broadcast для разнородных условий.

## 64. Отключение мультипрограммного режима работы ядра.
*Источник: robert_lav.md, Глава 10 «Отключение мультипрограммного режима работы ядра»*

### Что такое мультипрограммный режим работы ядра

Из robert_lav.md:
> «Поскольку ядро системы Linux является вытесняемым, выполнение некоторого процесса в ядре может быть приостановлено в произвольный момент времени и управление передано более высокоприоритетному процессу. Это означает, что новый процесс может начать своё выполнение в том же критическом участке кода, в котором выполнялся вытесненный процесс».

**Вытесняемое ядро** = ядро Linux, в котором планировщик может **переключить процесс прямо во время выполнения системного вызова**, если появится более приоритетный готовый процесс.

Это улучшает отзывчивость: пользовательский ввод не задерживается надолго даже если другой процесс в это время сидит в ядре.

### Преимущества вытесняемого ядра

- Лучше время отклика для интерактивных задач.
- Real-time возможности (с PREEMPT_RT).
- Справедливое планирование.

### Проблема

Вытеснение в середине критического участка → возможные конфликты с другими потоками на том же CPU.

### Когда нужно отключать вытеснение

**1. Защита через спинлоки** (неявно).

В Linux `spin_lock` неявно делает `preempt_disable`. Иначе:
- Поток A на CPU 0 захватил спинлок.
- Планировщик вытеснил A, начал поток B на CPU 0.
- B тоже хочет тот же спинлок.
- B активно ждёт. A никогда не получит CPU обратно (на однопроцессорной системе).
- Deadlock.

**2. Per-CPU данные.**

Из robert_lav.md:
> «Чаще всего такая ситуация возникает из-за доступа к данным, привязанным к определённому процессору (per-processor data). Если каждый процессор обращается только к своим уникальным данным, то защищать их с помощью блокировок нет никакой необходимости. Дело в том, что в подобных случаях к этим данным может обратиться только один процессор. Однако если никакая из блокировок не удерживается, ядро является вытесняемым».

Пример:

```c
int counters[NR_CPUS];

// Без preempt_disable:
counters[smp_processor_id()]++;
// 1. smp_processor_id() = 0
// 2. ВЫТЕСНЕНИЕ, поток переехал на другой CPU
// 3. Записываем в counters[0] — но мы уже на CPU 1!
```

С `preempt_disable`:

```c
preempt_disable();
counters[smp_processor_id()]++;
preempt_enable();
```

Между `disable` и `enable` поток гарантированно не мигрирует.

**3. Атомарные операции в ядре.**

Некоторые операции в ядре требуют, чтобы поток не переехал на другой CPU или не был вытеснен.

### Функции

```c
preempt_disable();           // отключить
preempt_enable();            // включить + проверить need_resched
preempt_enable_no_resched();  // включить без проверки
preempt_count();             // текущая глубина
preemptible();               // true если можно вытеснить
```

### Реализация

```c
struct thread_info {
    int preempt_count;
    // ...
};

#define preempt_disable() \
    do { current_thread_info()->preempt_count++; barrier(); } while (0)

#define preempt_enable() \
    do { \
        current_thread_info()->preempt_count--; \
        if (preempt_count() == 0 && need_resched()) \
            schedule(); \
    } while (0)
```

Планировщик проверяет `preempt_count` перед переключением:

```c
if (preempt_count() == 0) {
    // можно вытеснять
    schedule();
}
```

### Особенности

- **Рекурсивно:** disable можно вызывать вложенно, считается по счётчику.
- **Не отключает прерывания:** прерывания всё ещё приходят и обрабатываются.
- **Не работает на SMP** в смысле защиты от других CPU: запрет вытеснения только на нашем CPU.

### Пример: get_cpu() / put_cpu()

```c
int get_cpu(void) {
    preempt_disable();
    return smp_processor_id();
}

void put_cpu(void) {
    preempt_enable();
}

// Использование
int cpu = get_cpu();
counters[cpu]++;
put_cpu();
```

`get_cpu` атомарно возвращает CPU и фиксирует на нём.

### BKL — Big Kernel Lock (исторически)

Из robert_lav.md:
> «Большая блокировка ядра (Big Kernel Lock, или BKL) — это глобальная спин-блокировка, которая была создана специально для того, чтобы облегчить переход от первоначальной реализации SMP в операционной системе Linux к мелкоструктурным блокировкам».

История:
- **Ядро 2.0:** не SMP. Одна задача в ядре одновременно.
- **2.2:** введена SMP-поддержка. Чтобы избежать переписывания всего, ввели BKL — одна глобальная блокировка на всё ядро.
- **2.6 → 3.x:** постепенно заменяли BKL мелкоструктурными блокировками.
- **3.x:** BKL **удалён**.

Свойства BKL:
- **Рекурсивная** (в отличие от обычных спинлоков).
- **Автоматически освобождается** при переходе в sleep.
- **Только в контексте процесса.**

```c
lock_kernel();
// весь код ядра защищён
unlock_kernel();
```

Недостатки BKL:
- Очень грубая блокировка.
- Плохая масштабируемость.
- Заменена мелкоструктурными блокировками.

### preempt_disable vs spin_lock vs disable_irq

| Действие | preempt_disable | spin_lock | local_irq_disable |
|---|---|---|---|
| Защита от вытеснения (наш CPU) | Да | Да (неявно) | Да |
| Защита от других CPU | Нет | Да | Нет |
| Защита от прерываний (наш CPU) | Нет | Нет | Да |
| Накладные расходы | Минимум | Низкие | Минимум |
| Use case | Per-CPU data | Общие данные | Защита от IRQ |

### Применение

- **Per-CPU переменные.**
- **Внутри спинлоков** (неявно).
- **Read-side RCU** — `rcu_read_lock` неявно отключает вытеснение в classic RCU.
- **Получение `smp_processor_id()` с фиксацией.**

### Резюме

- Вытесняемое ядро → проблемы с per-CPU данными и спинлоками.
- `preempt_disable/enable` отключает вытеснение на текущем CPU.
- Прерывания продолжают обрабатываться.
- Не защищает от других CPU.
- Используется в спинлоках (неявно), per-CPU переменных, RCU.
- BKL — исторический «большой замок», удалён в современных ядрах.

## 65. Порядок выполнения операций и барьеры.
*Источник: rtos.md, Глава 2.2.5 «Модель памяти»*

### Проблема порядка операций

В программе мы пишем последовательность инструкций. Кажется, что они выполняются именно в этом порядке. На современных процессорах это **не так**.

### Источники переупорядочивания

**1. Компилятор.**

При оптимизации компилятор может:
- Переставить независимые операции.
- Кэшировать значение в регистре.
- Объединить запись (`a = 5; a = 6` → `a = 6`).

**2. Процессор.**

- **Out-of-order execution** — выполнение в порядке готовности данных.
- **Store buffer** — записи накапливаются в буфере, видны другим CPU позже.
- **Speculative execution** — выполнение «впрок» по предсказанию ветвления.

**3. Кэши.**

- Write-back: записи попадают в кэш, на шину выходят позже.
- Когерентность: разные CPU могут видеть значения в разном порядке.

### Модели памяти

**Модель памяти** = правила, по которым процессор разрешает переупорядочивать обращения.

Из rtos.md (2.2.5):
> «Если никакие оптимизации не выполняются, тогда все процессоры обращаются к памяти в том порядке, в котором обращается программа, а все обращения в целом формируют последовательность, которая получается путём чередования запросов от всех процессоров. Такая модель называется последовательно консистентной (sequentially consistent)».

**Последовательная согласованность** — идеал, но дорого.

**TSO (Total Store Order)** — x86. Запись может «опередить» чтение, но не наоборот. Записи каждого CPU видны другим в порядке их выполнения.

**Relaxed** — ARM, POWER. Допускаются почти любые переупорядочивания.

### Проблема на примере

```c
// Поток-производитель
data = compute_data();
ready = 1;

// Поток-потребитель
if (ready)
    use(data);
```

Производитель:
- На x86 (TSO): записи в порядке. Но компилятор может переставить.
- На ARM: записи могут наблюдаться в обратном порядке.

Потребитель видит `ready == 1`, но data ещё не обновлено!

Из rtos.md:
> «На практике, запись флага может быть переупорядочена таким образом, что другой процессор увидит флаг равный 1 до того, как будут присвоены переменные до этого, и, следовательно, обращение может вызвать ошибку».

### Барьеры памяти

**Барьер памяти** — специальная инструкция, **запрещающая переупорядочивание** в определённой точке.

Из rtos.md:
> «Должны использоваться барьеры памяти — специальные инструкции, которые вводят ограничения на то, как и какие оптимизации процессор может применять в данном участке кода, например ограничения на переупорядочивание».

### Типы барьеров

**Полный барьер** — все обращения завершены перед следующими.

**Read barrier** — все чтения завершены.

**Write barrier** — все записи завершены.

**Acquire barrier** — последующие обращения не могут «опередить» текущее.

**Release barrier** — предыдущие обращения завершены до этого.

### x86 барьеры

Из rtos.md:
- **`LFENCE`** — завершение всех чтений.
- **`SFENCE`** — завершение всех записей.
- **`MFENCE`** — полный барьер.

На x86 многие операции с префиксом `LOCK` неявно содержат полный барьер.

### ARM барьеры

Из rtos.md:
- **`DMB`** (Data Memory Barrier) — память.
- **`DSB`** (Data Synchronization Barrier) — память + инструкции.
- **`ISB`** (Instruction Synchronization Barrier) — конвейер.

ARM имеет много опций (например, `DMB ST` — только записи).

### Исправление примера

```c
// Источник: rtos.md, раздел 2.2.5
// Поток-производитель
data = compute_data();
mem_barrier();          // данные должны быть видны до флага
ready = 1;

// Поток-потребитель
flag = ready;
mem_barrier();          // чтение data только после флага
if (flag)
    use(data);
```

### В Linux

```c
mb();           // полный барьер
rmb();          // read barrier
wmb();          // write barrier
smp_mb();       // полный, только на SMP
smp_rmb();
smp_wmb();

barrier();      // компилятор-only (asm volatile)
```

`smp_*` версии становятся no-op на однопроцессорной конфигурации.

### Барьеры компилятора

`barrier()` в Linux:
```c
#define barrier() asm volatile("" ::: "memory")
```

Не генерирует кода CPU, но запрещает компилятору переставлять обращения через эту точку.

`volatile` в C тоже частично помогает, но это не замена настоящему барьеру.

### Применение

**1. Lock-free алгоритмы.**

```c
// Lock-free stack push
new_node->next = old_top;
smp_wmb();   // запись next должна быть видна
top = new_node;
```

**2. Передача данных через флаг.**

(см. пример выше)

**3. Inter-CPU коммуникация.**

```c
// CPU 0
write_data_to_shared_memory();
smp_wmb();
flag = 1;

// CPU 1
while (!flag) ;
smp_rmb();
read_data_from_shared_memory();
```

**4. Регистры устройств (MMIO).**

Из rtos.md:
> «Особую важность этот вопрос имеет при использовании отображённых на память устройств, которые требуют определённой последовательности доступа к своим регистрам».

```c
device_reg_setup = 1;
mmiowb();    // memory-mapped I/O write barrier
device_reg_trigger = 1;
```

**5. Реализация мьютексов, спинлоков, condvar.**

```c
// spin_unlock содержит release barrier
smp_store_release(&lock, 0);
```

### Acquire/Release семантика

Современный подход в C11/C++11:

```c
atomic_store_explicit(&flag, 1, memory_order_release);
// все предыдущие записи видны до этого

if (atomic_load_explicit(&flag, memory_order_acquire)) {
    // все последующие чтения после этого
    use_data();
}
```

Более тонкая, чем полный барьер.

### Важно для встроенных систем

Из rtos.md:
> «Для систем жёсткого реального времени использование процессоров с подобными оптимизациями нежелательно, однако разрабатываемое ПО, в том числе ядро ОС, должно оставаться функционально корректными при работе в любой системе».

То есть даже если ваш процессор простой (sequential), код должен быть корректным и для relaxed-моделей.

### Резюме

- Современные процессоры и компиляторы переупорядочивают обращения для производительности.
- Без барьеров возможны race conditions, даже при использовании атомарных операций.
- Барьеры — инструкции, запрещающие переупорядочивание.
- Типы: read, write, full; acquire, release.
- Особенно важны для lock-free, MMIO, SMP-коммуникации.

## 66. Задача производителей и потребителей. Пример на семафорах. Объяснить почему две переменные.
*Источник: tanenbaum.md, Глава 2.3.5*

### Постановка задачи

Из tanenbaum.md (2.3.4):
> «Два процесса используют общий буфер фиксированного размера. Один из них, производитель, помещает информацию в буфер, а другой, потребитель, извлекает её оттуда».

**Условия:**
- Буфер имеет вместимость N.
- Производитель блокируется, если буфер **полон**.
- Потребитель блокируется, если буфер **пуст**.
- Доступ к буферу — взаимно исключающий.

### Наивное решение и его проблемы

Tanenbaum рассматривает решение через `sleep/wakeup` с одной переменной `count`:

```c
int count = 0;

void producer() {
    while (TRUE) {
        item = produce_item();
        if (count == N) sleep();
        insert_item(item);
        count++;
        if (count == 1) wakeup(consumer);
    }
}
```

**Проблема — race condition с потерей сигнала:**

1. count = 0, потребитель проверил count == 0.
2. **Переключение** перед `sleep()`.
3. Производитель добавляет item, count = 1.
4. Производитель вызывает `wakeup(consumer)` — но потребитель ещё не спит → сигнал теряется.
5. Возврат к потребителю, он засыпает.
6. Потребитель никогда не проснётся.

### Решение через семафоры

Из tanenbaum.md (листинг 2.6):

```c
#define N 100
typedef int semaphore;
semaphore mutex = 1;     // взаимное исключение
semaphore empty = N;     // пустые ячейки
semaphore full = 0;      // заполненные ячейки

void producer(void) {
    int item;
    while (TRUE) {
        item = produce_item();
        down(&empty);        // подождать пустую ячейку
        down(&mutex);        // войти в CS
        insert_item(item);
        up(&mutex);          // выйти из CS
        up(&full);           // сигнал: новая запись
    }
}

void consumer(void) {
    int item;
    while (TRUE) {
        down(&full);         // подождать запись
        down(&mutex);
        item = remove_item();
        up(&mutex);
        up(&empty);          // освободить ячейку
        consume_item(item);
    }
}
```

### Разбор: семафоры

**3 семафора**, каждый со своей ролью:

- **`mutex`** (start = 1): взаимное исключение доступа к буферу.
- **`empty`** (start = N): счётчик пустых ячеек.
- **`full`** (start = 0): счётчик заполненных ячеек.

Инвариант: `empty + full = N` (если никто не в середине операции).

### Почему две переменные (empty и full)

**Очевидный вопрос:** зачем два счётчика, если они связаны?

**Ответ:** они отвечают за **разные ожидания**:
- `empty` нужен **производителю** для ожидания свободного места.
- `full` нужен **потребителю** для ожидания записи.

### Что нельзя сделать одним счётчиком

Если бы был один `count`:
- Производитель должен был бы ждать `count < N`.
- Потребитель должен был бы ждать `count > 0`.

Это **разные условия** на одну переменную. Семафор имеет одну операцию ожидания — `down`, которая декрементирует. Он не может выразить «ждать, пока ≥ X».

Можно было бы использовать два семафора с одним из них как «счётчик», но это и есть наше решение с `empty` и `full`.

### Можно ли без mutex

В однопроцессорной системе с **одним производителем и одним потребителем** mutex можно было бы убрать — они не одновременно в CS благодаря empty/full.

Но при **нескольких производителях** или **потребителях** — нет:
- Два производителя одновременно делают `insert_item` → коррупция.
- Mutex обязателен.

### Порядок захвата — важно!

**Правильный порядок:**
```c
down(&empty);    // сначала ресурс
down(&mutex);    // потом CS
```

**Неправильный порядок (deadlock!):**
```c
down(&mutex);
down(&empty);   // если буфер полон, ждём empty с захваченным mutex
                 // → потребитель не может войти в CS, чтобы освободить ячейку
```

### Поведение

**Сценарий 1: буфер пустой.**
- Потребитель: `down(&full)` — full=0, блокируется.
- Производитель: `down(&empty)` (empty=N→N-1), mutex, insert, `up(&full)` (0→1).
- Потребитель просыпается, читает.

**Сценарий 2: буфер полный.**
- Производитель: `down(&empty)` — empty=0, блокируется.
- Потребитель: `down(&full)` (full=N→N-1), mutex, remove, `up(&empty)` (0→1).
- Производитель просыпается.

**Сценарий 3: несколько производителей и потребителей.**

Несколько производителей могут одновременно делать `down(&empty)` (если есть несколько пустых ячеек). Но только один за раз войдёт в CS благодаря mutex. То же для потребителей.

### Аналогия

- `empty` — талоны на парковочные места.
- `full` — талоны на машины, ждущие забора.
- `mutex` — только один человек у регистратора одновременно.

Без всех трёх синхронизация была бы неполной.

### Применение

Producer-consumer — фундаментальный паттерн:
- **Веб-серверы:** запросы в очередь, обрабатываются воркерами.
- **Логирование:** треды кладут логи в очередь, фоновый поток пишет в файл.
- **Сетевые стеки:** пакеты в буферах.
- **Pipe в Unix:** `cmd1 | cmd2` — внутри pipe такая структура.
- **Channels в Go.**
- **Akka actor mailbox.**

### Резюме

- Producer-consumer — классическая задача синхронизации.
- Решение: 3 семафора (mutex, empty, full).
- Две переменные `empty` и `full` — для двух разных ожиданий.
- Порядок захвата: сначала ресурс, потом mutex.
- Применяется массово в реальных системах.

## 67. Задача писателей и читателей. Пример на семафорах.
*Источник: tanenbaum.md, Глава 2.5.2*

### Постановка задачи

Из tanenbaum.md (2.5.2):
> «Другая общеизвестная задача касается читателей и писателей (Courtois et al., 1971). В ней моделируется доступ к базе данных. Представим, к примеру, систему бронирования авиабилетов, в которой есть множество соревнующихся процессов, желающих обратиться к ней для чтения и записи».

**Правила:**
- **N читателей одновременно — да.**
- **1 писатель — да, остальные ждут.**
- **Читатель и писатель одновременно — нет.**

Идея: читатели не мешают друг другу, потому что не изменяют данные. Писатели требуют эксклюзивного доступа.

### Решение (Tanenbaum, листинг 2.15)

```c
typedef int semaphore;
semaphore mutex = 1;    // защита счётчика rc
semaphore db = 1;       // доступ к БД
int rc = 0;             // число активных читателей

void reader(void) {
    while (TRUE) {
        down(&mutex);
        rc = rc + 1;
        if (rc == 1)
            down(&db);           // первый читатель захватывает БД
        up(&mutex);

        read_data_base();         // КС чтения

        down(&mutex);
        rc = rc - 1;
        if (rc == 0)
            up(&db);             // последний читатель отпускает
        up(&mutex);

        use_data_read();
    }
}

void writer(void) {
    while (TRUE) {
        think_up_data();
        down(&db);                // эксклюзивный доступ
        write_data_base();
        up(&db);
    }
}
```

### Разбор

**Семафоры:**
- `mutex` (start = 1) — защищает счётчик `rc` от одновременного изменения.
- `db` (start = 1) — взаимное исключение между группой читателей и писателем.

**Целочисленная переменная:**
- `rc` — счётчик активных читателей.

### Логика читателей

1. Атомарно увеличить `rc`.
2. Если я **первый** (rc стал 1) — захватить `db`. Этим заблокирую писателей.
3. Освободить `mutex` (другие читатели могут пройти).
4. Прочитать БД.
5. Атомарно уменьшить `rc`.
6. Если я **последний** (rc стал 0) — отпустить `db`. Теперь писатель может работать.

**Ключевая идея:** не каждый читатель захватывает `db`, а только первый и последний.

### Логика писателей

Просто:
1. Захватить `db`.
2. Писать.
3. Отпустить.

Писатель полностью эксклюзивен. Если активны читатели, он ждёт.

### Сценарий

**1. Только читатели:**
- R1: rc=1, захватывает db.
- R2: rc=2.
- R3: rc=3.
- R1 заканчивает: rc=2.
- R2 заканчивает: rc=1.
- R3 заканчивает: rc=0, отпускает db.

Все три читали параллельно.

**2. Читатели + ждущий писатель:**
- R1 читает (rc=1, db захвачен).
- W пытается: `down(&db)` блокируется.
- R2 приходит: rc=2 (не пытается захватить db — он уже захвачен).
- R3 приходит: rc=3.
- R1, R2, R3 заканчивают: rc=0, db освобождён.
- W получает db, пишет.

**3. Пишет писатель, приходит читатель:**
- W захватил db.
- R: `down(&mutex)`, rc=1, `down(&db)` блокируется.
- Когда W заканчивает, R получает db.

### Проблема: голодание писателей

Из tanenbaum.md:
> «Если новый читатель будет прибывать, скажем, каждые 2 с и каждый читатель затратит на свою работу по 5 с, писатель доступа никогда не дождётся».

**Сценарий голодания:**
1. R1 читает.
2. W ждёт db.
3. R2 приходит — rc=2 (не пытается захватить db).
4. R1 заканчивает (rc=1), R2 продолжает.
5. R3 приходит, R4...

Пока есть активные читатели, rc > 0, db **не освобождается**. Писатель ждёт **навсегда**.

### Решение: приоритет писателю

Из tanenbaum.md:
> «Когда писатель находится в состоянии ожидания, то вновь прибывающий читатель не получает немедленного доступа, а приостанавливает свою работу и встает в очередь за писателем. При этом писатель должен дождаться окончания работы тех читателей, которые были активны при его прибытии, и не должен пережидать тех читателей, которые прибывают после него».

Добавляется ещё один семафор:

```c
semaphore writer_pending = 1;

void reader() {
    down(&writer_pending);   // блокирует, если ждёт писатель
    up(&writer_pending);
    // ... как раньше ...
}

void writer() {
    down(&writer_pending);    // блокирует новых читателей
    // захватить db и писать
    up(&writer_pending);
}
```

Цена: меньше параллелизма (новые читатели ждут писателя).

### Решение от Courtois (1971)

Cathedral solution или Bathroom solution — пара алгоритмов:
- **First problem (preference to readers)** — наш базовый, голодание писателей.
- **Second problem (preference to writers)** — приоритет писателям, голодание читателей возможно.

Более продвинутые решения дают справедливость без голодания.

### Применение

- **Базы данных:** read locks vs write locks.
- **Файловые системы:** одновременное чтение, эксклюзивная запись.
- **Сетевые роутинговые таблицы.**
- **In-memory caches.**
- **POSIX `pthread_rwlock_t`.**
- **Linux `rw_semaphore`** (с приоритетом читателей).

### Резюме

- Reader-writer — задача доступа к БД.
- Решение: 2 семафора (mutex, db) + счётчик rc.
- Параллельное чтение, эксклюзивная запись.
- Базовое решение голодает писателей; есть варианты с приоритетом писателям.
- Массовое применение в БД, файловых системах.

## 68. Задача обедающих философов. Пример на семафорах.
*Источник: tanenbaum.md, Глава 2.5.1*

### Постановка задачи

Из tanenbaum.md (2.5.1):
> «В 1965 году Дейкстра сформулировал, а затем решил проблему синхронизации, названную им задачей обедающих философов. Пять философов сидят за круглым столом, и у каждого из них есть тарелка спагетти. Эти спагетти настолько скользкие, что есть их можно только двумя вилками. Между каждыми двумя тарелками лежит одна вилка».

```
       Phil 0
   Fork4    Fork0
Phil 4              Phil 1
   Fork3    Fork1
       Phil 3
       Fork2
       Phil 2
```

Каждый философ:
1. Размышляет.
2. Голодает → пытается взять две вилки.
3. Ест.
4. Возвращает вилки → размышляет.

### Простая попытка

```c
void philosopher(int i) {
    while (TRUE) {
        think();
        take_fork(i);           // левая
        take_fork((i+1) % N);   // правая
        eat();
        put_fork(i);
        put_fork((i+1) % N);
    }
}
```

**Проблема — deadlock.** Если все 5 одновременно возьмут левую вилку:
- P0 взял F0, ждёт F1.
- P1 взял F1, ждёт F2.
- P2 взял F2, ждёт F3.
- P3 взял F3, ждёт F4.
- P4 взял F4, ждёт F0.

Цикл → все ждут навсегда.

### Попытка с возвратом

«Взять левую → если правая занята → положить левую → подождать → повторить».

**Проблема — livelock (или голодание).** Если все синхронно делают то же самое:
- Все берут левую.
- Все видят, что правая занята.
- Все одновременно кладут левую и ждут.
- Все одновременно снова берут левую.
- ... и так до бесконечности.

Из tanenbaum.md:
> «При некоторой доле невезения все философы могут приступить к выполнению алгоритма одновременно и, взяв левые вилки и увидев, что правые вилки недоступны, опять одновременно положить на место левые вилки, и так до бесконечности».

Случайное время ожидания спасает в среднем, но не гарантированно.

### Решение с одним мьютексом

```c
semaphore mutex = 1;

void philosopher(int i) {
    while (TRUE) {
        think();
        down(&mutex);
        // взять обе вилки
        eat();
        // положить обе вилки
        up(&mutex);
    }
}
```

**Корректно**, нет deadlock. Но **плохо**: за столом ест только один философ за раз. С 5 вилками могли бы есть двое.

### Правильное решение (Tanenbaum, листинг 2.14)

```c
#define N 5
#define LEFT  (i+N-1) % N
#define RIGHT (i+1) % N
#define THINKING 0
#define HUNGRY   1
#define EATING   2

typedef int semaphore;
int state[N];
semaphore mutex = 1;
semaphore s[N];           // по одному семафору на философа

void philosopher(int i) {
    while (TRUE) {
        think();
        take_forks(i);
        eat();
        put_forks(i);
    }
}

void take_forks(int i) {
    down(&mutex);
    state[i] = HUNGRY;
    test(i);              // попытаться поесть
    up(&mutex);
    down(&s[i]);          // если test не разрешил — ждём
}

void put_forks(int i) {
    down(&mutex);
    state[i] = THINKING;
    test(LEFT);           // может, сосед слева может теперь есть
    test(RIGHT);
    up(&mutex);
}

void test(int i) {
    if (state[i] == HUNGRY && state[LEFT] != EATING && state[RIGHT] != EATING) {
        state[i] = EATING;
        up(&s[i]);        // разрешить i-му философу есть
    }
}
```

### Разбор

**Структуры:**
- `state[N]` — состояние каждого: THINKING, HUNGRY, EATING.
- `mutex` — защита массива state.
- `s[N]` — индивидуальный семафор каждого. На нём философ блокируется, если не может есть прямо сейчас.

### Логика take_forks

1. Захватить mutex.
2. Объявить себя HUNGRY.
3. Вызвать `test(i)`:
   - Если соседи не едят → сам становлюсь EATING и `up(&s[i])`.
   - Иначе → оставляю себя HUNGRY.
4. Отпустить mutex.
5. `down(&s[i])`:
   - Если test разрешил — счётчик стал 1 (от up), down немедленно проходит.
   - Если нет — блокируемся, ждём, пока сосед разбудит.

### Логика put_forks

1. Захватить mutex.
2. Стать THINKING.
3. Проверить соседей — может, теперь они могут есть?
   - Если левый сосед HUNGRY и его соседи не едят → дать ему есть.
   - То же для правого.
4. Отпустить mutex.

### Сценарий

Все философы начинают, состояние THINKING. P0 голодает:

1. P0: take_forks → mutex → state[0]=HUNGRY → test(0): P4 не ест, P1 не ест → state[0]=EATING, up(s[0]). → mutex → down(s[0]) проходит сразу. P0 ест.

2. P2 голодает аналогично — соседи (P1, P3) не едят → P2 ест.

3. P1 голодает: → test(1): P0 ест (LEFT) → не разрешено, P1 остаётся HUNGRY. P1 → down(s[1]) → блокируется.

4. P0 заканчивает: put_forks → state[0]=THINKING → test(LEFT=4) (не голоден) → test(RIGHT=1): P1 HUNGRY, P0 не ест, P2 ест... но P2 — это RIGHT P1 = LEFT P1 → P2 — RIGHT для P1, который — RIGHT для P0. RIGHT(0) = 1, LEFT(1) = 0, RIGHT(1) = 2. P2 EATING. Значит test(1) не разрешит. P1 продолжает ждать.

5. P2 заканчивает: put_forks → state[2]=THINKING → test(LEFT=1): P1 HUNGRY, P0 не ест, P2 не ест → разрешено! state[1]=EATING, up(s[1]). P1 просыпается, ест.

Никто не голодает, максимум параллелизма.

### Почему это работает

- **Нет deadlock:** философ становится EATING только если соседи не едят. Невозможен цикл «все берут левую вилку».
- **Нет starvation** (в этом базовом варианте): если соседи едят, философ ждёт; когда они закончат, его проверят.
- **Параллелизм:** 0 и 2 могут есть одновременно, 1 и 3, 1 и 4 (но не все 5 сразу — только 2 «через одного»).

### Тонкости

Решение не идеально:
- Возможно theoretical starvation если «соседи» по очереди едят.
- Реальные системы используют priority или fairness.

### Применение

- **Иллюстрация проблем с многими ресурсами** в задачах синхронизации.
- **Стандартный тест** для новых примитивов синхронизации.
- **Аналогия:** доступ к множественным базам данных, маршрутным таблицам, GPU ресурсам.

### Резюме

- Обедающие философы — классическая задача с deadlock и livelock.
- Простые решения имеют проблемы.
- Правильное решение: state[], mutex, индивидуальные семафоры.
- Максимум параллелизма (2 одновременно), нет deadlock.
- Демонстрирует тонкости синхронизации с несколькими ресурсами.

## 69. RW-блокировки. Блокировки памяти.
*Источник: robert_lav.md, Глава 10*

### RW-блокировки — что это

**RW-lock (reader-writer lock)** — разделение на читателей и писателей:
- N читателей одновременно.
- Только 1 писатель.
- Читатель и писатель — нет.

См. задачу читателей-писателей (вопрос 67) для теории.

### Типы в Linux

**1. RW спин-блокировки (`rwlock_t`).**

Для коротких CS:

```c
rwlock_t mr_rwlock = RW_LOCK_UNLOCKED;

read_lock(&mr_rwlock);
// CS чтения
read_unlock(&mr_rwlock);

write_lock(&mr_rwlock);
// CS записи
write_unlock(&mr_rwlock);

write_lock_irqsave(&mr_rwlock, flags);
write_unlock_irqrestore(&mr_rwlock, flags);
```

Преимущество — параллельные читатели.
Недостаток — больше накладные расходы, чем у обычного спинлока.

**2. RW семафоры (`rw_semaphore`).**

Для длинных CS (могут блокироваться):

```c
DECLARE_RWSEM(my_rwsem);

down_read(&my_rwsem);
// ...
up_read(&my_rwsem);

down_write(&my_rwsem);
// ...
up_write(&my_rwsem);

downgrade_write(&my_rwsem);   // write → read
```

### Применение RW-блокировок

**Хорошо для:**
- **Read-mostly данных** (читается часто, пишется редко).
- **Структур с долгим чтением** и редкой записью.

Примеры:
- Конфигурация системы.
- Маршрутные таблицы.
- Файловые системы (для метаданных).
- DNS-кэши.

**Плохо для:**
- Чтение и запись примерно одинаково часто.
- Малое число одновременных читателей.
- Накладные расходы превышают выгоду.

### Особенность Linux: читатели приоритетны

Из robert_lav.md:
> «Блокировка по чтению всегда имеет большее преимущество по сравнению с блокировкой по записи».

Сценарий: есть читатель, ждёт писатель. Приходит новый читатель → проходит сразу, писатель продолжает ждать.

Преимущество: высокая пропускная способность для чтения.
Недостаток: возможно голодание писателей.

### downgrade_write

Уникальная функция RW-семафора:
```c
down_write(&sem);
// сделать модификацию
downgrade_write(&sem);
// теперь читать параллельно с другими
up_read(&sem);
```

Атомарно превращает write-lock в read-lock. Без неё нужно было бы:
```c
up_write(&sem);
down_read(&sem);    // между этим — гонка!
```

### Блокировки памяти (memory locking)

**mlock/munlock** — закрепить страницы в физической памяти, **запретить swap**.

```c
#include <sys/mman.h>

mlock(addr, len);                       // закрепить регион
munlock(addr, len);                     // освободить
mlockall(MCL_CURRENT | MCL_FUTURE);     // всё АП
munlockall();
```

### Зачем закреплять память

**1. Real-time системы.**

Page fault от swap — задержка от десятков микросекунд (SSD) до миллисекунд (HDD). Это катастрофа для жёсткого RT.

Тип `MCL_FUTURE` гарантирует, что и **будущие** аллокации не будут выгружены.

**2. Безопасность (crypto).**

Не дать ключам шифрования попасть на диск:
- Swap-файл может быть прочитан после извлечения диска.
- Hibernation файл хранит всё АП.

```c
struct key {
    char data[32];
};
struct key *k = malloc(sizeof(*k));
mlock(k, sizeof(*k));   // не свопить
// ... использовать k ...
memset(k, 0, sizeof(*k));   // обнулить перед освобождением
munlock(k, sizeof(*k));
free(k);
```

**3. Гарантия производительности.**

DMA-буферы, регионы, к которым часто обращаются.

### Ограничения

**1. Привилегированная операция.** Нужно:
- `CAP_IPC_LOCK` capability.
- Или быть root.

Раньше любой пользователь мог mlock небольшие регионы (`RLIMIT_MEMLOCK`).

**2. Лимит `RLIMIT_MEMLOCK`.** Ограничение на закреплённую память на пользователя:
```bash
ulimit -l       # показать лимит
ulimit -l 1048576   # 1 GB
```

**3. Влияние на систему.** Чем больше закреплено, тем меньше памяти для свопинга других процессов.

### Page-level vs region-level

- `mlock(addr, len)` — закрепить конкретные страницы.
- `mlockall(MCL_CURRENT)` — всё текущее АП.
- `mlockall(MCL_FUTURE)` — и будущие аллокации тоже.

### Типичный код для RT-приложения

```c
#include <sys/mman.h>
#include <sched.h>

int main() {
    // 1. Заранее выделить и закрепить весь стек
    mlockall(MCL_CURRENT | MCL_FUTURE);

    // 2. Прикоснуться к стеку для физического выделения
    char dummy[1024 * 1024];
    memset(dummy, 0, sizeof(dummy));

    // 3. Использовать FIFO scheduling
    struct sched_param sp = { .sched_priority = 80 };
    sched_setscheduler(0, SCHED_FIFO, &sp);

    // 4. Реальный RT-код
    rt_main_loop();
}
```

### Page locking в ядре

В ядре Linux есть API `get_user_pages` для закрепления страниц пользователя в RAM:

```c
struct page **pages = ...;
int n = get_user_pages(addr, count, FOLL_WRITE, pages, NULL);
// страницы зафиксированы — DMA в них безопасно
// ...
put_page(pages[i]);   // освободить
```

Используется в драйверах для DMA.

### Резюме

- RW-блокировки разделяют чтение (N параллельно) и запись (эксклюзивно).
- Хороши для read-mostly данных.
- В Linux: читатели приоритетны (голодание писателей).
- `downgrade_write` атомарно превращает write в read.
- Блокировки памяти (mlock) закрепляют страницы, запрещая swap.
- Применение: RT-системы, криптография, гарантия производительности.

## 70. Неблокирующие операции. RCU. Пример.
*Источник: rtos.md, Глава 2.2.4; robert_lav.md (RCU)*

### Зачем нужны неблокирующие алгоритмы

Из rtos.md (2.2.4):
> «Хотя спин-блокировки позволяют создавать критические секции, очевидно, такое решение страдает не очень хорошей масштабируемостью. Чем больше в алгоритме критических секций, тем больше другие процессоры будут тратить времени на ожидание. Начиная с какого-то момента, блокировка станет бутылочным горлышком, поэтому особый интерес представляют алгоритмы, которые не используют такие блокировки, а пользуются только атомарными инструкциями».

Lock-based алгоритмы имеют проблемы:
- **Deadlock.**
- **Priority inversion.**
- **Convoy effect** (зависимость от самого медленного).
- **Голодание.**
- **Плохая масштабируемость** на больших SMP.

Неблокирующие алгоритмы решают все эти проблемы.

### Иерархия неблокирующих гарантий

**1. Obstruction-free (слабейшая).**

Из rtos.md:
> «Процессор закончит выполнение алгоритма за ограниченное время, при условии, что остальные процессоры не будут обращаться к разделяемым данным».

Если другой CPU «помешал» — наша операция может не завершиться.

**2. Lock-free (среднее).**

> «Конечное количество процессоров закончат выполнение алгоритма за конечное время (по крайней мере 1 процессор будет продвигаться в каждый момент времени)».

Хотя бы один CPU делает прогресс. Конкретный CPU может «проигрывать» долго.

**3. Wait-free (сильнейшее).**

> «Конечное число процессоров закончат выполнение алгоритма за фиксированное время (все процессоры в каждый момент совершают прогресс)».

Каждый CPU имеет ограниченное худшее время выполнения.

Иерархия: Wait-free ⊂ Lock-free ⊂ Obstruction-free.

### Свойства неблокирующих алгоритмов

**Преимущества:**
- **Нет deadlock.**
- **Нет priority inversion.**
- **Хорошая масштабируемость** (на больших SMP лучше lock-based).
- **Безопасны в обработчиках прерываний.**
- **Толерантны к падению потока** — другие могут продолжить.

**Недостатки:**
- **Сложность** — даже простые структуры сложно реализовать.
- **Проблема ABA** (вопрос 54).
- **Требуют барьеров памяти.**
- **Часто хуже на низкой нагрузке** — overhead на атомарные операции.

### Типичные lock-free структуры

- **Lock-free queue** (Michael & Scott, 1996).
- **Lock-free stack** (Treiber, 1986).
- **Lock-free hash map** (Cliff Click).
- **Lock-free linked list** (Harris, 2001).
- **Lock-free skip list.**

### RCU — Read-Copy-Update

**RCU** — техника синхронизации для **read-mostly** данных. Появилась в Linux в начале 2000-х (Paul McKenney).

**Идея:**
1. Читатели **никогда не блокируются** — просто читают.
2. Писатель создаёт **копию** данных, изменяет её.
3. Атомарно меняет указатель на новую версию.
4. Старая версия живёт ещё некоторое время — пока есть читатели, которые её видят.
5. **После** того как все «текущие» читатели закончили — старая версия удаляется.

### Анатомия RCU

**Читатель:**
```c
rcu_read_lock();                    // отметить, что читаем
ptr = rcu_dereference(global_ptr);
use(ptr);
rcu_read_unlock();                  // закончили
```

**Писатель:**
```c
new = kmalloc(sizeof(*new), GFP_KERNEL);
memcpy(new, old, sizeof(*old));     // copy
modify(new);                         // update
rcu_assign_pointer(global_ptr, new); // атомарно поменять
synchronize_rcu();                   // дождаться читателей
kfree(old);                          // безопасно освободить
```

### Ключевые элементы

**`rcu_read_lock()` / `rcu_read_unlock()`:**

В classic RCU (kernel) это просто `preempt_disable() / preempt_enable()` — никаких атомарных операций!

Принцип: пока читатель не вытесняем, мы знаем точно, когда он закончит.

**`rcu_dereference(ptr)`:**

Чтение указателя с барьером памяти:
```c
#define rcu_dereference(p) \
    ({ \
        typeof(p) _p = ACCESS_ONCE(p); \
        smp_read_barrier_depends(); \
        _p; \
    })
```

**`rcu_assign_pointer(ptr, v)`:**

Запись указателя с барьером:
```c
#define rcu_assign_pointer(p, v) \
    ({ \
        smp_wmb(); \
        ACCESS_ONCE(p) = (v); \
    })
```

**`synchronize_rcu()`:**

Ждёт «период благодати» (grace period) — пока все CPU, которые **до этого** были в `rcu_read_lock`, не выйдут.

В Linux реализация очень умная: ядро проверяет на каждом CPU квант от time tick — если процесс вытеснен, значит, на нём не идёт read-side CS. Когда все CPU прошли через переключение контекста или вход в idle — grace period закончился.

### Преимущества RCU

- **Читатели абсолютно бесплатны** — нет атомарных операций, нет блокировок, нет барьеров (кроме лёгкого `read_barrier_depends`).
- **Отличная масштабируемость** — добавление CPU не увеличивает overhead для читателей.
- **Никогда не блокирует читателей.**
- **Безопасно из любого контекста.**

### Недостатки RCU

- **Сложнее в использовании** — нужно думать о grace period.
- **Большие задержки удаления** старых версий.
- **Дополнительная память** на копии во время grace period.
- **Не подходит для write-heavy workloads.**

### Применение в Linux

RCU массово используется в ядре Linux:
- **Маршрутные таблицы** (`fib_table`).
- **Списки задач** (`task_list`).
- **VFS** (dentry cache, inode cache).
- **Network protocol handlers.**
- **Modules** (динамическая загрузка).
- **SELinux policy.**

### Пример: маршрутная таблица

```c
struct route {
    u32 dest;
    u32 gw;
    struct route *next;
};
struct route *route_table;

// Поиск (читателей миллионы в секунду)
struct route *route_lookup(u32 dest) {
    struct route *r;
    rcu_read_lock();
    r = rcu_dereference(route_table);
    while (r) {
        if (r->dest == dest) {
            rcu_read_unlock();
            return r;   // ВНИМАНИЕ: ссылка живёт только в read-side CS
        }
        r = rcu_dereference(r->next);
    }
    rcu_read_unlock();
    return NULL;
}

// Добавление (редкое)
void route_add(u32 dest, u32 gw) {
    struct route *new = kmalloc(sizeof(*new), GFP_KERNEL);
    new->dest = dest;
    new->gw = gw;

    // Atomic insert в начало
    struct route *old;
    spin_lock(&route_lock);   // защита от других писателей
    new->next = rcu_dereference(route_table);
    rcu_assign_pointer(route_table, new);
    spin_unlock(&route_lock);
}

// Удаление (редкое)
void route_del(u32 dest) {
    struct route *r, *prev = NULL;
    spin_lock(&route_lock);
    r = route_table;
    while (r) {
        if (r->dest == dest) {
            if (prev)
                rcu_assign_pointer(prev->next, r->next);
            else
                rcu_assign_pointer(route_table, r->next);
            spin_unlock(&route_lock);
            synchronize_rcu();   // подождать читателей
            kfree(r);             // безопасно
            return;
        }
        prev = r;
        r = r->next;
    }
    spin_unlock(&route_lock);
}
```

### Варианты RCU

- **Classic RCU** — на основе `preempt_disable`.
- **Sleepable RCU** — позволяет блокировку в read-side.
- **Tree RCU** — масштабируемый для больших SMP.
- **Tiny RCU** — для маленьких систем.

### Альтернативы для решения ABA

Кроме RCU:
- **Hazard pointers** (Michael, 2002).
- **Reference counting** + atomic operations.
- **Garbage collection** (в managed-языках).

### Резюме

- Неблокирующие алгоритмы — без mutex, на атомарных операциях.
- Иерархия: obstruction-free ⊂ lock-free ⊂ wait-free.
- Преимущества: нет deadlock, priority inversion, лучшая масштабируемость.
- Недостатки: сложность, ABA, требуют барьеров.
- RCU — техника для read-mostly: copy-on-write + grace period.
- Читатели абсолютно бесплатны.
- Массово применяется в Linux ядре (маршруты, VFS, списки задач).
