## 71. Реализация и виртуализация потоков. Разделяемое ядро.
*Источник: rtos.md, Глава 3.3.5 «Архитектуры ОС», Глава 5 «Исполнительная подсистема»*

### Что значит «реализовать поток»

Из теоретического (вопрос 25): поток = последовательность инструкций + контекст + стек. На уровне реализации нужно:

**1. Структура данных TCB (Thread Control Block).**

Хранит всё о потоке:
- ID.
- Состояние (READY, RUNNING, BLOCKED, TERMINATED).
- Приоритет.
- Сохранённый контекст (когда не выполняется).
- Указатель стека.
- Принадлежность к процессу.
- Ссылки на структуры синхронизации, если ждёт.

**2. Создание/уничтожение потоков.**
```c
thread_t* thread_create(void (*entry)(void*), void *arg, int priority);
void thread_destroy(thread_t *t);
```

**3. Переключение контекста.**

Низкоуровневая функция, сохраняющая регистры одного потока и восстанавливающая регистры другого.

```c
void context_switch(thread_t *old, thread_t *new);
```

Реализуется на ассемблере, специфична для архитектуры.

**4. Планировщик.**

Выбирает следующий поток для выполнения. Многоуровневая приоритетная очередь.

**5. Очереди ожидания.**

Структуры, на которых блокируются потоки.

**6. Синхронизация.**

Мьютексы, семафоры, условные переменные — построены на TCB и очередях.

### Виртуализация потоков

Каждый поток видит **виртуальный процессор**. Создаётся иллюзия, что у потока есть свой CPU.

Из rtos.md (3.3.2):
> «Поток можно также определить как виртуальный процессор, который имеет сохраняемый в памяти контекст. Инфраструктура поддержки потоков должна по очереди загружать контексты потоков из памяти в регистры процессора, обеспечивая тем самым иллюзию параллельного исполнения потоков таким образом, как если бы каждый из них выполнялся на выделенном процессоре».

Виртуализация = переключение контекста + планирование.

### Уровни реализации потоков

**1. Потоки пользовательского пространства (Green threads, fibers).**

Полностью в библиотеке, ядро ничего не знает. Преимущество — дешёвое переключение. Недостаток — блокирующий syscall парализует весь процесс.

**2. Потоки ядра (kernel threads).**

Ядро знает о каждом потоке. Дороже переключение, но реальный параллелизм.

**3. Гибридные (M:N модель).**

M пользовательских потоков на N ядрерных. Сложнее реализация. Применялся в Solaris, ранние реализации Go.

В современных Linux: 1:1 — каждый поток приложения = поток ядра.

### Разделяемое ядро

Из rtos.md (3.3.5):
> «ОСРВ представляет собой библиотеку, статически компонуемую с приложением и использующуюся для реализации потоков и прочих сервисов, так называемую libOS… В этом случае граница между ОС и приложением практически стирается».

**Архитектура libOS:**
```
┌──────────────────────────┐
│  Приложение               │
├──────────────────────────┤
│  ОСРВ (libOS) — потоки,  │
│  планировщик, sync       │
├──────────────────────────┤
│  HAL — аппаратный        │
│  абстракционный слой     │
└──────────────────────────┘
        ↕ напрямую
    [ Железо ]
```

ОСРВ компилируется **вместе** с приложением в единый бинарный образ. Загружается в Flash микроконтроллера.

### Размер libOS

> «Размер библиотеки, реализующей функции ОС в этом варианте исчисляется единицами и десятками килобайт. Подобные ОС часто называются ядрами (RTOS kernels)».

Например, FreeRTOS — ядро ~10-20 КБ. Помещается даже в небольшие микроконтроллеры.

### Преимущества libOS

- **Очень быстро:** нет переключения режимов, нет системных вызовов в традиционном смысле.
- **Минимум памяти.**
- **Простая разработка** — нет границы пользователь/ядро.
- **Подходит для микроконтроллеров.**

### Недостатки libOS

- **Нет изоляции.** Ошибка в приложении валит ОС.
- **Нет защиты памяти** (без MMU/MPU).
- **Все имеют полный доступ** к оборудованию.

### Архитектуры ядер — обзор

**Монолитное ядро.**

Все системные сервисы в режиме ядра.
- Linux, FreeBSD, Windows NT (по сути).
- Высокая производительность.
- Низкая надёжность (ошибка в драйвере = крах).

**Микроядро.**

Из rtos.md:
> «Микроядро прежде всего обозначает именно архитектуру ядра, когда на ядро возлагается, в основном, только поддержка исполняемых сущностей, таких как потоки и организация межпроцессного взаимодействия, позволяющая все остальные части ядра выполнять в виде изолированных процессов».

- QNX, L4, MINIX 3.
- Высокая надёжность (изоляция сервисов).
- Низкая производительность (много IPC).

**Гибридное ядро.**

- macOS XNU, Windows NT.
- Компромисс.

**Экзоядро.**

> «В случае экзоядра, ядро системы, выполняющееся в привилегированном режиме процессора, отвечает только за выделение аппаратных ресурсов, таких как области памяти или время процессора. Все остальные сервисы ядра реализуются как libOS на стороне пользователя».

- MIT Exokernel (research).
- Максимальная гибкость.
- Эпизодически: Unikernels (MirageOS).

### Разделяемое ядро в системе с процессами

Когда есть MMU и несколько процессов:
- Ядро **отображается** в верхней половине ВАП каждого процесса.
- Это **один и тот же** физический код во всех процессах.
- Системные вызовы — без смены адресного пространства.
- Защита через права доступа MMU.

Преимущества:
- Экономия памяти (одна копия кода).
- Быстрые системные вызовы.
- Совместное использование данных ядра между процессами безопасно.

### Резюме

- Реализация потоков: TCB, переключение контекста, планировщик, очереди ожидания.
- Виртуализация: каждый поток видит свой CPU.
- libOS — статическая библиотека для микроконтроллеров.
- Архитектуры ядер: монолитное, микро, гибрид, экзо.
- Разделяемое ядро в верхней половине ВАП каждого процесса.

## 72. Прерывания и их обработка. Аппаратные и программные прерывания. Контроллер прерываний. Обработчики прерываний.
*Источник: rtos.md, Глава 2.1.4 «Прерывания»*

### Зачем нужны прерывания

Без прерываний CPU должен сам опрашивать устройства (polling): «есть данные? есть данные? есть данные?». Это:
- **Расходует CPU** — миллиарды проверок в секунду.
- **Плохая реакция** — между опросами устройство может ждать долго.
- **Зависит от частоты опроса.**

Из rtos.md (2.1.4):
> «В идеале, для достижения наилучшей скорости реакции, хотелось бы проверять состояние устройств после каждой выполненной инструкции. Поэтому решение заключается в том, чтобы встроить логику проверки состояния устройств непосредственно в процессор и сделать ее частью цикла выборки и исполнения команд».

**Прерывание** = аппаратный механизм, при котором устройство **само** сигнализирует процессору о готовности. Процессор прерывает текущую работу и переходит к обработчику.

### Цикл с прерываниями

```
Загружай инструкцию
Выполняй инструкцию
Если есть прерывание:
    Перейти на обработчик
Иначе:
    Следующая инструкция
```

Проверка происходит **после каждой инструкции**, очень дёшево (один регистр).

### Типы прерываний

**1. Аппаратные (внешние).**

- От устройств: диск, таймер, сеть, клавиатура.
- **Асинхронные** — могут прийти в любой момент.
- Идентифицируются вектором.

**2. Программные.**

- Инициируются программой:
  - `INT 0x80` на x86 (Linux syscall).
  - `SYSCALL` / `SYSENTER` на x86-64.
  - `SVC` на ARM.
- Используются для перехода в режим ядра.
- **Синхронные** (заранее запрограммированы в коде).

**3. Исключения (Exceptions).**

- Связаны с конкретной инструкцией:
  - Деление на 0.
  - Page fault.
  - Недопустимая инструкция.
  - Trap (точка останова).
- **Синхронные.**
- Обрабатываются аналогично прерываниям.

### Контроллер прерываний

Из rtos.md:
> «Контроллер прерываний — это интерфейсное устройство для сопряжения внешних устройств с разными процессорами».

Без контроллера каждое устройство нужно было бы подключать напрямую к процессору. С ростом числа устройств это невозможно. Решение — **контроллер прерываний**.

Структура:
```
[Устр. 1]   ┐
[Устр. 2]   ├→ [Контроллер прерываний] ──→ [Процессор]
[Устр. N]   ┘
```

Контроллер:
- Принимает запросы от устройств.
- Назначает им приоритеты.
- Сообщает процессору номер вектора.
- Маскирует прерывания при необходимости.
- На многоядерных — направляет на конкретный CPU.

### Примеры контроллеров

- **PIC (8259)** — старый x86.
- **APIC** — современный x86 (Advanced PIC).
- **NVIC** — ARM Cortex-M (Nested Vectored Interrupt Controller).
- **GIC** — ARM Cortex-A (Generic Interrupt Controller).

### Действия при прерывании

Из rtos.md:
> «При наличии активного запроса прерывания, процессор производит действия, эквивалентные вызову подпрограммы: помещает адрес возврата в стек и переходит к выполнению функции-обработчика прерывания, адрес которой предварительно должен быть сообщен процессору».

Шаги:
1. Окончание текущей инструкции.
2. Сохранение адреса возврата.
3. Сохранение регистра состояния.
4. Получение вектора от контроллера.
5. Поиск адреса обработчика в таблице векторов.
6. Переход на обработчик.

### Сохранение контекста

Из rtos.md:
> «Часто для сохранения адреса возврата используется стек, но могут быть также специальные выделенные ресурсы специально для прерываний, в частности, специальные регистры, в которых сохраняется адрес возврата. Использование регистров предпочтительно, потому что обработка прерываний использует меньше обращений к памяти».

Подходы:
- **Стек:** х86, ARM Cortex-A.
- **Регистры:** SPARC (регистровые окна), MIPS (k0, k1).
- **Часть в регистры, часть в стек:** ARM Cortex-M (часть сохраняется аппаратно в стек).

### Обработчик прерывания (ISR)

**ISR (Interrupt Service Routine)** — функция, вызываемая в ответ на прерывание.

Структура типичного ISR:
```c
void my_isr(void) {
    // 1. Подтвердить прерывание (acknowledge)
    device_clear_irq_flag();

    // 2. Обработка
    process_device_event();

    // 3. Возможно — запланировать дополнительную работу
    schedule_softirq();
}
```

### Ограничения ISR

Обработчики прерываний выполняются в **особом контексте**:

- **Не связаны с процессом** — `current` не определён.
- **Нельзя блокироваться** — нет планировщика для них.
- **Нельзя делать переключение контекста** — нет потока.
- **Должны быть короткими** — задерживают другие прерывания.
- **Должны быть атомарны** — могут быть вытеснены ещё более приоритетным прерыванием.

### Идиоматические правила

- **Не использовать malloc.**
- **Не блокироваться на семафорах** (можно `down_trylock`).
- **Не использовать обычные функции работы со временем.**
- **Использовать атомарные операции** для модификации общих данных.

### Прерывания и энергосбережение

Из rtos.md:
> «Помимо гибких возможностей реагировать на события и улучшения времени реакции системы, по сравнению с использованием полинга, прерывания дают возможность реализовать в процессоре различные режимы энергосбережения. Например, система команд может содержать специальную инструкцию останова, которая приостанавливает выборку и исполнение инструкций процессором до получения прерывания».

`WFI`/`HLT` — процессор спит до прерывания. Прерывание не нужно для этого «активного» опроса — оно может прийти когда угодно.

### Резюме

- Прерывание — аппаратный механизм реакции на события устройств.
- Виды: аппаратные (от устройств), программные (syscall), исключения (ошибки).
- Контроллер прерываний — мост между устройствами и процессором.
- Обработчик прерывания (ISR) — функция, выполняющаяся в ответ.
- ISR должны быть короткими, не могут блокироваться.

## 73. Прерывания и их обработка. Вектор прерываний, таблица векторов прерываний. Приоритеты прерываний.
*Источник: rtos.md, Глава 2.1.4*

### Вектор прерываний

Из rtos.md:
> «Поскольку устройств в наличии обычно более одного, обработчику как-то надо узнать, от какого именно устройства пришел запрос. К решению этой задачи есть несколько подходов, но большинство из них связаны с векторами прерываний. Вектор — атрибут источника прерывания, который позволяет различать запросы от разных устройств. Разным векторам соответствуют разные функции-обработчики прерывания».

**Вектор** = уникальный номер, идентифицирующий источник прерывания. Передаётся от контроллера к процессору.

Типичные диапазоны:
- 0–31: исключения процессора (на x86).
- 32–255: аппаратные прерывания.

### Таблица векторов прерываний — подход 1

Из rtos.md:
> «Первый из них предполагает существование в памяти так называемой таблицы векторов прерываний (interrupt vector table, IVT). Она находится либо в фиксированном месте в памяти, о котором знает процессор, либо у последнего есть специальный регистр, который указывает на таблицу».

Структура:
```
Базовый адрес таблицы (в регистре IDTR/VTOR)
    ↓
+--------+--------+--------+
| addr0  | addr1  | addr2  | ...
+--------+--------+--------+
   ↑        ↑        ↑
 ISR 0    ISR 1    ISR 2
```

При прерывании:
1. Контроллер передаёт вектор N.
2. Процессор: addr = таблица[N].
3. Переход на addr.

### Реализации

**x86: IDT (Interrupt Descriptor Table).**
- 256 записей по 8/16 байт.
- Адрес — в регистре IDTR.
- Каждая запись содержит указатель на обработчик + права доступа + тип.

**ARM Cortex-M:**
- Таблица векторов в начале памяти (или адрес настраивается через VTOR).
- Каждая запись — 4 байта (указатель на функцию).

**MIPS:**
- Все прерывания на одном фиксированном адресе (0xBFC00200).
- Программа сама смотрит регистр причины.

### Таблица векторов — подход 2

Из rtos.md:
> «Использование одного вектора на все аппаратные прерывания плюс возможность узнать текущий вектор уже внутри обработчика путём чтения либо регистров процессора, либо регистров контроллера прерываний».

Один обработчик для всех — внутри он определяет источник:
```c
void common_irq_handler(void) {
    int vector = read_current_vector();   // из регистра
    isr_table[vector]();                   // программная таблица
}
```

Удобно для инфраструктурных каркасов — общая обёртка с пред- и пост-обработкой.

### Комбинированный подход

Из rtos.md:
> «Допустимы также промежуточные варианты, например, у процессора может быть меньше векторов, чем у контроллера прерываний. Некоторые векторы могут вызываться напрямую, тогда как другие должны анализировать состояние контроллера, так как возможно несколько источников».

### Приоритеты прерываний

Из rtos.md:
> «Несколько источников вполне могут установить запрос прерывания одновременно. Возникающую при этом неопределенность: какой обработчик должен быть вызван, разрешают с помощью приоритета прерываний».

Каждому вектору назначается **приоритет** — число.

Реализация:
- **Статически:** прописано в железе.
- **Динамически:** программно настраиваемое.

При одновременных запросах контроллер выбирает самый приоритетный → передаёт его вектор процессору.

### Вложенные обработчики

Из rtos.md:
> «В зависимости от процессора, возможно не только определение порядка вызова при поступлении нескольких запросов, но и способность одних обработчиков прерывать другие: высокоприоритетные прерывания могут прерывать низкоприоритетные».

Схема вложения:
```
Прог. → ISR1 (низкий) → ISR2 (высокий) → ISR1 → Прог.
```

ISR1 был прерван ISR2, потом продолжился.

### Реализация вложения

Процессор поддерживает текущий уровень приоритета (CPL — Current Priority Level):
- При входе в обработчик: CPL = приоритет обработчика.
- Новые прерывания принимаются только если их приоритет > CPL.
- При выходе из обработчика: CPL восстанавливается.

### Где реализуются приоритеты

Из rtos.md:
> «Приоритеты могут реализовываться как самим процессором, так и контроллером прерываний».

- **Процессор:** маска приоритета.
- **Контроллер:** приоритизация запросов.

ARM Cortex-M / NVIC: 8–256 уровней приоритета, поддержка вложения «из коробки».

### Маска прерываний

Каждый бит регистра маски соответствует прерыванию. Бит = 1 → запрещено.

```c
disable_interrupt(N);   // установить маску для N
enable_interrupt(N);    // снять маску
```

Также есть глобальный флаг разрешения (на x86 — IF в EFLAGS, на ARM — PRIMASK).

### Пример: использование приоритетов в ОСРВ

```
Приоритет:   Прерывание:
   15 (max)  Таймер RTOS (тактирование)
   12        Аварийный датчик
    8        Сетевая карта
    4        UART (отладочный)
    0 (min)  Кнопка
```

При нажатии кнопки во время передачи UART — кнопочный обработчик ждёт окончания UART.

### Резюме

- Вектор — идентификатор источника прерывания.
- Таблица векторов сопоставляет вектор с адресом обработчика.
- Подходы: прямая таблица или единый обработчик с диспетчеризацией.
- Приоритеты разрешают одновременные запросы, поддерживают вложение.
- Маска прерываний — для критических секций.

## 74. Прерывания и их обработка. Запуск обработчика прерываний. Блокировка обработчика прерываний. Запрет прерываний.
*Источник: rtos.md, Глава 2.1.4, 2.1.6, 4.4.6*

### Запуск обработчика — полный алгоритм

Когда контроллер прерываний сигнализирует процессору:

**1. Окончание текущей инструкции.**

Прерывания проверяются между инструкциями. Текущая инструкция должна завершиться атомарно.

Некоторые длинные инструкции (например, REP MOVS на x86) могут проверять прерывания внутри.

**2. Проверка маски.**

Процессор сравнивает приоритет нового прерывания с текущей маской:
- Если приоритет ≤ маски → отложить, не вызывать.
- Если > маски → продолжить.

Также проверяется глобальный флаг разрешения.

**3. Сохранение контекста.**

Что сохраняется — зависит от архитектуры:
- **x86:** автоматически в стек: EFLAGS, CS, EIP. Если переход в режим ядра — сначала переключение на стек ядра, потом сохранение SS, ESP, EFLAGS, CS, EIP.
- **ARM Cortex-M:** автоматически в стек: R0-R3, R12, LR, PC, xPSR.
- **MIPS:** ничего автоматически, ПО само сохраняет.

**4. Переключение на стек прерываний** (опционально).

В некоторых архитектурах (ARM Cortex-M, x86 при IST) есть отдельный стек для прерываний.

**5. Получение вектора.**

Контроллер сообщает процессору номер вектора.

**6. Поиск адреса обработчика.**

Процессор использует вектор как индекс в таблице векторов.

**7. Переход на обработчик.**

PC устанавливается на адрес обработчика. Начинается выполнение ISR.

### Возврат из обработчика

Специальная инструкция:
- **x86:** `IRET` (или `IRETQ` на 64-битах).
- **ARM Cortex-M:** возврат на специальный адрес `0xFFFFFFFx`.
- **ARM Cortex-A:** `ERET` или специальная форма возврата.

Инструкция атомарно:
- Восстанавливает регистры из стека.
- Восстанавливает PC.
- Восстанавливает регистр состояния.
- При необходимости — переключает на пользовательский стек.

### Запрет прерываний

Из rtos.md (2.1.6):
> «Нужен способ приостановить получение новых прерываний во время работы с разделяемыми данными из приложения. Для реализации такой возможности, процессор поддерживает специальные инструкции, с помощью которых можно разрешать или запрещать получение прерываний. Блок кода, выполняемый с запрещенными прерываниями называется критической секцией».

```c
// Источник: rtos.md, раздел 2.1.6
disable_interrupts();
<код критической секции>
enable_interrupts();
```

### Инструкции запрета по архитектурам

**x86:**
- `CLI` (Clear Interrupt Flag) — запретить.
- `STI` (Set Interrupt Flag) — разрешить.
- Влияет на маскируемые прерывания. NMI продолжает работать.

**ARM Cortex-M:**
- `CPSID i` — запретить.
- `CPSIE i` — разрешить.
- Или запись в `PRIMASK`.

**ARM Cortex-A:**
- Установка/очистка бита I в `CPSR`.

**MIPS:**
- Запись в регистр Status.

### NMI — Non-Maskable Interrupt

Особое прерывание, которое **нельзя замаскировать**. Используется для:
- ECC ошибка памяти.
- Hardware watchdog.
- Управление питанием.

NMI всегда обрабатывается, даже при общем запрете.

### Что блокируется при запрете

При `CLI`/аналог:
- Маскируемые прерывания **не вызывают обработчик**.
- Контроллер прерываний продолжает их **накапливать**.
- При `STI`/аналог — прерывания обрабатываются немедленно (в порядке приоритета).

### Маска приоритета

Из rtos.md (4.4.6):
> «Уровень маски прерываний» — текущий приоритет в процессоре. Прерывания с приоритетом ниже маски не вызывают обработчик.

Это **более тонкий** механизм, чем полный запрет:
- Высокоприоритетные продолжают приниматься.
- Низкоприоритетные ждут.

Реализация:
- **ARM Cortex-M:** регистр `BASEPRI` — задаёт минимальный приоритет.
- **x86 APIC:** регистр TPR (Task Priority Register).

Применение:
- Критические секции между обработчиками разных приоритетов.
- Защита от прерываний определённого уровня без полного запрета.

### Вложение и счётчик

Из rtos.md (2.1.6):
> «Если используемый процессор допускает вложение обработчиков прерываний, сразу же возникает вопрос относительно атомарности процедуры входа в обработчик. Атомарность передачи управления обработчику обычно поддерживается аппаратно, но ОС может требоваться выполнить одновременно с этим дополнительные действия. Например, может потребоваться вести счётчик вложения прерываний».

```c
int intr_nesting = 0;

void interrupt_entry(void) {
    intr_nesting++;
    // ... обработка ...
    intr_nesting--;
    if (intr_nesting == 0) {
        // Все обработчики закончены — можно вызвать планировщик
        check_for_reschedule();
    }
}
```

### Атомарность входа

Из rtos.md:
> «Возможна ситуация, когда сразу же после входа в один обработчик, возникает более приоритетное прерывание, до выполнения первой инструкции в первом обработчике».

Проблема: вложенный обработчик может «обогнать» инкремент счётчика.

Решения:
- **x86:** аппаратный запрет прерываний при входе в обработчик. Обработчик должен сам решить, разрешать ли вложение.
- **ARM Cortex-M:** запрет прерываний того же или меньшего приоритета.

### Применение запрета прерываний

**1. Критические секции в обработчике** — синхронизация с программой.

```c
volatile int flag = 0;

void main_loop() {
    disable_interrupts();
    if (flag) {
        // обработка
        flag = 0;
    }
    enable_interrupts();
}

void isr() {
    flag = 1;
}
```

**2. Защита разделяемых структур данных.**

**3. Реализация атомарных операций на однопроцессорной системе.**

### Недостатки запрета прерываний

- **Время реакции страдает** — прерывания не обрабатываются.
- **Не работает на SMP** — запрет только на нашем CPU.
- **Опасно использовать в больших участках кода** — может сломать систему.

### Резюме

- При прерывании: окончание инструкции → проверка маски → сохранение контекста → получение вектора → поиск обработчика → переход.
- Возврат через `IRET`/`ERET` — атомарная инструкция.
- Запрет прерываний: `CLI`/`STI`, `CPSID`/`CPSIE`, `PRIMASK`.
- NMI нельзя замаскировать.
- Маска приоритета — тоньше полного запрета.
- Счётчик вложения для корректной обработки вложенных прерываний.

## 75. Прерывания и их обработка. Исключения, их отличия от прерываний. Первичные и вторичные обработчики (верхняя и нижняя половина). Переключение контекста.
*Источник: rtos.md, Глава 2.1.5, 4.4.2, 5.5.2*

### Исключения

Из rtos.md (2.1.5):
> «Идею проверки некоторых условий перед или после выполнения каждой инструкции можно развить дальше. Например, процессор может проверять корректность аргументов инструкций. В случае обнаружения каких-либо аномалий, могут происходить действия аналогичные прерыванию: вызов процедуры-обработчика ошибки. Такой тип прерываний называется исключениями или исключительными ситуациями».

**Исключение** — реакция процессора на **аномальную ситуацию** при выполнении конкретной инструкции.

### Отличия от прерываний

| Параметр | Прерывания | Исключения |
|---|---|---|
| Источник | Внешнее устройство | Конкретная инструкция |
| Тип | Асинхронные | Синхронные |
| Повторяемость | Зависит от устройства | Та же инструкция → то же исключение |
| Назначение | Реакция на события | Ошибки, спец. ситуации |
| Контроллер | Используется | Не используется (rtos.md: «Исключения реализуются самим процессором ядром, контроллер прерываний для их обработки не используется») |
| Маскирование | Можно | Обычно нельзя |

Из rtos.md:
> «Прерывания и исключения являются близкими понятиями, и их обработка процессором зачастую идентична, отличается только природа их источника. Эти соображения приводят к тому, что в большинстве процессоров нет различий в прерываниях и исключениях с точки зрения программиста: для исключений выделено несколько векторов в той же таблице, которая используется и для прерываний».

### Типы исключений

**Faults (ошибки) — исправимые.**
- Возврат **на ту же инструкцию** (повторное выполнение).
- Примеры: page fault, division by zero, general protection fault.

**Traps (ловушки) — преднамеренные.**
- Возврат **на следующую инструкцию**.
- Примеры: точка останова, syscall.

**Aborts — серьёзные ошибки.**
- Возврат невозможен, обычно крах.
- Примеры: machine check exception, тройная ошибка (triple fault).

### «Нормальные» исключения

Из rtos.md:
> «Хотя, как будет показано далее, в разделе про реализацию процессором защиты памяти, в некоторых случаях исключения можно считать нормальным ходом выполнения программы».

Page fault — пример «нормального» исключения:
- Программа обращается к выгруженной странице.
- ОС подгружает её с диска.
- Возвращает управление на ту же инструкцию.
- Программа не «знает», что произошло.

Аналогично: copy-on-write, демандовая загрузка.

### Системные вызовы как исключения

Из rtos.md (про x86):
> «Использование одного вектора на все аппаратные прерывания плюс возможность узнать текущий вектор уже внутри обработчика».

`INT 0x80` — программное прерывание, технически исключение типа trap. Процессор атомарно:
- Переключается в режим ядра.
- Сохраняет контекст.
- Переходит на адрес обработчика из таблицы.

### Верхняя и нижняя половина обработчика

Из rtos.md (5.5.2): концепция **отложенных процедур (DPC)**.

**Проблема:** обработчик прерывания должен быть коротким для хорошего времени реакции на другие прерывания. Но обработка события может требовать долгой работы (сетевой пакет — TCP/IP стек, файл с диска — VFS).

**Решение — двухуровневая обработка:**

**Верхняя половина (top half) / первичный обработчик / ISR:**
- Выполняется немедленно при прерывании.
- Очень короткая.
- Подтверждает устройство.
- Берёт минимально необходимые данные.
- Запланивает «отложенную процедуру».

**Нижняя половина (bottom half) / вторичный обработчик / DPC / softirq:**
- Выполняется **позже**, в более удобном контексте.
- Может быть длинной.
- Может быть прервана аппаратными прерываниями.
- Может использовать больше функционала.

### Реализация в rtos.md

```
Время:
0  ┐  Прерывание → ISR (короткая)
   │      ↓
   │  Возврат из ISR
1  │      ↓
   │  Срабатывает DPC (программное прерывание низкого приоритета)
   │      ↓
   │  DPC выполняется
2  │      ↓
   │  Возврат к прерванному коду
   ┘
```

### Реализация в Linux

**Tasklets:**
```c
void my_tasklet_fn(unsigned long data);
DECLARE_TASKLET(my_tasklet, my_tasklet_fn, 0);

void my_isr(int irq, void *dev_id) {
    // быстрая обработка
    tasklet_schedule(&my_tasklet);   // отложить
}
```

**Softirqs:** более общий механизм, на котором построены tasklets.

**Workqueues:** для долгой работы, выполняются в **потоках ядра** — могут блокироваться:
```c
DECLARE_WORK(my_work, my_work_fn);
schedule_work(&my_work);
```

### Контекст обработчика и переключение

Из rtos.md (4.4.3):
> «В случае использования концепции потоков, каждый из них имеет собственный контекст, в состав которого входит текущая вершина стека, счётчик инструкций и, возможно, аргумент, передаваемый в функцию потока».

При прерывании пользовательского потока:
1. Контекст потока сохраняется в его стек ядра.
2. Обработчик выполняется на стеке прерываний (или стеке ядра потока).
3. После обработчика — возможно, переключение контекста на другой поток.
4. Возврат в новый поток.

### Переключение контекста через DISPATCH

Из rtos.md (4.4.3):
> «Переключать контекст внутри прерываний нельзя, поскольку обработчики могут быть вложенными, но, тем не менее, в результате обработки прерывания могут потребоваться переключения контекста. Такое может произойти в том случае, если обработчик прерывания переводит в состояние готовности к выполнению поток с приоритетом выше чем у текущего потока, который был прерван прерыванием. Нужно как-то организовать асинхронный вызов функций ядра, занимающихся планированием и переключением контекста (функций планировщика), который произошёл бы после выхода из всех вложенных прерываний».

Подходы:
- **Счётчик вложения:** «кто сделал последний декремент и достиг нуля, тот вызывает планировщик».
- **Программное прерывание DISPATCH:** обработчик запрашивает программное прерывание низкого приоритета; оно срабатывает после всех аппаратных, делает планирование.

### Резюме

- Исключения — синхронные «прерывания», связанные с конкретной инструкцией.
- Faults / Traps / Aborts по поведению.
- Используются для page fault, syscall, ошибок.
- Верхняя половина — короткий ISR; нижняя — отложенная обработка.
- В Linux: tasklets, softirqs, workqueues.
- Переключение контекста — после всех вложенных прерываний.

## 76. Обработка прерываний и исключений в пользовательском процессе.
*Источник: rtos.md, Глава 7.8.1, 7.8.2; общие принципы*

### Сценарий

Пользовательский процесс выполняет код в режиме пользователя. Происходит:
- Аппаратное прерывание (от устройства).
- Исключение (например, page fault).
- Системный вызов.

### Действия при прерывании пользовательского кода

**1. Аппаратное переключение в режим ядра.**

Атомарно:
- Сохранение PSW (Program Status Word).
- Сохранение PC.
- Переключение на **стек ядра** потока (адрес в TSS на x86, или вручную на MIPS).
- Переход на обработчик.

**2. Сохранение полного контекста.**

Обработчик-обёртка сохраняет все регистры пользователя в стек ядра.

**3. Запуск C-обработчика.**

Высокоуровневый обработчик выполняется в режиме ядра.

**4. Обработка события.**

В зависимости от типа:
- Прерывание устройства: обработать, возможно разблокировать ожидающий поток.
- Page fault: возможно подгрузить страницу или поднять SIGSEGV.
- Syscall: вызвать `sys_*` функцию.

**5. Проверка необходимости переключения контекста.**

Если другой поток стал готов и приоритетнее — нужно переключиться.

**6. Возврат в пользователя.**

- Восстановление регистров из стека ядра.
- Возможная доставка сигналов.
- Атомарный возврат через `IRET`/`ERET`.

### Особенности

**Переключение стека обязательно.**

Из rtos.md (4.6.1):
> «Атомарность нужна для того, чтобы исключить сценарий, когда точно в момент между переключением режима и переключением стека произойдёт аппаратное прерывание. Обработчик может решить, что раз выполнение происходит уже в режиме ядра, то контекст можно сохранить на текущий стек и тем самым дать пользовательским потокам доступ к потенциально критической информации».

Без переключения на стек ядра возможны уязвимости.

**Контекст сохраняется в стек ядра потока, не в стек прерываний.**

Это нужно, чтобы потом можно было переключить контекст на другой поток.

**Невидимость для пользователя.**

После возврата пользовательский код продолжает выполнение, как будто прерывания не было. Регистры, флаги, PC — те же.

### Page fault — пример «нормального» исключения

Если пользователь обращается к выгруженной странице:
1. Процессор поднимает page fault.
2. Обработчик в ядре определяет:
   - Это в АП процесса? — Да.
   - Страница на диске? — Да.
3. Подкачивает страницу.
4. Обновляет таблицу страниц.
5. Возвращается **к той же инструкции**, она повторяется.
6. Пользователь не знает, что произошло.

### Page fault — настоящая ошибка

Если пользователь обращается по адресу вне своего АП:
1. Page fault.
2. Обработчик: «не в АП».
3. Посылает сигнал SIGSEGV процессу.
4. Если процесс не перехватывает SIGSEGV → завершение.

### Доставка сигналов

При возврате из ядра в пользователя ядро проверяет ожидающие сигналы:

```c
// Псевдокод
if (signal_pending(current)) {
    deliver_signal_to_user();
}
```

**Доставка сигнала:**
1. На стеке пользователя резервируется место для контекста.
2. Сохраняется текущий контекст (регистры).
3. PC устанавливается на адрес обработчика сигнала.
4. SP корректируется.
5. Возврат в пользователя — выполняется обработчик сигнала.

**Возврат из обработчика сигнала:**
- В конце обработчика — системный вызов `sigreturn`.
- Восстанавливает оригинальный контекст.
- Программа продолжает с прерванной точки.

### Сигналы в многопоточной системе

Сигнал, направленный процессу:
- Доставляется **одному** из потоков.
- Выбор: либо тот, кто не маскирует, либо специальный поток-обработчик.

Сигнал, направленный потоку (`pthread_kill`):
- Доставляется именно ему.

Маска сигналов — у каждого потока своя (`pthread_sigmask`).

### Прерывание CPU-bound потока

Сценарий:
- Поток выполняет долгие вычисления.
- Приходит таймерное прерывание.
- Обработчик: «истёк квант, нужно переключить».
- Устанавливается `need_resched`.
- При возврате в пользователя — планировщик переключает на другой поток.

### Системные вызовы

Системный вызов — самое частое «прерывание» пользовательского кода (в смысле перехода в ядро):

1. Пользователь: `mov rax, NUM; syscall`.
2. Аппаратно: переключение режима, сохранение PC, переход на обработчик `syscall_handler`.
3. `syscall_handler`: сохранение регистров, проверка номера, вызов `sys_xxx`.
4. `sys_xxx` выполняет работу, возвращает значение.
5. Возврат: восстановление регистров, возврат в пользователя.

Отличие от обычного прерывания: **синхронное**, инициировано самим пользователем.

### Резюме

- Прерывания и исключения пользовательского процесса обрабатываются прозрачно.
- Контекст сохраняется в стек ядра потока.
- Page fault часто «нормальное» — для подкачки страниц.
- Сигналы доставляются при возврате из ядра в пользователя.
- Системные вызовы — частный случай «прерывания» от пользователя.

## 77. Старший и младший номера устройств. MKDEV. Получение и освобождение номеров устройств.
*Источник: robert_lav.md, Глава 13; Linux Device Drivers (LDD)*

### Файлы устройств

В Unix/Linux действует принцип «всё — файл». Устройства представлены файлами в `/dev`:

```
$ ls -l /dev/sda /dev/tty1
brw-rw---- 1 root disk     8,  0 ... /dev/sda
crw--w---- 1 root tty      4,  1 ... /dev/tty1
```

- **b** или **c** в начале — block или character device.
- `8, 0` или `4, 1` — major и minor номера.

### Major и minor номера

**Major (старший) номер.**
- Идентифицирует **драйвер**.
- Все устройства, обслуживаемые одним драйвером, имеют один major.

**Minor (младший) номер.**
- Идентифицирует **конкретное устройство** среди обслуживаемых драйвером.
- Драйвер сам интерпретирует minor.

Пример: драйвер SCSI дисков имеет major 8. Minor 0 = sda, 16 = sdb, 32 = sdc... Для разделов: 1 = sda1, 2 = sda2.

### Тип dev_t

В Linux:
- 12 бит — major.
- 20 бит — minor.
- Итого 32 бита.

```c
typedef unsigned int dev_t;
```

Раньше было 8+8 = 16 бит, расширили для поддержки большего числа устройств.

### Макросы

```c
#include <linux/kdev_t.h>

// Создать dev_t из major и minor
dev_t MKDEV(unsigned int major, unsigned int minor);

// Извлечь
unsigned int MAJOR(dev_t dev);
unsigned int MINOR(dev_t dev);
```

Пример:
```c
dev_t dev = MKDEV(240, 5);
// MAJOR(dev) = 240
// MINOR(dev) = 5
```

### Статическое выделение

```c
int register_chrdev_region(dev_t first, unsigned int count, char *name);
```

Зарезервировать диапазон от `first` до `first + count - 1` для драйвера.

Пример:
```c
dev_t dev = MKDEV(240, 0);
int err = register_chrdev_region(dev, 4, "mydriver");
if (err < 0) {
    printk(KERN_WARNING "Cannot register device numbers\n");
    return err;
}
// Зарезервированы (240, 0), (240, 1), (240, 2), (240, 3)
```

### Проблема статического выделения

- Какой major выбрать? Документация ядра содержит **список зарезервированных** major-номеров.
- Если выбрать занятый — конфликт.
- На разных системах могут быть разные занятые номера.

Из-за этого современные драйверы используют **динамическое** выделение (вопрос 78).

### Освобождение

```c
void unregister_chrdev_region(dev_t first, unsigned int count);
```

Вызывается при выгрузке драйвера:
```c
static void __exit my_driver_exit(void) {
    cdev_del(&my_cdev);
    unregister_chrdev_region(my_dev, 4);
}
```

**Важно:** регион должен быть **тот же**, что и при регистрации (тот же first и count). Иначе ошибка.

### Просмотр зарегистрированных

```
$ cat /proc/devices
Character devices:
  1 mem
  4 /dev/vc/0
  4 tty
  5 /dev/tty
  5 /dev/console
 10 misc
...

Block devices:
  1 ramdisk
  8 sd
  9 md
```

### Создание файла устройства

После регистрации номера нужен файл в `/dev`:

**Вручную:**
```bash
mknod /dev/mydriver c 240 0
chmod 666 /dev/mydriver
```

**Через udev:**
```c
struct class *my_class;

static int __init my_driver_init(void) {
    register_chrdev_region(MKDEV(240, 0), 1, "mydriver");

    my_class = class_create(THIS_MODULE, "mydriver");
    device_create(my_class, NULL, MKDEV(240, 0), NULL, "mydriver");
    return 0;
}
```

udev автоматически создаст `/dev/mydriver`.

### Резюме

- Файлы устройств в `/dev` имеют major+minor номера.
- Major — драйвер, minor — конкретное устройство.
- `MKDEV(maj, min)` — создать dev_t; `MAJOR(d)`, `MINOR(d)` — извлечь.
- `register_chrdev_region` — статически зарезервировать диапазон.
- `unregister_chrdev_region` — освободить.
- Статика проблемна — лучше динамика.

## 78. Старший и младший номера устройств. Динамическое выделение старших номеров. alloc_chrdev_region.
*Источник: robert_lav.md, Глава 13; LDD*

### Зачем динамическое выделение

Проблемы статического (см. вопрос 77):
- Нужно знать заранее свободный major.
- На разных системах разные занятые номера.
- Возможны конфликты при установке.

Современные драйверы используют **динамическое выделение**: ядро само выбирает свободный major.

### Функция alloc_chrdev_region

```c
int alloc_chrdev_region(dev_t *dev, unsigned int firstminor,
                        unsigned int count, char *name);
```

Параметры:
- `dev` — **выходной**. Будет содержать назначенный dev_t.
- `firstminor` — начальный minor (обычно 0).
- `count` — сколько minor-номеров нужно.
- `name` — имя для `/proc/devices`.

### Полный пример использования

```c
#include <linux/init.h>
#include <linux/module.h>
#include <linux/fs.h>
#include <linux/cdev.h>

#define MYDRV_COUNT 4

static dev_t my_dev;
static struct cdev my_cdev;

static int __init my_driver_init(void) {
    int result;

    // 1. Динамически получить диапазон номеров
    result = alloc_chrdev_region(&my_dev, 0, MYDRV_COUNT, "mydriver");
    if (result < 0) {
        printk(KERN_WARNING "Cannot allocate dev numbers\n");
        return result;
    }
    printk(KERN_INFO "Got major %d\n", MAJOR(my_dev));

    // 2. Инициализировать структуру cdev
    cdev_init(&my_cdev, &my_fops);
    my_cdev.owner = THIS_MODULE;

    // 3. Зарегистрировать устройство
    result = cdev_add(&my_cdev, my_dev, MYDRV_COUNT);
    if (result) {
        unregister_chrdev_region(my_dev, MYDRV_COUNT);
        return result;
    }

    return 0;
}

static void __exit my_driver_exit(void) {
    cdev_del(&my_cdev);
    unregister_chrdev_region(my_dev, MYDRV_COUNT);
}

module_init(my_driver_init);
module_exit(my_driver_exit);
MODULE_LICENSE("GPL");
```

### Преимущества

- **Нет конфликтов** с другими драйверами.
- **Портируемость** — работает на любой системе.
- **Стандартная практика** — рекомендация документации.

### Недостатки

- **Major номер не известен заранее.**

### Как пользователь узнает major

Несколько способов:

**1. Через `/proc/devices`.**
```bash
$ cat /proc/devices | grep mydriver
240 mydriver
```

**2. Через `sysfs` (если используется class_create + device_create).**
```bash
$ cat /sys/class/mydriver/mydriver/dev
240:0
```

**3. Через `udev`.**

udev автоматически создаёт `/dev/mydriver` без участия администратора.

### Скрипт автоматического создания узла

Классический способ — скрипт после загрузки модуля:

```bash
#!/bin/bash
module="mydriver"
device="mydriver"
mode="664"

# Загрузить модуль
/sbin/insmod ./$module.ko

# Получить major
major=$(awk "\$2==\"$module\" {print \$1}" /proc/devices)

# Создать узел
mknod /dev/${device} c $major 0
chmod $mode /dev/${device}
```

Сейчас используется реже — udev делает это сам.

### Через udev (современный способ)

```c
#include <linux/device.h>

static struct class *my_class;

static int __init my_driver_init(void) {
    alloc_chrdev_region(&my_dev, 0, 1, "mydriver");

    my_class = class_create(THIS_MODULE, "mydriver");
    if (IS_ERR(my_class)) {
        unregister_chrdev_region(my_dev, 1);
        return PTR_ERR(my_class);
    }

    device_create(my_class, NULL, my_dev, NULL, "mydriver");

    cdev_init(&my_cdev, &my_fops);
    cdev_add(&my_cdev, my_dev, 1);

    return 0;
}

static void __exit my_driver_exit(void) {
    cdev_del(&my_cdev);
    device_destroy(my_class, my_dev);
    class_destroy(my_class);
    unregister_chrdev_region(my_dev, 1);
}
```

udev увидит уведомления от `device_create` и автоматически:
- Создаст `/dev/mydriver`.
- Установит права.
- Применит правила.

### Освобождение

То же, что и для статического выделения:
```c
unregister_chrdev_region(my_dev, MYDRV_COUNT);
```

### Резюме

- `alloc_chrdev_region` динамически выделяет major+minor.
- Стандартная практика для новых драйверов.
- Major номер узнаётся через `/proc/devices`, `sysfs`, или автоматически через udev.
- Освобождается через `unregister_chrdev_region`.

## 79. Файловые операции. int (*flush) (struct file *);
*Источник: robert_lav.md, Глава 13 «Объект file_operations»; LDD*

### Структура file_operations

Из robert_lav.md (Глава 13):
> «Объект `file_operations` (операции с файлами) содержит методы, которые ядро может вызывать для манипуляций с объектом файла».

Это **таблица функций**, реализующих интерфейс файла. Драйвер заполняет нужные поля своими функциями, остальные оставляет NULL.

```c
// Источник: robert_lav.md, Глава 13
struct file_operations {
    struct module *owner;
    loff_t (*llseek) (struct file *, loff_t, int);
    ssize_t (*read) (struct file *, char __user *, size_t, loff_t *);
    ssize_t (*write) (struct file *, const char __user *, size_t, loff_t *);
    int (*readdir) (struct file *, void *, filldir_t);
    unsigned int (*poll) (struct file *, struct poll_table_struct *);
    int (*ioctl) (struct inode *, struct file *, unsigned int, unsigned long);
    long (*unlocked_ioctl) (struct file *, unsigned int, unsigned long);
    int (*mmap) (struct file *, struct vm_area_struct *);
    int (*open) (struct inode *, struct file *);
    int (*flush) (struct file *, fl_owner_t id);
    int (*release) (struct inode *, struct file *);
    int (*fsync) (struct file *, struct dentry *, int datasync);
    int (*fasync) (int, struct file *, int);
    int (*lock) (struct file *, int, struct file_lock *);
    // ... ещё много методов
};
```

### Зачем такая таблица

Один и тот же системный вызов `read()` должен работать для разных типов файлов:
- Обычный файл на ext4.
- Файл на NTFS.
- Файл устройства.
- FIFO.
- Сокет.

Принцип VFS — единый интерфейс через таблицу. Системный вызов `read(fd, buf, n)`:

```c
// Псевдокод ядра
ssize_t sys_read(int fd, char __user *buf, size_t count) {
    struct file *file = fget(fd);
    return file->f_op->read(file, buf, count, &file->f_pos);
}
```

`f_op` — указатель на `file_operations`, конкретный для типа файла.

### Метод flush

```c
int (*flush) (struct file *file, fl_owner_t id);
```

**Что делает:** вызывается **при каждом** `close()` пользователем, **до** вызова `release`.

### Когда вызывается flush

Из стандартной семантики Linux:
- Каждый раз когда процесс вызывает `close(fd)`.
- Если файл открыт несколько раз (через `dup`, `fork`), `flush` вызывается **при каждом close**, а `release` — только при последнем.

Пример:
```c
int fd = open("/dev/mydev", O_RDWR);  // open
int fd2 = dup(fd);                     // ещё один fd на тот же file

close(fd);    // flush
close(fd2);   // flush, потом release (это последний)
```

### Чем flush отличается от release

| Параметр | flush | release |
|---|---|---|
| Когда | Каждый close | Только последний close |
| Параметры | file, fl_owner_t id | inode, file |
| Назначение | Сброс буферов, ожидание | Освобождение ресурсов |
| Возврат ошибки | Передаётся пользователю | Игнорируется |

### Назначение flush

**1. Сброс буферов.**

Если драйвер буферизует данные перед записью на устройство, `flush` гарантирует, что данные выйдут:

```c
int my_flush(struct file *file, fl_owner_t id) {
    struct mydev *dev = file->private_data;
    return flush_buffer_to_device(dev);
}
```

**2. Ожидание завершения.**

Подождать, пока все ранее отправленные данные действительно записались.

**3. Финализация записи.**

Например, для сетевых файлов — гарантия, что данные доставлены.

### Чем отличается от fsync

```c
int (*fsync) (struct file *, struct dentry *, int datasync);
```

- **fsync** — вызывается явно через `fsync(fd)`. Гарантирует запись данных на диск.
- **flush** — вызывается автоматически при `close`.

`fsync` сильнее: пользователь явно требует синхронизации с диском.

### Применение

В большинстве драйверов символьных устройств `flush` **не используется** — поле равно NULL. Это нормально.

Используется в:
- **FIFO/Pipe:** чтобы убедиться, что данные доставлены другой стороне.
- **NFS:** для синхронизации с сервером.
- **TTY:** сброс буферов терминала.
- **Сетевые драйверы:** flush очередей.

### Пример реализации

```c
struct mydev {
    spinlock_t lock;
    struct list_head pending_writes;
    wait_queue_head_t wait;
};

int my_flush(struct file *file, fl_owner_t id) {
    struct mydev *dev = file->private_data;

    // Подождать, пока все pending_writes уйдут
    spin_lock(&dev->lock);
    while (!list_empty(&dev->pending_writes)) {
        spin_unlock(&dev->lock);
        wait_event_interruptible(dev->wait, list_empty(&dev->pending_writes));
        spin_lock(&dev->lock);
    }
    spin_unlock(&dev->lock);

    return 0;
}
```

### Регистрация file_operations

В драйвере:
```c
static const struct file_operations my_fops = {
    .owner   = THIS_MODULE,
    .open    = my_open,
    .release = my_release,
    .read    = my_read,
    .write   = my_write,
    .flush   = my_flush,    // если нужен
    // .ioctl = my_ioctl,
};
```

Затем связывается с устройством:
```c
cdev_init(&my_cdev, &my_fops);
```

### Резюме

- `file_operations` — таблица методов работы с файлом.
- `flush` вызывается при каждом close, до release.
- Назначение: сброс буферов, ожидание завершения.
- Отличается от release (только при последнем close) и fsync (явный sync).
- В большинстве драйверов не используется.

## 80. Структура inode
*Источник: robert_lav.md, Глава 13 «Объект inode»*

### Что такое inode

Из robert_lav.md (Глава 13):
> «Для каждого файла в файловой системе существует представляющий его объект `inode`. Однако сам файловый индекс создается в памяти только в момент доступа к файлу. Это справедливо и для специальных файлов, таких как файлы устройств или конвейеры».

**inode** = **i**ndex **node** (индексный узел). Хранит **метаданные** файла, **независимо** от его имени.

### inode vs filename

Имя файла — это всего лишь **указатель** на inode:
```
имя "foo.txt" (dentry) → inode #42 → данные
имя "bar.txt" (другой dentry) → тот же inode #42 → те же данные (hard link)
```

Жёсткая ссылка — ещё одно имя для одного inode. Размер файла, права, время — всё в inode.

### Структура

```c
// Источник: robert_lav.md, Глава 13
struct inode {
    /* Идентификация и связи */
    struct hlist_node       i_hash;          /* хеш-таблица */
    struct list_head        i_list;
    struct list_head        i_sb_list;       /* список суперблока */
    struct list_head        i_dentry;        /* связанные dentries */
    unsigned long           i_ino;            /* номер индекса */

    /* Счётчики */
    atomic_t                i_count;         /* счётчик ссылок ядра */
    unsigned int            i_nlink;         /* количество жёстких ссылок */

    /* Владелец и права */
    uid_t                   i_uid;            /* user id */
    gid_t                   i_gid;            /* group id */
    umode_t                 i_mode;           /* права + тип */

    /* Размер и времена */
    loff_t                  i_size;           /* размер файла */
    struct timespec         i_atime;          /* последний доступ */
    struct timespec         i_mtime;          /* последнее изменение */
    struct timespec         i_ctime;          /* изменение метаданных */
    unsigned int            i_blkbits;
    blkcnt_t                i_blocks;

    /* Операции */
    struct inode_operations *i_op;            /* операции с индексом */
    struct file_operations  *i_fop;           /* операции с файлом */

    /* Связи */
    struct super_block      *i_sb;            /* суперблок файловой системы */
    struct file_lock        *i_flock;         /* список блокировок */
    struct address_space    *i_mapping;       /* отображение страниц */

    /* Тип-специфические данные */
    kdev_t                  i_rdev;           /* номер устройства */
    union {
        struct pipe_inode_info *i_pipe;        /* для FIFO */
        struct block_device    *i_bdev;        /* блочное устройство */
        struct cdev            *i_cdev;        /* символьное устройство */
    };

    /* Состояние */
    unsigned long           i_state;          /* флаги состояния */
    unsigned long           dirtied_when;     /* время первого изменения */
    unsigned int            i_flags;
    atomic_t                i_writecount;
    void                    *i_security;      /* для LSM */
    void                    *i_private;       /* для файловой системы */
};
```

### Основные поля

**Идентификация:**
- `i_ino` — номер индекса в файловой системе. Уникален.
- `i_sb` — суперблок (описание файловой системы).

**Связи:**
- `i_dentry` — список имён (dentries), указывающих на этот inode.
- `i_nlink` — число жёстких ссылок.

**Счётчики:**
- `i_count` — атомарный счётчик использования в ядре. Когда становится 0 и nlink=0 — освобождение.

**Владение:**
- `i_uid`, `i_gid` — владелец.
- `i_mode` — права (rwxrwxrwx) + тип файла.

**Размер и времена:**
- `i_size` — размер в байтах.
- `i_atime` — время последнего доступа (read, exec).
- `i_mtime` — время последней модификации содержимого (write).
- `i_ctime` — время изменения метаданных (chmod, chown).

**Операции:**
- `i_op` — операции с индексом (создание подфайлов, lookup, unlink).
- `i_fop` — операции с открытым файлом (read, write, ioctl).

### Типы файлов

`i_mode` содержит и права, и тип:
- `S_IFREG` — обычный файл.
- `S_IFDIR` — каталог.
- `S_IFLNK` — символическая ссылка.
- `S_IFBLK` — блочное устройство.
- `S_IFCHR` — символьное устройство.
- `S_IFIFO` — FIFO/pipe.
- `S_IFSOCK` — сокет.

Макросы проверки:
```c
S_ISREG(i->i_mode)
S_ISDIR(i->i_mode)
S_ISLNK(i->i_mode)
// ...
```

### Type-specific union

Из robert_lav.md:
> «Эти три поля сохраняются в структуре inode в виде объединения (union), поскольку один конкретный индекс в данный момент времени может быть связан только с одним из перечисленных выше типов устройств (либо ни с одним из них)».

Один inode — один тип:
- FIFO → `i_pipe`.
- Блочное устройство → `i_bdev`.
- Символьное устройство → `i_cdev`.

### Операции с inode

Поле `i_op` — таблица функций:

```c
struct inode_operations {
    int (*create)  (struct inode *, struct dentry *, int, struct nameidata *);
    struct dentry *(*lookup) (struct inode *, struct dentry *, struct nameidata *);
    int (*link)    (struct dentry *, struct inode *, struct dentry *);
    int (*unlink)  (struct inode *, struct dentry *);
    int (*symlink) (struct inode *, struct dentry *, const char *);
    int (*mkdir)   (struct inode *, struct dentry *, int);
    int (*rmdir)   (struct inode *, struct dentry *);
    int (*mknod)   (struct inode *, struct dentry *, int, dev_t);
    int (*rename)  (struct inode *, struct dentry *, struct inode *, struct dentry *);
    int (*readlink)(struct dentry *, char __user *, int);
    void (*truncate) (struct inode *);
    int (*permission)(struct inode *, int);
    // ...
};
```

Это операции **с самим inode** или **с его содержимым на уровне ФС** (например, создать подфайл в каталоге).

### Жёсткие ссылки

Несколько dentry → один inode:
```
"/home/user/file" → inode #42 → данные
"/home/user/link" → inode #42 → те же данные
```

`i_nlink` = 2. При `unlink` одного — становится 1. При unlink последнего и `i_count=0` — освобождение inode.

### Жизненный цикл inode

1. **Создание:** при первом доступе к файлу VFS создаёт inode в памяти, заполняет из диска.
2. **Использование:** атомарный счётчик `i_count` инкрементируется на каждое использование.
3. **Кеширование:** даже если `i_count = 0`, inode может оставаться в кеше для быстрого доступа.
4. **Освобождение:** при удалении файла (`i_nlink = 0`) и отсутствии ссылок.

### Кеш inodes (icache)

Из robert_lav.md (Глава 12):
> «В качестве примера рассмотрим структуры inode, которые представляют в оперативной памяти файловые индексы... Структуры inode выделяются из кеша inode_cachep».

Inode кэшируются в slab-кеше для быстрого повторного доступа. При обращении к файлу:
1. Поиск в icache.
2. Если есть — взять оттуда.
3. Если нет — прочитать с диска.

### Применение в драйверах

Для символьных устройств:
- При open ядро находит/создаёт inode.
- `i_cdev` указывает на `struct cdev` — структуру драйвера.
- `i_fop` копируется из `cdev->ops` — методы драйвера.

```c
// В open ядра:
filp->f_op = inode->i_fop;
return filp->f_op->open(inode, filp);
```

### Резюме

- inode — структура с метаданными файла, независимая от имени.
- Содержит: ID, владельца, права, размер, времена, операции.
- В union — тип-специфичные данные (pipe, block, char device).
- `i_count` — счётчик ссылок, `i_nlink` — жёсткие ссылки.
- Кешируется для быстрого доступа.
- Связывается с драйвером через `i_cdev` и `i_fop`.
