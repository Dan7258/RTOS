## 91. Выделение памяти kmalloc(). Флаги gfp_mask и модификаторы операций, зон. Модификаторы типов. __GFP_WAIT. GFP_NOIO. GFP_ATOMIC.
*Источник: robert_lav.md, Глава 12 «Управление памятью»*

### Прототип

```c
void *kmalloc(size_t size, gfp_t flags);
```

### gfp_mask — структура флагов

Флаги делятся на **3 категории**:
- **Модификаторы операций** — какие действия можно выполнять.
- **Модификаторы зон** — из какой зоны памяти выделять.
- **Модификаторы типа** — комбинации часто используемых вариантов.

### Модификаторы операций

| Флаг | Назначение |
|---|---|
| `__GFP_WAIT` | Может блокироваться (засыпать на ожидании памяти) |
| `__GFP_IO` | Может инициировать ввод-вывод (swap out) |
| `__GFP_FS` | Может вызывать операции файловой системы |
| `__GFP_HIGH` | Высокий приоритет, можно использовать резервы |
| `__GFP_COLD` | Получать «холодную» (не в кэше) страницу |
| `__GFP_NOWARN` | Не выводить предупреждения при ошибке |
| `__GFP_REPEAT` | Повторять попытки при ошибке |
| `__GFP_NOFAIL` | Никогда не возвращать ошибку (зацикливаться) |
| `__GFP_NORETRY` | Не повторять при ошибке |
| `__GFP_COMP` | Добавить метаданные для compound page |
| `__GFP_ZERO` | Обнулить страницу |

### Модификаторы зон

| Флаг | Зона |
|---|---|
| `__GFP_DMA` | ZONE_DMA — для DMA-совместимых устройств (нижние 16 МБ) |
| `__GFP_DMA32` | ZONE_DMA32 — нижние 4 ГБ |
| `__GFP_HIGHMEM` | ZONE_HIGHMEM — верхняя память (> 896 МБ на 32-bit) |

### Модификаторы типов

Удобные комбинации:

| Тип | Эквивалент | Применение |
|---|---|---|
| `GFP_ATOMIC` | `__GFP_HIGH` | Высокий приоритет, **не блокируется**. Для IRQ, удержанных спинлоков |
| `GFP_NOWAIT` | 0 | Как ATOMIC, но без приоритета |
| `GFP_NOIO` | `__GFP_WAIT` | Может блокироваться, но **не инициирует I/O** |
| `GFP_NOFS` | `__GFP_WAIT \| __GFP_IO` | Может I/O, но **не FS** |
| `GFP_KERNEL` | `__GFP_WAIT \| __GFP_IO \| __GFP_FS` | Обычное использование |
| `GFP_USER` | Аналогично KERNEL | Для аллокаций «от имени» пользователя |
| `GFP_HIGHUSER` | `GFP_USER \| __GFP_HIGHMEM` | Верхняя память для пользователя |
| `GFP_DMA` | `__GFP_DMA` | DMA-память |

### Когда что использовать

**GFP_KERNEL** — почти везде в обычном коде ядра:
```c
struct dog *p = kmalloc(sizeof(*p), GFP_KERNEL);
```

**GFP_ATOMIC** — когда нельзя блокироваться:
- В обработчике прерывания.
- Под спинлоком.
- В контексте softirq/tasklet.
- В обработчике RCU.

```c
spin_lock(&lock);
ptr = kmalloc(size, GFP_ATOMIC);
if (!ptr) { spin_unlock(&lock); return -ENOMEM; }
// ...
spin_unlock(&lock);
```

**GFP_NOIO / GFP_NOFS** — в коде самой подсистемы I/O или FS:
- Чтобы не вызвать рекурсию (FS-код вызывает kmalloc, который вызывает FS-код для swap).

**GFP_DMA** — для буферов DMA на старых устройствах:
```c
buf = kmalloc(size, GFP_KERNEL | GFP_DMA);
```

### __GFP_WAIT

Из robert_lav.md:
> «`__GFP_WAIT` — означает, что распределитель памяти может переходить в состояние ожидания».

Если установлен → можно блокироваться для освобождения памяти (swap out, sync).
Если НЕ установлен → должна вернуть результат сразу, либо NULL.

### GFP_NOIO

«Нельзя инициировать I/O при выделении памяти.»

Используется в коде драйверов блочных устройств. Если выделение вызовет I/O для swap, а I/O идёт через тот же драйвер — deadlock.

### GFP_ATOMIC

«Не блокировать, выполнить быстро.»

- Используется в IRQ-контексте.
- Может потерпеть неудачу (NULL).
- Имеет доступ к специальным резервам памяти.

### Пример из robert_lav.md

```c
// Источник: robert_lav.md, Глава 12
ptr = kmalloc(size, __GFP_WAIT | __GFP_IO | __GFP_FS);
// Эквивалентно: GFP_KERNEL
```

## 92. Заготовленные кэши. Использование кэшей в scull. void *kmem_cache_alloc (kmem_cache_t *cache, int flags); scullc_cache = kmem_cache_create("scullc", scullc_quantum, 0, SLAB_HWCACHE_ALIGN, NULL, NULL);
*Источник: robert_lav.md, Глава 12 «Уровень блочного распределения памяти»*

### Зачем заготовленные кэши

Из robert_lav.md (Глава 12):
> «Выделение и освобождение памяти, занимаемой структурами данных, — одна из самых частых операций, которые выполняются в любом ядре. Для того чтобы упростить процедуру частого выделения и освобождения данных, программисты обычно ведут списки свободных ресурсов (free list)».

**Заготовленный (slab) кэш** — пул объектов **одного размера**, оптимизированный для быстрого выделения/освобождения.

### Преимущества slab

1. **Быстрое выделение** — нет поиска подходящего блока, есть готовые.
2. **Уменьшение фрагментации** — все объекты одного размера.
3. **Кэширование инициализации** — освобождённый объект можно использовать без полной реинициализации.
4. **Лучшее использование процессорного кэша.**

### Создание кэша

```c
kmem_cache_t *kmem_cache_create(const char *name,
                                 size_t size,
                                 size_t align,
                                 unsigned long flags,
                                 void (*ctor)(void *),
                                 void (*dtor)(void *));   // dtor устарел
```

Параметры:
- `name` — имя для отладки (виден в `/proc/slabinfo`).
- `size` — размер одного объекта.
- `align` — выравнивание (обычно 0 = по умолчанию).
- `flags` — флаги.
- `ctor` — конструктор для новых объектов (или NULL).
- `dtor` — деструктор (исторически, теперь NULL).

### Флаги создания кэша

| Флаг | Назначение |
|---|---|
| `SLAB_HWCACHE_ALIGN` | Выравнивать по линии аппаратного кэша |
| `SLAB_POISON` | Заполнять «мусором» для отладки |
| `SLAB_RED_ZONE` | Защитные зоны для детекта переполнения |
| `SLAB_PANIC` | Падать в panic при ошибке создания |
| `SLAB_CACHE_DMA` | Использовать DMA-совместимую память |

### Использование в scull (вариация scullc)

```c
static kmem_cache_t *scullc_cache;

// В init модуля
scullc_cache = kmem_cache_create("scullc",
                                  scullc_quantum,
                                  0,
                                  SLAB_HWCACHE_ALIGN,
                                  NULL, NULL);
if (!scullc_cache) {
    /* error */
    return -ENOMEM;
}
```

### Выделение объекта

```c
void *kmem_cache_alloc(kmem_cache_t *cache, int flags);
```

Возвращает указатель на объект указанного кэшем размера. `flags` — обычные GFP_*.

```c
quantum = kmem_cache_alloc(scullc_cache, GFP_KERNEL);
if (!quantum)
    goto out;
```

### Освобождение объекта

```c
void kmem_cache_free(kmem_cache_t *cache, void *obj);
```

```c
kmem_cache_free(scullc_cache, quantum);
```

### Уничтожение кэша

```c
int kmem_cache_destroy(kmem_cache_t *cache);
```

Возвращает 0 при успехе, ошибку если есть невозвращённые объекты.

```c
if (scullc_cache)
    kmem_cache_destroy(scullc_cache);
```

### scullc vs scull

scullc — модифицированная версия scull, использующая slab-кэш:
- Вместо `kmalloc(quantum)` — `kmem_cache_alloc(scullc_cache, GFP_KERNEL)`.
- Вместо `kfree(quantum)` — `kmem_cache_free(scullc_cache, quantum)`.

Преимущество: все кванты одного размера → быстрее, меньше фрагментации.

### Слой kmalloc и slab

`kmalloc` сам использует slab-аллокатор. Внутри есть несколько готовых кэшей:
- `kmalloc-8`, `kmalloc-16`, `kmalloc-32`, `kmalloc-64`, ..., `kmalloc-131072`.

`kmalloc(size, flags)` округляет size до следующего кэша и выделяет из него.

Для специализированных размеров (как в scullc) — создание собственного кэша экономит память (нет округления).

### /proc/slabinfo

Просмотр всех кэшей:
```
$ cat /proc/slabinfo
# name <active_objs> <num_objs> <objsize> <objperslab> <pagesperslab>
inode_cache    12345  20000   720  22  4
dentry         98765  150000  192  21  1
kmalloc-256    5000   7000    256  16  1
scullc         100    200     4000  1   1
```

### Применение

Все часто-используемые структуры в Linux ядре имеют свой slab-кэш:
- `inode_cache`, `dentry`, `task_struct`.
- `vm_area_struct`, `page`.
- Драйверные структуры.

## 93. Основы USB устройства. Структура драйвера. Оконечные точки. BULK. ISOCHRONOUS.
*Источник: LDD3; tanenbaum.md, Глава 1*

### USB — обзор

**USB (Universal Serial Bus)** — централизованная шина с одним host-контроллером.

Из tanenbaum.md (1.3.5):
> «Шина USB была разработана для подключения к компьютеру всех медленных устройств ввода-вывода вроде клавиатуры и мыши… USB является централизованной шиной, в которой главное (корневое) устройство один раз в миллисекунду опрашивает все устройства ввода-вывода».

Скорости:
- USB 1.0: 12 Мбит/с.
- USB 2.0: 480 Мбит/с.
- USB 3.0: 5 Гбит/с.
- USB 3.1: 10 Гбит/с.

### Структура USB-устройства

Иерархия:
```
USB device
  └── Configuration (1 или больше)
       └── Interface (1 или больше)
            └── Endpoint (1 или больше)
```

### Оконечные точки (endpoints)

**Endpoint** — точка передачи данных в USB-устройстве. Каждая идентифицируется:
- **Адресом** (0..15).
- **Направлением** (IN — к хосту, OUT — от хоста).
- **Типом** передачи.

Endpoint 0 всегда есть — для управляющих команд (control transfers).

### Типы передачи

**1. Control** — управление, конфигурирование.
- Двунаправленные.
- Гарантия доставки.
- Для setup-команд.

**2. Bulk** — большие объёмы данных, без жёстких временных ограничений.
- Гарантия доставки.
- Гарантия отсутствия ошибок (CRC).
- Без гарантии задержки.
- Применение: принтеры, флешки, сканеры.

**3. Interrupt** — небольшие данные с гарантией latency.
- Гарантия частоты опроса.
- Применение: HID-устройства (клавиатуры, мыши).

**4. Isochronous** — стабильный поток.
- Гарантированная пропускная способность.
- **Без гарантии доставки** — потерянные пакеты не повторяются.
- Применение: аудио, видео, телефония.

### BULK подробнее

- Максимальный размер пакета: до 64 байт (USB 1.0), 512 байт (USB 2.0).
- Если пропускная способность доступна — пакеты идут быстро.
- При загруженной шине — ждут.

### ISOCHRONOUS подробнее

- Зарезервированное время на каждом цикле USB.
- Подходит для real-time потоков (audio, video).
- Если пакет потерян — не повторяется (лучше пропустить, чем задержать).
- Применение: USB-аудиокарта, USB-камера.

### Структура драйвера USB

```c
#include <linux/usb.h>

// Идентификация устройств
static struct usb_device_id my_table[] = {
    { USB_DEVICE(VENDOR_ID, PRODUCT_ID) },
    { }
};
MODULE_DEVICE_TABLE(usb, my_table);

// Структура драйвера
static struct usb_driver my_driver = {
    .name       = "my_usb_driver",
    .id_table   = my_table,
    .probe      = my_probe,
    .disconnect = my_disconnect,
};

// Регистрация
module_usb_driver(my_driver);
```

### probe и disconnect

```c
static int my_probe(struct usb_interface *interface,
                    const struct usb_device_id *id);
static void my_disconnect(struct usb_interface *interface);
```

**probe** вызывается при подключении совместимого устройства:
- Проверить, что устройство подходит.
- Инициализировать.
- Зарегистрировать символьное устройство (если нужно).

**disconnect** вызывается при отключении:
- Освободить ресурсы.
- Уведомить пользователей.

### URB — USB Request Block

```c
struct urb *usb_alloc_urb(int iso_packets, gfp_t mem_flags);
int usb_submit_urb(struct urb *urb, gfp_t mem_flags);
void usb_free_urb(struct urb *urb);
```

URB — единица передачи данных в USB. Для асинхронной работы.

### Применение

- **Bulk:** USB-флешки, принтеры, сетевые адаптеры.
- **Interrupt:** клавиатуры, мыши, USB-HID устройства.
- **Isochronous:** USB-микрофоны, веб-камеры.
- **Control:** инициализация любого устройства.

## 94. Основы USB устройства. Структура драйвера. Интерфейсы. Конфигурации. struct usb_host_interface *altsetting
*Источник: LDD3*

### Иерархия USB-устройства

```
Устройство (USB Device)
  ├── Конфигурация 1 (Configuration)
  │    ├── Интерфейс 1 (Interface)
  │    │    ├── Альт. настройка 0 (altsetting)
  │    │    │    ├── Endpoint 1
  │    │    │    └── Endpoint 2
  │    │    └── Альт. настройка 1
  │    │         └── Endpoint 1 (другие параметры)
  │    └── Интерфейс 2
  └── Конфигурация 2
```

### Конфигурация

**Конфигурация** — глобальный набор параметров устройства. Одно устройство может иметь несколько конфигураций.

Пример: устройство-комбо (флешка + читатель карт) может иметь:
- Конфигурация 1: только флешка.
- Конфигурация 2: только читатель карт.
- Конфигурация 3: оба активны.

Активна **одна** конфигурация в каждый момент.

### Интерфейс

**Интерфейс** — функциональная группа endpoint'ов. Соответствует **одной функции** устройства.

Пример: USB-аудио карта может иметь интерфейсы:
- Audio Control.
- Audio Streaming.
- Microphone.

В Linux **каждый интерфейс** обрабатывается **отдельным драйвером**. То есть один USB-устройство может «обслуживать» несколько драйверов.

### Альтернативные настройки (altsetting)

Один интерфейс может иметь несколько **альтернативных настроек** (alternate settings). Они отличаются параметрами endpoint'ов.

Применение: переключение между качеством потока.

Пример: USB-камера:
- altsetting 0: нет потока (для экономии bandwidth при инициализации).
- altsetting 1: 320×240 @ 15 fps.
- altsetting 2: 640×480 @ 30 fps.
- altsetting 3: 1280×720 @ 30 fps.

Хост выбирает нужный altsetting через `usb_set_interface`.

### struct usb_host_interface

В ядре Linux:
```c
struct usb_host_interface {
    struct usb_interface_descriptor desc;
    struct usb_host_endpoint *endpoint;
    char *string;
    /* ... */
};
```

### struct usb_interface

```c
struct usb_interface {
    struct usb_host_interface *altsetting;    // массив altsettings
    struct usb_host_interface *cur_altsetting; // текущий
    unsigned num_altsetting;                   // число
    int minor;
    enum usb_interface_condition condition;
    unsigned sysfs_files_created:1;
    /* ... */
};
```

`altsetting` — указатель на **массив** структур `usb_host_interface`, размер `num_altsetting`.

### Доступ к endpoint'ам интерфейса

```c
struct usb_host_interface *iface = intf->cur_altsetting;
struct usb_endpoint_descriptor *ep;

for (int i = 0; i < iface->desc.bNumEndpoints; i++) {
    ep = &iface->endpoint[i].desc;
    // обрабатывать ep
}
```

### Дескрипторы

USB-устройство содержит дескрипторы:
- **Device descriptor** — общая инфа (VID, PID, версия).
- **Configuration descriptor** — конфигурации.
- **Interface descriptor** — интерфейсы.
- **Endpoint descriptor** — endpoint'ы.
- **String descriptor** — текстовые строки.

Они читаются хостом при подключении (enumeration).

### Регистрация драйвера

```c
static struct usb_driver my_driver = {
    .name       = "my_driver",
    .id_table   = my_table,
    .probe      = my_probe,
    .disconnect = my_disconnect,
};
```

`probe` принимает `struct usb_interface*` — драйвер обрабатывает **интерфейс**, не устройство целиком.

### Пример probe

```c
static int my_probe(struct usb_interface *interface,
                    const struct usb_device_id *id)
{
    struct usb_host_interface *iface;
    struct usb_endpoint_descriptor *ep;
    int i;

    iface = interface->cur_altsetting;

    for (i = 0; i < iface->desc.bNumEndpoints; i++) {
        ep = &iface->endpoint[i].desc;
        if (usb_endpoint_is_bulk_in(ep)) {
            // нашли bulk-in endpoint
        }
    }

    return 0;
}
```

### Helper-макросы

```c
usb_endpoint_is_bulk_in(ep)
usb_endpoint_is_bulk_out(ep)
usb_endpoint_is_int_in(ep)
usb_endpoint_is_isoc_in(ep)
// ...
```

## 95. Пример изохорной точки в whireshark. Устройство отправляющее изохорные точки.
*Источник: LDD3; общие знания о USB-протоколе*

### Что такое isochronous endpoint

См. вопрос 93. Изохронные передачи — для **стабильного потока** реального времени.

### Устройства, отправляющие изохронные данные

- **USB-микрофон** — поток аудио (PCM-данные).
- **USB-камера** — поток видео.
- **USB-аудио** (наушники с микрофоном) — звук.
- **USB-телефония** (VoIP-гарнитуры).

### Пример: USB-микрофон

USB-аудио устройство соответствует **USB Audio Class** (UAC). Имеет:
- Audio Control interface — управление.
- Audio Streaming interface — потоки данных.

Streaming interface содержит **изохронный endpoint**, через который **постоянно** идут пакеты аудио.

### Пакет в Wireshark

Wireshark отображает USB-трафик при наличии USBPcap (Windows) или usbmon (Linux).

Типичный пакет:
```
URB type: COMPLETE
URB transfer type: ISOCHRONOUS (0x00)
Endpoint: 0x81 (IN, EP 1)
Device: 4
URB packet length: 192
ISO descriptor:
    Frame number: 12345
    Status: 0
    Length: 192
Data: 88 92 7B 6D ... (PCM data)
```

### Характеристики пакета

- **Endpoint type:** ISOCHRONOUS.
- **Direction:** IN (от устройства к хосту).
- **Packet length:** соответствует параметрам потока. Для 48kHz 16-bit моно: 96 байт/мс.
- **Status:** 0 = успех.

### Поток данных

Изохронный endpoint:
- Опрашивается **каждый кадр** USB (1 мс для full-speed, 125 мкс для high-speed).
- Передаёт фиксированное число байт.
- При потере — не повторяется.

### Wireshark — практика

В Linux:
```bash
sudo modprobe usbmon
sudo wireshark
# выбрать usbmonX (X = номер хост-контроллера)
# подключить USB-микрофон
# в фильтре: usb.transfer_type == 0x00
```

В Windows: USBPcap + Wireshark.

### Анализ потока

В Wireshark видны:
- Регулярные пакеты (каждый ~1 мс).
- Фиксированный размер.
- Endpoint 0x81 (IN).
- Frame numbers последовательны.

При проблемах:
- Status != 0 — ошибка.
- Длина 0 — потерянный пакет.
- Большие промежутки между пакетами — задержки.

### Структура аудио-устройства

USB Audio Class 1.0:
- Sample rate указан в дескрипторе.
- Формат: PCM 16-bit, 24-bit, или другие.
- Каналы: моно, стерео, многоканальный.

При isoc-передаче кадр USB (1 мс) содержит данные точно на 1 мс аудио.

## 96. Пример данных получаемых от мыши в whireshark.
*Источник: LDD3; HID-протокол*

### USB-мышь

USB-мышь — это **HID-устройство** (Human Interface Device). Использует **interrupt endpoint** (не bulk, не isochronous).

### Характеристики Interrupt-передачи

- Периодический опрос (обычно 8 мс или быстрее для gaming-мышей).
- Гарантия latency.
- Малое количество данных.
- Гарантия доставки.

### Структура отчёта (HID report)

Стандартный HID-репорт мыши — 4 байта:

```
Байт 0: кнопки
        бит 0 - левая
        бит 1 - правая
        бит 2 - средняя
        биты 3-7 - дополнительные кнопки
Байт 1: dX (смещение по X, signed)
Байт 2: dY (смещение по Y, signed)
Байт 3: dWheel (прокрутка колеса, signed)
```

### Wireshark — захват

```bash
sudo modprobe usbmon
sudo wireshark
# выбрать usbmonX
# фильтр: usb.transfer_type == 0x01 (interrupt)
```

### Пример пакета (мышь не двигается, кнопки не нажаты)

```
URB type: COMPLETE
URB transfer type: INTERRUPT (0x01)
Endpoint: 0x81 (IN, EP 1)
Device: 5
URB packet length: 4
Data: 00 00 00 00
```

### Пример пакета (нажата левая кнопка)

```
Data: 01 00 00 00
       │
       └ бит 0 = 1 (левая кнопка)
```

### Пример пакета (движение вправо-вверх)

```
Data: 00 0A FB 00
       │  │  │  │
       │  │  │  └ колесо не крутится
       │  │  └ dY = 0xFB = -5 (вверх)
       │  └ dX = 0x0A = +10 (вправо)
       └ кнопки не нажаты
```

(Координаты Y инвертированы — увеличение Y вниз.)

### Пример: прокрутка колеса вниз

```
Data: 00 00 00 FF
                │
                └ dWheel = -1 (вниз)
```

### Что Wireshark показывает

```
HID Data:
  Buttons: 0x00
  X-axis: 0
  Y-axis: 0
  Wheel: 0
```

Wireshark часто **парсит** HID-репорт и показывает поля в человекочитаемом виде.

### Опрос мыши

Мышь опрашивается каждые N мс (N = bInterval из endpoint descriptor):
- USB 1.x: N в мс.
- USB 2.0 HS: N в микрокадрах (125 мкс).

Стандартная мышь — 8-10 мс. Gaming-мыши — 1 мс (1000 Hz polling).

### При отсутствии активности

Если ничего не меняется, мышь шлёт «пустые» отчёты (нули) каждый интервал. Хост получает их, но не реагирует.

При движении — данные отличны от нуля. Драйвер передаёт пользовательскому приложению.

### Boot Protocol vs Report Protocol

- **Boot protocol** — простой 3-байтный или 4-байтный отчёт (как описано выше). Поддерживается BIOS.
- **Report protocol** — может быть сложнее, с дополнительными полями (программируемые кнопки, наклон колеса).

При запуске мышь обычно в Boot protocol. ОС может переключить в Report.

## 97. Интерфейс PCI. Адресация. Регистры конфигурации. DeviceID и VendorID. PCI_DEVICE
*Источник: LDD3; tanenbaum.md*

### PCI — обзор

**PCI (Peripheral Component Interconnect)** — параллельная шина для подключения периферии.

Из tanenbaum.md (1.3.5):
> «Наиболее старой из них является PCI (Peripheral Component Interconnect — интерфейс периферийных компонентов)».

PCIe — современный преемник, последовательный.

### Адресация PCI

Каждое устройство адресуется тройкой:
- **Bus** (0..255) — номер шины.
- **Device** (0..31) — номер устройства на шине.
- **Function** (0..7) — номер функции (одно устройство может иметь несколько функций).

Итого: 256 × 32 × 8 = 65536 функций (на одну PCI-иерархию).

В Linux обозначение: `BB:DD.F` (например, `00:1f.2`).

### Просмотр устройств

```bash
$ lspci
00:00.0 Host bridge: Intel ...
00:02.0 VGA compatible controller: Intel ...
00:14.0 USB controller: Intel ...
00:1f.2 SATA controller: Intel ...
03:00.0 Network controller: Intel ...
```

### Конфигурационное пространство

Каждая PCI-функция имеет **256 байт** (PCI) или **4096 байт** (PCIe) конфигурационных регистров.

Стандартный header (первые 64 байта):
```
Offset  Field
0x00    Vendor ID    (2 байта)
0x02    Device ID    (2 байта)
0x04    Command      (2 байта)
0x06    Status       (2 байта)
0x08    Revision ID  (1 байт)
0x09    Class code   (3 байта)
0x0C    Cache Line Size, Latency, Header Type, BIST
0x10    BAR0         (4 байта) — Base Address Register
0x14    BAR1
0x18    BAR2
0x1C    BAR3
0x20    BAR4
0x24    BAR5
0x2C    Subsystem Vendor ID
0x2E    Subsystem Device ID
0x3C    Interrupt Line, Pin
```

### Vendor ID и Device ID

**Vendor ID** — идентификатор производителя. Выдаётся PCI-SIG (https://pcisig.com).
Примеры:
- 0x8086 — Intel.
- 0x10DE — NVIDIA.
- 0x1022 — AMD.

**Device ID** — конкретная модель устройства. Назначает сам производитель.

Пара (VID, DID) однозначно идентифицирует тип устройства.

### Просмотр VID/DID

```bash
$ lspci -nn
00:1f.2 SATA controller [0106]: Intel Corporation [8086:8c02]
                                                  ^^^^^^^^^^^
                                                  VID:DID
```

### BAR — Base Address Register

BAR содержат адреса, по которым устройство отображено в адресное пространство:
- I/O ports.
- Memory.

Содержимое BAR ОС изменяет при загрузке (PCI configuration).

### Регистры в Linux

API для чтения конфигурационного пространства:
```c
int pci_read_config_byte(struct pci_dev *dev, int where, u8 *val);
int pci_read_config_word(struct pci_dev *dev, int where, u16 *val);
int pci_read_config_dword(struct pci_dev *dev, int where, u32 *val);
int pci_write_config_byte(...);
// и т.д.
```

### Идентификация устройств — PCI_DEVICE

Макрос для таблицы поддерживаемых устройств:

```c
#define PCI_DEVICE(vend, dev) \
    .vendor = (vend), .device = (dev), \
    .subvendor = PCI_ANY_ID, .subdevice = PCI_ANY_ID
```

Использование:
```c
static struct pci_device_id my_pci_ids[] = {
    { PCI_DEVICE(0x8086, 0x1234) },
    { PCI_DEVICE(0x10DE, 0x5678) },
    { 0, }    // окончание
};
MODULE_DEVICE_TABLE(pci, my_pci_ids);
```

`MODULE_DEVICE_TABLE` создаёт информацию для **hotplug** — автоматической загрузки драйвера при подключении устройства.

### Class code

3-байтное поле, классификация устройства:
- 0x01: Mass Storage.
- 0x02: Network.
- 0x03: Display.
- 0x04: Multimedia.
- 0x0C: Serial bus (USB controllers).

Внутри класса — подкласс и интерфейс.

### Применение

VID/DID — основа выбора драйвера:
- Драйвер регистрирует таблицу поддерживаемых VID/DID.
- При обнаружении устройства ядро ищет драйвер с подходящим VID/DID.
- Вызывает probe этого драйвера.

## 98. Интерфейс PCI. Регистрация PCI драйвера. *probe, *remove, int (*suspend) (struct pci_dev *dev, u32 state);
*Источник: LDD3*

### struct pci_driver

```c
struct pci_driver {
    const char *name;
    const struct pci_device_id *id_table;
    int (*probe)   (struct pci_dev *dev, const struct pci_device_id *id);
    void (*remove) (struct pci_dev *dev);
    int (*suspend) (struct pci_dev *dev, pm_message_t state);
    int (*resume)  (struct pci_dev *dev);
    void (*shutdown) (struct pci_dev *dev);
    /* ... */
};
```

### Регистрация

```c
static struct pci_driver my_driver = {
    .name     = "my_pci_driver",
    .id_table = my_pci_ids,
    .probe    = my_probe,
    .remove   = my_remove,
    .suspend  = my_suspend,
    .resume   = my_resume,
};

static int __init my_init(void) {
    return pci_register_driver(&my_driver);
}

static void __exit my_exit(void) {
    pci_unregister_driver(&my_driver);
}

module_init(my_init);
module_exit(my_exit);
```

Или коротко:
```c
module_pci_driver(my_driver);
```

### probe

```c
int (*probe)(struct pci_dev *dev, const struct pci_device_id *id);
```

Вызывается при **обнаружении** устройства, которое поддерживает драйвер.

Задачи:
1. Включить устройство (`pci_enable_device`).
2. Запросить регионы BAR (`pci_request_regions`).
3. Отобразить регистры (`pci_iomap`, `ioremap`).
4. Получить IRQ (`request_irq`).
5. Зарегистрировать связанные интерфейсы (символьное устройство, сетевой интерфейс).
6. Сохранить указатель в `pci_set_drvdata`.

Возвращает 0 при успехе.

Пример:
```c
static int my_probe(struct pci_dev *pdev, const struct pci_device_id *id)
{
    int err;

    err = pci_enable_device(pdev);
    if (err)
        return err;

    err = pci_request_regions(pdev, "my_driver");
    if (err)
        goto out_disable;

    void __iomem *regs = pci_iomap(pdev, 0, 0);   // BAR0
    if (!regs) {
        err = -ENOMEM;
        goto out_release;
    }

    pci_set_drvdata(pdev, regs);

    err = request_irq(pdev->irq, my_irq_handler, IRQF_SHARED, "my_driver", pdev);
    if (err)
        goto out_unmap;

    return 0;

out_unmap:
    pci_iounmap(pdev, regs);
out_release:
    pci_release_regions(pdev);
out_disable:
    pci_disable_device(pdev);
    return err;
}
```

### remove

```c
void (*remove)(struct pci_dev *dev);
```

Вызывается при **отключении** устройства или **выгрузке** драйвера.

Зеркальная probe — освободить всё:

```c
static void my_remove(struct pci_dev *pdev)
{
    void __iomem *regs = pci_get_drvdata(pdev);

    free_irq(pdev->irq, pdev);
    pci_iounmap(pdev, regs);
    pci_release_regions(pdev);
    pci_disable_device(pdev);
}
```

### suspend

```c
int (*suspend)(struct pci_dev *dev, pm_message_t state);
```

Вызывается при **переходе системы в спящий режим**. Задачи:
- Сохранить состояние устройства.
- Перевести устройство в low-power.
- Прекратить ввод-вывод.

```c
static int my_suspend(struct pci_dev *pdev, pm_message_t state)
{
    pci_save_state(pdev);                     // сохранить конфигурацию
    pci_set_power_state(pdev, PCI_D3hot);     // в D3 (low power)
    return 0;
}
```

### resume

```c
int (*resume)(struct pci_dev *dev);
```

Вызывается при **выходе из сна**:
- Восстановить состояние устройства.
- Включить устройство.

```c
static int my_resume(struct pci_dev *pdev)
{
    pci_set_power_state(pdev, PCI_D0);    // включить
    pci_restore_state(pdev);              // восстановить
    pci_enable_device(pdev);
    return 0;
}
```

### Состояния питания

| State | Назначение |
|---|---|
| D0 | Полностью включено |
| D1 | Лёгкий сон |
| D2 | Глубже |
| D3hot | Питание есть, устройство выключено |
| D3cold | Без питания |

### shutdown

Вызывается при **выключении системы**. Гарантирует, что устройство не повредит данные.

### Hotplug

Через `MODULE_DEVICE_TABLE(pci, my_pci_ids)` ядро знает, какие устройства драйвер поддерживает. udev/systemd автоматически загружают модуль при обнаружении нового устройства.

### Тип pm_message_t

```c
typedef struct pm_message {
    int event;
} pm_message_t;
```

События:
- `PM_EVENT_SUSPEND` — обычный suspend.
- `PM_EVENT_HIBERNATE` — гибернация.
- `PM_EVENT_FREEZE` — заморозка для гибернации.

## 99. Файловые системы. Непрерывное размещение файлов. Достоинства и недостатки. Пример. Применение. Атрибуты файла при непрерывном размещении файлов.
*Источник: tanenbaum.md, Глава 4 «Файловые системы»*

### Непрерывное размещение

Из tanenbaum.md (4.3.2):
> «Простейшая схема размещения заключается в хранении каждого файла на диске в виде непрерывной последовательности блоков. Таким образом, на диске с блоками, имеющими размер 1 Кбайт, файл размером 50 Кбайт займет 50 последовательных блоков».

Каждый файл — **непрерывная** последовательность блоков на диске.

### Атрибуты файла

Минимальные для непрерывного размещения:
- **Адрес первого блока.**
- **Количество блоков (или размер).**
- + обычные: имя, размер, время, права.

Зная начало и длину — можно вычислить адрес любого блока: `block_n = start + n`.

### Пример

Диск с блоками по 1 КБ:
- Файл A: 4 блока, начало 0. Блоки: 0, 1, 2, 3.
- Файл B: 6 блоков, начало 4. Блоки: 4, 5, 6, 7, 8, 9.
- Файл C: 3 блока, начало 10. Блоки: 10, 11, 12.

### Достоинства

**1. Простота реализации.**

Из tanenbaum.md:
> «Его просто реализовать, поскольку отслеживание местонахождения принадлежащих файлу блоков сводится всего лишь к запоминанию двух чисел: дискового адреса первого блока и количества блоков в файле».

**2. Превосходная производительность чтения.**

> «Весь файл может быть считан с диска за одну операцию. Для нее потребуется только одна операция позиционирования (на первый блок). После этого никаких позиционирований или ожиданий подхода нужного сектора диска уже не потребуется».

Идеально для последовательного чтения больших файлов.

**3. Быстрый произвольный доступ.**

Чтобы найти n-й байт — `start + n / block_size`. O(1) вычисление, без чтения каких-то служебных структур.

### Недостатки

**1. Фрагментация.**

Из tanenbaum.md:
> «Со временем диск становится фрагментированным».

При удалении файлов появляются «дыры». При создании нового файла нужно искать достаточно большую непрерывную область.

**2. Нужно знать размер заранее.**

> «Но при создании нового файла необходимо знать его окончательный размер, чтобы выбрать подходящий для размещения участок».

Если файл должен расти — придётся либо резервировать с запасом (теряем место), либо перемещать файл при росте (медленно).

Это **смертельный** недостаток для типичных файлов общего назначения.

**3. Уплотнение медленное.**

Для устранения фрагментации нужно перемещать миллионы блоков — часы или дни.

### Где применяется

Из tanenbaum.md:
> «Есть одна сфера применения, в которой непрерывное размещение вполне приемлемо и все еще используется на практике, — это компакт-диски. Здесь все размеры файлов известны заранее и никогда не изменяются».

Применение:
- **CD-ROM, DVD-ROM, Blu-ray.** Файлы записываются один раз, не меняются.
- **UDF** на DVD. Большие фильмы могут разбиваться на куски (экстенты).
- **Tape (магнитные ленты)** — последовательная запись.
- **Архивы** (CPIO, TAR).

### Возвращение идеи

> «Непрерывное размещение благодаря своей простоте и высокой производительности использовалось в файловых системах магнитных дисков много лет назад… Затем из-за необходимости задания конечного размера файла при его создании эта идея была отброшена. Но неожиданно с появлением компакт-дисков, DVD, Blu-ray-дисков и других однократно записываемых оптических носителей непрерывные файлы снова оказались весьма кстати».

### Экстенты — компромисс

Современные ФС (ext4, NTFS, XFS) используют **экстенты** — описание непрерывных участков:

```
Экстент: (start_block, length)
Файл = последовательность экстентов
```

Это компромисс:
- В пределах одного экстента — преимущества непрерывности.
- Между экстентами — гибкость.
- Меньше метаданных, чем у списка блоков.

### Резюме

- Непрерывное размещение: блоки файла подряд.
- Атрибуты: адрес первого блока + длина.
- Простота, отличная производительность чтения.
- Недостатки: фрагментация, нужен заранее известный размер.
- Применение: CD/DVD/Blu-ray, ленты, архивы.
- Современные ФС используют экстенты как компромисс.

## 100. Файловые системы. Размещение файлов с использованием связанного списка. Достоинства и недостатки. Пример. FAT. Достоинства и недостатки. Пример.
*Источник: tanenbaum.md, Глава 4 «Файловые системы»*

### Связанный список

Из tanenbaum.md (4.3.2):
> «Второй метод хранения файлов заключается в представлении каждого файла в виде связанного списка дисковых блоков. Первое слово каждого блока используется в качестве указателя на следующий блок, а вся остальная часть блока предназначается для хранения данных».

Структура блока:
```
+----------+----------------------+
| next ptr |     данные           |
+----------+----------------------+
```

В каталоге хранится только **адрес первого блока**. Остальное — по цепочке.

### Достоинства связанного списка

**1. Нет фрагментации.**

> «В этом методе может быть использован каждый дисковый блок. При этом потери дискового пространства на фрагментацию отсутствуют».

Любой свободный блок можно использовать — не нужна непрерывная область.

**2. Простой каталог.**

Только адрес первого блока. Размер файла — можно вычислить по цепочке.

**3. Динамический рост.**

Добавление блока в конец — просто добавить указатель.

### Недостатки

**1. Медленный произвольный доступ.**

> «Чтобы добраться до блока n, операционной системе нужно начать со стартовой позиции и прочитать поочередно n − 1 предшествующих блоков. Понятно, что осуществление стольких операций чтения окажется мучительно медленным».

O(n) для произвольного доступа.

**2. Размер блока не кратен степени 2.**

> «Объем хранилища данных в блоках уже не кратен степени числа 2, поскольку несколько байтов отнимает указатель… Многие программы ведут чтение и запись блоками, размер которых кратен степени числа 2».

Если блок 1024 байт и указатель 4 байта — данных только 1020 байт. Неудобно.

**3. Уязвимость к ошибкам.**

Если потерян один указатель — потерян весь хвост файла.

### FAT — улучшение

Из tanenbaum.md:
> «Оба недостатка размещения с помощью связанных списков могут быть устранены за счет изъятия слова указателя из каждого дискового блока и помещения его в таблицу в памяти. Такая таблица, находящаяся в оперативной памяти, называется FAT (File Allocation Table — таблица размещения файлов)».

**Идея:** все указатели — в **отдельной таблице** в памяти. Блоки целиком используются под данные.

### Структура FAT

Таблица — массив записей, по одной на каждый блок диска:
```
Block 0: FAT[0] — указатель на следующий блок (или EOF, или FREE)
Block 1: FAT[1] — указатель
...
Block N: FAT[N] — указатель
```

Значения:
- **Конкретное число:** номер следующего блока.
- **0:** свободен.
- **-1 (или 0xFFFF):** конец файла (EOF).
- **-2:** плохой блок.

### Пример

Файл A: блоки 4, 7, 2, 10, 12.
В каталоге: A → block 4.
В FAT:
```
FAT[4]  = 7
FAT[7]  = 2
FAT[2]  = 10
FAT[10] = 12
FAT[12] = EOF
```

Чтобы прочитать файл — идём по цепочке в **FAT** (в памяти, быстро), читаем блоки.

### Достоинства FAT

**1. Произвольный доступ — без обращения к диску.**

Цепочка в памяти → пройти по ней быстро. Найти n-й блок — O(n), но **без I/O**.

**2. Блок целиком — данные.**

Размер блока кратен степени 2.

**3. Нет фрагментации.**

Как и обычный связанный список.

**4. Простота.**

### Недостатки FAT

**1. Таблица должна быть в памяти.**

Из tanenbaum.md:
> «Основным недостатком этого метода является то, что для его работы вся таблица должна постоянно находиться в памяти».

**2. Размер таблицы пропорционален размеру диска.**

> «Для 1-терабайтного диска, имеющего блоки размером 1 Кбайт, потребовалась бы таблица из 1 млрд записей, по одной для каждого из 1 млрд дисковых блоков. Каждая запись должна состоять как минимум из 3 байт. Для ускорения поиска размер записей должен быть увеличен до 4 байт. Таким образом, таблица будет постоянно занимать 3 Гбайт или 2,4 Гбайт оперативной памяти».

Плохо масштабируется на большие диски.

**3. Произвольный доступ всё ещё O(n).**

Хотя без I/O.

### Версии FAT

- **FAT12** — 12-бит на запись. Старая, для дискет.
- **FAT16** — 16 бит. MS-DOS, ранние Windows. Максимум 64K блоков.
- **FAT32** — 32 бита (фактически 28). Современные Windows, флешки, SD-карты.
- **exFAT** — расширение для флеш-носителей и больших размеров.

### Применение

- **MS-DOS** — оригинальная FAT.
- **Windows 9x, NT/XP** — поддержка FAT.
- **Современные:** USB-флешки, SD-карты, карты памяти камер.
- **EFI System Partition** — UEFI BIOS требует FAT.

Причина широкой поддержки: **простота**, **переносимость**, **отсутствие лицензионных платежей** (открытая спецификация).

### Сравнение

| Параметр | Связ. список | FAT | i-узлы |
|---|---|---|---|
| Фрагментация | Нет | Нет | Нет |
| Произвольный доступ | O(n) с I/O | O(n) без I/O | O(1)/O(log n) |
| Размер служебной информации | По указателю в каждом блоке | Таблица пропорциональна диску | По i-узлу для каждого открытого файла |
| Подходит для больших дисков | Плохо | Плохо | Отлично |
| Применение | Историческое | FAT-носители | Unix, Linux |

### Резюме

- Связанный список: указатель в каждом блоке. Простота, но медленный random access и нестандартный размер блока.
- FAT: указатели в отдельной таблице в памяти. Быстрее, но нужно много RAM для больших дисков.
- FAT применяется на флеш-носителях и в EFI.
- i-узлы (вопрос 101) — лучшее решение для больших дисков.
