## 101. Файловые системы. i-узлы. Достоинства и недостатки. Пример.
*Источник: tanenbaum.md, Глава 4.3.2; Глава 4.5.2 «Файловая система UNIX V7»*

### Понятие i-узла

Из tanenbaum.md (4.3.2):
> «Последним из рассматриваемых методов отслеживания принадлежности конкретного блока конкретному файлу является связь с каждым файлом структуры данных, называемой i-узлом (index-node — индекс-узел), содержащей атрибуты файла и дисковые адреса его блоков».

**i-узел** = индексный узел = структура с **атрибутами файла** + **массив адресов** его блоков.

### Структура i-узла UNIX V7

Из tanenbaum.md (рис. 4.10, 4.28):

```c
struct inode_v7 {
    // Атрибуты
    mode_t  mode;          // тип и права доступа
    uid_t   uid;           // владелец
    gid_t   gid;           // группа
    off_t   size;          // размер в байтах
    time_t  atime, mtime, ctime;
    nlink_t nlink;         // количество связей

    // Адреса блоков (13 элементов)
    block_t addr[10];      // прямые адреса первых 10 блоков
    block_t single_indirect; // адрес однократного косвенного блока
    block_t double_indirect; // двукратного косвенного
    block_t triple_indirect; // трёхкратного косвенного
};
```

### Косвенные блоки

При размере блока 1 КБ и адресе блока 4 байта:
- **Прямые 10 блоков:** до 10 КБ напрямую.
- **Однократный косвенный:** один блок содержит 256 адресов → +256 КБ.
- **Двукратный косвенный:** блок с 256 адресами на блоки по 256 адресов → +64 МБ.
- **Трёхкратный косвенный:** 256³ блоков → до 16 ГБ.

Маленькие файлы — быстрый доступ через прямые адреса. Большие — через косвенные.

### Главное преимущество перед FAT

Из tanenbaum.md (4.3.2):
> «Большим преимуществом этой схемы перед связанными списками, использующими таблицу в памяти, является то, что i-узел должен быть в памяти только в том случае, когда открыт соответствующий файл».

Расход RAM пропорционален **числу открытых файлов**, а не размеру диска. Если открыто 1000 файлов и i-узел занимает 128 байт → 128 КБ. На любом диске.

В FAT — пропорционально размеру диска (на 1 ТБ диск ~ 4 ГБ FAT).

### Достоинства

**1. Эффективное использование памяти.**

i-узлы открытых файлов кэшируются, остальные хранятся на диске.

**2. Быстрый доступ к маленьким файлам.**

Большинство файлов помещаются в 10 прямых блоков (40 КБ для блока 4 КБ).

**3. Поддержка очень больших файлов.**

Через косвенные блоки.

**4. Поддержка жёстких ссылок.**

Несколько имён → один i-узел. Счётчик `nlink` отслеживает.

**5. Атрибуты в одном месте.**

`stat()` читает один i-узел, не каталог.

### Недостатки

**1. Косвенные блоки = дополнительные обращения к диску.**

Для блока 100 КБ нужно прочитать косвенный блок + сам блок. 2 операции вместо 1.

**2. Сложнее реализация** по сравнению с FAT.

**3. Размер файла ограничен** размером трёхкратного косвенного.

### Пример: открытие /usr/ast/mbox (UNIX V7)

Из tanenbaum.md (рис. 4.29):

1. **i-узел корневого каталога** — в фиксированном месте.
2. По нему → корневой каталог.
3. Поиск имени `usr` → номер i-узла /usr.
4. По номеру → i-узел /usr → блоки каталога.
5. Поиск `ast` → i-узел /usr/ast.
6. По нему → каталог /usr/ast.
7. Поиск `mbox` → i-узел файла mbox.
8. i-узел /usr/ast/mbox считывается в память и хранится, пока файл открыт.

### Применение

i-узлы — основа всех современных Unix/Linux ФС:
- **ext2/3/4**
- **XFS**
- **JFS**
- **ZFS** (модифицированная идея)
- **Btrfs** (с экстентами)
- **NTFS** (похожая концепция — MFT records).

### Современные расширения

Современные ФС (ext4, XFS) дополнили i-узлы **экстентами**:
- Вместо массива адресов отдельных блоков — массив экстентов.
- Экстент = (start_block, length).
- Меньше метаданных для непрерывных файлов.

## 102. Файловые системы. Реализация каталогов в FAT и i-узлах. Атрибуты файлов при FAT и i-узлах.
*Источник: tanenbaum.md, Глава 4.3.3 «Реализация каталогов»; 4.5.1, 4.5.2*

### Запись каталога — что хранить

Из tanenbaum.md (4.3.3):
> «Основной функцией системы каталогов является преобразование ASCII-имени файла в информацию, необходимую для определения местоположения данных».

Минимум:
- **Имя файла.**
- **Способ найти содержимое** (зависит от схемы размещения).

### Каталоги в FAT (MS-DOS)

В FAT атрибуты хранятся **прямо в записи каталога** (рис. 4.11, а):

```
Запись каталога MS-DOS (32 байта):
┌─────────────────────────────────────┐
│ Имя (8 байт) + расширение (3 байта) │   = 11 байт
├─────────────────────────────────────┤
│ Атрибуты (1 байт)                    │   read-only, hidden, system, ...
├─────────────────────────────────────┤
│ Зарезервированно (10 байт)          │
├─────────────────────────────────────┤
│ Время изменения (2 байта)            │
├─────────────────────────────────────┤
│ Дата изменения (2 байта)             │
├─────────────────────────────────────┤
│ Номер первого кластера (2 байта)    │
├─────────────────────────────────────┤
│ Размер файла (4 байта)               │
└─────────────────────────────────────┘
```

Имя в формате **8.3** — 8 символов + 3 расширение. Без точки. Только верхний регистр.

### Атрибуты файла в FAT

1 байт битовых флагов:
- **R** (Read-only).
- **H** (Hidden).
- **S** (System).
- **V** (Volume label).
- **D** (Directory).
- **A** (Archive — изменён).

### Длинные имена в FAT (VFAT, LFN)

Microsoft расширила FAT для имён до 255 символов. Длинные имена хранятся в **специальных записях каталога**, выглядящих как «странные файлы» для старых ОС. Связаны со стандартной 8.3-записью.

### Каталоги с i-узлами (UNIX)

В UNIX каталог содержит только **имя файла** и **номер i-узла** (рис. 4.11, б):

```
Запись каталога UNIX V7 (16 байт):
┌─────────────────┐
│ Номер i-узла    │   2 байта
├─────────────────┤
│ Имя (14 байт)   │   14 байт
└─────────────────┘
```

Атрибуты — **в i-узле**, не в каталоге.

### Современный Unix (ext2/3/4)

Запись каталога переменной длины:
```c
struct ext2_dir_entry {
    __u32 inode;        // номер i-узла
    __u16 rec_len;      // длина записи
    __u8  name_len;     // длина имени
    __u8  file_type;    // тип
    char  name[];       // имя переменной длины
};
```

### Сравнение

| Параметр | FAT | i-узлы (UNIX) |
|---|---|---|
| Атрибуты | В каталоге | В i-узле |
| Размер записи | Фиксированный (32 байта) | Переменный |
| Имя | 8.3 (или LFN) | Произвольное (до 255) |
| Жёсткие ссылки | Невозможны | Возможны |
| Найти атрибуты | Прямо из каталога | Прочитать i-узел |
| Хранение каталога | Похож на FAT-файл | Обычный файл с i-узлом |

### Преимущества i-узлов

**1. Жёсткие ссылки.**

Несколько записей каталогов могут указывать на один i-узел. Атрибуты общие. Изменения видны всем «именам».

**2. Эффективное переименование/перемещение.**

При переименовании или перемещении меняется только запись каталога. Содержимое и i-узел не трогаются.

**3. Меньший размер каталога.**

Каталог только имена + номера, а не дублирует атрибуты.

### Преимущества FAT-каталога

**1. Простота.**

Все нужное в одном месте.

**2. Меньше disk seeks** при чтении атрибутов.

Без открытия файла — `dir` уже показывает атрибуты. В UNIX — нужно прочитать i-узел.

### Реализация жёстких ссылок в UNIX

```c
link("a.txt", "b.txt");   // создать ссылку
```

Что происходит:
1. Найти i-узел `a.txt`.
2. В каталоге создать запись `b.txt` → тот же i-узел.
3. `nlink++` в i-узле.

После удаления `a.txt`:
1. Удалить запись из каталога.
2. `nlink--`.
3. Если nlink > 0 — i-узел остаётся (по `b.txt`).
4. Если nlink == 0 и нет открытых fd — освободить.

В FAT жёсткие ссылки невозможны — атрибуты в каталоге, при удалении одной записи теряются.

## 103. Журналируемые файловые системы. Иденпотентность операций.
*Источник: tanenbaum.md, Глава 4.3.4-4.3.5; общие принципы*

### Проблема: непротиворечивость после сбоя

При записи файла нужны несколько операций:
1. Найти свободный блок.
2. Обновить bitmap свободных блоков.
3. Записать данные в блок.
4. Обновить i-узел (адрес блока, размер).
5. Записать i-узел.

Если в **середине** последовательности отключится питание — ФС в противоречивом состоянии.

Из tanenbaum.md (4.4.3):
> «Если, к примеру, после успешной операции записи новых данных в блок происходит отказ питания… могут образоваться плохие сектора и появиться отсылки на несуществующие блоки».

### Традиционный fsck

Из tanenbaum.md:
> «В системе UNIX есть fsck, а в системе Windows — sfc… Эта утилита может быть запущена при каждой загрузке системы, особенно после сбоя».

fsck проверяет:
- Все ли блоки учтены (в файлах или в свободных).
- Соответствие nlink количеству ссылок из каталогов.
- Корректность структуры каталогов.

Проблема — fsck **долго работает** на больших дисках (часы).

### Журналируемые файловые системы

**Идея:** перед изменением структуры записать намерение в **журнал**. При сбое — повторить из журнала.

Этапы записи:
1. **Записать в журнал:** «собираюсь обновить блок X и i-узел Y».
2. **Записать на диск:** обновить X и Y.
3. **Отметить в журнале:** «выполнено».

При сбое:
- Если в журнале есть незавершённая запись → повторить операции.
- Не нужно сканировать всю ФС.

### Идемпотентность операций

**Идемпотентная операция** — повторение даёт тот же результат, что и однократное выполнение.

Примеры:
- ✅ «Записать в блок X данные D» — идемпотентна.
- ✅ «i-узел Y → размер = 100, блок 0 = X».
- ❌ «Увеличить размер на 100» — НЕ идемпотентна (повторение даст +200).
- ❌ «Добавить блок в конец списка» — НЕ идемпотентна.

**Требование к журналируемой ФС:** все операции должны быть **идемпотентными**.

Тогда при восстановлении можно безопасно **повторить** все операции из журнала.

### Журналирование данных vs метаданных

**Полное журналирование (full journaling):** в журнал пишутся и метаданные, и сами данные.
- Безопасно.
- Медленно (двойная запись).

**Метаданные (metadata journaling):** в журнал только метаданные.
- Быстрее.
- Данные могут быть потеряны, но структура ФС целостна.

**Ordered:** метаданные в журнале, данные записываются **до** записи метаданных в журнал.
- Компромисс.
- ext3/4 по умолчанию.

### Журналируемые ФС

- **ext3, ext4** (Linux).
- **NTFS** (Windows).
- **JFS** (IBM).
- **XFS** (SGI/Linux).
- **HFS+ Journaled** (macOS).
- **ReiserFS.**

### Преимущества

- **Быстрое восстановление** после сбоя — секунды вместо часов.
- **Целостность** метаданных гарантирована.
- **Журнал** компактен (десятки МБ).

### Недостатки

- **Накладные расходы** при записи.
- **Износ** SSD из-за двойной записи в журнал.

### Log-structured FS (LFS)

Радикальный подход: вся ФС — это **один большой журнал**.

Из tanenbaum.md (4.3.5):
> «Идея была в том, чтобы превратить весь диск в один большой журнал».

Все записи → в конец журнала. Существующие данные не модифицируются, новые версии записываются как новые блоки.

Применение:
- **NILFS** (Linux).
- **F2FS** (для SSD).
- ZFS и Btrfs используют похожие идеи (copy-on-write).

### Резюме

- Журналируемые ФС записывают намерения в журнал перед выполнением.
- Гарантируют целостность при сбое.
- Требуют **идемпотентности** операций для безопасного повтора.
- Разные уровни: full / metadata / ordered.

## 104. Ввод-вывод, отображаемый на пространство памяти.
*Источник: tanenbaum.md, Глава 5.1.3*

### Способы доступа к устройствам

Устройства управляются через **регистры** контроллера. Есть два способа доступа:

**1. Раздельное пространство I/O (port-mapped I/O).**

Специальные инструкции IN/OUT (x86):
```asm
in  al, dx     ; прочитать байт из порта DX в AL
out dx, al     ; записать байт из AL в порт DX
```

Адреса I/O — отдельное пространство (0..65535).

**2. Memory-mapped I/O.**

Регистры устройств **отображены на адреса памяти**. Обычные инструкции LD/ST.

Из tanenbaum.md (5.1.3):
> «На некоторых компьютерах регистры устройств отображаются в адресное пространство памяти».

### Memory-mapped I/O — подробно

Каждый регистр устройства имеет физический адрес. Обращение к этому адресу — прямое обращение к регистру.

```c
#define UART_DATA   0xF1000000
#define UART_STATUS 0xF1000004

volatile uint32_t *data   = (uint32_t *)UART_DATA;
volatile uint32_t *status = (uint32_t *)UART_STATUS;

// Запись байта в UART
*data = 0x41;   // 'A'

// Чтение статуса
while ((*status & 0x01) == 0)
    ;
```

### Преимущества memory-mapped I/O

**1. Не нужны специальные инструкции.** Обычные LD/ST.

**2. Поддержка C/C++.** Указатели работают как с памятью.

**3. Гибкость.** MMU позволяет defer доступ к устройствам.

**4. Защита.** Через таблицы страниц — пользователю недоступно.

**5. Кеширование можно отключить** для региона.

### Недостатки

**1. Адресное пространство расходуется** на устройства.

На 32-битной системе с 4 ГБ ОЗУ и устройствами на 1 ГБ — реально доступно только 3 ГБ ОЗУ.

**2. Кеширование может вызвать проблемы.** Запись в регистр устройства может «попасть в кэш» и не дойти до устройства.

### Применение в современных системах

Все современные процессоры (ARM, RISC-V, новые x86) предпочитают memory-mapped I/O.

На x86 раздельные I/O порты остаются для совместимости (для PIC, PIT, CMOS).

### В Linux: ioremap

Чтобы работать с memory-mapped регистрами в драйвере:

```c
void __iomem *regs;
regs = ioremap(physical_addr, size);

// Использовать через readl/writel (помнят про барьеры)
u32 val = readl(regs + 0x10);
writel(0x1234, regs + 0x10);

iounmap(regs);
```

`__iomem` — аннотация для sparse: «это I/O память, не RAM».

### Барьеры памяти

Для memory-mapped I/O **обязательны барьеры памяти** при записи нескольких регистров в определённом порядке:

```c
writel(setup_value, regs + REG_SETUP);
wmb();   // запись завершена
writel(GO, regs + REG_CONTROL);
```

Без барьера CPU может изменить порядок записи → устройство получит команду до настройки.

### Резюме

- Memory-mapped I/O — регистры устройств в адресном пространстве памяти.
- Обращение через обычные LD/ST или `readl/writel`.
- Преимущества: простота, гибкость, поддержка C.
- Минусы: расходует адресное пространство, нужны барьеры.
- В Linux: `ioremap` для отображения.

## 105. Программный ввод-вывод. Опрос устройства.
*Источник: tanenbaum.md, Глава 5.2.2*

### Что такое программный I/O (PIO)

Из tanenbaum.md (5.2.2):
> «Простейшая форма ввода-вывода — programmed I/O — заключается в том, что центральный процессор делает всю работу».

CPU **постоянно опрашивает** устройство, ждёт готовности, передаёт данные.

### Пример: вывод строки на принтер

```c
void print_string(char *str, int len)
{
    for (int i = 0; i < len; i++) {
        // Ждать готовности (busy wait)
        while ((*printer_status & READY) == 0)
            ;
        // Передать байт
        *printer_data = str[i];
    }
}
```

### Опрос (polling)

**Опрос:** периодически проверять регистр статуса устройства.

```c
while (!device_ready()) {
    // ничего не делаем (или другая работа)
}
read_data();
```

### Достоинства программного I/O

- **Простота.** Минимум кода.
- **Низкая латентность.** Реакция мгновенная.
- **Не требует прерываний.**

### Недостатки

- **CPU занят 100%.** Не делает полезной работы пока ждёт.
- **Очень неэффективно** при медленных устройствах.

### Когда применяется

- **Очень простые системы** (микроконтроллеры без ОС).
- **Быстрые устройства**, где ожидание короткое.
- **Системы реального времени**, где предсказуемость важнее экономии CPU.
- **Внутри обработчиков прерываний** (короткое ожидание).

### Аналогия

CPU = ребёнок, ждущий маму с работы:
- Каждые 30 секунд: «Мама пришла? Мама пришла? Мама пришла?»
- Не делает домашнее задание.
- В конце концов мама придёт, но день потерян.

### Сравнение с прерываниями

Из tanenbaum.md (5.2.3):
> «Прерывания обладают небольшим временем задержки… В то же время при использовании современных центральных процессоров прерывания приводят к существенным издержкам в силу необходимости переключения контекста и их влияния на конвейерную обработку, буфер TLB и кэш».

При коротких ожиданиях polling быстрее, при длинных — прерывания.

## 106. Ввод-вывод, управляемый прерываниями. Точные и неточные прерывания. Достоинства и недостатки точных и неточных прерываний. Синхронизация устройств.
*Источник: tanenbaum.md, Глава 5.2.3, 5.1.4*

### I/O с прерываниями

Из tanenbaum.md (5.2.3):
> «Чтобы позволить центральному процессору заниматься другой работой во время ожидания готовности печати… можно использовать прерывания».

**Идея:**
1. CPU инициирует операцию.
2. CPU занимается **другой работой**.
3. Устройство, закончив, шлёт **прерывание**.
4. Обработчик прерывания передаёт следующую порцию или завершает операцию.

### Пример

```c
// Инициация
*printer_data = str[0];
strpos = 1;
enable_interrupt(PRINTER_IRQ);

// CPU свободен для других задач

void printer_isr() {
    if (strpos < len) {
        *printer_data = str[strpos++];
    } else {
        signal_completion();
    }
    acknowledge_interrupt();
}
```

### Преимущества

- **CPU занят полезной работой.**
- **Эффективное использование** ресурсов.
- **Хорошая отзывчивость** на события.

### Недостатки

- **Накладные расходы** на каждое прерывание (контекст, TLB, кэш).
- **Сложнее** код.
- **Возможны гонки** (race conditions).

### Точные прерывания (precise interrupts)

Из tanenbaum.md (5.1.4):

**Точное прерывание** — состояние процессора при прерывании **точно определено**:
- PC указывает на следующую инструкцию.
- Все инструкции **перед** PC выполнены полностью.
- Никакие инструкции **после** PC не начаты.

Это важно для:
- **Page fault.** Обработчик должен выполнить инструкцию заново.
- **Отладка.** Можно знать, где именно произошло прерывание.
- **Виртуальная память.**

### Достоинства точных прерываний

- **Простота программирования** обработчиков.
- **Корректность** виртуальной памяти.
- **Возможность отладки.**

### Недостатки

- **Сложность реализации** в процессорах с конвейером.
- **Возможное снижение производительности** — нужно дождаться завершения «в полёте».

### Неточные прерывания (imprecise interrupts)

В **out-of-order** процессорах с конвейером инструкции могут быть в разной стадии:
- Некоторые после PC уже выполнены.
- Некоторые до PC ещё в конвейере.

При прерывании состояние **«некрасивое»** — приходится сохранять полный snapshot конвейера.

### Достоинства

- **Высокая производительность** процессора.
- **Меньшая сложность** обработки самого прерывания.

### Недостатки

- **Сложно отлаживать.**
- **Нельзя использовать для page fault** (нужно перезапускать).
- **Сложная реализация ОС** — нужно восстанавливать состояние.

### Современные процессоры

x86 и ARM поддерживают точные прерывания **для архитектурно важных событий** (page fault, syscall). Для машинно-зависимых исключений (machine check) могут быть неточные.

### Синхронизация устройств

Из tanenbaum.md: после прерывания нужно **синхронизировать** устройство и драйвер:
- Acknowledge прерывания.
- Подготовка к следующей операции.
- Уведомление пользовательского процесса (например, через `wake_up`).

### Гонки

При работе с прерываниями возможны гонки:
- Драйвер читает регистр статуса.
- Между чтением и обработкой статус меняется.

Решение — критические секции с запретом прерываний или спинлоки.

## 107. Устройства ввода-вывода. Ввод-вывод с использованием DMA.
*Источник: tanenbaum.md, Глава 5.1.4, 5.2.4*

### Проблема прерываний для быстрых устройств

Сетевая карта 10 Гбит/с при пакетах 1500 байт — миллион пакетов в секунду. Один миллион прерываний → CPU занят только их обработкой.

### DMA — Direct Memory Access

Из tanenbaum.md (5.1.4):
> «Многие компьютеры используют для оптимизации работы возможности прямого доступа к памяти (Direct Memory Access (DMA)), который может управлять потоком битов между памятью и некоторыми контроллерами без постоянного вмешательства центрального процессора».

**DMA-контроллер** — отдельное устройство, которое передаёт данные между устройством и памятью **без участия CPU**.

### Принцип работы

Из tanenbaum.md (5.2.4):

1. CPU **настраивает** DMA: «передать N байт от устройства X в память Y».
2. CPU занимается **другими задачами**.
3. DMA пересылает данные.
4. По завершении DMA шлёт **одно** прерывание.

### Преимущества DMA

- **CPU свободен** во время передачи.
- **Меньше прерываний** — одно на блок, а не на байт.
- **Эффективнее на больших объёмах.**

### Недостатки

- **Сложность.**
- **Расход памяти** на DMA-дескрипторы.
- **Cache coherency** проблемы (см. ниже).

### Типы DMA

**1. Block transfer.** Передаётся блок, потом одно прерывание.

**2. Cycle stealing.** DMA «крадёт» циклы шины, передаёт данные понемногу.

**3. Burst.** Передаются пачки байт.

**4. Scatter-gather.** Один запрос — несколько участков памяти.

### Cache coherency

Проблема: CPU имеет кэш данных. Если DMA пишет в память, CPU может видеть старые данные из кэша.

Решения:
- **Snooping:** кэш отслеживает DMA-транзакции и инвалидирует свои строки.
- **Software:** драйвер вручную флэшит/инвалидирует кэш до/после DMA.
- **Non-cacheable** регион для DMA-буферов.

### В Linux: DMA API

```c
// Когерентный DMA-буфер (доступен и CPU, и устройству)
dma_addr_t dma_handle;
void *cpu_addr = dma_alloc_coherent(&pdev->dev, size, &dma_handle, GFP_KERNEL);

// ... настроить устройство для записи в dma_handle ...

// ... позже ...
dma_free_coherent(&pdev->dev, size, cpu_addr, dma_handle);
```

### Streaming DMA

Для однократных передач:
```c
dma_handle = dma_map_single(&pdev->dev, buffer, size, DMA_TO_DEVICE);
// устройство читает из buffer
dma_unmap_single(&pdev->dev, dma_handle, size, DMA_TO_DEVICE);
```

### Применение

- **Сетевые карты.**
- **SCSI/SATA/NVMe контроллеры.**
- **Видео захват.**
- **Звуковые карты.**

Большинство современных устройств высокой производительности используют DMA.

### Bus mastering vs Third-party DMA

**Bus mastering:** устройство **само** управляет DMA (современный подход, PCI/PCIe).

**Third-party DMA:** отдельный DMA-контроллер (старый ISA-DMA).

### Резюме

- DMA — передача данных без CPU.
- Контроллер настраивается, выполняет передачу, шлёт прерывание по окончании.
- Снижает нагрузку на CPU при больших объёмах данных.
- Проблемы: cache coherency, сложность.
- В Linux: `dma_alloc_coherent`, `dma_map_single`.

## 108. Уровни программного обеспечения ввода-вывода. Реентерабельность драйверов.
*Источник: tanenbaum.md, Глава 5.3 «Уровни программного обеспечения ввода-вывода»*

### Уровни

Из tanenbaum.md (5.3):

```
┌────────────────────────────────────────────┐
│  Пользовательские программы                 │   уровень 4
├────────────────────────────────────────────┤
│  Программное обеспечение, не зависящее      │   уровень 3
│  от конкретных устройств                    │
├────────────────────────────────────────────┤
│  Драйверы устройств                         │   уровень 2
├────────────────────────────────────────────┤
│  Обработчики прерываний                     │   уровень 1
├────────────────────────────────────────────┤
│  Аппаратное обеспечение                     │   уровень 0
└────────────────────────────────────────────┘
```

Каждый уровень предоставляет **абстракции** уровню выше и пользует абстракции уровня ниже.

### Уровень 1: Обработчики прерываний

См. вопросы 72-75. Очень низкий уровень, скрывают аппаратные прерывания от верхних уровней.

### Уровень 2: Драйверы устройств

Управляют конкретными типами оборудования.

Особенности:
- **Зависят от устройства.**
- Часто **реализованы как модули** ядра.
- **Регистрируют операции** в file_operations (символьные) или другие интерфейсы.

### Уровень 3: Device-independent software

Предоставляет **единый интерфейс** к разным устройствам.

Из tanenbaum.md (5.3.3) функции:
- Единое именование устройств.
- Защита.
- Буферизация.
- Сообщение об ошибках.
- Распределение/освобождение устройств.

### Уровень 4: User-space

Пользовательские программы используют ВВ через стандартные библиотеки (libc, stdio).

### Реентерабельность драйверов

Из tanenbaum.md (5.3.2):
> «Драйверы должны быть реентерабельны (повторновходимы)».

**Реентерабельная функция** — может быть **вызвана повторно** до завершения первого вызова, без проблем.

### Зачем реентерабельность

- В Linux несколько процессов могут одновременно открыть одно устройство.
- В SMP драйвер может выполняться на разных CPU.
- Прерывания могут прерывать сами себя (на разных CPU или приоритетах).

### Требования к реентерабельной функции

- **Не использует глобальные переменные** без синхронизации.
- **Не использует статические локальные переменные.**
- **Не возвращает указатель на статические данные.**
- **Не вызывает не-реентерабельные функции.**

### Пример НЕ-реентерабельной функции

```c
char buffer[256];

void format_message(int n) {
    sprintf(buffer, "Value: %d", n);
    print(buffer);
}
```

Два потока одновременно → portion одного перезаписывается другим.

### Реентерабельная версия

```c
void format_message(int n) {
    char buffer[256];   // на стеке, у каждого потока свой
    sprintf(buffer, "Value: %d", n);
    print(buffer);
}
```

### Синхронизация в драйверах

Для общих данных — спинлоки, мьютексы, атомарные операции.

```c
struct mydev {
    spinlock_t lock;
    // shared data
};

ssize_t my_read(struct file *filp, char __user *buf, size_t count, loff_t *ppos)
{
    struct mydev *dev = filp->private_data;
    unsigned long flags;

    spin_lock_irqsave(&dev->lock, flags);
    // CS
    spin_unlock_irqrestore(&dev->lock, flags);

    // I/O без блокировок
}
```

### Резюме

- Уровни I/O: обработчики прерываний → драйверы → device-independent → user.
- Драйверы должны быть реентерабельными.
- Реентерабельность требует синхронизации общих данных.
- Локальные переменные на стеке — реентерабельны естественно.

## 109. Уровни программного обеспечения ввода-вывода. Программное обеспечение, не зависящее от конкретных устройств. Двойная буферизация.
*Источник: tanenbaum.md, Глава 5.3.3*

### Device-independent layer

Из tanenbaum.md (5.3.3):
> «Программное обеспечение, не зависящее от конкретных устройств, выполняет ту работу для ввода-вывода данных, которая является общей для всех устройств, и предоставляет единый интерфейс программному обеспечению уровня пользователя».

Функции:
- **Единое именование.**
- **Защита** через права доступа.
- **Буферизация.**
- **Размер блока** независим от устройства.
- **Распределение** буферов.
- **Сообщение об ошибках.**

### Единое именование

Пример Unix: устройства видны как **файлы**:
- `/dev/sda` — диск.
- `/dev/tty` — терминал.
- `/dev/null` — null-устройство.

Можно использовать обычные `open/read/write`.

### Буферизация

**Зачем буферизация:**
- Уменьшить число системных вызовов.
- Сгладить разницу скоростей устройств.
- Принять данные асинхронно.

### Простейший случай: без буферизации

Каждый байт от пользователя → системный вызов → I/O. Очень медленно.

### Одинарная буферизация

Буфер в ядре:
- Пользователь → буфер ядра → ядро принимает решение, когда писать на устройство.

Например, при выводе на принтер:
- Программа: `write(fd, byte1)`, `write(fd, byte2)`, ...
- Все байты → буфер.
- Когда буфер полон или явный flush → отправка на устройство.

### Двойная буферизация

Из tanenbaum.md (5.3.3):
> «Можно использовать двойную буферизацию: пока заполняется один буфер, опустошается второй, и наоборот».

```
Время:
┌────────────┐
│ Буфер A    │  → передача на устройство
└────────────┘
┌────────────┐
│ Буфер B    │  ← заполнение от пользователя
└────────────┘
```

После переполнения A:
- Поменять роли.
- Передача продолжается без задержки.

### Применение двойной буферизации

- **Видео:** front buffer (отображается) + back buffer (рисуется).
- **Аудио:** играется один буфер, заполняется второй.
- **Сетевые потоки.**
- **Принтеры.**
- **Графические интерфейсы** (избегают мерцания).

### Circular buffer (round-robin)

Развитие — **N буферов** в кольце:
- Производитель пишет в свободный.
- Потребитель читает из заполненного.
- Когда кольцо полно — производитель ждёт.

Используется в:
- Linux DMA ring buffers (сетевые карты).
- USB ring buffers.

### Защита

Device-independent уровень проверяет:
- Может ли пользователь открыть устройство?
- Может ли читать/писать?

Через стандартные права файлов в `/dev`.

### Сообщения об ошибках

Стандартные коды ошибок (`errno`):
- `EACCES`, `ENODEV`, `EBUSY`, `EIO`, `ENOSPC`.

Независимо от того, какое устройство.

### Размер блока

Разные устройства имеют разные «естественные» размеры:
- Диск: 512 байт или 4 КБ.
- TCP: ~1500 байт.
- USB: 64-1024 байт.

Device-independent уровень предоставляет **единый размер блока** (обычно размер страницы памяти, 4 КБ).

### Резюме

- Device-independent уровень даёт единый интерфейс.
- Двойная буферизация: пока один буфер передаётся, второй заполняется.
- Кольцевые буферы для нескольких уровней.
- Защита, именование, обработка ошибок.

## 110. Нижняя и верхняя половина обработчика прерываний. Запрет прерываний. Что делает верхняя половина прерывания для сетевого драйвера.
*Источник: robert_lav.md, Глава 7 «Прерывания и их обработка»*

### Верхняя и нижняя половина

См. также вопрос 75. Здесь — подробнее по контексту Linux.

Из robert_lav.md:
> «Процесс обработки прерываний разбивается на две части, или половины. Обработчик прерывания относится к верхней половине (top half). Он запускается сразу после получения прерывания и выполняет только ту работу, которая критична к задержкам по времени, такую, как отправка подтверждения о получении прерывания или сброс аппаратного устройства. Работа, которую можно выполнить позже, откладывается до запуска нижней (или основной) половины (bottom half)».

### Верхняя половина — задачи

1. **Acknowledge** устройства.
2. **Минимальная** обработка.
3. **Сохранение состояния** для нижней половины.
4. **Запуск** нижней половины.

### Нижняя половина — задачи

1. **Длинная** обработка.
2. **Может прерываться** другими прерываниями.
3. **Не может** блокироваться (для softirq/tasklet) или может (для workqueue).

### Что делает верхняя половина для сетевого драйвера

Из robert_lav.md:
> «Когда сетевая плата получает прибывшие из сети пакеты, она должна уведомить ядро о том, что доступны новые данные… Обработчик прерывания запускается, аппаратному обеспечению направляется подтверждение о получении прерывания, пакеты копируются в основную память, и после этого сетевой адаптер снова готов к получению новых пакетов».

Шаги верхней половины сетевого драйвера:

1. **Прочитать статус** сетевой карты.
2. **Подтвердить прерывание** записью в регистр устройства.
3. **Скопировать пакеты** из DMA-буфера в socket buffer (sk_buff).
4. **Освободить DMA-слот** для следующих пакетов.
5. **Запустить нижнюю половину** (softirq) для обработки протоколов.

Из robert_lav.md:
> «Ядро должно очень быстро скопировать сетевой пакет в основную память компьютера, поскольку буфер данных сетевой платы имеет небольшой фиксированный размер… Любая задержка с копированием пакета может привести к ситуации переполнения буфера».

### Нижняя половина сетевого драйвера

В **NET_RX_SOFTIRQ**:
- Разбор Ethernet-заголовка.
- Маршрутизация (IP).
- Передача в TCP/UDP.
- Доставка приложению через сокет.

### Регистрация обработчика

```c
int request_irq(unsigned int irq,
                irq_handler_t handler,
                unsigned long flags,
                const char *name,
                void *dev);
```

### Флаги

Из robert_lav.md (Глава 7):
- **`IRQF_DISABLED`** — запретить **все** прерывания на время.
- **`IRQF_SAMPLE_RANDOM`** — вносить вклад в pool энтропии.
- **`IRQF_TIMER`** — таймерное прерывание.
- **`IRQF_SHARED`** — линия IRQ может разделяться несколькими драйверами.

### Запрет прерываний

В обработчике прерывания обычно прерывания **той же линии** запрещены автоматически.

`IRQF_DISABLED` запрещает **все** прерывания.

Из robert_lav.md:
> «В большинстве обработчиков прерываний данный флаг не устанавливается, поскольку запрещать все прерывания — это плохая идея. Данный флаг используется для критичных по времени обработчиков прерываний, которые должны выполняться очень быстро».

### Прототип обработчика

```c
typedef irqreturn_t (*irq_handler_t)(int, void *);

irqreturn_t my_handler(int irq, void *dev_id);
```

Возврат:
- `IRQ_HANDLED` — мы обработали.
- `IRQ_NONE` — не наш (для shared lines).
- `IRQ_WAKE_THREAD` — разбудить threaded handler.

### Освобождение

```c
void free_irq(unsigned int irq, void *dev_id);
```

### Резюме

- Верхняя половина: быстрая обработка, ACK, копирование данных.
- Нижняя половина: длинная обработка протоколов.
- Для сетевой карты: ACK + копирование пакета в RAM + запуск softirq.
- Флаги: SHARED для shared IRQ.

## 111. Передача данных в прерывании. Определение процесса и его виртуального адресного пространства. Копирование данных в виртуальное адресное пространство процесса.
*Источник: rtos.md, Глава 7.8.1; LDD3; общие принципы*

### Проблема

Прерывание происходит **в произвольный момент**. Текущий процесс на CPU может быть **не тот**, кому нужны данные.

Сценарий:
- Процесс A открыл сокет и ждёт данных (заблокирован в `read`).
- Процесс B сейчас на CPU.
- Приходит пакет для A.
- Прерывание сетевой карты.
- **Обработчик не может писать в АП A**, потому что:
  - На CPU сейчас АП B.
  - Не известно, где буфер A.

### Решение: ядро как промежуточный буфер

1. Обработчик прерывания копирует данные в **буфер ядра**.
2. Помечает процесс A как «есть данные».
3. Будит A.
4. A в своём контексте делает `copy_to_user` из буфера ядра в свой буфер.

### Определение процесса

Прерывание не имеет «своего» процесса. `current` в обработчике — это **тот, кто был на CPU**, не «владелец» данных.

Чтобы найти **владельца**:
- Драйвер хранит ссылку на структуру `file` или `socket`.
- При прерывании смотрит, кто связан с устройством.

### Где данные

Чтобы передать данные процессу:
- **НЕ напрямую в его АП** (АП может быть не загружено).
- **В буфер ядра**, связанный с открытым файлом/сокетом.
- Процесс при возврате в свой контекст забирает.

### Копирование в АП процесса

Происходит **не в прерывании**, а позже — в контексте процесса. Когда A проснётся в своём системном вызове `read`:

```c
ssize_t my_read(struct file *filp, char __user *buf, size_t count, loff_t *ppos)
{
    struct my_dev *dev = filp->private_data;

    // подождать данных (в очереди ожидания)
    wait_event_interruptible(dev->wait, dev->data_ready);

    // СЕЙЧАС мы в контексте процесса A, АП A загружено
    if (copy_to_user(buf, dev->buffer, count))
        return -EFAULT;

    return count;
}
```

`copy_to_user` работает потому, что:
- Мы в контексте процесса A.
- CR3 указывает на АП A.
- Указатель `buf` валиден в этом АП.

### Что делает обработчик прерывания

```c
irqreturn_t my_isr(int irq, void *dev_id)
{
    struct my_dev *dev = dev_id;

    // 1. Acknowledge
    writel(IRQ_ACK, dev->regs + REG_IRQ);

    // 2. Скопировать в буфер ядра
    memcpy(dev->buffer, hardware_data, len);
    dev->data_ready = 1;

    // 3. Разбудить ожидающих
    wake_up_interruptible(&dev->wait);

    return IRQ_HANDLED;
}
```

### Кольцевой буфер

Часто в драйвере используется кольцевой буфер:
- Обработчик прерывания пишет.
- read() читает.
- При переполнении — либо потеря, либо блокировка.

### DMA — устройство пишет прямо в RAM

С DMA устройство пишет **прямо в физическую память** (буфер ядра, выделенный заранее).

Обработчик прерывания только:
- Узнаёт, что DMA закончил.
- Будит ожидающие процессы.

### Передача в АП процесса с zero-copy

В современных Linux есть механизмы передачи данных без копирования:
- **mmap** — отображение DMA-буфера в АП пользователя.
- **splice** — передача между fd без копирования.

Тогда обработчик ставит флаг готовности, а пользователь читает прямо из памяти, куда писало устройство.

### Резюме

- В прерывании нельзя писать прямо в АП процесса.
- Копирование в буфер ядра.
- При возврате в контекст процесса — `copy_to_user`.
- DMA пишет прямо в буфер ядра.
- Zero-copy техники — mmap, splice.

## 112. Прерывание. Блокировка вызвавшего процесса, возврат из прерывания. «Пробуждение» пользовательского процесса.
*Источник: rtos.md, Глава 7.8.1; LDD3; robert_lav.md*

### Сценарий: блокирующий read

Полный цикл:

1. **Пользовательский процесс P** делает `read(fd, buf, count)`.
2. **Системный вызов** → `sys_read` → драйвер `my_read`.
3. В драйвере нет данных:
   ```c
   wait_event_interruptible(dev->wait, dev->data_ready);
   ```
4. **P засыпает** — состояние TASK_INTERRUPTIBLE, удалён из runqueue.
5. **CPU занят другими процессами.**
6. **Внешнее событие** — приходит пакет.
7. **Прерывание сетевой карты.**
8. **Обработчик прерывания:**
   - Копирует данные в буфер ядра.
   - `wake_up_interruptible(&dev->wait)`.
9. **Процесс P** переводится в TASK_RUNNING, добавляется в runqueue.
10. **Планировщик** выбирает P (если приоритет позволяет).
11. **P продолжает** выполнение в своём `my_read`:
    - `copy_to_user(buf, kernel_buffer, count)`.
12. **Возврат в пользователя** — `read()` возвращает count.

### Шаг 3: блокировка процесса

Из robert_lav.md (Глава 4):
- `wait_event_interruptible` атомарно ставит процесс в очередь и засыпает.
- Состояние → TASK_INTERRUPTIBLE.
- `schedule()` вызывается → планировщик выбирает другой процесс.

### Шаг 6-8: прерывание и пробуждение

**Прерывание** аппаратно:
- Текущий поток приостанавливается.
- Контекст сохраняется в стек.
- Запускается обработчик.

**В обработчике** — `wake_up_interruptible`:
```c
void wake_up_interruptible(wait_queue_head_t *q)
{
    // Найти процессы в очереди
    // Для каждого в TASK_INTERRUPTIBLE:
    //   состояние → TASK_RUNNING
    //   добавить в runqueue
}
```

### Шаг 9-10: возврат из прерывания

После обработчика:
- Возврат к прерванному контексту.
- Проверка `need_resched` — если true, вызывается планировщик.

`wake_up` может установить `need_resched`, если разбуженный процесс приоритетнее текущего.

### Шаг 11: P продолжает

Когда планировщик выбирает P:
- Контекст P восстанавливается.
- P продолжает с того места, где был в `schedule()` внутри `wait_event_interruptible`.
- Цикл проверяет условие — оно теперь true.
- Выход из wait, продолжение драйвера.

### Возврат к пользователю

После `copy_to_user`:
- Драйвер возвращает count.
- Возврат к `sys_read`.
- Возврат к syscall обёртке.
- Возврат в пользовательский режим.
- Программа продолжает после `read()`.

### Пользователь не знает

Из его POV — `read()` просто вернул данные. Сколько он ждал — может быть проверено по часам, но не из самой логики.

### Диаграмма

```
Время →
User:    [read()]                                              [продолжение]
                ↓                                              ↑
Kernel:  [sys_read][wait_event_int]                            [copy_to_user]
                          ↓                                    ↑
                       P засыпает                          P пробуждается
                          │                                    │
                          ↓                                    │
                     планировщик                               │
                          ↓                                    │
                    [процесс Q]                                │
                          ...                                  │
                          ↓                                    │
                     IRQ от устройства                          │
                          ↓                                    │
                    [обработчик]                               │
                       wake_up ────────────────────────────────┘
                          ↓
                     возврат из IRQ
                          ↓
                     планировщик
                          ↓
                     процесс P
```

### Возврат после сигнала

Если до прихода данных P получил сигнал (например, Ctrl+C):
- `wait_event_interruptible` возвращает `-ERESTARTSYS`.
- Драйвер возвращает -ERESTARTSYS.
- `sys_read` обрабатывает: либо рестарт, либо возврат -EINTR пользователю.

### Возврат после таймаута (с timeout-версией)

`wait_event_interruptible_timeout`:
- При таймауте возвращает 0.
- Драйвер может вернуть -ETIMEDOUT или 0 (EOF), в зависимости от семантики.

### Резюме

- Блокирующий read: процесс → wait → засыпает → планировщик переключает.
- Прерывание устройства → обработчик → wake_up.
- Процесс пробуждается → продолжает с того места → copy_to_user → возврат пользователю.
- Прозрачно для пользователя: просто `read()` вернул данные.
- При сигнале → -ERESTARTSYS → возможно рестарт syscall.
