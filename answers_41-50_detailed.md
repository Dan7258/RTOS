## 41. Синхронизация через разделяемые объекты. Синхронный обмен сообщений.
*Источник: rtos.md, Глава 8.7 «Особенности реализации синхронного обмена сообщениями»*

### Постановка задачи

В асинхронном обмене (вопрос 40) отправитель посылает сообщение в очередь и **продолжает работу**. Иногда это нежелательно:

- Отправитель хочет **знать**, что сообщение принято.
- Не хочется тратить память на буфер.
- Нужна жёсткая синхронизация скоростей.

Синхронный обмен решает эти задачи.

### Идея синхронного обмена

Отправитель и получатель **встречаются**: оба блокируются и ждут, пока другая сторона не будет готова. Сообщение передаётся напрямую — без буфера.

**Аналогия:** встреча двух человек в назначенном месте. Один приходит раньше, ждёт. Когда приходит второй — они обмениваются (передают сообщение) и расходятся.

### Также называется рандеву (rendezvous)

Термин из французского «встреча» (rendez-vous). В программировании — точка встречи двух процессов/потоков.

### Семантика

- `send(msg)`: блокируется, пока кто-то не сделает `recv()`. Затем передаёт сообщение и возвращается.
- `recv()`: блокируется, пока кто-то не сделает `send()`. Возвращает полученное сообщение.

Кто пришёл первым — ждёт второго.

### Реализация через семафоры

```c
typedef struct {
    void *msg;
    semaphore_t can_send;     // initial = 0
    semaphore_t can_recv;     // initial = 0
} channel_t;

void send(channel_t *c, void *msg) {
    sem_wait(&c->can_send);   // ждать сигнала "можно отправлять"
    c->msg = msg;
    sem_post(&c->can_recv);   // "сообщение готово"
}

void *recv(channel_t *c) {
    sem_post(&c->can_send);   // "я готов принять"
    sem_wait(&c->can_recv);   // ждать сообщение
    return c->msg;
}
```

Поток выполнения:
1. Получатель приходит первым: `sem_post(can_send)` → `sem_wait(can_recv)` блокируется.
2. Отправитель приходит: `sem_wait(can_send)` проходит (счётчик стал 0 после post) → копирует msg → `sem_post(can_recv)`.
3. Получатель просыпается → читает msg.

Если порядок обратный (отправитель первым) — он блокируется на `can_send` до прихода получателя.

### Преимущества

**1. Гарантированная доставка.** Возврат из `send` означает, что сообщение принято. В асинхронной модели возврат означает только «положено в буфер».

**2. Минимум памяти.** Не нужен буфер — только указатель.

**3. Естественная синхронизация скоростей.** Если получатель медленный — отправитель ждёт. Нет переполнения буфера.

**4. Простота семантики.** Нет вопросов «потерялось ли сообщение», «достигло ли».

**5. Естественное ограничение нагрузки.** Под нагрузкой отправители ждут — система не «захлёбывается».

### Недостатки

**1. Низкая пропускная способность.** Каждое сообщение требует встречи.

**2. Чувствительность к задержкам.** Если получатель медленно обрабатывает — все отправители стоят.

**3. Невозможно сглаживать пики.** В асинхронной модели буфер поглощает всплеск; в синхронной — нет.

**4. Сложнее композиция систем.** Зависимости по времени между компонентами.

**5. Риск взаимоблокировки.** Если A ждёт B, а B ждёт A — оба заблокированы.

### Применение

**1. Caml / OCaml / Erlang / Go.**

Go — известный пример. Канал без буфера в Go:
```go
ch := make(chan int)    // буфер размера 0 = синхронный

go func() {
    ch <- 42            // блокируется до recv
}()

x := <-ch               // блокируется до send, получает 42
```

**2. CSP (Communicating Sequential Processes)** — формальная модель Tony Hoare. Основа Erlang, Go.

**3. Язык Ada** — рандеву встроено в синтаксис:
```ada
task body Server is
begin
    accept request(x : in Integer) do
        -- обработка
    end request;
end Server;

-- Клиент:
Server.request(42);    -- блокируется до accept
```

**4. ОСРВ** — между задачами с жёсткими требованиями (нет потери, известное время).

### Сравнение с асинхронным

| Параметр | Асинхронный | Синхронный |
|---|---|---|
| Буфер | Конечный | Нет |
| Блокировка отправителя | При полном буфере | Всегда |
| Гарантия доставки | До буфера | До получателя |
| Пропускная способность | Высокая (буферизация) | Ограниченная |
| Память | Размер буфера × N | Минимум |
| Под нагрузкой | Сглаживание | Естественное замедление |
| Чувствительность к задержкам | Низкая | Высокая |
| Простота семантики | Сложнее | Проще |

### Выбор между ними

- **Синхронный** — для строгого взаимодействия, RPC, систем с обратной связью.
- **Асинхронный** — для очередей задач, логирования, систем с разной скоростью производства/потребления.

В сложных системах часто **комбинируют**: где нужна гарантия — синхронный, где нужна производительность — асинхронный.

## 42. Критическая секция. Условия возникновения взаимоблокировок. Моделирование блокировок. Граф.
*Источник: tanenbaum.md, Глава 6 «Взаимоблокировка», 6.2*

### Что такое критическая секция

В многозадачной системе несколько потоков/процессов могут одновременно работать с **общими данными** (переменными, файлами, устройствами). Без специальных мер это вызывает **race condition** — результат зависит от порядка выполнения.

Из tanenbaum.md (2.3.2):
> «Та часть программы, в которой используется доступ к общей памяти, называется критической областью или критической секцией. Если бы удалось всё выстроить таким образом, чтобы никакие два процесса не находились одновременно в своих критических областях, это позволило бы избежать состязаний».

Решение — **взаимное исключение**: в критической секции одновременно может быть только один.

### Что такое взаимоблокировка

Реализация взаимного исключения обычно требует **захвата ресурса** (мьютекса, семафора). Когда процессу нужны несколько ресурсов, может возникнуть **тупик**.

Из tanenbaum.md (6.0):
> «Если два процесса одновременно выводят информацию на принтер, то получается полная тарабарщина… Поэтому все операционные системы способны временно предоставлять процессу исключительные права доступа к конкретным ресурсам».

### Классический пример

Tanenbaum описывает (6.0):

> «Процесс A запрашивает разрешение на использование сканера и получает его. Процесс B запрограммирован по-другому: сначала он запрашивает разрешение на использование пишущего привода Blu-ray-дисков и также получает это разрешение. Теперь A запрашивает разрешение на использование пишущего привода Blu-ray-дисков, но запрос отклоняется до тех пор, пока это устройство не будет освобождено процессом B. К сожалению, вместо того чтобы освободить привод, B запрашивает разрешение на использование сканера. И в этот момент оба процесса оказываются заблокированными навсегда».

В коде:
```c
// Процесс A
sem_wait(&scanner);
sem_wait(&blu_ray);
// делать работу
sem_post(&blu_ray);
sem_post(&scanner);

// Процесс B
sem_wait(&blu_ray);
sem_wait(&scanner);
// делать работу
sem_post(&scanner);
sem_post(&blu_ray);
```

A берёт сканер, B берёт Blu-Ray. A ждёт Blu-Ray (у B), B ждёт сканер (у A). Deadlock.

### Формальное определение

Из tanenbaum.md:
> «Взаимоблокировка в группе процессов возникает в том случае, если каждый процесс из этой группы ожидает события, наступление которого зависит исключительно от другого процесса из этой же группы».

Ключевое:
- Группа процессов.
- Все блокированы.
- Каждый ждёт другого в группе.
- Никто не может прогрессировать.

### Четыре условия Коффмана (1971)

Для возникновения **ресурсной** взаимоблокировки должны выполняться **все четыре**:

**1. Взаимное исключение.** Каждый ресурс либо выделен ровно одному процессу, либо доступен.

Если ресурс может использоваться одновременно несколькими (например, файл для чтения) — deadlock невозможен из-за этого ресурса.

**2. Удержание и ожидание.** Процесс, удерживающий ресурс, может запрашивать новый.

Если процесс должен сразу запросить все нужные ресурсы — нет «удержания при ожидании».

**3. Невыгружаемость.** Ресурс нельзя отобрать у процесса — он должен освободить сам.

Если ОС может «забрать» ресурс — deadlock прерывается принудительно.

**4. Циклическое ожидание.** Существует кольцо процессов P1 → R(a) → P2 → R(b) → ... → Pn → R(z) → P1, где Pi удерживает R(j-1) и ждёт R(j).

Если запрос ресурсов в строгом порядке (по нумерации) — цикл невозможен.

Если **хоть одно** условие не выполняется → deadlock структурно невозможен.

### Моделирование графом (Holt, 1972)

Из tanenbaum.md (6.2.2):
> «У графов имеется два вида узлов: процессы, показанные окружностями, и ресурсы, показанные квадратами».

**Узлы:**
- Процесс — круг.
- Ресурс — квадрат.

**Рёбра:**
- **Ресурс → Процесс** — ресурс выделен процессу.
- **Процесс → Ресурс** — процесс ждёт ресурс.

### Примеры графов

**1. Ресурс R выделен процессу A:**
```
[ R ] ────→ ( A )
```

**2. Процесс B ждёт ресурс S:**
```
( B ) ────→ [ S ]
```

**3. Deadlock между C и D:**
```
    [ T ] ←──── ( C )
      │           ▲
      ▼           │
    ( D ) ────→ [ U ]
```

C удерживает U, ждёт T. D удерживает T, ждёт U. Цикл C → T → D → U → C.

### Поиск deadlock по графу

При **одном экземпляре каждого ресурса** правило простое: если в графе есть цикл → есть deadlock.

При **нескольких экземплярах** ресурса (например, 3 принтера) цикл сам по себе не означает deadlock — нужен более сложный алгоритм (вопрос 43).

### Использование графов

**1. Обнаружение.** Периодически строится граф, ищется цикл (DFS).

**2. Уклонение.** Перед каждым выделением — симуляция: добавить ребро, проверить цикл. Если будет — отказать.

**3. Анализ.** Изучение последовательностей запросов на этапе проектирования системы.

### Резюме

- **Критическая секция** — область кода с разделяемыми данными.
- **Взаимоблокировка** — группа процессов навсегда блокирована, каждый ждёт другого.
- **Условия Коффмана** (4): взаимное исключение, удержание+ожидание, невыгружаемость, циклическое ожидание. Все 4 обязательны.
- **Граф ресурсов** — модель для обнаружения: процессы (круги), ресурсы (квадраты), рёбра «удерживает» и «ждёт».
- **Цикл в графе = deadlock** при одном экземпляре каждого ресурса.

## 43. Стратегии для борьбы с взаимоблокировками. Страусинный алгоритм. Обнаружение взаимоблокировок с использованием одного типа ресурса/нескольких ресурсов каждого типа.
*Источник: tanenbaum.md, Глава 6.3 «Страусиный алгоритм», 6.4 «Обнаружение взаимоблокировок»*

### Четыре стратегии

Из tanenbaum.md (конец 6.2):
> «Чаще всего для борьбы с взаимными блокировками используются четыре стратегии:
> 1. Игнорирование проблемы.
> 2. Обнаружение и восстановление.
> 3. Динамическое уклонение от них за счёт тщательного распределения ресурсов.
> 4. Предотвращение за счёт структурного подавления одного из четырёх условий, необходимых для их возникновения».

Эта глава вопроса покрывает (1) и (2). (3) — вопрос 44, (4) — вопрос 45.

### Страусинный алгоритм

Шутка: страус, видя опасность, прячет голову в песок. На самом деле страусы так не делают (см. сноску в tanenbaum), но термин прижился.

**Идея:** ничего не делать. Если deadlock редок — игнорировать.

Из tanenbaum.md:
> «Математики считают её неприемлемой и говорят, что взаимоблокировки следует предотвращать любой ценой. Инженеры спрашивают, как часто ожидается возникновение проблемы, как часто система даёт сбой по другим причинам и насколько серьёзны последствия взаимоблокировки. Если взаимоблокировка возникает в среднем один раз в пять лет, а система раз в неделю сбоит из-за технических отказов и дефектов операционной системы, большинство инженеров не захотят платить за избавление от взаимоблокировок существенным снижением производительности или удобства использования».

**Где применяется:**
- **Linux, Windows, macOS** — для большинства ресурсов.
- Системы общего назначения.
- При редкой вероятности deadlock и простой реакции (перезагрузка).

**Где не применяется:**
- Системы реального времени.
- Авиация, медицина, атомные станции.
- Системы с большой ценой простоя.

### Обнаружение при одном экземпляре каждого ресурса

Если ресурсов по одному, deadlock = цикл в графе. Нужен алгоритм обнаружения циклов.

Из tanenbaum.md (6.4.1), пример с 7 процессами и 6 ресурсами:

```
1. A удерживает R, хочет S
2. B хочет T
3. C хочет S
4. D удерживает U, хочет S и T
5. E удерживает T, хочет V
6. F удерживает W, хочет S
7. G удерживает V, хочет U
```

Граф:
- R → A, A → S
- B → T
- C → S
- U → D, D → S, D → T
- T → E, E → V
- W → F, F → S
- V → G, G → U

**Цикл:** D → T → E → V → G → U → D. Процессы D, E, G в deadlock.

A, C, F **не в deadlock** — после освобождения S (например, никто не удерживает S) они могут продолжить.

### Алгоритм поиска цикла

Из tanenbaum.md (6.4.1) — DFS с памятью посещений:

1. Для каждого узла N графа:
   a. Очистить список L.
   b. Запустить DFS с N как корня.
2. На каждом шаге добавлять текущий узел в L.
3. Если узел появился в L дважды — найден цикл, остановиться.
4. Если все узлы из текущего достижимы без повторения — нет цикла из N.

**Пример выполнения** на графе выше, начиная с R:
- L = [R], идём к A.
- L = [R, A], идём к S.
- L = [R, A, S], S — тупик (нет исходящих рёбер).
- Возврат к A, нет других рёбер. Возврат к R. Поиск с R закончен.

Перезапуск с B:
- L = [B, T, E, V, G, U, D].
- D имеет рёбра к S и T. Выбираем S → тупик. Возврат к D, выбираем T.
- L = [B, T, E, V, G, U, D, T]. T уже в L! **Цикл найден.**

### Обнаружение при нескольких экземплярах каждого ресурса

Если есть, скажем, 3 принтера, цикл в графе ещё не означает deadlock — кто-то из других процессов может освободить принтер.

Используются **векторы и матрицы**:

- **E** — вектор существующих ресурсов. `E[j]` — сколько экземпляров ресурса j всего.
- **A** — вектор доступных. `A[j]` — сколько свободно.
- **C** — матрица выделения. `C[i][j]` — сколько ресурса j удерживает процесс i.
- **R** — матрица запросов. `R[i][j]` — сколько процессу i нужно ресурса j ещё.

**Инвариант:** для каждого j: `sum(C[*][j]) + A[j] = E[j]`.

### Алгоритм

Идея: пытаемся «прокрутить» систему вперёд. Если есть процесс, чьи запросы можно удовлетворить, — пусть он завершится и вернёт ресурсы. Это может освободить другие процессы.

```
mark all processes as unmarked

while есть непомеченный процесс P, у которого R[P] <= A:
    A = A + C[P]      // P завершается, отдаёт всё
    mark P

если остались непомеченные процессы — они в deadlock
```

Сравнение векторов: `R[P] ≤ A` означает `R[P][j] ≤ A[j]` для каждого j.

### Пример (Tanenbaum, рис. 6.5)

3 процесса, 4 типа ресурсов (ленты, плоттеры, сканеры, Blu-Ray).

```
E = (4, 2, 3, 1)    // всего: 4 ленты, 2 плоттера, 3 сканера, 1 Blu-Ray

C (выделено):           R (нужно ещё):
P1: 0 0 1 0            P1: 2 0 0 1
P2: 2 0 0 1            P2: 1 0 1 0
P3: 0 1 2 0            P3: 2 1 0 0

Занято: 2 1 3 1   →   A = E - занято = (2, 1, 0, 0)
```

**Шаг 1:** ищем P с `R[P] ≤ A = (2,1,0,0)`.
- P1: R=(2,0,0,1). Нужен 1 Blu-Ray, доступно 0. Не подходит.
- P2: R=(1,0,1,0). Нужен 1 сканер, доступно 0. Не подходит.
- P3: R=(2,1,0,0). Нужно 2 ленты (есть), 1 плоттер (есть). Подходит!

Пометить P3, `A = A + C[P3] = (2,1,0,0) + (0,1,2,0) = (2, 2, 2, 0)`.

**Шаг 2:** ищем среди P1, P2.
- P1: R=(2,0,0,1). Не подходит — нет Blu-Ray.
- P2: R=(1,0,1,0). Подходит!

Пометить P2, `A = (2,2,2,0) + (2,0,0,1) = (4, 2, 2, 1)`.

**Шаг 3:** P1: R=(2,0,0,1). Всё доступно. Подходит.

Все процессы помечены — **deadlock нет**.

### Когда запускать обнаружение

- **Каждый запрос:** точно, но нагрузка.
- **Каждые k минут:** компромисс.
- **При низкой загрузке CPU:** если многие процессы заблокированы, deadlock более вероятен.

В базах данных (Oracle, PostgreSQL) часто проверка каждую секунду или при определённых событиях.

### Выход из обнаруженного deadlock

**1. Приоритетное овладение (preemption).**

Отобрать ресурс у одного процесса, отдать другому. Можно для немногих ресурсов:
- Память — выгрузить процесс на диск.
- CPU — снять с выполнения.

Не подходит для:
- Принтер во время печати.
- Запись в БД во время транзакции.

**2. Откат через контрольные точки (rollback).**

Процессы периодически сохраняют состояние в контрольные точки. При deadlock — откатить процесс к точке до получения ресурса.

Сложно для:
- Внешних эффектов (отправленные сообщения).
- Записи в БД (нельзя «откатить» уже отправленное письмо).

**3. Уничтожение процесса.**

Грубо, но просто:
- Убить процесс в цикле.
- Если не помогло — убить следующего.

Выбор жертвы:
- Меньше уже сделанной работы.
- Легко перезапускается.
- Меньшее количество ресурсов удерживает.

Не подходит для:
- Процессов, которые трудно перезапустить (БД, обновляющая записи).

## 44. Стратегии для борьбы с взаимоблокировками. Выход из взаимоблокировок. Уклонение от взаимоблокировок. Безопасное и небезопасное состояние. Алгоритм банкира.
*Источник: tanenbaum.md, Глава 6.5 «Уклонение от взаимоблокировок»*

### Уклонение — третья стратегия

Не дать deadlock возникнуть, тщательно распределяя ресурсы. При каждом запросе:
- Проверить, не приведёт ли выделение к небезопасному состоянию.
- Если приведёт — отложить запрос.

Требует **знания будущих запросов**.

### Безопасное и небезопасное состояние

**Определение (Tanenbaum 6.5.2):**

> «Состояние считается безопасным, если существует какой-то порядок планирования, при котором каждый процесс может доработать до конца, даже если все процессы внезапно и срочно запросят максимальное количество ресурсов».

Безопасное — есть **гарантированный** способ всех успешно завершить, даже в худшем случае.

Небезопасное — гарантии нет. **Не значит deadlock неизбежен** — но может случиться.

### Пример безопасного состояния

Один ресурс, 10 экземпляров. Процессы:
- A: удерживает 3, максимум 9 (нужно ещё ≤6).
- B: удерживает 2, максимум 4 (нужно ≤2).
- C: удерживает 2, максимум 7 (нужно ≤5).

Доступно: 10 − (3+2+2) = 3.

**Возможный план:**
1. Дать 2 процессу B (доступно 3 ≥ 2). B доработает, отдаст 4.
2. Доступно теперь 5. Дать C ещё 5. C доработает, отдаст 7.
3. Доступно 7. A нужно 6. Дать. A доработает.

Все завершились → состояние безопасное.

### Пример небезопасного состояния

Та же ситуация, но A удерживает 4 (а не 3). Доступно: 10 − (4+2+2) = 2.

Запросы максимумов:
- A может запросить ещё 5.
- B может запросить ещё 2.
- C может запросить ещё 5.

**Попытка плана:**
1. Дать B 2. Доступно 0. B доработает, отдаст 4.
2. Доступно 4. Никто не пройдёт: A нужно 5, C нужно 5.

Тупик. **Состояние небезопасное.**

Хотя сам по себе deadlock не возник — A или C может ещё не запросить максимум, и тогда обойдётся. Но **гарантии** нет.

### Алгоритм банкира Дейкстры (1965)

Аналогия с банком:
- Банкир имеет ограниченные средства.
- Клиенты могут получать кредиты, но известен **максимум** каждого клиента.
- При запросе кредита банкир проверяет: остаётся ли возможность последовательно удовлетворить всех клиентов?

В ОС:
- Банкир = ОС.
- Клиенты = процессы.
- Деньги = ресурсы.
- Максимум = заявленный максимум ресурсов процесса.

### Алгоритм для одного ресурса

При каждом запросе:
1. **Симуляция:** временно удовлетворить запрос.
2. **Проверка безопасности:**
   - Найти процесс P, для которого `максимум - удерживаемое ≤ доступно`.
   - Симулировать его завершение: `доступно += удерживаемое P`.
   - Повторять.
   - Если все процессы пройдут — состояние безопасно.
3. Если **безопасно** — выделить ресурс по-настоящему.
4. Если **небезопасно** — заблокировать запрос, оставить процесс в ожидании.

### Алгоритм для нескольких типов

Используются те же матрицы E, A, C, R из вопроса 43. Алгоритм проверки безопасности:

```
mark all unmarked

while существует непомеченный процесс P с R[P] <= A:
    A = A + C[P]
    mark P

если все помечены → безопасно, иначе → небезопасно
```

При запросе процесса i на ресурсы `Req[i]`:
1. Если `Req[i] > R[i]` — ошибка (запрос больше максимума).
2. Если `Req[i] > A` — ждать (нет ресурсов).
3. Симулировать: `A -= Req[i]`, `C[i] += Req[i]`, `R[i] -= Req[i]`.
4. Проверить безопасность. Если **да** — оставить так. Если **нет** — откатить, заблокировать.

### Пример (Tanenbaum, рис. 6.10)

5 процессов, 4 типа ресурсов:

```
E = (6, 3, 4, 2)    // 6 лент, 3 плоттера, 4 принтера, 2 Blu-Ray
```

Матрица C (текущее выделение):
```
A: 3 0 1 1
B: 0 1 0 0
C: 1 1 1 0
D: 1 1 0 1
E: 0 0 0 0
```

Сумма по столбцам = занято = (5, 3, 2, 2).

```
A = E - занято = (1, 0, 2, 0)
```

Матрица R (ещё нужно):
```
A: 1 1 0 0
B: 0 1 1 2
C: 3 1 0 0
D: 0 0 1 0
E: 2 1 1 0
```

**Проверка безопасности:**
- D: R=(0,0,1,0). A=(1,0,2,0). R≤A. Подходит! A += (1,1,0,1) → A=(2,1,2,1).
- A: R=(1,1,0,0). A=(2,1,2,1). Подходит! A += (3,0,1,1) → A=(5,1,3,2).
- E: R=(2,1,1,0). A=(5,1,3,2). Подходит! A += (0,0,0,0) → A=(5,1,3,2).
- B: R=(0,1,1,2). A=(5,1,3,2). Подходит! A += (0,1,0,0) → A=(5,2,3,2).
- C: R=(3,1,0,0). A=(5,2,3,2). Подходит!

Все процессы пройдут → **безопасно**.

Теперь B запрашивает 1 принтер:
- `Req = (0,0,1,0)`. `A = (1,0,2,0) ≥ Req`.
- Симуляция: `A = (1,0,1,0)`, `C[B] = (0,1,1,0)`, `R[B] = (0,1,0,2)`.
- Проверка безопасности — пройдёт.
- → Выделить.

Если бы E попросил последний принтер при `A = (1,0,1,0)`:
- Симуляция: `A = (1,0,0,0)`. Никто не пройдёт (всем нужен принтер).
- → Отказать, заблокировать E.

### Практическая ценность

Из tanenbaum.md:
> «Хотя алгоритм замечателен в теории, на практике он по существу бесполезен, поскольку нечасто можно определить заранее, каковы будут максимальные потребности процессов в ресурсах. Кроме того, количество процессов не фиксированно, оно динамически изменяется по мере входа пользователей в систему и выхода их из неё. И более того, ресурсы, считавшиеся доступными, могут внезапно пропасть (накопитель на магнитной ленте может сломаться)».

**Реально используется** редко. Применяется **дросселирование** — эвристика похожего рода: «не давать больше 70% буфера на одного пользователя».

## 45. Стратегии для борьбы с взаимоблокировками. Выход из взаимоблокировок. Предотвращение взаимоблокировок.
*Источник: tanenbaum.md, Глава 6.6 «Предотвращение взаимоблокировки»*

### Идея предотвращения

Структурно гарантировать, что **одно из 4 условий Коффмана** никогда не выполнится. Тогда deadlock невозможен в принципе.

Из tanenbaum.md:
> «Если мы сможем гарантировать, что хотя бы одно из этих условий никогда не будет выполнено, то взаимоблокировки станут структурно невозможными».

### Атака условия 1: взаимное исключение

Сделать ресурс **разделяемым**. Например, файлы для чтения — могут открыть несколько процессов одновременно.

Для разделяемого ресурса deadlock из-за него невозможен.

**Проблема:** не все ресурсы разделяемы. Принтер — нет.

**Спулинг (spooling).** Виртуализация принтера через очередь:
- Процессы пишут вывод в файл-очередь.
- Реальный принтер обслуживает только демон.
- Процессы не «удерживают» принтер.

Из tanenbaum.md:
> «В этой модели единственным процессом, который фактически запрашивает физический принтер, является демон принтера».

**Проблема спулинга:** ограничение очереди. Если двое заполнили половину буфера каждый и ждут, чтобы закончить — deadlock смещается на дисковое пространство.

### Атака условия 2: удержание и ожидание

**Запретить** удержание одного ресурса при ожидании другого. Варианты:

**Вариант A: Заранее запрашивать всё.** Процесс должен указать **все** нужные ресурсы при старте. Получает либо все, либо ждёт.

**Проблемы:**
- Многие процессы не знают заранее.
- Ресурсы простаивают (плоттер удерживается, хотя нужен в конце).

Использовалось в пакетных системах, где задание описывало нужные ресурсы.

**Вариант B: Отказ перед запросом нового.** Перед запросом ресурса процесс должен **освободить** все уже удерживаемые. Затем запросить всё заново.

**Проблемы:**
- Сложность.
- Возможна потеря состояния (если ресурс — например, частично сделанная работа).

### Атака условия 3: невыгружаемость

Сделать так, чтобы ресурс **можно было отобрать**. Это виртуализация.

Пример:
- Память — выгружаемая (своппинг).
- CPU — выгружаемый (вытесняющее планирование).
- Принтер — через спулинг.

**Не подходит для:**
- Запись в БД (нельзя отменить уже выполненные изменения).
- Открытые сетевые соединения.

### Атака условия 4: циклическое ожидание — самый практичный подход

Два варианта:

**Вариант A: одного ресурса.** Процесс может иметь только **один ресурс одновременно**. Чтобы взять новый — освободить старый.

Слишком ограниченно. Не подходит для процесса, копирующего с диска на принтер (нужны оба одновременно).

**Вариант B: глобальная нумерация ресурсов.**

Все ресурсы пронумерованы:
```
1: принтер
2: плоттер
3: сканер
4: Blu-Ray
5: лента
```

**Правило:** процесс может запрашивать ресурсы **только в порядке возрастания номеров**.

Если процессу нужны плоттер (2) и сканер (3), он сначала берёт плоттер, потом сканер. Никогда нельзя взять сканер, потом плоттер.

### Почему это исключает циклы

Доказательство (Tanenbaum 6.6.4):

Предположим есть цикл `P_a → R(i) → P_b → R(j) → ... → P_a`. То есть P_a удерживает R(j) (последний в цикле) и просит R(i). По правилу `i > j` (запрос только возрастающего).

Аналогично, P_b удерживает R(i) и просит R(j_next). По правилу `j_next > i`.

Продолжая цикл: номера должны постоянно возрастать. Но мы возвращаемся к P_a и нужен номер `> j`. Получается, что в цикле номера строго возрастают, но мы вернулись в начало → противоречие.

Поэтому **циклов нет** → **deadlock невозможен**.

### Применение

Glob nbalanya нумерация широко используется:

**1. Linux kernel:** правила порядка захвата мьютексов и спинлоков. Описаны в `Documentation/locking/`.

**2. Базы данных:** деревья блокировок (например, всегда `BEGIN → table → row`).

**3. Графические системы:** строгий порядок захвата ресурсов GPU.

### Слегка ослабленный вариант

Tanenbaum:
> «При незначительном изменении этого алгоритма исключается требование приобретения ресурсов в строго возрастающем порядке и просто требуется, чтобы ни один...»

Можно сделать порядок не строго возрастающим, а просто: «ресурс с большим номером можно взять только если все удерживаемые имеют меньшие». То же доказательство работает.

### Резюме сравнения 4 стратегий

| Условие | Атака | Применение |
|---|---|---|
| 1: Взаимное исключение | Спулинг | Принтеры, иногда файлы |
| 2: Удержание+ожидание | Заранее запросить всё | Пакетные системы |
| 3: Невыгружаемость | Виртуализация | Память, CPU |
| 4: Циклическое ожидание | Нумерация | **Ядра ОС, БД** |

Самый практичный — **нумерация ресурсов**.

### Резюме главы 6

| Стратегия | Когда применяется | Цена |
|---|---|---|
| Игнорирование (страус) | Linux, Windows, macOS | Редкие сбои |
| Обнаружение и восстановление | БД, критические системы | Сложность, откаты |
| Уклонение (банкир) | Редко | Знание будущего |
| Предотвращение (нумерация) | Ядра, БД | Дисциплина программистов |

## 46. Критические секции и доступ к данным. Отличие в процессах и потоках.
*Источник: rtos.md, Глава 6.1; общие принципы*

### Что общее

И в потоках, и в процессах **критическая секция** имеет одно и то же значение: участок кода, где работа с разделяемыми данными должна быть взаимно исключающей.

Цель — избежать **race condition**: ситуации, когда результат зависит от порядка выполнения.

### Принципиальное отличие

**Потоки одного процесса:** **по умолчанию** делят АП. Все глобальные переменные, статические данные, куча — общие.

**Процессы:** **по умолчанию** изолированы. Никаких общих данных, пока явно не создан общий регион (shared memory).

### Следствия

#### 1. Что защищать

**Потоки:**
- Любая глобальная переменная.
- Куча (если разделяется через указатели).
- Любая статическая переменная функции.

**Процессы:**
- Только явно созданные общие данные.
- IPC-объекты (очереди, каналы).
- Файлы (если открыты для записи).

#### 2. Примитивы

**Потоки:**
- `pthread_mutex_t`, `pthread_rwlock_t`, `pthread_cond_t`.
- `atomic_int`, atomic-операции.
- Spinlock (внутри ядра).
- Реализованы через futex — быстрые, не требуют системных вызовов в нерасторопном случае.

**Процессы:**
- POSIX semaphore (`sem_open` — именованные).
- System V semaphore (`semget`).
- File locks (`flock`, `fcntl`).
- Mutex в shared memory с атрибутом `PSHARED`.

#### 3. Производительность

**Потоки:** очень быстро. `pthread_mutex_lock` без конкуренции — 20-30 наносекунд.

**Процессы:** дольше — есть системные вызовы. POSIX semaphore через shared memory — на порядок медленнее.

#### 4. Защита от ошибок

**Потоки:** **никакой защиты** от ошибок. Один поток может затереть данные другого:
- Дикий указатель → SEGFAULT всего процесса.
- Переполнение буфера → коррупция данных.

**Процессы:** **полная защита** через MMU. Один процесс не может «увидеть» данные другого, кроме как через явные общие регионы. Сбой не вредит другим.

### Пример кода для потоков

```c
#include <pthread.h>

pthread_mutex_t counter_lock = PTHREAD_MUTEX_INITIALIZER;
int counter = 0;

void *worker(void *arg) {
    for (int i = 0; i < 1000; i++) {
        pthread_mutex_lock(&counter_lock);
        counter++;            // критическая секция
        pthread_mutex_unlock(&counter_lock);
    }
    return NULL;
}

int main() {
    pthread_t t1, t2;
    pthread_create(&t1, NULL, worker, NULL);
    pthread_create(&t2, NULL, worker, NULL);
    pthread_join(t1, NULL);
    pthread_join(t2, NULL);
    printf("counter = %d\n", counter);   // 2000
}
```

Без мьютекса — counter был бы непредсказуем (race condition).

### Пример для процессов через shared memory

```c
#include <sys/mman.h>
#include <pthread.h>
#include <fcntl.h>

typedef struct {
    pthread_mutex_t lock;
    int counter;
} shared_t;

int main() {
    // Создание разделяемой памяти
    int fd = shm_open("/myshm", O_CREAT | O_RDWR, 0666);
    ftruncate(fd, sizeof(shared_t));
    shared_t *s = mmap(NULL, sizeof(shared_t),
                       PROT_READ | PROT_WRITE,
                       MAP_SHARED, fd, 0);

    // Инициализация мьютекса как разделяемого
    pthread_mutexattr_t attr;
    pthread_mutexattr_init(&attr);
    pthread_mutexattr_setpshared(&attr, PTHREAD_PROCESS_SHARED);
    pthread_mutex_init(&s->lock, &attr);
    s->counter = 0;

    if (fork() == 0) {
        for (int i = 0; i < 1000; i++) {
            pthread_mutex_lock(&s->lock);
            s->counter++;
            pthread_mutex_unlock(&s->lock);
        }
        return 0;
    }
    // ... другой процесс ...
}
```

Ключевые отличия от потоков:
- Нужно создать **общую память** (`shm_open` + `mmap`).
- Мьютекс должен быть **разделяемым** (`PSHARED`).
- Мьютекс **в самой общей памяти**.

### Что выбрать

**Использовать потоки:**
- Нужна быстрая коммуникация.
- Много обмена данными.
- Не критично, если один баг валит всё.
- Веб-серверы, БД, игры.

**Использовать процессы:**
- Нужна изоляция (защита от ошибок).
- Безопасность (sandbox).
- Распределение нагрузки между ядрами/машинами.
- Chrome, Apache (prefork).

### Гибридный подход

Многие современные приложения используют **и то, и другое**:
- Несколько процессов (по числу ядер) для безопасности.
- В каждом процессе — пул потоков для производительности.
- Между процессами — IPC через сокеты, общую память.

Пример: Chrome — каждая вкладка в отдельном процессе, внутри процесса много потоков.

## 47. Критическая секция. Запрещение прерываний. Достоинства и недостатки. Пример.
*Источник: rtos.md, Глава 6.2.1 «Запрет прерываний и вытеснения»; tanenbaum.md, Глава 2.3.3*

### Идея

В **однопроцессорной** системе переключение контекста происходит только в результате прерывания:
- Таймер прерывает поток → планировщик переключает.
- I/O прерывание разблокирует поток → планировщик переключает.
- Системный вызов — но это уже сознательное действие потока.

Если **запретить прерывания** перед критической секцией:
- Таймер не прервёт → не будет вытеснения.
- I/O не приведёт к переключению.
- Критическая секция выполняется атомарно для других потоков.

И **обработчики прерываний** тоже не запустятся → нет конкуренции с ними.

### Реализация

На уровне процессора:
- **x86:** инструкция `CLI` (clear interrupt flag), `STI` (set).
- **ARM Cortex-M:** запись в регистр `PRIMASK`.
- **MIPS:** изменение бита IE в регистре Status.

В C:
```c
// Linux kernel, на любой архитектуре
unsigned long flags;
local_irq_save(flags);   // запретить прерывания + сохранить старое состояние
// критическая секция
local_irq_restore(flags); // восстановить состояние
```

В rtos.md — через HAL:
```c
// Источник: rtos.md, раздел 6.2.1
spl_t prev_spl = raise_spl(SPL_SYNC);
// критическая секция
lower_spl(prev_spl);
```

`raise_spl(SPL_SYNC)` устанавливает уровень маски прерываний так, чтобы ничего не прерывало нашу работу.

### Что защищает

**От потоков:** да. Без прерывания таймера нет вытеснения.

**От обработчиков прерываний:** да. Сами прерывания не приходят.

**От SMP:** нет. Запрет на одном CPU не влияет на других. Если на другом CPU работает обработчик или поток — он продолжает.

### Достоинства

**1. Очень быстро.** Одна инструкция, единицы тактов.

**2. Минимальные накладные расходы.** Нет управляющих структур.

**3. Простота.** Идея и реализация банальны.

**4. Защищает от всех источников.** Без других мер обеспечивает взаимное исключение.

**5. Подходит для очень коротких секций.** Где даже мьютекс был бы оверкилом.

Из rtos.md:
> «Из-за того, что в большинстве процессоров запрет прерываний выполняется всего одной инструкцией, этот метод отличается исключительной скоростью и отсутствием накладных расходов».

### Недостатки

**1. Не работает на SMP.** Критически важный недостаток в эпоху многоядерных систем.

**2. Грубая гранулярность.**

Из rtos.md:
> «Поскольку запрет прерываний влияет на весь процессор, то использование такого подхода влияет не только на объекты, вовлечённые во взаимодействие, но и на всё, что может выполняться на данном процессоре. Это можно сравнить с системой управления светофорами в городе, которая имеет всего два положения "зелёный свет всем, кто едет с севера на юг и обратно, красный — тем кто едет с запада на восток и обратно" и его противоположность».

**3. Влияет на время реакции.**

Если прерывания запрещены долго → пропадает реакция системы. Прерывание от датчика не обрабатывается → задержка.

В РВ-системах это критично: нельзя предсказать время реакции, если длительность критической секции переменная.

**4. Не подходит для пользовательского кода.** Запрет прерываний — привилегированная операция. Пользователь не может её выполнить.

**5. Опасно для отладки.** Если кто-то забыл `enable_interrupts()` после критической секции — система зависнет.

### Пример: атомарная вставка в связный список

```c
typedef struct node {
    int value;
    struct node *next;
} node_t;

node_t *head = NULL;

void push(int v) {
    node_t *n = malloc(sizeof(node_t));
    n->value = v;

    unsigned long flags;
    local_irq_save(flags);
    n->next = head;
    head = n;
    local_irq_restore(flags);
}

int pop() {
    int val = -1;
    unsigned long flags;
    local_irq_save(flags);
    if (head) {
        val = head->value;
        head = head->next;
        // ... освободить старый head ...
    }
    local_irq_restore(flags);
    return val;
}
```

Без запрета прерываний:
- Поток начал `push`: установил `n->next = head`.
- Прерывание тоже делает `push` — добавляет свой узел.
- Возврат в поток: `head = n` — теряем узел, добавленный прерыванием.

С запретом — атомарно.

### Применение

- **Простые RTOS** на однопроцессорных микроконтроллерах.
- **Обработчики прерываний** ядра (для синхронизации с другими прерываниями).
- **Очень короткие критические секции** в ядре.
- **Embedded** без MMU и SMP.

В Linux на SMP этот подход **недостаточен** — нужны спинлоки, дополнительно отключающие прерывания (`spin_lock_irqsave`).

### Современная реальность

Чисто запрет прерываний без других мер — устарел. На SMP-системах используется комбинация:
- Спинлок для защиты от других CPU.
- Запрет прерываний для защиты от обработчиков на нашем CPU.

```c
spin_lock_irqsave(&lock, flags);
// критическая секция
spin_unlock_irqrestore(&lock, flags);
```

Это даёт защиту и от потоков на других CPU (через спинлок), и от прерываний на нашем (через `irqsave`).

## 48. Критическая секция. Запрещение вытеснения. Достоинства и недостатки. Пример.
*Источник: rtos.md, Глава 5.3.6 «Блокирование вытеснения»; robert_lav.md, Глава 4*

### Идея

Запрет прерываний (вопрос 47) запрещает **всё**: и прерывания, и вытеснение. А что, если запретить только **вытеснение планировщиком**, а прерывания пусть продолжают приходить?

Тогда:
- Прерывания обрабатываются нормально → не страдает время реакции.
- Текущий поток выполняется монопольно до явного освобождения CPU.

### Реализация

В Linux:
```c
preempt_disable();
// критическая секция
preempt_enable();
```

Внутри обычно:
```c
static inline void preempt_disable(void) {
    current_thread_info()->preempt_count++;
}

static inline void preempt_enable(void) {
    current_thread_info()->preempt_count--;
    if (preempt_count == 0 && need_resched)
        schedule();
}
```

Идея: счётчик `preempt_count`. Планировщик проверяет: если `preempt_count > 0` — не переключать.

В rtos.md: `fx_sched_lock()` / `fx_sched_unlock()`.

### Что защищает

**От потоков на нашем CPU:** да. Планировщик не вытеснит.

**От обработчиков прерываний:** **нет**. Прерывания приходят и обрабатываются.

**От SMP:** **нет**. Другие CPU не знают о нашем `preempt_disable`.

### Достоинства

**1. Прерывания работают нормально.** Время реакции на прерывания не ухудшается.

**2. Менее агрессивно**, чем запрет прерываний.

**3. Простая реализация** через счётчик.

**4. Лучше для систем с критическим временем реакции.**

**5. Безопасно для длинных секций** — не блокирует другие CPU и обработчики.

### Недостатки

**1. Не защищает от обработчиков прерываний.** Если разделяемые данные доступны и из прерывания — недостаточно.

**2. Не работает на SMP сама по себе.** Другие CPU могут одновременно работать с данными.

**3. Длинная критическая секция = плохая отзывчивость.** Высокоприоритетный поток ждёт.

### Пример: модификация структуры, не доступной прерываниям

```c
struct list_head my_list;

void add_to_list(struct item *item) {
    preempt_disable();
    list_add_tail(&item->node, &my_list);
    preempt_enable();
}
```

Если `my_list` модифицируется только потоками (не из прерываний), `preempt_disable` достаточно.

### Связь со спинлоками

В Linux **`spin_lock()` неявно делает `preempt_disable`**. Это потому, что:
- Если бы планировщик мог переключить поток, удерживающий спинлок, другой поток на том же CPU попытался бы взять этот спинлок → активное ожидание навсегда (на однопроцессорной системе).

Поэтому при удержании спинлока ядро Linux **никогда не переключает поток**.

Если нужна защита от прерываний — `spin_lock_irqsave()` (то и то).

### Современное применение

В Linux:
- **`preempt_disable`** часто используется внутри ядра для коротких операций.
- **`get_cpu()` / `put_cpu()`** — закрепляет процесс за CPU, тоже отключает вытеснение.
- **Спинлоки** — основной механизм; неявно отключают вытеснение.

В пользовательском пространстве аналога нет (нельзя запретить ядру переключать поток).

### Сравнение с запретом прерываний

| Параметр | Запрет прерываний | Запрет вытеснения |
|---|---|---|
| От потоков (тот же CPU) | + | + |
| От обработчиков | + | – |
| От SMP | – | – |
| Время реакции | Ухудшается | Не страдает |
| Требует привилегий | Да (kernel) | Да (kernel) |
| Накладные расходы | Минимум | Минимум |

## 49. Критическая секция на основе условий и переменных. Проблемы.
*Источник: tanenbaum.md, Глава 2.3.3; общие принципы синхронизации*

### Программные попытки взаимного исключения

До появления аппаратных примитивов (TSL, CAS) делались попытки реализовать критическую секцию **только** на обычных операциях с памятью. Результат — серия алгоритмов с **тонкими проблемами**.

### Попытка 1: одиночная переменная-флаг

```c
volatile int lock = 0;

void enter_cs() {
    while (lock != 0) { /* spin */ }
    lock = 1;
}

void exit_cs() {
    lock = 0;
}
```

**Проблема:**
- Поток A: проверил `lock != 0` → 0, готов входить.
- **Переключение.**
- Поток B: проверил `lock != 0` → всё ещё 0, готов входить. `lock = 1`. В CS.
- Возврат в A: `lock = 1`. В CS.

**Оба** в CS. Race condition.

Проблема — отсутствие атомарности «проверь и установи». Между проверкой и установкой возможно переключение.

### Попытка 2: переменная очереди (строгое чередование)

```c
volatile int turn = 0;

// Поток 0
void enter_cs() {
    while (turn != 0) { /* spin */ }
}
void exit_cs() {
    turn = 1;
}

// Поток 1 — аналогично с turn != 1
```

**Корректно** в смысле взаимного исключения, но плохо:

- **Жёсткое чередование.** Поток 0 → Поток 1 → Поток 0 → ...
- Если поток 1 не хочет в CS — поток 0 тоже не может!
- Это нарушает условие 3 для корректного решения (Tanenbaum, 2.3.2):
  > «Никакие процессы, выполняемые за пределами своих критических областей, не могут блокироваться любым другим процессом».

### Алгоритм Декера (1965)

Корректный для двух потоков, но сложный. Использовал две переменные `flag` (хочет ли) и `turn` (чья очередь).

### Алгоритм Петерсона (1981)

Упрощённый Декер. Корректный для 2 потоков:

```c
volatile bool interested[2] = {false, false};
volatile int turn;

void enter_cs(int self) {
    int other = 1 - self;
    interested[self] = true;        // (1) объявить интерес
    turn = self;                     // (2) уступить очередь другому
    while (interested[other] && turn == self) { /* spin */ }
}

void exit_cs(int self) {
    interested[self] = false;
}
```

**Корректность:** если оба интересуются, последний установивший `turn` уступает другому. Гарантируется взаимное исключение и прогресс.

**Но:**

**1. Активное ожидание.** Поток крутит цикл, тратит CPU.

**2. Только для 2 потоков.** Обобщения существуют (Lamport's bakery algorithm), но сложны.

**3. Не работает с современным железом.** Компилятор и CPU могут **переупорядочить** операции:
- `interested[self] = true; turn = self;` — может быть выполнено в обратном порядке.
- Записи могут не быть видны сразу другим CPU.

Нужны **барьеры памяти** (memory barriers).

### Условные переменные

Альтернативный подход — **блокирующее ожидание** на условии. Используются вместе с мьютексом.

```c
pthread_mutex_t mtx = PTHREAD_MUTEX_INITIALIZER;
pthread_cond_t cv = PTHREAD_COND_INITIALIZER;
int condition_met = 0;

// Поток-ожидающий
void wait_for_event() {
    pthread_mutex_lock(&mtx);
    while (!condition_met)
        pthread_cond_wait(&cv, &mtx);
    // обработка
    pthread_mutex_unlock(&mtx);
}

// Поток-сигнализирующий
void signal_event() {
    pthread_mutex_lock(&mtx);
    condition_met = 1;
    pthread_cond_signal(&cv);
    pthread_mutex_unlock(&mtx);
}
```

**Как работает `pthread_cond_wait`:**
1. Атомарно: разблокировать мьютекс + поместить в очередь ожидания + заснуть.
2. При сигнале: проснуться + заблокировать мьютекс.

Важно: эти три действия — **атомарно**.

### Проблемы условных переменных

**1. Spurious wakeup (фиктивное пробуждение).**

Поток может проснуться **без сигнала** — из-за внутренних оптимизаций POSIX или сигнала ОС.

Решение: проверка условия в `while`, не `if`:

```c
while (!condition_met)        // НЕ if!
    pthread_cond_wait(&cv, &mtx);
```

Если разбудились без причины — снова в `wait`.

**2. Lost wakeup (потерянный сигнал).**

Если `signal` придёт **до** того, как ждущий поток дошёл до `wait` — сигнал теряется.

Поэтому **проверка условия и `wait` обязательно под мьютексом**, и `signal` тоже под мьютексом (или под тем же), что и проверка/установка условия.

Атомарность `unlock-mutex + wait` в `pthread_cond_wait` гарантирует, что между разблокировкой и сном никто не успеет послать сигнал, который мы пропустим.

**3. Thundering herd.**

`pthread_cond_broadcast` будит **всех** ожидающих. Часто только один может реально продолжить — остальные проверят условие и снова заснут.

Лучше использовать `pthread_cond_signal` (будит одного), если нет специальной причины для broadcast.

**4. Самоблокировка.**

Если поток держит мьютекс и пытается ждать условие, которое может выполнить только он сам, — тупик.

**5. Зависимость от мьютекса.**

Условная переменная без мьютекса бессмысленна — нужна атомарность «проверить + ждать». Этим занимается мьютекс.

### Принцип

**Без аппаратной поддержки** надёжно реализовать взаимное исключение **невозможно**. Все программные алгоритмы имеют тонкости с переупорядочиванием, барьерами, видимостью.

Современные мьютексы используют **аппаратные** атомарные операции:
- TSL, XCHG (вопрос 52).
- CAS (вопрос 54).
- LL/SC (вопрос 55).

Условные переменные — высокоуровневый примитив на основе мьютексов.

### Резюме

- Программные алгоритмы (Петерсон, Декер) корректны теоретически, но имеют проблемы практического применения.
- Активное ожидание неэффективно.
- Не работают с современным железом без барьеров памяти.
- Условные переменные — практичный высокоуровневый примитив, но требуют дисциплины (проверка в цикле, синхронизация под мьютексом).
- Современные системы используют аппаратные примитивы как основу мьютексов.

## 50. Понятие синхронизации. Уведомления, события. Двухстороннее рандеву.
*Источник: rtos.md, Глава 6.1 «Задача синхронизации», 6.2.5 «Синхронизация с прямым уведомлением», 6.3 «События»*

### Понятие синхронизации

**Синхронизация** — координация работы нескольких потоков/процессов/устройств для обеспечения корректности.

Из rtos.md (6.1):
> «Во всех ситуациях, где присутствует некоторый общий для нескольких пользователей ресурс, всегда неизбежно возникает проблема синхронизации доступа к этому ресурсу».

### Два класса синхронизации

**1. Взаимное исключение (mutual exclusion).**

Только один может работать с ресурсом одновременно. Цель — избежать race condition.

Примитивы:
- Мьютексы.
- Бинарные семафоры.
- Спинлоки.
- TSL, CAS (аппаратно).

**2. Уведомление (notification, signalling).**

Один сообщает другому о событии. Получатель блокируется до сигнала. Цель — координация во времени.

Примитивы:
- События.
- Условные переменные.
- Сигналы (signals).
- Семафоры (тоже).
- Очереди сообщений.

### Свойства корректных механизмов синхронизации

**Безопасность (safety):**
- Корректность результата.
- Защита инвариантов общих данных.

**Жизнеспособность (liveness):**
- Прогресс системы.
- Отсутствие deadlock.
- Отсутствие starvation (голодания).

Из rtos.md:
> «Если свойства надёжности или жизнеспособности для механизма синхронизации не выполняются (в зависимости от чередования потоков нарушаются соглашения, которые определены для задачи), такой механизм называется содержащим гонки или условия гонок (race condition)».

### События

Простейший объект уведомления. Из rtos.md (6.3):

> «Событие имеет внутреннее состояние в виде бинарного значения, указывающего на то, находится ли объект в активном состоянии или нет, а также 3 основных метода, не считая конструктора и деструктора».

```c
int event_set(event_t *event);              // сделать активным
int event_reset(event_t *event);            // сбросить
int event_timedwait(event_t *event, uint32_t timeout); // ждать
```

**Поведение:**
- Событие в **активном** состоянии: попытки `wait` сразу возвращаются.
- Событие в **неактивном** состоянии: поток блокируется.
- `set` пробуждает **всех** ожидающих, переводит в активное состояние.
- `reset` переводит в неактивное.

**Аналогия:** дверь.
- Открытая дверь — все проходят.
- Закрытая — ждут, пока кто-то откроет.

### Пример использования (rtos.md, 6.3)

```c
// Источник: rtos.md, раздел 6.3 «События»
static event_t event;

void ISR(void *arg) {
    /* Отключение прерываний от устройства */
    dpc_insert(&dpc, DPC, NULL);
}

void DPC(void *arg) {
    event_set(&event);
}

void Thread(void *arg) {
    do {
        int error = event_timedwait(&event, ...);
        /* ... */
        event_reset(&event);
        /* Обработка прерывания */
        /* Включение прерываний от устройства */
    } while(1);
}
```

Поток ждёт событие → обработчик прерывания через DPC активирует → поток просыпается, обрабатывает.

### Достоинства событий

**1. Простая семантика.**

**2. Широковещательность.** Все ожидающие просыпаются. Из rtos.md:
> «Главное достоинство события в том, что это широковещательный (то есть переводящий в состояние готовности все ожидающие потоки, синхронизация типа 1 к N) объект, то есть может использоваться в том случае, когда количество ожидающих не известно заранее».

**3. Можно ждать с таймаутом.**

**4. Универсальный 1-к-N паттерн.**

### Недостатки

**1. Повторные сигналы теряются.**

Из rtos.md:
> «Данный пример не будет работать, в случае, если новые прерывания могут приходить во время обработки текущего, поскольку попытки установить в активное состояние событие, которое уже установлено, теряются».

**2. Не подходит для взаимного исключения.** Нет счётчика — нельзя реализовать ограниченные ресурсы.

**3. Время `event_set` зависит от числа ожидающих.** Не детерминировано.

Поэтому в РВ-системах для нотификации предпочитают **семафоры** (детерминированное время).

### Прямое уведомление

Из rtos.md (6.2.5):
> «Часто встроенная система представляет собой набор обработчиков внешних событий, для чего требуется обеспечить возможность обработчика прерывания уведомить поток о возникающих событиях. Такой тип взаимодействия также может быть реализован с помощью интерфейса потоков без использования дополнительных объектов и называется синхронизацией с прямым уведомлением (direct notification)».

Без отдельного объекта синхронизации — через приостановку/возобновление потока:
- Поток приостанавливает сам себя (`thread_suspend`).
- Обработчик возобновляет (`thread_resume`).

Пример (rtos.md):

```c
void ISR(void *arg) {
    dpc_insert(&dpc, DPC, NULL);
}

void DPC(void *arg) {
    thread_resume(&thread);
}

void Thread(void *arg) {
    do {
        /* Обработка прерывания */
        sched_lock(...);
        /* Включение прерываний от устройства */
        thread_suspend();
        sched_unlock(...);
    } while(1);
}
```

### Однонаправленное рандеву (unilateral rendezvous)

Из rtos.md:
> «Механизм, когда один агент (в данном случае, обработчик прерывания) сигнализирует другому (потоку) о наличии некоторой работы и продолжает выполнение без ожидания ответа называется односторонним рандеву (unilateral rendezvous)».

Отправитель **не ждёт** ответа. Получатель просто узнаёт, что что-то произошло.

Примеры:
- Прерывание → поток. Прерывание не ждёт обработки.
- Событие пользователя → главный цикл. Пользователь не ждёт.

### Двухстороннее рандеву (bilateral rendezvous)

Отправитель **ждёт ответа** от получателя.

Из rtos.md:
> «Два события могут использоваться для синхронизации между клиентом и сервером. В этом случае каждый из них использует свой объект для ожидания, и второй — для уведомления другой стороны. Такой тип взаимодействия предусматривает не только уведомление о каком-либо запросе, но и ожидание ответа, поэтому называется двухсторонним рандеву (bilateral rendezvous)».

### Реализация двухстороннего рандеву

```c
event_t request, response;
data_t shared_data;

void Client() {
    shared_data = prepare_request();
    event_set(&request);                  // отправить запрос
    event_timedwait(&response, ...);      // ждать ответ
    event_reset(&response);
    process(shared_data);                 // обработать ответ
}

void Server() {
    while (1) {
        event_timedwait(&request, ...);   // ждать запрос
        event_reset(&request);
        shared_data = process_request(shared_data);
        event_set(&response);             // отправить ответ
    }
}
```

Полный цикл:
1. Клиент готовит запрос, сигналит `request`, ждёт `response`.
2. Сервер ждал `request`, проснулся, обработал, сигналит `response`.
3. Клиент проснулся, использует ответ.

### Применение двухстороннего рандеву

**1. RPC (Remote Procedure Call).** Клиент вызывает функцию на сервере, ждёт результат.

**2. Producer-consumer с подтверждением.** Производитель ждёт, что потребитель забрал данные.

**3. Распределённые системы.** Request-reply паттерн.

**4. Multithreaded handoff.** Один поток передаёт работу другому и ждёт результат.

### Сравнение

| Параметр | Уведомление | Однонаправл. рандеву | Двухнаправл. рандеву |
|---|---|---|---|
| Объект | Сигнал | Один event | Два event |
| Отправитель | Не ждёт | Не ждёт | Ждёт ответа |
| Получатель | Может не быть | Ждёт уведомления | Обязателен |
| Применение | События, прерывания | Уведомление о работе | RPC, запрос-ответ |
| Сложность | Низкая | Низкая | Средняя |

### Резюме

- Синхронизация делится на взаимное исключение и уведомления.
- События — простой 1-к-N объект уведомления; есть проблемы с потерей повторных сигналов.
- Прямое уведомление — через suspend/resume без объекта синхронизации.
- Однонаправленное рандеву — отправитель не ждёт.
- Двухнаправленное рандеву — отправитель ждёт ответа. Основа RPC и request-reply паттернов.
- Реализуется парой событий: одно для запроса, второе для ответа.
