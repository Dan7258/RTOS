## 21. Алгоритмы планирования SJF и SRTF. Примеры.
*Источник: tanenbaum.md, Глава 2.4.2 «Планирование в системах пакетной обработки» (Сначала самое короткое задание; Приоритет наименьшему времени выполнения)*

**SJF (Shortest Job First — «сначала самое короткое задание»).** Непреемптивный алгоритм для пакетных систем. Среди готовых заданий планировщик выбирает то, у которого наименьшее ожидаемое время выполнения. Предполагается, что время выполнения **известно заранее** (например, по статистике повторяющихся однотипных задач — пакет страховых заявлений, рендер кадра).

**Доказательство оптимальности.** Для N заданий с временами `a₁, a₂, ..., aₙ` среднее оборотное время = `(N·a₁ + (N-1)·a₂ + ... + aₙ)/N`. Время самого первого задания `a₁` входит с максимальным весом N, поэтому первым должно быть самое короткое. Аналогично для остальных. Алгоритм оптимален **только если все задания доступны одновременно**.

**Пример SJF.** Задания A, B, C, D с длительностями 8, 4, 4, 4 мин, поступили одновременно.
- В порядке поступления (FCFS): среднее оборотное время = (8 + 12 + 16 + 20)/4 = **14 мин**.
- По SJF (B, C, D, A): среднее = (4 + 8 + 12 + 20)/4 = **11 мин**.

**Контрпример (Tanenbaum).** Пять заданий A, B, C, D, E с временами 2, 4, 1, 1, 1 и временами прихода 0, 0, 3, 3, 3.
- SJF выберет A (короче, чем B на момент 0) → A, B, C, D, E: среднее ожидание = 4,6.
- Порядок B, C, D, E, A: среднее = 4,4 — лучше! Потому что C, D, E не были доступны на момент 0.

**SRTF (Shortest Remaining Time First — «приоритет наименьшему оставшемуся времени»).** Преемптивная версия SJF. Планировщик всегда выбирает процесс с **наименьшим оставшимся временем**. При поступлении нового задания его полная длительность сравнивается с оставшимся временем текущего; если новое короче — текущее вытесняется и запускается новое.

Это позволяет коротким новым задачам не ждать длинных, уже выполняющихся.

**Пример SRTF.** Задание X (длительность 10) запущено в момент 0. В момент 2 приходит Y (длительность 3). У X осталось 8, у Y — 3. SRTF вытесняет X, запускает Y. Y завершается в момент 5. X дорабатывает до момента 13.

**Достоинства SJF/SRTF:**
- Минимальное среднее оборотное время.
- Хорошо для пакетных систем с предсказуемыми задачами.

**Недостатки:**
- **Требуется знание времени выполнения** — на практике редко выполнимо.
- **Голодание** длинных задач: если короткие приходят постоянно, длинная может ждать вечно.
- Не подходит для интерактивных систем.

В реальных ОС используется приближение: ОС оценивает «характер» процесса по истории (I/O-bound vs CPU-bound) и динамически меняет приоритет.

## 22. Алгоритм планирования RMS и EDF. Примеры.
*Источник: rtos.md, Глава 5.3.5 «Планирование»; tanenbaum.md, Глава 2.4.4 «Планирование в системах реального времени»*

Алгоритмы для систем реального времени с **периодическими задачами**. Задача характеризуется периодом `Pᵢ` и временем выполнения `Cᵢ`.

**Критерий планируемости** (Tanenbaum): система планируема, если сумма загрузок не превышает 1:
`Σ (Cᵢ / Pᵢ) ≤ 1`

**RMS (Rate-Monotonic Scheduling).** Статический алгоритм с приоритетами по периоду: **чем меньше период, тем выше приоритет**. Приоритет назначается один раз при создании задачи. Планировщик вытесняющий: всегда выполняется самая приоритетная готовая задача.

**Теорема Liu & Layland (1973):** RMS гарантирует выполнение всех периодических задач, если их суммарная загрузка не превышает:
`Σ (Cᵢ / Pᵢ) ≤ N·(2^(1/N) − 1)`

При N → ∞ предел стремится к `ln(2) ≈ 0,693`. То есть RMS гарантированно работает при загрузке до ~69%.

**Пример RMS.**
- Задача T1: P=4, C=1 (загрузка 0,25);
- Задача T2: P=5, C=2 (загрузка 0,4);
- Задача T3: P=20, C=5 (загрузка 0,25).
- Суммарная загрузка 0,9. Граница Liu-Layland для N=3: ≈ 0,779. **Не гарантировано** теоремой, но возможно при удачном расположении дедлайнов (нужна проверка временной диаграммой).

Приоритеты по RMS: T1 (период 4) > T2 (период 5) > T3 (период 20).

**EDF (Earliest Deadline First).** **Динамический** алгоритм: в каждый момент выполняется задача с **ближайшим дедлайном**. Приоритеты меняются по мере приближения дедлайнов задач.

**EDF оптимален** для систем реального времени с одним процессором: планируемо всё, что в принципе можно спланировать, при условии `Σ Cᵢ/Pᵢ ≤ 1`.

**Пример EDF.** Те же T1 (P=4, C=1), T2 (P=5, C=2). В момент 0 приходят оба. Дедлайны: T1 в 4, T2 в 5 → выполняется T1 (1 ед.) до момента 1. Далее T2 до 3. T1 приходит снова с дедлайном 8, T2 — следующий период с дедлайном 10 → T1 ближе. И т.д.

**RMS vs EDF:**

| Параметр | RMS | EDF |
|---|---|---|
| Приоритеты | Статические | Динамические |
| Загрузка ≤ | ~69% (для большого N) | 100% |
| Накладные расходы | Низкие | Высокие (пересчёт дедлайнов) |
| Предсказуемость | Высокая | Средняя |
| Применение | Промышленные ОСРВ | Реже, академические |

В большинстве промышленных ОСРВ используется **RMS** (как частный случай приоритетного планирования) — из-за простоты, предсказуемости и минимальных накладных расходов. EDF теоретически оптимален, но требует постоянного пересчёта дедлайнов, а сами дедлайны бывает трудно определить.

## 23. Очереди ожидания. Попадание процесса в очередь ожидания и выход из нее.
*Источник: robert_lav.md, Глава 4 «Системный планировщик и диспетчеризация процессов», раздел «Очереди ожидания»*

**Очередь ожидания (wait queue)** — список процессов, ожидающих наступления определённого события (завершения ввода-вывода, освобождения семафора, поступления данных). В ядре Linux представлена типом `wait_queue_head_t`.

**Попадание процесса в очередь ожидания** (рекомендованный паттерн в Linux):

```c
// Источник: robert_lav.md, Глава 4 «Очереди ожидания»
DEFINE_WAIT(wait);
add_wait_queue(q, &wait);
while (!condition) {
    prepare_to_wait(&q, &wait, TASK_INTERRUPTIBLE);
    if (signal_pending(current))
        /* обработка сигнала */;
    schedule();
}
finish_wait(&q, &wait);
```

Шаги:
1. **`DEFINE_WAIT(wait)`** — создать элемент очереди.
2. **`add_wait_queue(q, &wait)`** — добавить в очередь.
3. **`prepare_to_wait()`** — установить состояние `TASK_INTERRUPTIBLE` (можно разбудить сигналом) или `TASK_UNINTERRUPTIBLE` (нельзя).
4. **Проверить условие** в цикле (важно — между установкой состояния и засыпанием условие могло выполниться).
5. **`schedule()`** — отдать CPU. Задача удаляется из runqueue, помещается в очередь ожидания.
6. После пробуждения снова проверить условие; если выполнено — выйти из цикла.
7. **`finish_wait()`** — установить `TASK_RUNNING` и удалить себя из очереди.

**Выход из очереди ожидания.** Происходит через функцию **`wake_up()`**, вызываемую кодом, который обработал событие. Она:
1. Вызывает `try_to_wake_up()` для задач в очереди.
2. Устанавливает состояние `TASK_RUNNING`.
3. Вызывает `enqueue_task()` — добавляет в красно-чёрное дерево готовых.
4. Устанавливает `need_resched`, если разбуженная задача приоритетнее текущей.

**Пример из VFS:** после чтения данных с диска подсистема VFS вызывает `wake_up()` для очереди, которая содержит ожидающие данных процессы.

**Фиктивная активация (spurious wakeup).** Задача может быть разбужена сигналом, а не реальным событием. Поэтому условие **обязательно проверяется в цикле**.

**Гонка между проверкой и засыпанием.** Если условие проверить до помещения в очередь, оно может стать истинным между проверкой и `schedule()`, и `wake_up()` ничего не разбудит. Паттерн `prepare_to_wait` + проверка `condition` устраняет гонку: задача уже в очереди до проверки условия, поэтому `wake_up()` её разбудит.

## 24. Особенности работы планировщика при энергосбережении. Работа с главным циклом/акторами и потоками.
*Источник: rtos.md, Глава 3.2.3 «Синхронизированный цикл», Глава 3.3 «Использование ОСРВ»; Глава 5 «Исполнительная подсистема»*

**Главная задача планировщика при энергосбережении** — между событиями уводить процессор в режим пониженного энергопотребления. Это достигается через специальные конструкции: ожидание прерывания (`WFI`/`HLT`), поток простоя, tickless idle.

### Главный цикл / синхронизированный цикл

Простейший планировщик в синхронизированном цикле:
- В конце итерации — `stop()` с проверкой «нет работы» под запретом прерываний → `WFI`.
- При приходе прерывания обработчик ставит флаг и возвращает управление.
- Цикл просыпается, обрабатывает события, опять засыпает.

### Акторы

Когда все акторы неактивны:
- Главный цикл выполняет `wait_for_interrupt()`.
- Прерывание активирует актор через `actor_activate()`.
- После выхода из обработчика прерывания планировщик акторов запускает обработчик.
- После завершения обработчика, если других активных акторов нет — снова `WFI`.

### Потоки

Для потоков вводится **поток простоя (idle thread)** с минимальным приоритетом:

```c
void idle_thread(void) {
    while (1) {
        wait_for_interrupts();
    }
}
```

Когда нет ни одного готового рабочего потока, планировщик выбирает поток простоя. Прерывание может разбудить ожидающий поток, после чего планировщик переключается на него.

### Дополнительные техники

- **Tickless idle.** В обычной ОС таймер прерывает CPU каждые 1 мс (tick) → процессор не может спать долго. В tickless-режиме перед сном ОС вычисляет ближайшее запланированное событие и программирует таймер на этот срок. Процессор спит до реальной работы.
- **Динамическое масштабирование частоты (DVFS).** При низкой нагрузке снижается частота и напряжение CPU.
- **Глубокие уровни сна.** ОС выбирает уровень от лёгкого сна (быстрое пробуждение) до глубокого (большая экономия, но дольше пробуждаться) в зависимости от ожидаемой длительности простоя.

### Особенности для ОСРВ

В жёстком реальном времени глубокий сон может ухудшить время реакции. Поэтому:
- Используются только лёгкие уровни сна.
- Между событиями засыпание короткое — преимущество в энергии минимальное, зато реакция предсказуемая.

## 25. Отличие потоков от процессов. Планировщик потоков или процессов. Отличия.
*Источник: rtos.md, Глава 3.3 «Использование ОСРВ» (3.3.2 «Потоки», 3.3.3 «Процессы»); robert_lav.md, Глава 3 «Управление процессами», раздел «Реализация потоков в ядре Linux»; tanenbaum.md, Глава 2.4.6 «Планирование потоков»*

**Процесс** = один или более потоков + набор сопоставленных с ними ресурсов (адресное пространство, открытые файлы, права).

**Поток** = последовательность инструкций + аппаратный контекст (регистры + стек). Виртуальный процессор внутри процесса.

### Ключевые отличия

| Параметр | Процесс | Поток |
|---|---|---|
| Адресное пространство | Своё | Общее с другими потоками процесса |
| Стек | (свой у каждого потока) | Свой |
| Открытые файлы | Свои | Общие с другими потоками |
| Создание | Дорогое (адресное пространство) | Дешёвое |
| Переключение контекста | Дорогое (смена адресного пространства, TLB) | Дешёвое (только регистры) |
| Изоляция при ошибке | Полная (сегфолт не валит других) | Нет (валит весь процесс) |

### Реализация в Linux

Linux уникален: **нет отдельной концепции потоков**. Поток — это процесс, разделяющий ресурсы с другими процессами. Каждый поток имеет свой `task_struct`. Реализуется через `clone()` с флагами совместного использования (`CLONE_VM`, `CLONE_FILES`, ...).

Это противоположно Windows/Solaris, где есть отдельный концепт «процесса с быстрым переключением контекста» (lightweight process).

### Планировщик потоков или процессов

**В большинстве систем (включая Linux):** планируются потоки. Процесс не имеет приоритета — он только контейнер ресурсов. Все потоки от всех процессов рассматриваются единообразно: выбирается самый приоритетный готовый поток (rtos.md называет это «сквозным планированием»).

**Альтернативный подход — иерархическое планирование (rtos.md, рис. 22):** применяется в ответственных системах. На этапе проектирования составляется расписание процессов: каждый получает временной слот, а внутри слота уже планируются его потоки. Удобно для гарантированного распределения времени между независимыми задачами в одной системе.

### Потоки на уровне пользователя vs на уровне ядра (Tanenbaum 2.4.6)

**На уровне пользователя:**
- Ядро не знает о потоках. Видит только процесс.
- Внутри процесса собственный планировщик потоков (часто реализуется библиотекой).
- Преимущества: дешёвое переключение, может учитывать особенности приложения.
- Недостаток: блокирующий вызов из одного потока блокирует весь процесс.

**На уровне ядра:**
- Ядро знает о каждом потоке отдельно.
- Преимущества: блокирующий вызов из одного потока не блокирует другие; ядро может оптимизировать планирование (например, отдать предпочтение потоку из того же процесса, что и предыдущий).
- Недостаток: дороже переключение, чем чисто пользовательских потоков.

### Производительность

В Linux переключение между потоками одного процесса дешевле, чем между процессами, потому что не нужно менять адресное пространство (CR3) и TLB сохраняет валидность. Однако в Linux даже переключение процессов очень быстрое из-за общей реализации.

## 26. Стеки потоков. Организация памяти для потоков, отличие от процессов.
*Источник: rtos.md, Глава 3.3.2 «Потоки»; Глава 4.6.1 «Стек ядра»; robert_lav.md, Глава 3 «Управление процессами»*

### Что такое стек потока

Поток = последовательность инструкций + контекст (регистры процессора) + стек. Стек хранит:
- активационные записи функций (локальные переменные, адреса возврата, сохранённые регистры);
- параметры функций;
- временные значения вычислений.

Без собственного стека поток не может вызывать функции с состоянием — поэтому у каждого потока стек **обязательно свой**.

### Сравнение с акторами

В архитектуре акторов все обработчики используют **общий стек** — потому что каждый запускается с начала и не сохраняет состояние между активациями. Поток же должен сохранять контекст, чтобы возобновляться с того места, где остановился, — это требует **своего стека**.

### Организация памяти для потоков одного процесса

Все потоки процесса разделяют:
- **Сегмент кода (.text)** — один на процесс.
- **Сегмент глобальных данных (.data, .bss)** — общий.
- **Кучу (heap)** — общая; `malloc()` из любого потока возвращает указатель, доступный всем.
- **Открытые файловые дескрипторы.**

Уникально для каждого потока:
- **Стек.** Выделен в виртуальном адресном пространстве процесса в отдельном регионе. По умолчанию в Linux — 8 МБ.
- **Регистры процессора** (включая SP, PC) — сохраняются в `task_struct` при переключении.
- **TLS (Thread-Local Storage)** — переменные с модификатором `__thread`/`thread_local` имеют свою копию у каждого потока.

### Типичная карта адресного пространства многопоточного процесса

```
0xFFFFFFFF  ┌────────────────────────┐
            │  Область ядра          │
0xC0000000  ├────────────────────────┤
            │  Стек потока 1 (main)  │
            ├────────────────────────┤
            │  Стек потока 2          │
            ├────────────────────────┤
            │  Стек потока 3          │
            ├────────────────────────┤
            │  Libraries (.so), mmap │
            ├────────────────────────┤
            │  Куча (общая)          │
            ├────────────────────────┤
            │  .bss, .data           │
            ├────────────────────────┤
            │  .text                 │
0x08048000  └────────────────────────┘
```

### Отличие организации памяти от процессов

| Параметр | Процессы | Потоки одного процесса |
|---|---|---|
| Адресное пространство | Каждому своё | Общее |
| Стек | Один на процесс (исторически) | По одному на каждый поток |
| Куча | Своя | Общая |
| Глобальные данные | Свои | Общие |
| Передача данных | IPC (shared memory, сокеты, сообщения) | Через общие переменные |
| Защита от ошибок | MMU защищает один процесс от другого | Нет защиты |

### Стек ядра для каждого потока

Если поток выполняется в системе с разделением привилегий, у него **два стека** (rtos.md, 4.6.1):
- **Пользовательский стек** — используется в непривилегированном режиме.
- **Стек ядра** — используется при системных вызовах и обработке прерываний для этого потока.

При системном вызове аппаратура атомарно переключает SP на стек ядра потока. В Linux стек ядра небольшой и фиксированный — 8 КБ на 32-битах, 16 КБ на 64-битах (см. вопрос 30).

### Тонкости стеков потоков

- **Размер стека ограничен.** Переполнение → segmentation fault. По умолчанию в Linux 8 МБ; можно изменить через `pthread_attr_setstacksize()` или `ulimit`.
- **Защитная страница (guard page)** — между стеками потоков обычно есть невыделенная страница, попадание на которую вызывает page fault — защита от переполнения.
- **Рекурсия и большие локальные массивы** опасны — могут переполнить стек.

## 27. Уровни приоритетов планирования. Планирование прерываний, акторов и потоков в ОСРВ.
*Источник: rtos.md, Глава 3.3 «Использование ОСРВ»; Глава 4.4.1 «Аппаратные прерывания»; Глава 5.3 «Проектирование интерфейса»*

В ОСРВ существуют несколько уровней планирования — от самого высокого приоритета (аппаратные прерывания) до самого низкого (поток простоя).

### Уровневая иерархия (от высшего к низшему)

1. **Аппаратные прерывания** (с собственной иерархией приоритетов внутри).
2. **Программные прерывания / DPC** (отложенные процедуры).
3. **Акторы** или **DISPATCH-обработчик** ОСРВ.
4. **Потоки** (по приоритетам, обычно несколько уровней).
5. **Поток простоя** (idle).

Более высокий уровень вытесняет любой более низкий **немедленно**, как только его «триггер» сработал.

### Планирование прерываний

- **Векторы.** Каждое прерывание имеет свой вектор в таблице прерываний.
- **Приоритеты прерываний.** Каждому вектору сопоставлен приоритет — статически или динамически. Высокоприоритетные прерывания могут вытеснять обработчики низкоприоритетных (вложенные обработчики).
- **Маска приоритета.** Процессор поддерживает текущий уровень маски — прерывания с приоритетом ≤ маски игнорируются. Повышение уровня маски используется в критических секциях.
- **Контроллер прерываний** (NVIC на ARM, APIC на x86) аппаратно реализует приоритизацию.

Время реакции на прерывание самого высокого приоритета — фиксированное и минимальное.

### Планирование акторов

После выхода из всех обработчиков прерываний планировщик акторов выбирает **самый приоритетный активный** актор и запускает его обработчик. Приоритеты акторов задаются при инициализации (`actor_init(&actor, priority, ...)`).

Акторы планируются:
- В порядке приоритета.
- Внутри одного приоритета — обычно FIFO.
- Выполняются на общем стеке, до завершения (run-to-completion).

### Планирование потоков

Потоки имеют диапазон приоритетов (например, 0–31 или 0–255). Планировщик потоков:
- Выбирает самый приоритетный готовый поток.
- При одинаковых приоритетах — round robin с квантами, либо кооперативно (rtos.md, 5.3.5).
- Если поток с более высоким приоритетом разблокируется → немедленно вытесняет текущий.

### Цикл обработки события в ОСРВ

```
1. Аппаратное прерывание → сохранение контекста потока в его стек ядра
2. Запуск обработчика прерывания (на стеке прерываний)
3. Обработчик переводит ожидающий поток/актор в READY,
   запрашивает программное прерывание DISPATCH
4. Выход из обработчика прерывания
5. Срабатывает DISPATCH-обработчик (программное прерывание
   низкого приоритета)
6. Планировщик выбирает самый приоритетный готовый поток
7. Переключение контекста → исполнение нового потока
```

Эта схема позволяет:
- **Прерывания не задерживаются** — они короткие, не делают переключения контекста.
- **Переключение контекста происходит после всех вложенных прерываний** — нет рекурсивных вызовов планировщика.

### Уровни маскирования

Между уровнями прерываний и уровнями приоритетов потоков ОС вводит логическую границу:
- **IRQ-уровень** — выше → могут вытеснять обработчики с более низким IRQ-уровнем.
- **DISPATCH-уровень** — на нём работает планировщик; код потоков работает ниже него.
- **PASSIVE-уровень** — обычные потоки.

Поднятие IRQL запрещает определённые операции (например, нельзя ждать на семафоре).

## 28. Режимы пользователя и режимы ядра.
*Источник: rtos.md, Глава 2.3.1 «Режимы исполнения», 2.3.2 «Переключения между режимами»; tanenbaum.md, Глава 1 «Введение»; robert_lav.md, Глава 2 «Начальные сведения о ядре Linux»*

### Зачем разделение режимов

В системе могут работать программы разной степени надёжности и важности. Ошибка в неважной программе (например, в фоновом логировании) не должна валить всю систему. Решение — **разделение привилегий**: код выполняется в одном из двух режимов.

### Режим ядра (kernel mode, supervisor mode)

- Полный доступ ко всем инструкциям процессора.
- Прямой доступ ко всему аппаратному обеспечению.
- Возможность выполнять привилегированные инструкции: установка таблиц страниц, изменение масок прерываний, инструкции ввода-вывода.
- Доступ ко всей памяти.

В режиме ядра работают:
- Ядро ОС.
- Драйверы устройств.
- Обработчики прерываний.

### Режим пользователя (user mode)

- Доступно подмножество инструкций.
- Запрещены команды управления оборудованием.
- Доступ к памяти ограничен своим адресным пространством (через MMU).
- Запрещены инструкции запрета прерываний, изменения таблиц трансляции.

Нарушение → исключение, которое перехватывается ядром.

### Переключение между режимами

**Из пользователя в ядро** возможно только в **контролируемых точках**:
- **Системный вызов** — специальная инструкция (`SVC` на ARM, `INT 0x80` или `SYSCALL` на x86). Процессор атомарно переключается в режим ядра и передаёт управление обработчику, адрес которого настроен ядром.
- **Прерывание** — асинхронно, инициируется устройством.
- **Исключение** — например, page fault, деление на 0.

Из rtos.md (2.3.2): «Непривилегированная программа не может напрямую вызывать код привилегированной, вместо этого используется специальная инструкция, называемая системным вызовом или программной ловушкой (software trap)».

**Из ядра в пользователя** — через специальную инструкцию возврата (`IRET`, `ERET`, `RFE`). Восстанавливаются регистры состояния процессора, режим понижается.

### Хранение текущего режима

Текущий режим — поле в специальном регистре процессора (CPSR на ARM, CS/RFLAGS на x86). Доступ к этому регистру привилегированный.

### Расширенные модели

x86 поддерживает **4 уровня (Ring 0–3)**: Ring 0 — ядро, Ring 3 — пользователь, Ring 1–2 теоретически для драйверов и расширений, но почти не используются (из-за непереносимости).

ARM Cortex-M имеет 2 уровня: Privileged и Unprivileged.

ARM Cortex-A имеет до 4 уровней (EL0–EL3): EL0 пользователь, EL1 ядро, EL2 гипервизор, EL3 безопасный монитор.

### Производительность

Переключение между режимами имеет накладные расходы:
- Сохранение/восстановление регистров.
- Возможно — переключение стека (на стек ядра).
- Сброс конвейера процессора.

Для оптимизации существуют:
- **SYSENTER/SYSEXIT** на x86 — быстрый системный вызов через регистры.
- **vDSO** — некоторые системные вызовы реализованы в пользовательском режиме через специальную страницу, которую ядро отображает в каждый процесс.

## 29. Потоки ядра. Адресное пространство потока. Демоны.
*Источник: robert_lav.md, Глава 3 «Управление процессами», раздел «Потоки в ядре»*

### Понятие потока ядра

**Поток ядра (kernel thread)** — обычный процесс, который выполняется **исключительно в пространстве ядра**. Используется для выполнения фоновых операций ядра: сброс грязных страниц на диск, обработка отложенных сигналов, мягкие прерывания, и т.д.

**Существенное отличие от обычных процессов:** у потока ядра **нет своего адресного пространства**. Поле `mm` в `task_struct` равно `NULL`. Поток ядра не имеет пользовательской части и не переключается в пользовательский режим.

### Адресное пространство потока ядра

Когда выполняется поток ядра:
- Используется адресное пространство **предыдущего** процесса (или произвольного — для ядра неважно, потому что ядерные адреса видны во всех АП).
- Это поле называется `active_mm` в Linux.

Это работает потому, что часть АП, отображённая на ядро (верхняя), одинакова во всех процессах.

### Планирование потоков ядра

Потоки ядра планируются и вытесняются **так же, как обычные процессы**. Каждый имеет свой `task_struct` и стек ядра.

### Создание потока ядра

В Linux все потоки ядра порождаются специальным процессом **`kthreadd`** (PID 2). Интерфейс:

```c
// Источник: robert_lav.md, Глава 3 «Потоки в ядре»
struct task_struct *kthread_create(
    int (*threadfn)(void *data),
    void *data,
    const char namefmt[], ...);

struct task_struct *kthread_run(
    int (*threadfn)(void *data),
    void *data,
    const char namefmt[], ...);

int kthread_stop(struct task_struct *k);
```

- `kthread_create()` — создаёт в незапущенном состоянии. Нужно вызвать `wake_up_process()`.
- `kthread_run()` — создаёт и сразу запускает (макрос).
- `kthread_stop()` — завершает поток.

Внутри `kthread_create` через `clone()` создаётся новый `task_struct`, в котором будет выполняться функция `threadfn` с аргументом `data`.

### Демоны

**Демон** — фоновый процесс, выполняющий определённую задачу системы. В Linux демоны делятся:

- **Потоки ядра.** Самые важные: `flush` (запись грязных страниц), `ksoftirqd` (обработка soft IRQ), `kswapd` (освобождение памяти под подкачку), `kthreadd` (порождение других потоков ядра). Видны через `ps -ef` — обычно в квадратных скобках.

- **Пользовательские демоны.** Обычные процессы, работающие в фоне в пользовательском режиме: `sshd`, `systemd`, `cron`. Они имеют адресное пространство, работают как обычные программы, но запускаются автоматически и не имеют управляющего терминала.

### Завершение потока ядра

Потоки ядра завершаются через:
- Вызов `do_exit()` внутри самого потока.
- Вызов `kthread_stop(k)` снаружи — внутри потока цикл должен проверять `kthread_should_stop()`.

После запуска поток обычно крутится в цикле, обрабатывая поступающие задания:

```c
int my_thread(void *data) {
    while (!kthread_should_stop()) {
        wait_for_work();
        do_work();
    }
    return 0;
}
```

## 30. Стек ядра. Назначение, применение, особенности взаимодействия.
*Источник: robert_lav.md, Глава 2 «Начальные сведения о ядре Linux», Глава 12 «Управление памятью» (раздел «Статическое выделение памяти в стеке»); rtos.md, Глава 4.6.1 «Стек ядра»*

### Назначение стека ядра

Когда поток выполняется в режиме ядра (обрабатывает системный вызов, прерывание, исключение), ему нужен стек для:
- Локальных переменных функций ядра.
- Адресов возврата из вложенных функций.
- Сохранения регистров при переключениях контекста и вложенных вызовах.

Использовать пользовательский стек **нельзя**: это вопрос безопасности.

> «Самый простой вариант, как может быть реализована поддержка процессов: системный вызов используется для того чтобы повысить потоку привилегии и далее он выполняет код ОС так же, как и привилегированный поток. Некоторые ОС реализуют такой тип защиты, однако подобный подход может быть полезным только при отладке. Многие структуры данных, включая информацию о привилегиях, хранятся на стеке потока, поэтому любой поток может изменить структуры данных ОС на стеке и тем самым вызвать крах. Кроме того, такой дизайн имеет много проблем с безопасностью: путём модификации стека и аргументов, передаваемых через него, можно влиять на выполнение кода ОС, а через него и на другие процессы, что расходится с концепцией процессов как механизма защиты и изоляции».

Поэтому каждому потоку выделяется **отдельный стек ядра**.

> «Решением данной проблемы является выделение непривилегированному потоку двух стеков. Потоки, выполняемые в привилегированном режиме, используют как и ранее один стек, потоки же, выполняемые в пользовательском режиме, имеют два стека: один используется в непривилегированном режиме, а второй используется в то время, когда поток выполняется в режиме ядра, во время обработки системного вызова — он называется стеком ядра потока. Прерывания обрабатываются на отдельном стеке, как и ранее. Таким образом, в любой момент времени выполнение кода может происходить на одном из трёх стеков: стек прерываний, стек ядра текущего потока и пользовательский стек текущего потока (Рис. 38)».

```
// Источник: rtos.md, Глава 4.6.1, Рис. 38

  Стек прерываний                              ┌─────────────┐
                                               │ Обработчик  │
                                               │ прерываний  │
                                               └──▲────────┬─┘
                                                  │        │
  Стек ядра текущего потока       ┌────┐          │        ▼          ┌────┐
                                  │ ОС ├──────────┘        └──────────┤ ОС │
                                  └──▲─┘                              └──┬─┘
                                     │ системный вызов                   │
                                     │                                   ▼
  Пользовательский стек потока  ┌────┴───┐                          ┌────────┐
                                │ Поток  │                          │ Поток  │
                                └────────┘                          └────────┘
                                                                                ──► t
```

### Размер стека ядра

Стек ядра **мал и фиксирован**. В Linux:
- **8 КБ (2 страницы по 4 КБ)** на 32-битных платформах x86 — исторически.
- **16 КБ (2 страницы по 8 КБ)** на 64-битных.
- Опционально **одностраничный стек** — 4 или 8 КБ.

Причины малого размера (robert_lav.md, гл. 12):
1. Экономия памяти — стек выделяется на каждый процесс.
2. Сложно найти две физически смежные страницы при долгой работе системы — фрагментация физической памяти.

### Стек прерываний

При переходе на одностраничные стеки выяснилось, что обработчикам прерываний не хватает места в маленьком стеке прерванного процесса. Решение — **отдельный стек прерываний** (один на CPU). Размер — одна страница на CPU.

Из robert_lav.md, гл. 12: «при переходе на одностраничные стеки ядра обработчикам прерываний были назначены собственные стеки».

Таким образом, в любой момент времени поток может выполняться на одном из стеков:
- **Пользовательский стек** — в пользовательском режиме.
- **Стек ядра** — в режиме ядра (системные вызовы, исключения).
- **Стек прерываний** — при обработке прерывания (опционально).

### Особенности взаимодействия

**1. Аппаратное переключение на стек ядра при системном вызове.** Когда поток выполняет инструкцию системного вызова, процессор атомарно:
- Переключается в режим ядра.
- Переключает SP на стек ядра потока.

Атомарность важна: иначе прерывание между двумя действиями могло бы прийти с уже привилегированным режимом, но пользовательским стеком → угроза безопасности.

**2. Структура thread_info в стеке.** В Linux в самом конце стека ядра располагается структура `thread_info`, которая содержит указатель на `task_struct`. Это позволяет очень быстро находить дескриптор текущего процесса по указателю стека:

```asm
movl $-8192, %eax    ; маска для 8KB стека
andl %esp, %eax      ; обнуляем младшие 13 бит SP
                     ; → получили адрес thread_info
```

**3. Переполнение стека ядра незаметно.** Из robert_lav.md, гл. 12: «при его переполнении новые данные просто перезапишут данные, находящиеся за пределами стека. В первую очередь пострадает структура `thread_info`».

Поэтому в коде ядра:
- **Не объявлять большие локальные массивы и структуры.**
- Суммарный объём локальных переменных функции должен быть в пределах сотен байтов.
- Большие объекты — через `kmalloc()` (см. вопрос 90-91).
- Глубокая рекурсия запрещена.
- Использование `alloca()` не рекомендуется.

**4. Стек ядра потока vs стек прерываний.**
- Стек ядра — свой у каждого потока (на каждый `task_struct`).
- Стек прерываний — один на CPU. Когда происходит прерывание, текущий поток приостанавливается, его регистры сохраняются (часто в стек ядра потока), и обработчик начинает работу на стеке прерываний.

**5. Реализация в HAL ОСРВ (rtos.md, 4.6.1).** При прерывании из пользовательского режима первичный обработчик HAL должен сохранить контекст в стек ядра прерванного потока. Некоторые процессоры (ARM Cortex-M) делают это аппаратно; другие (например, ARM Cortex-A, MIPS) требуют программной реализации в HAL.

### Резюме

Стек ядра — маленький (4–16 КБ) фиксированный стек, выделяемый каждому потоку и используемый при работе в режиме ядра. В самом конце находится структура `thread_info` (Linux) с указателем на `task_struct`. Переключение на стек ядра при системном вызове — аппаратная атомарная операция. Из-за малого размера в коде ядра запрещены большие локальные переменные и глубокая рекурсия. Обработчики прерываний могут использовать либо стек прерванного потока, либо отдельный стек прерываний.
